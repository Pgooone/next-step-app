# Next-Step · 新手引导与使用指南

> 给**第一次打开 Next-Step**的你。读完这份指南，你会明白它能帮你做什么、为什么值得用，并能亲手跑通一条「让 AI 帮你写文档、但每一处改动都由你逐块拍板」的完整工作流。
>
> 全程约 15 分钟。看不懂的地方可以先跳过，照着「5 分钟跑起来」动手，边做边懂。

---

## 一、Next-Step 是什么

**Next-Step 是一个跑在你自己电脑上的「多 Agent 软件工厂」。**

你可以把它理解成：一个**项目工作台**，里面住着若干个你亲手调教的 **AI Agent**（每个有自己的模型、技能、角色），你派它们干活——而它们产出的**文档**，会经过一套**像 Word 修订 / Notion 评论一样的「逐块审核」机制**，最后才落成你项目目录里一份真实的 `.md` 文件。

一句话：**让 AI 替你写，但白纸黑字落盘前，每一段都得你点头。**

它有四个招牌能力：

| 能力 | 说明 |
|---|---|
| 🗂️ **项目即工作区** | 每个项目独立——会话、Agent、产物互不串台。打开应用先选项目，进去就是一个干净的工作台。 |
| 🤖 **可自定义多 Agent** | 给项目建专属 Agent，分别配模型、技能、角色、思考强度。像组建一支各有专长的小团队。 |
| 🔗 **多 Agent 协作** | 设一个目标，让多个 Agent 串行接力产出文档；或在主对话里 `@` 一下，把上下文转交给某个 Agent。 |
| ✅ **文档改动逐块确认** | AI 改文档不会直接覆盖你的文件，而是把改动变成「待确认变更」——你在并排 Diff 里逐块 ✓/✗，确认后才写盘、留版本。 |

**它适合谁**：想用 AI 批量产出**需求 / 设计 / 分析 / 方案类文档**，又**不放心让 AI 直接改你文件**的人。如果你受够了「AI 一把梭把整个文档重写、好的坏的全混在一起、想回退都难」，Next-Step 就是冲这个痛点来的。

**它的边界**：本地单用户、纯文件落盘（无数据库、无登录、无云端），同时进行的会话建议 ≤ 3。它不是聊天玩具，而是一个**把 AI 产出纳入人工审核闭环**的生产工具。

## 二、为什么是 Next-Step（核心优势）

市面上让 AI 写东西的工具很多，Next-Step 的不同在这四点：

### 1. AI 只能「提议」，不能「直接改」你的文件 —— 审核权牢牢在你手里

这是最关键的一条。当你让一个 Agent 改受管文档时，它**结构上就没有直接写盘的能力**：系统只给它一套「只读探索 + 提议」的工具，没有 `write`/`edit`。所以它能做的，只有把「我想这样改」提交成一条**待确认变更**，等你审。

**这意味着**：AI 再怎么放飞，也碰不到你已落盘的内容；任何一个字进入你的文件，都经过你这一关。对「不敢把重要文档交给 AI 乱动」的人，这是定心丸。

### 2. 逐块确认，而不是「全盘接受 / 全盘拒绝」

AI 改文档常常是「重写整篇」，但你往往只想要其中一两处。Next-Step 会用差异算法把「旧内容 → 新内容」**切成一个个变化块**（新增 / 删除 / 修改），在正文里用**绿 / 红 / 黄行内高亮**标出，你可以：

- 逐块 **✓ 接受 / ✗ 拒绝**；
- 按 `D` 看**并排 Diff**；
- 全部处理完，才**一次性物化写盘 + 生成新版本**。

「只改一段」你就只会看到那一块，不会满屏噪声。这正是 Word「修订模式」、Notion「建议编辑」的体验，只是这里的「修订者」是 AI。

### 3. 真实文件 + 版本历史 —— 产出是你的，不锁在工具里

受管文档物化后，就是你**项目目录里一份真实的 `.md`**，用任何编辑器 / git 都能打开。同时每次确认都留一个**版本**，你能切到历史版只读查看、也能**回滚**（回滚同样要二次确认、并生成新版，不会偷偷覆盖）。

数据是你的、是纯文本、在你硬盘上——不绑定云端、不锁定格式。

### 4. 一支 AI 小团队，而不是一个全能助手

你可以建多个 Agent，各配角色（「资深需求分析师」「严谨的架构师」「挑刺的评审」），用 `@` 把活转交给最合适的那个，或设个目标让它们**串行接力**。比起反复给同一个对话框换 prompt，这更接近「带团队」。

## 三、5 分钟跑起来

### 1. 启动

```bash
npm install
npm run dev      # 默认端口 30141
```

浏览器打开 **http://localhost:30141**。

> 需要 Node 环境；首次 `npm install` 稍久。生产构建用 `npm run build && npm run start`。

### 2. 建第一个项目

打开应用，你会看到**项目卡片墙**。

![打开应用后看到的项目卡片墙](docs/images/01-项目墙.png)

- 点「**新建项目**」，填**名称** + 一个**本地路径**（这个项目的工作目录），确定后它就入墙了。
- 点项目卡 → 进入**工作台**。

### 3. 发出第一条消息

进项目默认落在「**主对话**」。直接在输入框打字、回车——一个通用 Agent 就会回答你。

![工作台主界面：左侧会话栏 + 中间主对话](docs/images/02-工作台.png)

> 主对话不预建空会话——**你发出第一条消息，它才被认定为这个项目的「主对话」**。

到这里，你已经会用最基础的 Next-Step 了。下面是真正让它发光的部分。

## 四、招牌工作流：让 AI 帮你写文档，每一段都你说了算

这是 Next-Step 最值得体验的一条路径。跟着做一遍：

### 第 1 步：建一个有「人设」的 Agent

顶栏点 **Agents**（需先选好项目）→ 进入 **Agent 管理**：一面**玻璃质感的正方形卡片墙**，每个 Agent 一张卡。

- 点网格里的「**+**」空卡 → 填**名称**（必填）→ 给它写个**角色描述**，比如「资深需求分析师，擅长把模糊想法拆成清晰的功能点」。
- **卡片本身就是入口**：点任意卡片，就地弹出该 Agent 的面板——上方**起会话**、中间**配置**（模型 / 技能 / 角色 / **思考强度**，改完即存）、下方**删除**。

![点开一张 Agent 卡：起会话 + 完整配置（模型 / 技能 / 角色 / 思考强度）一览](docs/images/03-agent玻璃卡.png)

### 第 2 步：起它的会话，直接说需求

点卡片 →「**起会话**」，开场白直接说你要什么，例如：

> 「帮我新建一个《用户登录需求》文档，包含账号密码、短信验证码、第三方登录三个功能点。」

Agent 会自己调用 **`create_artifact`** 工具，建好文档、落第一版，并**物化成你项目根目录里一份真实的 `.md`**。同时它出现在左栏 **EXPLORER → 「受管文档」分组**里，点开就进入右侧的**文档面板**（像一个 Notion 页）。

### 第 3 步：让它改，见证「逐块确认」

接着说：

> 「把第 2 点改成『语音验证码登录』，其它不变。」

这一次见证奇迹：Agent 调用 **`propose_edit`**，**不会直接改你的文件**，而是把改动变成一条**待确认变更**。文档面板里：

- 正文出现**行内高亮**（绿 / 红 / 黄），右上角提示「**N 处待确认**」；
- 左侧浮出 **逐块确认卡**，你对每一块点 **✓ / ✗**；
- 按 **`D`** 可看**并排 Diff**。

![核心体验：行内高亮 + 逐块确认 + 并排 Diff](docs/images/04-块级确认.png)

哪怕 Agent 内部回的是「整篇新全文」，系统也只会把**真正变化的那一块**切出来给你——所以「只改第 2 点」你就只看到第 2 点那一块。

### 第 4 步：确认 → 落盘 → 留版本

逐块处理完（或点「**全部 ✓ / 全部 ✗**」，高危操作有**二次确认**），系统才：

1. **物化写盘**——更新项目目录里那份真实 `.md`；
2. **生成新版本**——面板头部的版本下拉里多一条历史。

至此，一条「AI 提议 → 你逐块审 → 落盘留版本」的闭环就跑通了。**这套机制跨会话闲置、甚至开发服务器重启都依然可靠**——过会儿回来继续让它改，提议工具不会失效。

## 五、功能详解

### 项目与项目墙
- 打开应用先见**项目卡片墙**，每张卡显「项目名 + 路径 + 创建于」。
- **建**：填名称 + 本地路径。**进**：点卡片。**回墙**：工作台内点「回到项目墙」。
- **删项目**：二次确认，**只移除登记、不删磁盘上的 `.pi/` 数据**；重新添加同一路径即恢复（本地单用户下这是特性，不是 bug）。

### 主对话与会话栏
- 每个项目有唯一的「**主对话**」。左侧会话栏按归属分三区：**主对话** / **各 Agent 分组**（带名字 + 色点 + 计数，可折叠）/ **其它会话**。切项目不串显。

### 建 / 配 Agent
- 字段速记：**角色描述** = 职责与风格；**思考强度** = 是否启用深层思考；**模型**留空 = 全局默认；**技能** = 内核内置 + 项目 `.pi/agent/skills` + `.claude/skills`。
- 配置即时保存，无需「保存」按钮。

### `@` 转交：把上下文甩给某个 Agent
- 在**主对话**输入框打 **`@`**（行首或空格后）→ 弹出该项目的 **Agent 待选列表**（输入框提示「输入消息，或用 @ 转交给 Agent」）。
- 选中 Agent → 弹「**转交内容勾选**」：默认勾选【全部主对话历史】+【已附文件】，可现场增减。
- 点「转交」→ 打包成带来源标注的上下文 + 文件附件，**起该 Agent 一条新会话**并自动切过去。

![主对话里 @ 转交给某个 Agent](docs/images/05-转交.png)

> `@` 只在主对话生效；每次转交起的是**新会话**。

### 多 Agent 派发（Dispatch）
- 设一个目标 → 派发面板让多个 Agent **串行接力**，每个产出落在 `.pi/artifacts/` 下。
- 和 `@` 转交是两套并存：**派发 = 自动接力**；**`@`转交 = 手动把上下文交给单个 Agent**。

### 上传文本文件
- 输入框左下的**文件按钮**（标题写明「附加图片或文本文件」）→ 选文件，内容在发送时内联进消息，让 Agent 读懂。
- 支持：`.md .txt .json .csv .log .py .ts .tsx .js .jsx .html .css .yaml .yml .xml .sql .sh`。单文件 > 256KB 有软提示。
- **PDF / Word / Excel 暂不支持**，请先转成文本。

### 版本、回滚与删除
- **版本**：文档面板头部版本下拉，切到任意历史版（**只读快照**）。
- **回滚**：回到某个历史版（**二次确认**，并生成一个新版本，不覆盖历史）。
- **删除受管文档**：两个入口——① 受管分组里**悬停文档行 → 垃圾桶**；② 打开文档后**面板头部「删除」**。均两步确认，且是**彻底删除**（文档 + 全部版本历史 + 待确认变更 + 磁盘上的真实 `.md`，**不可恢复**）。

## 六、一个完整的例子

把上面串起来，一个典型的半天工作：

1. **建项目**「电商后台」，进工作台。
2. 主对话里聊两句，理清你大概要做什么。
3. 建两个 Agent：「需求分析师」和「技术评审」。
4. 起「需求分析师」会话：「写一份《订单退款需求》，覆盖整单退、部分退、超时自动退三种。」→ 它 `create_artifact` 出 v1。
5. 你审一遍，让它改：「超时阈值从 7 天改成 15 天。」→ `propose_edit` → 你在 Diff 里**只看到那一处改动** → ✓ 确认 → v2 落盘。
6. 主对话里 `@技术评审`，把这份需求转交过去：「挑挑这份需求的漏洞。」→ 它给出评审意见。
7. 回需求文档，根据评审再让分析师改几处，逐块确认 → v3。

全程：**AI 干了大部分体力活，但每一版落盘的内容，都是你逐块点过头的。**

## 七、新手须知（当前边界）

Next-Step 仍在迭代，有几条边界先知道，免得困惑：

- **逐块确认目前覆盖「Agent 单独会话」**——也就是上面招牌工作流走的路径。**主对话**和**派发**产出的文档暂不进这套逐块确认（派发产物是轻量普通 `.md`）。想要「AI 改文档 → 逐块审」，请走「建 Agent → 起它的会话」。
- **Agent 卡片里的「工具」配置当前不生效**——任何 Agent 起会话都会自动装上「文档型受限工具集」（只读探索 + 提议工具，无 write/edit/bash）。这是为了保证上一条的「AI 只能提议」。
- **外部直接改受管 `.md`**（用编辑器 / git 改那份真实文件）不会自动进版本；下次 AI 要改时会**检测到外部改动并拒绝覆盖**，防止你的手改被冲掉。
- **文件类型**：仅文本类，PDF / Word / Excel 需先转文本。
- 同时进行的会话建议 **≤ 3**（单机资源约束）。
