## T1 · `reattachProfileSession` + 单测
**做**：在 `lib/pi/profile-session-wiring.ts` 新增 `reattachProfileSession({sessionId, filePath, projectId, projectRoot, profile, ...di})`：`SessionManager.open(filePath)` 重开既有会话 → `assembleProfileSessionOptions`（带注入块 loader，`extraSkillDirs` 透传）→ **解构** `const { options: docOptions } = assembleDocSessionOptions({ projectId, sourceActor: profile.name, cwd: projectRoot, ...docDepsOverride })` → `createAgentSession({ ...options, ...docOptions, ...createOptionsOverride })`（spread 顺序不可调，删冗余 `sessionManager: sm`）→ `applyProfileRuntime` 重设 model → `registerInnerSession`。**不发首条 message**。
**DI 缝（major #2，hermetic 必需）**：追加可选 `sessionManager` / `createOptionsOverride` / `docDepsOverride` / `registerInnerSession`（默认走真实实现）。**最硬约束 = `docDepsOverride`**（否则 `buildDocTools` 默认 `new ProjectRegistry()` 读真实 `~/.pi`，单测非 hermetic）。
**spike 已澄清（无需 spike）**：systemPrompt = override（内核不持久化、永远由 resourceLoader 现算）；re-attach **必须**照常重建带注入块的 loader，否则角色/记忆静默丢失。初版「若叠加则只传 docOptions」预案已删。
**AC**：
- ① faux 单测：`reattachProfileSession` 后 `getActiveToolNames()` 恰=7 受限名（read/grep/find/ls + create_artifact/propose_edit/list_artifacts），不含 write/edit/bash（此断言即抓 major #1 错层 spread）。
- ② 重开的会话保留既有 jsonl 历史——用 `inner.sessionManager.buildSessionContext().messages.length > 0`（**不可**用 rpc wrapper 的 messageCount，它硬编码 0）；需 tmpdir 持久化 fixture（create+appendMessage→open）。
- ③ re-attach 后 systemPrompt 含 `<agent_profile>` 角色注入块（lead 自主加，防丢角色）。
- ④ lint+test+tsc 绿、零回归。

## T2 · 抽 `startRpcSessionInner` + `resolveOrReattachSession` 解析器 + 单测
**做（方案 A 前置）**：`lib/rpc-manager.ts` 把 `startRpcSession` 的 `:316-353` 建会话主体抽成**导出的不带锁内层** `startRpcSessionInner(sessionFile, cwd, toolNames?)`；`startRpcSession` 改为「existing 快路径 → inflight 检查 → `locks.set(startRpcSessionInner(...).finally(delete))`」（**行为不变**，`/api/agent/new` 继续用它）。
**做（解析器）**：新增 `lib/pi/session-reattach.ts`：`resolveOrReattachSession(sessionId, filePath, cwd, deps?)`——① 活会话快路径 `return {session, realSessionId}`；② `locks.get` 并发去重（复用 `__piStartLocks`）；③ Promise 内 `getOwner(cwd, sessionId)` 有 agentId → `try { projectId=new ProjectRegistry().list().find(p=>normalizeRoot(p.root)===normalizeRoot(cwd))?.id; if(projectId) profile=new AgentProfileStore().get(projectId,agentId) } catch { profile=null }` → `profile && projectId` 则 `reattachProfileSession`；否则 `startRpcSessionInner`（generic，不带锁）；④ `locks.set + finally delete`。三分支统一返回 `{session, realSessionId}`。DI 缝收成可选 deps。
**AC**：
- ① bySession 有 agentId 且 profile 在 → 走 reattach（断言激活集含 propose_edit）。
- ② sessionId=mainSessionId（无 bySession 项，`getOwner` 返 null）→ 走 generic（不含 doc 工具，符合主对话现状）。
- ③ agentId 在但 profile/项目被删（`get` throw NOT_FOUND）→ try/catch 容错落 generic、不抛错。
- ④ root→projectId 命中/未命中纯逻辑测（含 cwd 两侧 `normalizeRoot`）。
- ⑤ **并发去重**：同 sessionId 并发两次 `resolveOrReattachSession`，断言 `createAgentSession` 只调一次、registry 一个 wrapper。
- ⑥ **idle 销毁路径**：`destroy()` 摘除后 resolver 仍走 reattach（与 dev 重启同路收口）。
- ⑦ 零回归：`startRpcSession` 抽函数后既有 rpc-manager 测试不破（行为等价）。
