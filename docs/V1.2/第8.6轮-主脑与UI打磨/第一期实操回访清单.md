## 问题 1 · 主脑对话流：提交计划后对话「结束」+ 完成后不自动汇总　[🔴 承重 · P1]

**现象（用户原话转述）**：
- 主脑调 `submit_plan` 提交计划后，**对话就"结束"了**——用户还没选择确认/否决，就感觉对话完了。
- 更糟：用户点「确认放行」、队员跑完任务后，**还得再次发起对话（手动说"汇总一下"）才能拿到工作结果和总结**。
- 期望：① 提交计划后对话**不结束、一直等用户选择**、期间不再继续输出；② 放行后任务完成时**主脑自动给出结果总结**（不用用户再问）；③ 用户能查看中途工作过程（队员进度）加强参与感。

**触点（file:line 已坐实）**：
- `app/api/projects/[id]/mastermind/runs/[runId]/approve/route.ts`：`void runMastermind(...)` **fire-and-forget**（异步不 await）。
- `lib/domain/mastermind-orchestrator.ts`：runMastermind 全程起 **headless 队员 worker**、**主脑主会话 LLM 不在 run 中**（run 与主脑会话刻意解耦）。
- `lib/pi/orchestrator-session.ts:48` ORCHESTRATOR_SYSTEM_PROMPT：已引导「提交计划后**暂停等待**用户在计划卡上确认放行」。
- `hooks/useAgentSession.ts:346/522`：`handleSend` / `handleFollowUp` = **前端 programmatic send 口子**（option i 可行性锚点·lead 亲核）。
- `components/MastermindPollDriver.tsx` + `useMastermindStore`：前端**已轮询到 run 终态**（done/partial）。

**根因（拆两个子问题）**：
- **(a)「对话结束」= 部分是感知问题**：submit_plan 后主脑回合**内核正常结束**（计划卡 awaiting_plan_approval 态本就是"等待"）。prompt 已引导暂停，但主脑可能仍输出收尾文字让它显得"结束"而非"等待"。→ 等待态可在 **UI 层强化**（计划卡明确"待你确认"）。
- **(b)「不自动汇总」= 承重·架构级**：run 是 fire-and-forget headless、主脑主会话与 run **刻意解耦**（T6 设计），**无 run→主脑的 nudge 机制**。这正是 **ADR D-R8.6-13 明确 deferred 到二期的「自动 nudge」**（"汇总触发 = MVP A 用户触发/主脑本轮…自动 nudge 留二期"），现被实操推为真痛点。

**性质**：承重 UX 改进 + **已登记二期项**（回链 D-R8.6-13，非新发现）。**严重度/工作量**：P1 / 中~大（取决于触发形态）。

**候选修法（待用户先定形态、再 spike 验证）**：
- **i.〔推荐候选·爆炸半径小〕** 前端 run 轮询到终态 → 自动 `handleSend`/`handleFollowUp` 一条隐式"汇总 runId"prompt 唤起主脑回合（纯前端、**不碰服务端刻意解耦**；口子已存在 useAgentSession:346/522）。**待 spike 命门**：主会话是否还活/没切走/idle destroy 时序、不打断用户输入、自动 prompt 是否留可见痕迹。
- **ii.** 服务端 run 结束回调 → 向主脑会话 append 消息唤起。改动大、触碰刻意解耦的 run↔主脑边界。
- **iii.〔最保守·半自动〕** 保持手动，但把"汇总"做成计划卡/队员卡区的**显式按钮**。

**「对话不结束一直等」做到什么程度**（待用户确认期望深度）：(a) **仅 UI 层承载等待态**（计划卡明确"待确认"、回合仍结束但用户不困惑，最轻）vs (b) **真让主脑会话保持活跃/阻塞**直到用户选择（触碰会话生命周期、较重）。用户描述倾向 b，但 a 可能已满足感知诉求。

**建议处置**：🟡 承重、须用户先定 **触发形态（i/ii/iii）+ 等待深度（a/b）+ "自动"是否接受隐式痕迹**，再 spike + 拆三件套。

## 问题 2 · hover 浮框离卡片远、未紧贴　[🔴 回归 bug · P2]

**现象**：鼠标 hover 队员卡片时，速览浮框（StageHoverPreview）出现在**卡片右下角/离得远/甚至飘出视口顶**，应**紧贴卡片**。

**根因（真浏览器控制实验坐实·ns-hover-repro，getBoundingClientRect 实测）**：
- `.brow` 卡片上残留 **GSAP identity transform `matrix(1,0,0,1,0,0)`**——来自 `components/MastermindTeammateCards.tsx:62-68` 的 `gsap.from(cards,{y:8,...})` 入场补间，**tween 完成后 GSAP 不清 inline transform**。
- 任何非 `none` 的 transform 会让该元素成为其 **`position:fixed` 后代的包含块**。而 `StageHoverPreview` 渲在 `.brow` **内部**（`PipelineStageCard.tsx:154`），于是它的 `position:fixed` 不再相对视口、而相对 `.brow` → `computeFixedPopover` 算出的视口坐标被整体平移 → 浮框飞到卡片右下 **+320px / +442px**（去掉 transform 后 gap **精确回到 6px、left 精确对齐**）。
- **这是 T5 GSAP 动效引入的回归**（D-R8.6-12 我批的入场动效副作用）、**`popover-position.ts` 算法本身没错**。
- ⚠️ **教训**：ultracode 代码静态读把根因误判为「宽卡/maxHeight/滚动 stale」——runtime inline transform 静态读根本看不到，**真浏览器复现才揪出真因**（正印证用户要"复现截图查看"的必要）。
- **附带**：StageHoverPreview 头徽章名字显 `role-a1b2c3d4`（uuid8 未剥、漏用 `friendlyAgentName`）——T5 小遗留，一并修。

**性质**：T5 内联引入的回归 bug。**严重度/工作量**：P2 / 小。

**候选修法**：
- **①〔推荐·最稳〕** 浮框 **Portal 到 `document.body`**（`createPortal`）——脱离 `.brow`、`position:fixed` 恢复相对视口、`computeFixedPopover` 逻辑不动即对；**连带修 `StageSessionMenu`（T6 菜单同渲在 `.brow` 内、同坑）**。
- ② GSAP 入场后 `clearProps:"transform"` 去 identity transform。风险：reduced-motion 分支 `gsap.set(cards,{y:0})` 也写 transform 须一并清、未来给卡加 transform 动画会复发、没 Portal 稳。

**建议处置**：🟢 明确回归 bug、修法清晰（Portal）、低成本，**建议纳入**。

## 问题 3 · agent 卡片缺「框」凸显集群　[视觉增强 · P2]

**现象**：队员卡片在对话消息流里**松散漂浮**，缺一个**限定的框**凸显集群——参考 Kimi 实操：agent 集群被「⑂ Agent 集群｜N 个并行任务」头部 + 圆角细边框容器整体包起来。

**根因（ultracode P3 坐实·「内联版只搬了行、没搬壳与头」，非 bug、是 T5 spec 既定范围）**：
- `components/MastermindTeammateCards.tsx:80` 外层 `className={\`pipeline-board ${themeClass}\`}`——**只有 `pipeline-board` 基类、缺 `board` 修饰类**。而框样式 `PipelineBoardStyles.tsx:20` 是 `.pipeline-board.board{border-radius:14px;padding:.7rem;background}` （**双类选择器**）→ bg/圆角/内边距**全不生效**、外层是零样式透明 div。
- 同文件 `:21-24` 已定义好 `.hd/.hd1/.fork/.cnt` **集群头样式**（`⑂` fork 图标 + 名 + 右对齐计数），但 MastermindTeammateCards **从没渲任何 `.hd` 头**。
- 对照原版 `PipelineBoard.tsx:71` 用的是 `pipeline-board board`（**带 board**）+ 渲完整 `.hd` 头（与看板视觉定稿 kimi-v11.html 同源）。

**性质**：视觉增强。**严重度/工作量**：P2 / 小。

**候选修法**：复用 T7 已验收的 `.board/.hd/.fork/.cnt` + PipelineBoard 现成 git-branch SVG（与 Kimi 视觉定稿同源），但**用专属类名承接** border/填充避免波及共享看板 modal（爆炸半径收敛到零）；微调暗色 bg（现 `--bg:#0c0c0e` 比对话更黑=方向反、须改透明/`--container`）+ 补 border + 渲集群头（`⑂ Agent 集群 | N`：awaiting 用计划队员数、running 用已完成/总数）。

**建议处置**：🟢 低成本视觉增强、骨架现成，**建议纳入**。
