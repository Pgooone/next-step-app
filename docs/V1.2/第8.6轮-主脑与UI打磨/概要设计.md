## 一、技术地基结论（调研摘要）

**核心判断：主脑 = 主会话（已是 pi 内核 agent）+「总管能力薄壳」，约 80% 复用内核/现有，真新增集中三处**：派活工具注入路径、主脑编排器的失败处理与计划确认闸、卡片内联渲染点。

| 层 | 复用现成（已 file:line 验） | 真新增 |
|---|---|---|
| **lib/pi**（内核封装） | 系统提示注入 `appendSystemPromptOverride`(agent-profile-session.ts:138)、customTools `defineTool`(doc-tools.ts:104)、内核 compaction(rpc-manager.ts:163-186 已 wire) | 主脑会话装配模块 + 派活工具集 |
| **lib/domain**（领域层） | acquireSlot(并发门计 `__piSessions.size`)、runWorker、evictSession(按 sessionId)、PipelineRunStage 数据形态、run-controllers cancel、AgentProfileStore.create(:113)、ArtifactService | 主脑编排器（失败处理 + 计划确认前置 + 动态 run） |
| **app/api**（路由） | runs/route.ts 编排触发范式 | 主会话起会话链「主脑模式」分支 + 计划确认路由 |
| **components**（UI） | 第七轮卡片族(PipelineStageCard/StageHoverPreview/StageSessionMenu)、usePipelineStore「import type + fetch」范式 | 队员卡片内联锚点 + 计划卡 |

**长上下文（承重点）不需自研**：靠内核 compaction（现成）+ 编排层把队员拆「短命会话」隔离 + 主脑汇总走「轻读」（list_artifacts 元数据 + 按需 read，禁全文 N 篇拼接）。

## 二、路线 C 架构（工具派活 + 计划模式确认，对标 Claude Code）

```
用户说需求
   ▼
主会话（装「总管 system prompt」）理解 → 拆活
   ▼
主脑调【submit_plan 工具】(参数=结构化计划:队员/任务/验收点，内核 schema 校验)
   ▼
系统落盘 status=awaiting_approval + 暂停
   ▼
前端渲【计划卡 + 确认/否决/打回】(用结构化计划渲染，可靠)
   ▼ 用户确认
翻状态放行 → 主脑调【dispatch_task 工具】fire 编排
   ▼
runPipeline 族(动态 run) → 队员 worker(临时造角色/点名现有)
   ▼
队员卡片内联(乙·冒在主脑回复下) 实时状态 ← 复用第七轮卡片族
   ▼
队员产受管文档 → 主脑【check_status / list_artifacts】轻读 → 汇总
```

**关键**：计划与派活均走「工具调用参数」通道（内核 schema 校验、可靠），非路线 B 的「回复正文夹 JSON 让前端解析」。即 Claude Code 的 `ExitPlanMode→批准→工具执行` 流程（D-V1.2-84）。

## 三、模块划分

### M1 · 主脑会话装配（lib/pi）
- **职责**：给主会话装「总管 system prompt」(resourceLoader) +「派活工具集」(customTools)。
- **复用**：`buildProfileResourceLoader` 的 `appendSystemPromptOverride`(agent-profile-session.ts:116-156) + `defineTool` 工厂(doc-tools.ts:104-142) + 闭包注入 projectId/projectRoot。
- **新增**：`lib/pi/orchestrator-session.ts`（主脑装配）+ `lib/pi/mastermind-tools.ts`（派活工具集：submit_plan / dispatch_task / spawn_agent / check_status / list_teammates）。
- **承重**：D-V2-04 白名单——派活工具名**必须**进主会话激活集，否则内核 `_refreshToolRegistry` 按名过滤致静默失效（spike-1 验）。

### M2 · 主会话起会话链改造（app/api + hooks）
- **职责**：让主会话建立时能接 resourceLoader/customTools（现 `startRpcSessionInner`:336-385 无入口）。
- **新增**：`/api/agent/new` 加「主脑模式」分支（白名单=编码工具 ∪ 派活工具名）；`useAgentSession.ts:372` 首条消息分支透传主脑旗标。**走新分支、不污染普通主会话**。
- **承重**：spike-1 命门——碰 V2「不碰 /api/agent/new」红线、属本轮有意打破（D-R8.6-05）。

### M3 · 计划确认闸（lib/domain + app/api + components）
- **职责**：plan-mode 式确认——submit_plan 落计划 `awaiting_approval` + 暂停 → 前端计划卡 + 确认 → 放行。
- **新增**：计划状态机（前置 run 状态）+ 确认/否决/打回路由 + 计划卡 UI。借 PendingChange「先产提案→等确认→才执行」形态（非代码）。
- **承重**：spike-3——与 run-controllers globalThis 单例（dev 热重载丢 controller 已有先例）时序须验。

### M4 · 主脑编排器（lib/domain）
- **职责**：仿 `pipeline-orchestrator.ts` 骨架，但加**失败处理**（轻微自重试 1 次 → 暂停 → 给选项 重试/换人/跳过/中止）+ 计划确认前置 +**动态 run**（不落正式蓝图）。
- **复用**：acquireSlot(传 **Infinity** 避 60s 饿死)、runWorker、evictSession(本阶段 sessionId)、setOwner、累积喂下游、run-controllers cancel。
- **新增**：`lib/domain/mastermind-orchestrator.ts`——失败/暂停骨架 + run 终态扩展(done/failed/**paused/partial**)+ 动态 run 入口（守 agentId 预检、cwd 取项目 root）。
- **承重**：spike-2——给现有 fire-and-forget（建即跑、二元终态）引入「可暂停中断点」。
- **红线**：只走 **pipeline 族**，**否决 dispatch 链**（dispatch 无 cancel / completed worker 不 evict 还槽 / 上游单跳 / 默认 60s 超时 四坑）。

### M5 · 队员卡片内联（components）— 乙·对标 Kimi
- **职责**：卡片冒在主脑某条回复下（D-V1.2-83 乙）。
- **复用**：第七轮卡片族（PipelineStageCard/StageHoverPreview/StageSessionMenu）+ t-kimi 主题；队员数据适配成同构 `PipelineRunStage[]`。
- **新增**：「主脑派活内联锚点」（独立 store + 按 entryId 插卡，**禁塞进内核 messages 数组**）；数据契约字段边界（验收点 §2.9① / 失败选项态 §2.10🔴1 / 动态档案标记 §2.5 是现状无字段 → 扩 PipelineRunStage vs 卡片族吃超集，详设定）。
- **承重**：UI 无先例（ChatWindow.tsx:375-421 纯角色 `.map()`）；守 **D-R7B-07**（只 import type + fetch JSON、机制层留服务端、真浏览器验收兜底）。

### M6 · 临时造角色（lib/domain）
- **职责**：主脑临时造队员角色（D-V1.2-79 两种来源之②）。
- **复用**：`AgentProfileStore.create`(:113)。
- **新增**：避重名（:127 撞名 throw → 加序号/uuid 后缀）；第一期「存档进池」（零新增即得默认路径），清理/去重留二期。

### M7 · 长上下文管理（lib/pi + 编排层策略）
- **职责**：主脑长任务上下文不爆（§2.10🔵8 承重）。
- **复用**：内核 compaction（现成、零接线、起会话即生效）+ 前端 context% 指示 + 压缩按钮。
- **策略**（设计重点不在「实现压缩」而在隔离）：① 队员拆短命会话（每 worker fresh-session + 跑完即 evict、上游 accumulate-into-prompt）；② 主脑汇总走**轻读**（list_artifacts 元数据 + 按需 read + artifactId 链接，**禁全文 N 篇拼接**）。
- **承重**：spike-4——compaction 行为只能真模型端到端验（本机默认 faux 跳过），放最后。

### M8 · UI 打磨（需求 2，独立但卡片处与 M5 交汇）
- `.btn` 四型六态 token/类（P0 真缺陷 + P1）+ 修流水线卡片真 bug（queued 矛盾/双蓝割裂/hover 硬切/键盘不可达/reduced-motion）+ 补 a11y。P2 全量迁移移出本轮渐进。详见需求 §三。
- **与 M5 交汇**：同一套 PipelineStageCard——M5 负责搬运+数据适配+动效，M8 负责修 bug+a11y+打磨，**统一做、避免两次改同文件**。

## 四、承重点 + spike 计划（按调研定稿顺序）

| spike | 承重点 | 验什么 | 依赖 |

| **spike-3** | 计划确认闸（M3） | submit_plan→落盘 awaiting_approval(acquireSlot 零调用) / 确认否决打回翻状态 / 确认后才 fire / 不与 globalThis 单例时序纠缠 | spike-1 GO |

## 五、红线与护栏

- **不改内核**：compaction / 工具机制 / 系统提示注入全用现成钩子。
- **守 D-R7B-07**：所有新数据通道「只 import type + fetch JSON」，机制层（含 node:fs / store / 编排器）留服务端 lib/domain+lib/pi，前端触发仿 usePipelineStore；**真浏览器验收兜底**（lint/test/build 抓不到 node:fs 雷）。
- **打破 V2「不碰 /api/agent/new」红线**：属本轮有意决策（需求 2.1 主会话即总管），走**新分支隔离**普通主会话、ADR 已记（D-R8.6-05）。
- **并发≤3**：复用 acquireSlot（主脑 + 队员同计 `__piSessions.size`、超排队、传 Infinity 避饿死）；主脑占 1 槽 → 默认同时 2 队员（已向需求方知会）。
- **只走 pipeline 族**：否决 dispatch 链（四坑）。
- **流水线 evict 只 destroy 不碰 owner-map**（第五/八轮）。
- **临时造角色避重名**（序号/uuid）。
