# 第 8.6 轮第二期 · 主脑对话流与 UI 打磨 · 需求文档

> **来源**：第一期（T1~T7）收官后用户真跑 + 亲录 Kimi 实操 mp4 + 提供真实回放页深读 → 登记 [第一期实操回访清单](../第一期实操回访清单.md) → 用户圈定甲（开第二期三合一，D-V1.2-85）+ 定主脑运行模型（D-V1.2-86）。
> **决策依据**：QA D-V1.2-85（组织形态甲）、D-V1.2-86（主脑运行模型：先中间路→全 Kimi 升级作末步+确认闸保留+并发按任务调整不设默认）。
> **本文档 = vibe-coding 第一步（需求 proposal），确认后 → 概要设计 → 详细设计（承重 spike）→ 拆卡 → 实现。**

## 一、目标

让主脑派活体验对标 Kimi agent 集群「会话中间有 agent 干活 → 等待 → 自动继续」的参与感（真实回放页坐实 Kimi 主脑全程在场、边跑边旁白、下钻工作台、走马灯），**同时保留我们比 Kimi 更稳的计划确认闸**；并修复第一期实操暴露的回归 bug、补齐缺失的视觉/交互细节。**先做中间路拿九成体验、全 Kimi「主脑 run 期实时旁白」作本期最后一步升级。**

## 二、需求分解

### 需求一 · 主脑对话流（#1，🔴承重）

**1.1 计划卡等待态清晰化（确认闸保留）**
提交计划后计划卡明确渲「⏳ 等你确认放行」等待态、主脑不输出多余收尾文字 → 用户一眼看出「在等我」而非「对话结束了」。确认闸（approve/reject/revise）**保留**（D-V1.2-86：用户 #1 要「选择确认协作计划」、比 Kimi 无闸更稳）。

**1.2 中间路自动汇总（本期主体·承重）**
主脑退场跑队员（守 T6 架构、run fire-and-forget headless、不碰后台解耦）→ **前端按每个队员/批干完自动 nudge 主脑吐一句阶段小结**（模拟 Kimi「边跑边旁白自动接续」）→ 全干完主脑自动产总汇总受管文档。= 第一期 option i 的加强版（每阶段一次 nudge 而非只终态一次）。
- **承重待 spike**：前端多次 nudge 的时序（主会话是否还活/没切走/idle destroy）、不打断用户输入、nudge 回合在对话里的呈现（可见的阶段小结）。
- 复用第一期已有：run 轮询（MastermindPollDriver/useMastermindStore）、终态汇总工具（create_artifact/list_artifacts，第一期 T6 已装）。

**1.3 全 Kimi 升级（🔴承重·本期最后一步）**
主脑真在 run 期间实时旁白（触碰 run↔主脑刻意解耦边界、option ii）。**定为第二期最后一步**——中间路 + #2 + #3 + 细节增强全部收官后再做，须独立 spike 验证（run 状态→主脑会话实时推送、不破 T6 解耦红线的可行性）。若 spike 风险过大可留三期。

### 需求二 · hover 浮框贴卡修复（#2，🟢回归 bug）

第一期 T5 引入的回归：`.brow` 卡片残留 GSAP identity transform → 成 `position:fixed` 后代包含块 → StageHoverPreview 浮框飞到卡片右下 +320px（真浏览器控制实验坐实、去 transform 后精确回 6px）。
- **修法**：浮框 **Portal 到 `document.body`**（脱离 `.brow`、fixed 恢复相对视口、popover-position 算法不动）；**连带修 StageSessionMenu**（T6 菜单同渲在 `.brow` 内、同坑）。
- **顺带**：剥 uuid8（StageHoverPreview 现显 `role-a1b2c3d4`、漏用 friendlyAgentName）+ hover 预览卡内容对齐 Kimi（显队员**职衔** + 该子任务**验收点/成功标准**，呼应需求 2.9① 子任务带验收点）。

### 需求三 · agent 集群框（#3，🟢视觉）

队员卡片在对话流里松散漂浮、缺限定框。对标 Kimi「(节点树图标) Agent 集群 ｜ N 个并行任务」框（真实页 class `.multi-agent-block`：框头图标+文案+计数、框内内部滚动）。
- **修法**：MastermindTeammateCards 外层加 `board` 修饰类（现只有 `pipeline-board` 基类致框样式全不生效）+ 渲 `.hd` 集群头（复用 T7 已验收的 `.board/.hd/.fork/.cnt` + git-branch SVG、与看板视觉定稿 kimi-v11.html 同源），**用专属类名承接避免波及看板 modal**；微调暗色 bg（现 `--bg:#0c0c0e` 比对话更黑=方向反、改透明/`--container`）。
- **⚠️ 必须处理·queued 态**：Kimi 一次全并行无排队；我们并发受限 → 框里会同时有 running（亮绿点阵）+ queued（排队中）两态。集群框须定 queued 子代理呈现（灰态/点阵不亮），别照抄 Kimi 全 running 视觉。
- 集群头文案 + 计数语义（Kimi「Agent 集群｜N 个并行任务」vs Next-Step「主脑派活｜已完成 x/N」）详设定。

### 需求四 · 细节增强（missed-details·分优先级）

- **🟡 队员拟人友好名**：Kimi 给子代理中文人名（纳什/逸飞…）+ 职衔 + 编号三件套；我们现 uuid8 未剥。至少剥 uuid8（含 #2）；友好命名详设定。
- **🟡 主脑创建角色可见卡**（可选）：Kimi「👤 创建助手｜🤖 日本市场研究员」内联卡（增参与感、呼应 #1③查看中途过程）。
- **🔵 下钻工作台（轻量版）**：Kimi 点子代理→右侧「Kimi's Computer」私有全轨迹+产物+走马灯切换（重）。我们**接受轻量版**：点队员→复用第七轮 StageSessionMenu 看 transcript / 该队员产受管文档；**可选**底部走马灯横切看各队员产出。**不做** Kimi 那种重工作区面板。
- **🔵 全局进度头条**（可选）：若主脑支持多阶段顺序推进（需求 2.2 第6步一阶段喂下一阶段）才补轻量整体进度；单批扇出则不需。

### 并发上限（D-V1.2-86）

**按每次任务调整、不设默认值**（呼应 Kimi 每阶段 fan-out 数不同 11/7/3/2 + 资源实测活会话仅~1MB/HARD_CAP 过度保守/真天花板是 CPU+限流非内存）。改现有「全局默认 3 + factory-config HARD_CAP」的并发设计 → 派活时按本次任务定并行数。详设定落点（提交计划时带并行数？还是主脑声明？）。

## 四、红线与约束

- 守 T6 架构：run fire-and-forget headless、主脑主会话与 run 解耦（中间路只用前端 nudge、不碰后台解耦；全 Kimi 升级作末步、单独 spike）。
- 守 D-R7B-07（UI 只 import type + fetch）、第七轮卡片族 t-kimi token 命名空间（集群框用专属类名隔离、不波及看板 modal）、不改 pi 内核。
- 确认闸保留（不学 Kimi 无闸）。UI 卡验收走真浏览器。

## 五、验收方向

真浏览器端到端（对标 Kimi 回放）：说需求 → 计划卡「⏳等确认」→ 确认 → 集群框内队员卡片实时进度（running 亮绿点阵 / queued 灰态）→ **每个队员干完主脑自动冒阶段小结** → 全干完主脑自动产汇总受管文档；hover 队员卡浮框贴卡（6px）显职衔+验收点；pageErrors=0。全 Kimi 升级单独验。
