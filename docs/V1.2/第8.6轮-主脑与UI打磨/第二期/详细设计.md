## §1 M1 · 中间路 nudge 驱动（本期主体·🔴承重·设计评审）

### 1.1 目标与形态

主脑退场跑队员期间，前端**观察 run 阶段进度**，每当「一个队员/一批干完」→ 自动 nudge 主脑吐一句阶段小结；全干完 → nudge 主脑产总汇总。= 第一期 option i（终态一次）的加强版（每阶段一次），参与感接近 Kimi「边跑边旁白自动接续」，改动仍纯前端。

**关键：nudge = 前端 programmatic 追一条隐式 user 消息触发主脑回合**，走 `handleFollowUp`（`useAgentSession.ts:522`）。不碰后台 fire-and-forget（`approve/route.ts:63` / `runMastermind` 一字不改，守 T6 + D-R8.6-13）。

### 1.2 落点与接口

- **感知源**：`useMastermindStore`（`lib/stores/useMastermindStore.ts`）的 `runs: Record<string, MastermindRun>` + `MastermindPollDriver`（`components/MastermindPollDriver.tsx`）已每 2s 轮询 running run → 组件可订阅 `runs[runId].stages[].status` 的翻转（pending→running→done）。
- **触发口子**：`handleFollowUp(message)`（`useAgentSession.ts:522-540`）——往 `sessionIdRef.current`（当前主脑会话）追 user 消息 + `sendAgentCommand type follow_up`。**前提**：nudge 时该会话即主脑主会话（用户没切走、`sessionIdRef` 仍指主脑）。
- **新增**：一个前端 nudge 驱动器（hook 或组件，如 `useMastermindNudge` / 挂在 ChatWindow）——
  - 输入：当前会话的 run 集合（从 transcript 派生的 runId → `runs[runId]`，复用 `derivePlanRefsFromMessage`）+ `handleFollowUp`。
  - 逻辑：观察每个 running run 的 `stages`，记「已 nudge 过的 (runId, stageOrder)」集合去重；某 stage 首次翻到 `done` → `handleFollowUp("（第 N 个队员「XX」已完成，请简要汇报这一步的进展）")`；run 翻到 `done/partial` → `handleFollowUp("（所有队员已完成，请产出一份汇总受管文档并给我文字摘要）")`（复用第一期 T6 汇总纪律 prompt、主脑会自动 list_artifacts→read→create_artifact）。
  - 去重：`useRef<Set<string>>` 存 `${runId}:${stageOrder}` + `${runId}:__final__`；跨 run 天然不串（key 含 runId）。

### 1.3 承重前提（走设计评审证伪·非 spike）

> 按 cc-multi-agent-dev-flow「实现前证伪承重前提」：默认怀疑、派反驳者死磕 unhappy 路径。以下**全走 ultracode 设计评审**（读码 + 对抗），**不搭 spike**。

- **前提 M1-P1（会话活性）**：nudge 时主脑主会话仍活、`sessionIdRef.current` 仍指它。**反驳者死磕**：run 是异步跑（可能几分钟），期间用户可能①切到别的会话（ChatWindow 卸载/换 key、`sessionIdRef` 变）②主脑会话 idle 10min 被 destroy（`session-reattach` re-attach 后 sessionId 是否变？）。→ 评审须确认：nudge 驱动器**随会话切换卸载**（同 MastermindPollDriver 挂 ChatWindow）→ 切走则不 nudge（run 仍在后台跑、切回来轮询补齐终态、可靠）；idle destroy 后 follow_up 会触发 re-attach（`reattachOrchestratorSession`）、sessionId 复用同一个（jsonl 重开）→ handleFollowUp 的 `sessionIdRef.current` 仍有效。**结论待评审坐实**。
- **前提 M1-P2（不打断）**：nudge 是否打断用户正在输入/主脑正在跑的回合。**反驳者死磕**：`handleSend`/`handleFollowUp` 在 `agentRunning` 时的行为——`handleSend:348` `if (agentRunning) return`（跑时不发）；`handleFollowUp` 无此 guard（follow_up 语义 = 排队追问）。nudge 若在主脑回合进行中 fire，会不会叠加/丢失/打断用户输入框？→ 评审须定：nudge 只在 `!agentRunning`（主脑空闲）时 fire、否则延到本回合结束（观察 `agent_end`）；且 nudge 不动用户输入框（handleFollowUp 不碰 ChatInput 内容）。
- **前提 M1-P3（去重 + 多 run）**：阶段翻转 → nudge 严格一次、多 run 并存不串。**反驳者死磕**：轮询每 2s 产新 run 对象、React 并发渲染下 effect 可能重跑；一条对话可多个 submit_plan（打回后旧 failed + 新 run）。→ 评审确认 `Set<"${runId}:${stageOrder}">` 去重 + key 含 runId 天然隔离（同 derive-run-ids 的多 run 自证范式）。
- **前提 M1-P4（时序）**：nudge 回合与 run 轮询、与队员卡片渲染的并发。**反驳者死磕**：主脑 nudge 回合本身产 assistant 消息、可能又含 submit_plan（主脑误理解）？→ 评审确认 nudge prompt 措辞明确「简要汇报/产汇总」不诱发再 submit_plan；且 nudge 回合的 messages 正常进 transcript、不影响已渲的队员卡片（卡片按 runId 独立）。

**为何不 spike**：以上全是「读 useAgentSession/store/reattach 现有代码 + 推理失败模式」能 settle 的问题，无内核 runtime 未知行为。末尾**真浏览器端到端**（DeepSeek 真凭证跑多队员 run、看阶段完主脑真冒小结、pageErrors=0）是权威 oracle。

### 1.4 红线

- 只前端 nudge、**绝不改** `approve/route.ts` / `runMastermind` / T6 fire-and-forget 解耦。
- nudge 有可见痕迹（阶段小结是主脑真回合、渲在对话流，非无痕系统提示）——符合需求「用户能查看中途工作过程加强参与感」。
- 不改 pi 内核（走内核既有 follow_up 命令）。

## §3 M3 · hover 浮框贴卡修复（🟢回归·设计评审 + 真浏览器）

### 3.1 真根因（lead 真浏览器控制实验坐实）

`MastermindTeammateCards.tsx:62-68` 的 `gsap.from(cards,{y:8,...})` 入场补间**完成后不清 inline transform** → `.brow` 残留 identity `matrix(1,0,0,1,0,0)` → 成其 `position:fixed` 后代的**包含块** → `StageHoverPreview`（渲在 `.brow` 内、`PipelineStageCard.tsx:154-167`）+ `StageSessionMenu`（`:170-179`）的 `position:fixed` 不再相对视口 → `computeFixedPopover`（`popover-position.ts`）算的视口坐标被整体平移到卡片右下 +320px。**算法本身没错**（真浏览器验证：去 transform 后精确回 6px gap、left 对齐）。

### 3.2 修法：Portal 到 document.body

- **`StageHoverPreview.tsx`**：把 fixed 浮框的 return JSX 包 `createPortal(<div style={{position:fixed...}}>...</div>, document.body)`（`react-dom`）——脱离 `.brow` 包含块、fixed 恢复相对视口、`computeFixedPopover` 逻辑与 `anchorRef.getBoundingClientRect()`（仍是视口坐标）一字不动。
- **`StageSessionMenu.tsx`**：同样 createPortal 到 body（同坑、`:170-179` 渲在 `.brow` 内）。
- SSR 安全：两组件的坐标测量已在 `useLayoutEffect`（客户端）+ `visibility:hidden` 首帧兜底；createPortal 目标 `document.body` 仅在客户端调用（`"use client"` 组件、`typeof document` 客户端才有）→ 加 `typeof document !== "undefined"` 守卫或 mounted 态。

### 3.3 承重前提（走设计评审证伪·非 spike）

- **前提 M3-P1（token 作用域·隐藏命门）**：t-kimi CSS 变量（`--pop/--line/--task/--run-accent…`）定义在 `.pipeline-board.t-kimi-*` 作用域（`PipelineBoardStyles.tsx:17-18`）。浮框 Portal 到 `document.body` 后**脱离该作用域** → `var(--pop)` 等全失效 → 浮框白屏/无边框。**这是修复的隐藏命门**。**反驳者死磕**：Portal 修好位置却引入白屏 = 更糟。→ 评审须定处理：(a) Portal 目标不是裸 body、而是一个带 t-kimi 类的 wrapper（Portal 到 `.pipeline-board.t-kimi-*` 容器？但那正是包含块问题源）；(b) 浮框内联 style 里的 var() 改成不依赖 t-kimi 作用域的值（外提到 `:root` 或组件自带 style 变量）；(c) Portal wrapper 自己带 t-kimi 类（在 body 下建一个 `<div class="pipeline-board t-kimi-dark">` 容器承接浮框、但它无 transform 不成包含块）。**详设推荐 (c)**：createPortal 到一个动态创建/复用的 body 级 `.pipeline-board.t-kimi-{theme}` 容器（无 GSAP transform → 不成包含块 → fixed 相对视口 + token 有值），一举解决位置 + token。lead 定（§10-②）。
- **前提 M3-P2（事件冒泡）**：`StageSessionMenu` 外点关用 `document.addEventListener("pointerdown", ..., true)` 捕获（`:60-70`）、Esc 用 document 监听——Portal 后仍在 document 上、正常工作；但 Portal 脱离 React 树后**合成事件冒泡**是否影响父卡 `onClick`（`PipelineStageCard.tsx:115` setMenuOpen）？React Portal 保留 React 事件冒泡（冒泡到 React 父级、非 DOM 父级）→ 菜单内 `onClick stopPropagation`（`:156`）仍拦得住。评审确认。
- **前提 M3-P3（波及看板 modal）**：`PipelineBoard` 的浮框同样渲在 `.brow` 内（`PipelineStageCard` 共用）——但看板卡片**无 GSAP transform 入场**（`MastermindTeammateCards` 独有 `gsap.from`）→ 看板浮框本无此 bug。Portal 修改 `StageHoverPreview`/`StageSessionMenu` 会**同时作用于看板**（两处共用组件）→ 须确认 Portal 后看板浮框仍正确（token 作用域同样处理、位置不回归）。真浏览器验收须**看板 + 主脑内联两处都验**（回归防护）。

**为何不 spike**：Portal 波及面是「读组件 + React/CSS 语义推理」能 settle 的；真浏览器（getBoundingClientRect 实测 gap=6px + Portal 后 token 不白屏 + 看板不回归）是权威 oracle。

### 3.4 附带（并入 M5a）

`StageHoverPreview.tsx:156` 头徽章剥 uuid8（`friendlyAgentName`）+ 内容加职衔/验收点——见 M5a。**同文件、随 M3 一并改**（避免两次改 StageHoverPreview）。

### 3.5 红线

t-kimi token 命名空间不波及看板 modal（Portal wrapper 的 token 处理对看板浮框统一生效）；`popover-position.ts` 算法不动。

## §6 M6 · 并发按任务调整（🟡P1·设计评审 + 逻辑层）

### 6.1 现状

并发上限是**全局单值**：`acquireSlot`（`concurrency-gate.ts:34-52`）的 `limit` 默认 `readMaxConcurrent()`（`factory-config.ts:42` 读 `~/.pi/factory-config.json` 全局）。`runMastermind`（`mastermind-orchestrator.ts:169`）每阶段 `acquireSlot({timeoutMs:Infinity})`（不传 limit→全局）。

### 6.2 目标与开放问题（§10-⑧·lead 定核心）

「按每次任务调整、不设默认值」（D-V1.2-86）——须让本次 run 携带本次并行数、该 run 的 worker 用该值。**开放问题（承重·lead 定）**：
1. **并行数从哪来**：(i) 主脑在 submit_plan schema 声明（加字段，主脑按任务判断并行度）；(ii) 用户在计划卡确认时设（UI 加输入）；(iii) 派 N 个队员即 N 并行（N=teammates.length、clamp 到 HARD_CAP）。
2. **⚠️ 串行骨架的语义冲突**（关键）：当前编排器**串行 for 循环 + 累积喂下游**（`mastermind-orchestrator.ts:107/181` D-R8.6-11 Q6② MVP 串行、「同时 2 队员」靠 acquireSlot 排队物理并发自然发生）。「按任务调并行」在串行模型下语义受限——**调 acquireSlot 的 limit**（放宽排队上限、让更多 worker 物理并发，但编排逻辑仍串行发起）vs **真并行执行**（触碰串行骨架、爆炸半径大、失败/pause 语义在并行下复杂化）。**详设强烈倾向前者**（只放宽 limit、不动串行骨架）：即 run 携带 `maxConcurrent`、`acquireSlot({limit: run.maxConcurrent ?? readMaxConcurrent()})`——保持串行发起 + 累积喂下游语义，仅调排队墙。真并行留后续。lead 定。

### 6.3 承重前提（走设计评审证伪·非 spike）

- **前提 M6-P1（不破 acquireSlot 不变量）**：`acquireSlot` 计 `__piSessions.size`（主脑 + 队员 + 前端会话同计，`concurrency-gate.ts:8-11/21`）。改 limit 是否破「主脑占 1 槽 → 队员用剩余」不变量、是否与 M7 全 Kimi（主脑常驻）冲突。→ 评审确认：只调 limit 值、不动计数源；主脑常驻占 1 是既有事实（D-R8.6-13 已知会）。
- **前提 M6-P2（HARD_CAP 兜底）**：任意来源的并行数须 clamp 到 `[1, HARD_CAP]`（`factory-config.ts:62` 现成 clamp、复用）——防用户/主脑填天文数字。

**为何不 spike**：改并发是「读 acquireSlot/factory-config + 推理不变量」能 settle 的；逻辑层（并行数落点单测：submit_plan schema / clamp / acquireSlot limit 传递）+ 真浏览器（派活时并行数生效、queued 卡数正确）验。

### 6.4 红线

不改 pi 内核；HARD_CAP 兜底保留（资源实测 D-R7-09）；不改 acquireSlot 本体（只传 limit 参数、仿第七轮传 Infinity 范式）。
