## 二、本期落地流程（对标 Kimi 参与感、保留确认闸）

```
用户说需求
   ▼
主脑（主会话·总管）理解 → 调 submit_plan（第一期已有）
   ▼
计划卡渲「⏳ 等你确认放行」等待态（M2 强化：明确"在等我"、主脑不吐多余收尾）
   ▼ 用户点「确认放行」→ approve 路由 fire runMastermind（第一期已有）
   ▼

   ├─ running：亮绿点阵 / 呼吸（第一期已有）
   └─ queued：灰态 / 点阵不亮（M4 补·我们并发受限、Kimi 无此态）
   ▼
★ 每个队员 / 每批干完 → 前端 nudge 主脑吐一句阶段小结（M1·本期主体）
   （模拟 Kimi「边跑边旁白自动接续」，纯前端 handleFollowUp，不碰后台解耦）
   ▼
全干完 → 主脑自动产总汇总受管文档（M1 复用第一期 T6 汇总工具）
   ▼
hover 队员卡 → 浮框贴卡 6px（M3 Portal 修回归）显职衔 + 验收点（M5）
   ▼
```

## 三、模块划分

### M1 · 中间路 nudge 驱动（需求一 1.2，🔴承重·本期主体·走设计评审）

- **职责**：主脑退场跑队员期间，前端**按每个队员/批干完自动 nudge 主脑吐一句阶段小结**（模拟 Kimi 边跑边旁白自动接续），全干完 nudge 一次让主脑产总汇总。= 第一期 option i（终态一次 nudge）的加强版（每阶段一次）。
- **落点**（lead 已坐实）：
  - 触发口子 = `hooks/useAgentSession.ts:522 handleFollowUp`（前端 programmatic send，`sendAgentCommand type follow_up` 往当前主脑会话追一条 user 消息、触发主脑一个回合）。
  - 终态/阶段进度感知 = `useMastermindStore` 的 `runs` Record + `MastermindPollDriver` 轮询（`selectMastermindNeedsPolling` 仅 running；run 每 2s 刷新一次，可观察 `stages[].status` 从 pending→running→done 的翻转）。
  - 汇总能力 = 第一期 T6 已装 create_artifact/list_artifacts（`orchestrator-session-wiring.ts`）+ 总管 prompt 已含汇总纪律（`orchestrator-session.ts:50-56`）。
- **复用第一期**：轮询 store、汇总工具、总管 prompt 全现成；**新增仅前端「阶段完 → nudge」驱动器**（一个 hook/组件，观察 run 阶段翻转、去重后 handleFollowUp 一条隐式阶段小结 prompt）。
- **红线**：守 T6 架构——**只前端 nudge、绝不碰后台 fire-and-forget 解耦**（不改 approve 路由、不改 runMastermind）；nudge 是「用户视角的隐式 follow_up」、有可见痕迹（阶段小结是主脑真回合、渲在对话流里）。
- **承重前提**（详设 §识别、走设计评审证伪，非搭 spike）：① nudge 时主脑主会话是否还活/没被切走/idle 未 destroy；② nudge 是否打断用户正在输入/正在跑的回合；③ 阶段翻转 → nudge 的去重（一阶段只 nudge 一次、多 run 并存不串）；④ nudge 回合与 run 轮询、与队员卡片的并发时序。

### M6 · 并发按任务调整（需求一附带 D-V1.2-86，🟡P1）

- **职责**：改现有「全局默认 3 + factory-config HARD_CAP」→ 派活时按本次任务定并行数、不设默认值。
- **落点**：`lib/pi/factory-config.ts`（`readMaxConcurrent`/`DEFAULT_MAX`/`HARD_CAP`）+ `lib/pi/concurrency-gate.ts`（`acquireSlot` 的 `limit` 默认 `readMaxConcurrent()`）。
- **现状**：并发上限是**全局单值**（`readMaxConcurrent()` 读 `~/.pi/factory-config.json`），`acquireSlot` 每次读全局。要「按本次任务」需让 run 携带本次并行数、`acquireSlot` 该 run 的 worker 用该值。
- **待定落点**（详设定·lead 决定）：并行数从哪来——(i) 主脑在 submit_plan 里声明（加 schema 字段）；(ii) 用户在计划卡确认时设；(iii) 派 N 个队员即 N 并行（N=teammates.length、上限 clamp）。⚠️ **当前编排器是串行**（`mastermind-orchestrator.ts:107` for 顺序 + 累积喂下游），「并行数」在串行模型下语义受限（D-R8.6-11 Q6②「MVP 串行、同时 2 队员靠 acquireSlot 排队物理并发」）→ 详设须澄清「按任务调并行」是调 acquireSlot 的 limit（放宽排队上限）还是真并行执行（触碰串行骨架、爆炸半径大）。
- **承重前提**（走设计评审）：改并发上限是否破 acquireSlot 的「主脑 + 队员同计 `__piSessions.size`」不变量、是否与 M7 全 Kimi（主脑常驻占 1 槽）冲突。
- **红线**：不改 pi 内核；HARD_CAP 兜底保留（资源实测活会话仅 ~1MB、真约束是 CPU + LLM 限流，D-R7-09）；配置文件 MVP 无 GUI setter（D-V1.2-48）除非本期新增。

### M7 · 全 Kimi 末步·主脑 run 期实时旁白（需求一 1.3，🔴承重·唯一 spike·本期最后一步）

- **职责**：主脑真在 run 期间实时旁白（触碰 run↔主脑刻意解耦边界、option ii）。**本期最后一步**——M1~M6 全收官后再做；风险过大可留三期。
- **落点**（触碰 T6 解耦边界）：`.../approve/route.ts:63` 的 `void runMastermind` fire-and-forget（主脑主会话与 run 刻意解耦，D-R8.6-13）+ `orchestrator-session-wiring.ts`（主脑会话装配）+ `mastermind-orchestrator.ts`（run 循环）。要让 run 状态实时推到主脑主会话 → 须建 run→主脑会话的推送通道（服务端），破现有解耦。
