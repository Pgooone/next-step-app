## §0 承重优先原则

主脑大脑的**最终形态取决于 spike-1**（主会话能否装派活工具 + 走 A 分支还是 B）。故本详细设计**先把 spike-1 的承重前提/命门/判据钉死**，spike-1 GO 后再据实回填 M1/M2 最终方案；M3~M8 先给接口契约骨架，spike-1 后细化实现。

## §1 承重命门 · spike-1（派活工具注入路径）

### 1.1 要证伪的承重前提
> 按 cc-multi-agent-dev-flow「实现前证伪承重前提」：默认怀疑，派反驳者死磕 unhappy 路径。

- **前提 P1**：主会话起会话链能接 `resourceLoader`（注入总管 prompt）+ `customTools`（派活工具），且**不破坏普通主会话**现有行为。
- **前提 P2**（D-V2-04 命门，最尖锐）：派活 customTool 名进主会话**激活工具集**后，主脑 LLM 能**真正调到**该工具；**漏名则内核 `_refreshToolRegistry` 按名过滤致连注册都不到、静默无报错**。
- **前提 P3**：总管 prompt 经 `appendSystemPromptOverride` 注入后**真进系统提示**、不被内核 `_rebuildSystemPrompt` 事后覆盖（D-24 红线：禁事后改 `session.state.systemPrompt`）。
- **前提 P4**：主脑「既自己动手（编码工具 bash/write/edit）又派活（派活工具）」并存，**不被误判为文档会话**、不套 DOC_SESSION_TOOLS 受限集。

### 1.2 命门（绝不能漏的致命路径）
**P2 是命门**：若派活工具「装了但调不到」（白名单漏名），整个主脑哑火且无报错——最难查。**反驳者须死磕**：不能只断言「工具在 options.customTools 里」，必须断言「**经 createAgentSession 起会话后、工具在内核激活注册表里、faux 模型产该工具调用时 execute 真被触发**」（vacuous 防范：漏名负对照必须真的调不到）。

### 1.3 spike 方案（hermetic·faux 模型，无需真凭证）
落点 `next-step-V1.2/lib/pi/orchestrator-session.spike.test.ts`（spike 后删、命门事实迁详细设计 + 永久回归迁正式单测，仿第八轮 evict-session.spike）。

构造最小主脑装配：
1. `buildOrchestratorResourceLoader(injectionBlock)` —— 照 `agent-profile-session.ts:116-156` 造带 `appendSystemPromptOverride` 的 DefaultResourceLoader。
2. `MASTERMIND_TOOLS = [submit_plan, dispatch_task]`（defineTool，execute 打桩记录被调）。
3. 装配 options = `{ tools: [...编码工具名, 'submit_plan', 'dispatch_task'], customTools: MASTERMIND_TOOLS, resourceLoader }`。
4. `createAgentSession(options)` + faux 模型。

### 1.4 判据（5 断言全绿 = GO）
- **A1**（P3）起会话后 `session` 系统提示含总管注入块、且 `await loader.reload()` 后不被重建覆盖。
- **A2**（P2 命门·正）faux 模型产 `submit_plan` 工具调用 → 该工具 execute **真被触发**（桩记录命中）。
- **A3**（P2 命门·负对照）把 `submit_plan` 从 `tools` 白名单移除（仅留 customTools）→ faux 产同调用 → execute **不被触发**（坐实「漏名静默失效」、证 A2 非 vacuous）。
- **A4**（P4）装配后激活集**同时含**编码工具（bash/write/edit）+ 派活工具，且会话**未被判为 doc 会话**（无 DOC_SESSION_TOOLS 收窄）。
- **A5**（P1）该装配是**独立函数**，不改 `startRpcSessionInner` 普通分支签名（普通主会话零影响、用变异/对照证）。

### 1.5 两子路线落点评估（spike 附带产出）
spike-1 同时评估：**(A) 改 `/api/agent/new` + rpc-manager 加「主脑模式」分支** vs **(B) 纯 API 编排（主脑产计划经 submit_plan 工具，但派活走独立 API 不碰 rpc-manager 主会话装配）**。判据 = 哪条「装派活工具到主会话」爆炸半径小。**路线 C 要工具派活 → 倾向 A**，但若 A 改 rpc-manager 风险过大，B 可作「主会话只装 submit_plan（产计划）、dispatch 走确认后 API」的降级。spike 实测定。

## §2 M1 · 主脑会话装配（lib/pi，新增）

- `lib/pi/orchestrator-session.ts`：`buildOrchestratorResourceLoader(deps)` + `assembleOrchestratorSessionOptions(deps)` → `{ tools, customTools, resourceLoader }`。仿 `agent-profile-session.ts`。
- `lib/pi/mastermind-tools.ts`：派活工具集（defineTool）：
  - `submit_plan(plan: {teammates:[{name,role,subTask,acceptanceCriteria}], notes})` —— 落计划 awaiting_approval、返回 planId。promptGuidelines 挂「先 submit_plan 等确认、再 dispatch」纪律。
  - `dispatch_task(planId)` —— 确认后放行，fire 主脑编排器、返回 runId。
  - `check_status(runId)` —— 轻读 run 状态（队员卡片数据）。
  - `list_teammates(projectId)` —— 列已建 agent 池（供主脑选）。
  - （`spawn_agent` 临时造角色并入 dispatch_task 的 teammate.isNew 分支，见 M6）
- **闭包注入**：projectId / projectRoot / registerInnerSession（execute 服务端 import 编排器）。
- **白名单**：工具名进激活集（spike-1 A2/A3 命门）。

## §3 M2 · 起会话链改造（app/api + rpc-manager + hooks）

- 据 spike-1 选 A：`/api/agent/new` body 加 `mastermind?:boolean`；route → `startRpcSessionInner` 加「主脑模式」分支：装 `assembleOrchestratorSessionOptions` 的 resourceLoader+customTools，白名单=编码工具 ∪ 派活工具名。**新分支、普通主会话零改**（A5）。
- `hooks/useAgentSession.ts:372` 首条消息分支透传 `mastermind` 旗标。
- **自动判断当总管**（开放问题，见 §10-④）：旗标默认开（主会话即总管），由主脑大脑（总管 prompt）自判「大任务才拆活派人、小事直接答」。

## §4 M3 · 计划确认闸（lib/domain + app/api + components）

- **状态机**：run 增前置态 `awaiting_plan_approval`。`submit_plan` 落 `MastermindRun{ status:'awaiting_plan_approval', plan }`（**不 acquireSlot、不 fire**）。
- **路由**：`POST /api/projects/[id]/mastermind/runs/[runId]/approve | reject | revise`（翻状态）。approve → 才 `void runMastermind()` fire。
- **UI**：计划卡（内联，见 M5 同锚点）——渲 plan.teammates + 验收点 + 「确认 / 否决 / 打回」按钮。打回 = reject + 主脑据反馈重 submit_plan。
- **承重**（spike-3 后验）：与 run-controllers globalThis 单例时序（dev 热重载丢 controller，run-controllers.ts:8-11 先例）。

## §5 M4 · 主脑编排器（lib/domain，新增）

- `lib/domain/mastermind-orchestrator.ts`，仿 `pipeline-orchestrator.ts` 但：
  - **终态扩展**：`done | failed | paused | partial`（§10-⑤）。
  - **失败处理算法**（§2.10🔴1）：worker 失败/超时 → `retryCount<1` 则自重试 1 次；否则 `status='paused'` + 落 `failedTeammate` + 选项枚举（retry/reassign/skip/abort），**暂停等用户**（不 fail-fast 中止后续）。
  - **动态 run**：不读正式蓝图——`runMastermind(projectId, plan)` 直接据 plan.teammates 起 worker（每 teammate 经 AgentProfileStore 拿/造 agentId）。
  - **复用**：acquireSlot(**Infinity**)、runWorker、evictSession(本阶段 sessionId)、setOwner、累积喂下游、run-controllers cancel。**只走 pipeline 族**。
- **承重**（spike-2）：可暂停中断点 + 暂停期 evict 释槽。

## §6 M5 · 队员卡片内联（components）— 乙·对标 Kimi

- **内联锚点**：`ChatWindow.tsx` 在 `messages.map`（:375-421）按 entryId 旁挂「主脑派活块」——**不塞内核 messages 数组**，用独立 `useMastermindStore`（仿 usePipelineStore「import type + fetch」）按 entryId→runId 映射插卡。
- **数据契约**（§10-①）：`MastermindStage extends PipelineRunStage`（卡片族吃超集、不动 PipelineRunStage），加 `acceptanceCriteria? / failureOptions? / isDynamic?`。卡片族（PipelineStageCard 等）props 收超集、向后兼容。
- **动效**（D-R8.6-02）：GSAP stagger 入场 / 状态流转 / running 呼吸+LED / 完成打钩。守 prefers-reduced-motion。
- 守 **D-R7B-07**（只 import type + fetch、真浏览器验收兜底）。

## §7 M6 · 临时造角色（lib/domain）

## §8 M7 · 长上下文策略（lib/pi + 编排层）

- **复用内核 compaction**（零接线、起会话即生效）。

| # | 问题 | lead 决定 |
|---|---|---|
| ① | 数据契约：扩 PipelineRunStage vs 卡片族吃超集 | **卡片族吃超集**（`MastermindStage extends PipelineRunStage` + 可选新字段），不动 PipelineRunStage（避牵动第七轮回归/落盘）。 |
