## 一、背景与动机

Next-Step 已具备「多 Agent 派发（dispatch）」「可保存流水线 + 阶段看板（第七轮）」「会话按 agent 分组（第五轮）」等能力，但编排始终是**用户手动驱动**（手填子任务 / 手配流水线蓝图）。

用户希望引入「主脑」——让**主会话本身**成为一个 AI 总管：你说一句需求，它就理解、拆活、派 agent、盯进度、汇总，并像 Kimi agent 集群那样在主对话里实时显示子 agent 队员。

同时，现有按钮与流水线卡片交互「很烂」（按钮零统一、卡片 hover 生硬、状态有 bug、可访问性近零），需配套打磨——且打磨点正与主脑「把卡片搬进对话」交汇。

## 二、需求一 · 主脑（主会话即总管）

### 2.1 本体定位（D-V1.2-80 / D-R8.6-01）

**主会话本身 = 总管（主脑）**，不新建独立 orchestrator 实体、不另起名 / 造身份。你平时跟主会话聊天；丢给它大任务，它自动进入「总管」状态。

层级：**主会话（总管）= 父**；它派出的队员 = **子会话**（= 对话里那排队员卡片）。

### 2.2 工作流（7 步）

1. 你对主会话说需求（一句话，可简单可复杂）
2. **自动判断**：小事正常回；大任务则理解、拆成几步、每步配队员（队员从已建 agent 挑 或 临时造新角色）
3. **给你看派活计划，你点头**（计划确认闸）
4. 派活开干（同时最多 3，多了排队）
5. **对话里冒出队员卡片**：实时显示「在干 / 干完 / 在等」，点开看队员详情
6. 主脑边盯边推进（一阶段喂下一阶段）
7. 全干完，汇总给你

### 2.3 定位与自治度（已拍板）

- 定位 = **统一上层·主脑统领**（D-V1.2-74）：流水线、手动派发都成为主脑的下属手段。
- 自治度 = **计划确认制**（D-V1.2-75）：大动作前你拍板。
- 主脑**既能自己动手又能派活**（D-R8.6-03，继承主对话带工具能力 = Kimi 主脑自己搜/跑代码）。

### 2.4 要设置的（主会话当总管的行为规矩）

| 设置项 | 说明 |
|---|---|
| 能派哪些 agent | 圈定可调动的 agent 池 + **允不允许临时造新角色** |
| 派活前是否先给计划 | 默认开（计划确认闸） |
| 并发上限 | 默认 3（守项目规矩、可配） |
| （模型 / 说话风格） | 沿用主会话既有设置，非新增 |

> 注意：因「主会话即总管」，**没有「创建主脑 / 给主脑起名」这类身份设置**——只设「行为规矩」。

### 2.5 队员（子 agent）来源（D-V1.2-79）

**两种都要**：
- ① 从已建 agent 池**点名**（复用现有 dispatch）；
- ② 主脑**临时现造**新角色（如「日本市场研究员」，要 N 个就实例化 N 个，= Kimi 模式）。

> 衍生待定（留详细设计）：临时造出的 agent 用完即弃 vs 存为正式档案进池复用（lead 倾向后者，呼应「两种都要」闭环）。

### 2.6 队员可视化（D-V1.2-77 / D-R8.6-02）

**内联团队卡·长在主对话里**：主脑派活时在 ChatWindow 消息流内联渲染「队员卡片」（对标 Kimi「Agent 集群」块）。

- **复用第七轮卡片族**：`PipelineStageCard`(`.brow`) + `StageHoverPreview`（悬停浮窗）+ `StageSessionMenu`（点击进详情）+ 底部切换条——**不重画**。
- 本轮活 = ①把卡片从独立看板面板「搬」进对话流内联；②数据源从 `PipelineRunStage` 适配为「主脑派活的队员」；③加 GSAP **克制**流程动效（卡片 stagger 逐个入场 / 状态平滑流转 / running 呼吸+LED 渐亮 / 完成打钩 / 阶段衔接，守 Swiss + `prefers-reduced-motion`）。

### 2.7 第一期范围 / 二期（D-V1.2-76 修正版）

- **第一期做**：完整闭环（理解→计划→确认→派活→队员卡片→汇总）+ **两种队员来源（挑现有 + 临时造）** + 卡片内联对话 + 计划确认闸 + 守并发。
- **留二期**：把「启动一整条预设流水线」也作为主脑的派活手段、更深的多轮自动返工、临时造 agent 的高级管理。

### 2.8 红线与约束（D-R8.6-03）

- 主脑 + 队员都计入**并发 ≤3**（Kimi 一次 11 并行在 Next-Step **自动排队**，复用第七轮 queued 态 + 编排器并发门）。
- **不改 pi 内核**；纯文件无数据库；本地单用户。
- 复用第七轮编排器 / 看板机制，不另造。

### 2.9 主脑作为「完整 Agent」的 7 要素对照（D-V1.2-81）

> **前提**：主脑 = 主会话（已是 pi 内核 agent）+ 总管能力，**非从零造** → 多数要素内核 / 现有项目已提供，只补「总管特有」少数项（守「简单优先」红线，不重复造轮子）。

| 要素 | 现状 | 本轮处置 |
|---|---|---|
| ① 任务/边界/成功标准 | 边界已有（行为规矩+红线）；成功标准弱 | 🟡 **第一期增强**：派活计划为每个子任务带验收点（像 Kimi「输出要求+✓完成项」） |
| ② 系统提示词 | 计划内 | 🔵 必做（总管提示） |
| ③ think-act-observe 主循环 | 内核已有 | 🟢 复用 |
| ④ 至少一个工具 | 主会话有 + 派活工具 | 🔵 已计划（派活工具） |
| ⑤ 短/长期记忆 | 短期有（会话上下文）；长期专属无 | 🟢 短期复用；🟡 **长期记忆留二期**（第一期靠读项目文件） |
| ⑥ 护栏 + trace | 大部分有（计划确认+并发+受限工具 / transcript+run+版本） | 🟢 复用 + 🔵 计划确认闸 |
| ⑦ 评测用例 | 测试框架有（vitest+双层验收）；大脑拆活质量 LLM 非确定性、难自动测 | 🟡 **标杆场景**：sf-mini 全流程 → 「好看/好用/可运行的 UI 原型」作端到端 oracle；机制层走确定性测试 |

> 结论：主脑设计在「完整 Agent」7 要素上**全覆盖**——大半复用内核/现有，新增聚焦 ② 总管提示、④ 派活工具、⑥ 计划确认闸、① 子任务成功标准；⑤ 长期记忆与 ⑦ 自动评测框架留二期 / 不做。

### 2.10 设计完整性补充 · 7 要素外的 9 项缺口处置（D-V1.2-82）

> 进设计前完整性体检，补多 agent 编排常见缺口。

**🔴 第一期定调（影响骨架）**：
1. **失败处理**：队员失败 / 超时 / 部分成功 → 主脑轻微自重试 1 次，不行**停下上报 + 给选项**（重试 / 换人 / 跳过 / 中止），不反复重试（防烧钱 / 死循环）。
2. **开工后中途控制**：复用第七轮 run cancel（随时喊停整任务）；「改方向」第一期 = 停掉重说，执行中插话改向**留二期**。
3. **计划可编辑**：第一期「同意 / 否决 / 打回重拟」（不逐行编辑）；执行中主脑想改计划**要再确认**。
4. **信息传递**：复用第七轮「上游累积喂下游」+ 主脑给队员下发任务卡（背景 + 成功标准）+ 主脑读产物汇总。**详细设计重点**。

**🟡 设计时定（有现成复用）**：
5. 产物归属 → 复用**第四轮**受管文档；6. 队员工具/权限 → 复用**第六轮** doc/coding 模式；7.「自动判断当总管」边界 → 主脑判断 + 用户一句话覆盖。

**🔵 技术承重点（详细设计重点攻）**：
8. **长任务上下文管理**：主脑跑大任务对话会爆 → **承重点**，详设专门处理（内核 compaction）；9. 成本/时间感知 → 计划确认时提示「派 N 个队员」。

## 三、需求二 · UI 打磨（按钮 + 流水线卡片）（D-V1.2-78 / D-R8.6-04）

### 3.1 方向：不引组件库 · 自建 `.btn` + 修真 bug + 补 a11y

ultracode 9-agent 调查（含对抗式怀疑者复验全部 file:line + 7 个对比度数字）判定：本轮真痛点 = **按钮零统一 + 可访问性近零**，非缺库的行为层。引库会制造第二范式（内联 style vs className 实测 900:62）、违反三红线（简单优先 / 不为单次用途加抽象 / surgical-changes）、稀释 Swiss 语言。故**自建 `.btn` 语义类层 + 修卡片真 bug + 补 a11y**；组件库留未来弹层族（届时 Base UI 非 shadcn）。

### 3.2 诊断（现状痛点·摘要）

- **按钮**：182 个全内联手写、无统一 Button、borderRadius 16 种取值、danger 红 4 种写法、全项目仅 1 处 focus-visible、暗色主按钮文字对比度 2.54:1（不达 WCAG）。
- **流水线卡片**：hover 硬 outline 瞬切无过渡、状态徽章裸 emoji（⏳/✓/✕）、**queued 状态 bug（卡片显「排队中」、浮窗显「待执行」自相矛盾）**、双蓝割裂、键盘不可达、动画无 reduced-motion 守卫。

### 3.3 分档（P0 / P1 / P2）

- **P0（真缺陷·无争议）**：① 暗色 primary 文字对比度 → 新增 `--color-on-accent`（暗色深字 7.46:1）② danger 实心色 #ef4444(3.76 FAIL)→#dc2626(4.83) ③ `status-meta.ts` 补 queued 键 ④ StageHoverPreview 头徽章配色改用看板 token（消双蓝割裂）。
- **P1**：`globals.css` 落 `--btn-*` token + `.btn` 四型（primary/secondary/ghost/danger）× 六态（默认/hover/active/focus-visible/disabled/loading）；`PipelineBoard` `.brow` 交互升级（hover 过渡、emoji→chip、选中态）；三组件键盘 a11y（role/tabIndex/Enter-Space/Esc/外点关）；reduced-motion 守卫。
- **P2（移出本轮·后续多轮渐进）**：900 处内联 style 全量迁移；**只对新写 / 本就要改的按钮强制走 `.btn`，存量渐进共存、一个一个修**（用户原话「先建立自己的模板，一个一个修复」）。

### 3.4 不动

`PipelineBoard` `.brow` 的 t-kimi 独立 token 命名空间（只升交互层）、ProjectHome shader 首页、`.glass-card` / `.agent-card`、pi 内核。

## 四、两需求的交汇（队员卡片）

需求一「把卡片搬进对话 + 接主脑派活数据」与需求二「打磨卡片交互 + 补 a11y + 加微动效」在**流水线 / 队员卡片**处交汇——同一套 `PipelineStageCard`：

- 需求一负责「搬运 + 数据适配 + 流程动效」；
- 需求二负责「修真 bug + 补 a11y + 微交互打磨」。

**统一做、避免两次改同一文件**（守 surgical-changes）。

## 五、验收方向

- **主脑**：真浏览器端到端跑一个复杂需求（对标 Kimi 案例）——说需求 → 看计划 → 点头 → 队员卡片出现 + 实时状态 → 汇总；守并发（多于 3 自动排队、复用 queued 态）。
- **UI**：真浏览器验收（UI 卡必走真浏览器）——按钮四型六态在亮暗主题下 hover/focus/disabled/loading 正确；卡片 queued 状态一致、hover 过渡顺、键盘可达、Esc 关、reduced-motion 生效；`pageErrors=0`。
- 决策一致性自检；门禁（lint / test / tsc）通过。
