# V1.2 第一轮 · 多 Agent 管理 —— 需求文档（proposal）

> vibe-coding 第 1 步产物。本文把「做什么 / 为谁做 / 成功标准」钉死，作为后续概要 / 详细设计 / 任务划分的唯一约束源。

## 一、项目概述

- **目标**：把 Next-Step 的「多 Agent 管理 / 派发」界面，从当前朴素的 CRUD 卡片 + 实色面板，升级为有「软件工厂」质感的协作配置界面；以 sf-mini 原型（`ui原型参考/index.html`、`board.html`）的**视觉语言**为标杆。
- **性质**：本地单用户工具的迭代（V1.2 工作区，承接 V1.1 全部已收官能力的代码）。**内部工具向**，不是开源对外发布。
- **技术栈**（不变）：Next.js 16（App Router）+ React 19 + TypeScript strict；状态 Zustand；纯文件存储，无数据库 / 鉴权 / 多租户。
- **内核红线**（不变）：不 fork / 不改 pi 内核；artifact 改动必经 `PendingChange` → 按块确认；并发会话 ≤ 3。

## 二、本轮愿景（roadmap，分增量执行）

多 Agent 管理的**目标形态**（参照 sf-mini `board.html`，**仅取视觉 / 信息架构，不照搬其 agent 工作机制**）：

1. 角色化 agent 卡片：玻璃质感、信息富集、（可）分组泳道。
2. 二级菜单玻璃效果：点击卡片弹出的操作菜单 / 抽屉。
3. 发起多 Agent 派发的入口与表单升级。
4. 事件流 / 进度的呈现方式。

**执行顺序「先视觉后机制」**（用户明确「我们先做视觉方向的改进」）：

| 增量 | 范围 | 本轮 |
|---|---|---|
| **增量一（本次）** | 重构 **agent 管理卡片**样式（视觉 + 内容展示），**不改 agent 工作机制** | ✅ 现在做 |
| 增量二+ | 二级菜单玻璃化 / DispatchPanel 视觉统一 / 发起派发能力 / 事件流呈现 | 顺延（本轮内续做或下一轮） |

> 本需求文档第三~五节只约束**增量一**。增量二+ 的需求待增量一收口后另行细化。

## 三、增量一 · 功能需求（本次实现范围）

**现状**：`components/AgentManager.tsx` 的 `AgentCard`（约 :470）是正方形玻璃卡，仅展示「哈希色块首字母 + 名称」；用全局 `.glass-card`（`app/globals.css`，CSS 变量、亮/暗双主题）。入口 = 工作台左侧栏底部「Agents」按钮 → 全屏模态 `AgentManager`。

- **F1 视觉升级（玻璃质感）**：把 sf-mini 的玻璃拟态质感「翻译」进**我们现有 CSS 变量体系**，亮/暗双主题自适应、与 app 其余部分协调（**不割裂**）。强化 `.glass-card` / 设计 token：模糊（backdrop-blur）、半透明层、渐变强调、hover 反馈、圆角、（可选）发光/描边。
  - ← 用户拍板「**主题自适应玻璃**」（非深色指挥舱 / 非忠实照搬 sf-mini 深空霓虹）。
- **F2 内容展示升级**：卡片从「仅名称」升级为展示 agent **档案的关键信息**。
  - ⚠️ 关键约束：管理面板里的卡片是「**档案（profile）**」而非「**在跑的会话**」，因此**不展示伪造的运行态 / Critic / 待审**（那些 sf-mini 富信息需要实时会话数据，属增量二「发起派发 + 事件流」）。
  - 候选展示字段（**具体字段与布局在「详细设计 / QA」阶段拍板**）：头像 + 名称 + 角色 `role` + 模型 + 模式徽章（doc / coding）+ 技能 / 工具计数。
- **F3 不回归**：以下既有行为一律不破——卡片可点击进二级菜单（起会话 / 配置 / 删除）、hover 反馈、网格布局、「新建 Agent」空卡、亮/暗主题切换、CRUD 表单。

## 四、非功能需求

- **主题自适应**：亮 / 暗双主题都正确（复用 `--bg / --accent / --text / --border / --glass-*` 等 token，**禁止硬编码颜色**）。
- **不破红线**：纯**展示层**改动；机制层（`agent-profile-store` / `profile-session-wiring` / 提议工具 / dispatch / 内核）**一行不动**。
- **可测试 / 可验收**：真浏览器对照基线（亮/暗 × hover × 二级菜单 × 新建空卡，`pageErrors=0`）；`npm run lint && npm run test && npm run build` 三门禁全绿。
- **性能**：纯 CSS / 样式升级，不新增重渲染来源、不引入重型依赖。
- **模块独立**（vibe-coding 原则 2）：视觉 token 改动集中在 `globals.css`，卡片结构改动集中在 `AgentManager.tsx` 的卡片子组件，二者边界清晰、可分别验证。

## 五、明确不做（增量一）

- ❌ 不改 agent 工作机制 / 派发逻辑 / pi 内核。
- ❌ 不在卡片造「运行中 / Critic / 待审」等需要实时会话数据的状态。
- ❌ 不做 DispatchPanel / 泳道 / 事件流（顺延增量；本次至多在 QA 拍板后**顺带统一** DispatchPanel 选择卡的玻璃风，作为可选项）。
- ❌ 不引入 emoji 角色图标库（除非 QA 拍板；agent 为用户自定义、无 emoji 字段）。

## 六、参考与输入

- **sf-mini 视觉原型**：`next-step-V1.2/ui原型参考/{index.html, board.html}`——深空 + cyan/violet 霓虹的纯深色玻璃风（`.glass` = `rgba(255,255,255,.035)` + `backdrop-blur(8px)`、状态色芯片 + `pulse-dot`、滑入式玻璃抽屉、font-mono 事件流）。**仅视觉 / 信息架构参考。**
- **设计调研来源**：`../QA/开发/通用多Agent配置-sf-mini反推.md`（V1.1 第八轮调研，sf-mini → Next-Step 映射 + 4 形态对比）。
