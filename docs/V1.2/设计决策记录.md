## 第二轮 · 版本 diff 与历史

> 用户拍板见 `QA/第二轮-版本diff与历史决策.md`（D-V1.2-10~14）。承重 file:line 经 lead 亲读复验（`ArtifactPanel.tsx` / `anchor.ts` / `lcs.ts` / `artifact-service.ts` / `useArtifactStore.ts` / `degrade.ts` / `pending-change-service.ts`）。

- **D-R2-01 · 新建客户端安全版本 diff 聚块函数**：版本对比要在客户端把「两版 content」算成 `DiffBlock[]`，但现成的 `computeReplaceDiffBlocks`（`pending-change-service.ts:129`）在 **server-only 模块**（顶部值导入 `node:crypto/node:fs/node:path`，:1-14）。客户端值导入它会把 `node:fs` 拖进 bundle、Turbopack 全站 500（D-R7B-07 血泪、`lcs.ts:1-11` 记载）。故**新建 `lib/artifact-view/version-diff.ts`**：`computeVersionDiffBlocks(oldContent,newContent):DiffBlock[]`，照搬 `groupOpsToBlocks`（`pending-change-service.ts:94-123`）纯算法逻辑、聚块循环与 `anchor.ts:36-49` 逐字一致，**只从 `lib/domain/lcs.ts` 值导入** `lcsDiff/splitLines`、`DiffBlock` 仅 `import type`；block.id 用**确定性序号**（如 `v{old}-{new}-{idx}`，绝不用 `node:crypto.randomUUID`，id 只供 React key 与 `data-block-id`）。

- **D-R2-02 · 去 ☑️❌ 走现成只读路径、复用形态C，不改 DiffBlockCard 内部**：`DiffBlockCard` 的 ✓/✗ 整段受 `canResolveHere = isFullscreen && !resolved && changeId!=null && resolveBlock` 门控（`ArtifactPanel.tsx:551,595-616`），「已确认/已拒绝」状态标受 `isFullscreen && resolved` 门控（:618-622）。版本对比走「**不传 changeId / resolveBlock / isFullscreen**」即恒只读、无按钮无状态标——`DiffBlocksView`（:646-657，`<DiffBlockCard block={b}/>`）已是此先例（全屏下也不冒标，因 isFullscreen 未传 = undefined）。故**零改 DiffBlockCard**；只需把 `InlineDiffView` 参数化为接受裸 `(oldContent,newContent,diffBlocks)` + 可选 resolve 三件套，或抽只读兄弟组件。视觉**复用形态C混合**（equal=Markdown / change=DiffBlockCard），与受管文档现有行内 diff 一致。

- **D-R2-03 · 降级护栏按「版本 diff 总块数」另判**：现有降级判定 `countPendingBlocks`（`degrade.ts:19-21`）基于 `state==='pending'`。版本对比块非 pending（D-R2-01 给确定性序号、state 取非 pending），该计数恒 0、永不降级。故版本 diff 须**按总块数 > effectiveLimit** 单独判降级到并排 `DiffBlocksView`，复用 `INLINE_HL_LIMIT=25`（`degrade.ts:8`）/ `FULLSCREEN_INLINE_HL_LIMIT=80`（`ArtifactPanel.tsx:21`）阈值常量。

- **D-R2-04 · 取前驱版按 versions[] 排序取前一个、非「版号-1」**：版本号当前连续（`submitVersion`/`rollback` 恒 `currentVersion+1`），但「前驱 = 选中版号-1」是「版号连续」的隐含假设、未显式保证。**改为：versions[] 按 version 升序，取选中版在序列里的前一个元素**作 base，应对未来潜在版本号空洞。v1（序列首元素）无前驱 → 只读全文 + 提示。

- **D-R2-05 · diff 历史 = listVersions 懒算、覆盖 rollback、瞬态标量态**：时间线条目 = `store.versions`（`useArtifactStore.ts:68`，listVersions 已拉回含 content/note/author/createdAt，:209-220），**零新增存储**；点条目才算单条相邻版 diff（**懒算**，`lcsDiff` 是 O(n*m) 全表 DP、勿预算全部版本对）。范围**覆盖 rollback 版本**（note=`rollback to v{n}` 作语义标注）。时间线「当前展开条目」用**标量字段**（如 `expandedHistoryVersion: number|null`），**严禁**新增返回新数组/对象的派生 selector（否则 useShallow 漏用 → 无限重渲染崩、只真浏览器暴露，D-D3-10）。

- **D-R2-06 · diff 历史入口 = 独立 toggle 按钮 + 覆盖正文（甲）**：`[Diff历史]` 是 ArtifactPanel 工具栏一个 toggle 按钮（放版本下拉右侧、常驻可见），**不**塞进 `[行内│查看Diff]` 段（该段在历史/无 pending 时隐藏，会一并藏掉入口）。激活时主体区切成时间线、**覆盖正文**（非浮层，避开 A1 全屏浮层层级/手势冲突）；时间线模式下隐藏 `[行内│查看Diff]` 段；再点按钮或选版本下拉即回正文/版本 diff。

- **D-R2-07 · 推翻 D-D5-4（历史版纯只读、无 Diff）**：D-D5-4（祖先 `docs/V1.1/设计决策记录.md`）原定「看历史版 = 纯只读全文」。本轮（用户拍板 D-V1.2-11）改为「选历史版 = 该版 vs 上一版只读行内 diff」，**保留「回到只读全文」逃生开关**。这是有意识地扩展/推翻既有决策、非回归。

## 第三轮 · TOC diff 与文档性能

> 用户拍板见 `QA/第三轮-TOC-diff与文档性能决策.md`（D-V1.2-15~18）。承重 file:line + 性能 baseline 经 lead 亲读/亲测复核（chrome-devtools 实测）。调研 verdict GO·需修正 4 处，已全部纳入下列 ADR。

- **D-R3-01 · 新建客户端安全 `lib/artifact-view/toc-diff.ts`**：导出 `computeTocDiff(oldContent,newContent): TocDiffItem[]`（`TocDiffItem = TocItem + diffKind:'add'|'del'|'mod'|null + side:'base'|'target'`）。**只值导入 `lib/domain/lcs.ts` 与 `lib/artifact-view/toc.ts`**，绝不值导入 `pending-change-service.ts`/`artifact-service.ts`（含 node:fs，守 D-R7B-07）。纯函数、O(行数) 线性。

- **D-R3-02 · `parseTocWithLines` 不破 `parseToc` 契约**：归属需标题行号，但给原 `parseToc` 加 `line` 字段会打破 `toc.test.ts:28-32/:52` 的 `toEqual` 三字段严格断言。故**新建 `parseTocWithLines`**（或在 toc-diff.ts 内复刻 toc.ts 的 ATX_HEADING/FENCE 扫描），**原 `parseToc` 一字不改**。`parseToc` 行序与 `splitLines` 行序同源（都按 `\n` split），标题行号与 lcsDiff 的 ops 行号互通。

- **D-R3-03 · 章节归属算法（三步线性）**：① base/target 各 `parseTocWithLines` 拿标题行号；② 一趟 `lcsDiff(splitLines(base),splitLines(target))` 遍历累加 i(base行)/j(target行)，标 `baseChangedLines[i]`（del）/`targetChangedLines[j]`（add），equal 两侧未变；③ 每标题章节区间 = 从本标题到**下一个同级或更高级标题**之前；key 在 target 有 base 无→`add`、base 有 target 无→`del`、两侧都有且区间内有改动行→`mod`、否则 `null`。**子章节改动冒泡标父**（核实修正①：这是「按下一同级/更高级标题切区间」的**必然结果**——父区间天然含子章节行，非可选项）。

- **D-R3-04 · 对齐 key + 真同名兜底（核实修正②）**：对齐 key 用「**剥离去重后缀的原文 text + level**」（规避 `toc.ts:52-56` 的 `-1/-2` 后缀因增删错位误判）；但「**真·同名同级**章节」（合法）会撞 key，故对撞 key 的标题用**标题序列顺序配对**兜底（按出现顺序一一对应），不靠纯 key 集合。标题改名（`## 背景`→`## 项目背景`）**接受拆成 del+add 两条**，不做模糊匹配（成本高收益低）。

- **D-R3-05 · 删除章节占位 + 视觉=下方色线（D-V1.2-15/17）**：`computeTocDiff` 产出 **base/target 合并有序序列**（按位置交织），del 章节按 base 位置插入。TocSidebar 渲染：改动章节目录条加 `border-bottom: 2px solid <KIND_STYLE 色>`（add 绿 #4ade80 / mod 黄 #eab308 / del 红 #f87171）+ `marginBottom`/`paddingBottom` 留间距；**无圆点/图标**。del 条目额外：暗色文本（opacity 降低）+ **不可点击**（无 data-slug 落点，onClick no-op / cursor default）。未变章节渲染与现状完全一致（零回归）。

- **D-R3-06 · 性能=仅 memo 化（D-V1.2-18 + lead chrome-devtools baseline）**：**实测 baseline**（大文档 ~1200 行 / 16KB）——滚动流畅（avg 16.5ms、p95 16.7ms、**0 掉帧、0 long task**）；卡顿是**渲染瞬间 ~600ms long task**（只读 1 实例 593ms ≈ 版本 diff 11 实例 596ms，**实例数非放大器、总内容量相同**——调研「N 实例数十倍」热点被实测**推翻**，核实修正④）。故优化=**仅 memo 化**：`Markdown` 组件的 6 个 heading 工厂 + components 对象**提模块级常量**（不依赖 props）、`Markdown` 包 `React.memo`（children 字符串可浅比较）、InlineDiffView 段做 `React.memo` + key 从 index 改**段内容稳定 key**（保 D-R7B-04 对齐不漂移）、DiffBlockCard `React.memo`。目标=消除「内容不变时的冗余重解析」（store refresh / 父重渲染触发的 600ms）。**不虚拟化**（滚动已流畅、收益存疑、撞 TOC/A3 querySelector 跳转）、**不按长度降级**（损形态C体验）。

- **D-R3-07 · 不共享 ops、不碰 D-R7B-04（核实修正③）**：TOC 归属**独立跑一次 lcsDiff** + `useMemo` 缓存即可，**不拆 version-diff 的聚块纯函数、不动 buildLineDiffSegments/computeVersionDiffBlocks/groupOpsToBlocks 三处对齐**。理由：性能瓶颈实测是 react-markdown 渲染、**非 lcsDiff**（O(n*m) 在 1200 行约几十 ms、远小于 600ms 渲染），无需为省一次 lcsDiff 去碰对齐红线。

- **D-R3-08 · T0 baseline profiling 作承重墙先做、性能验收走真浏览器 Performance API**：性能优化前先实测 baseline（已由 lead chrome-devtools 完成）；优化后须真浏览器对比 render long task 下降（量化）。`build`/单测/lint 证明不了掉帧（D-R1-04 build 非 oracle）。**工具取舍（2026-06-22 用户拍板）**：验收默认 **browser-e2e**（脚本断言+截图+Performance API via page.evaluate，省上下文）；chrome-devtools 的 `take_snapshot` 太耗上下文，仅留作交互式调试备用。见 [[next-step-browser-e2e]]。

- **D-R3-09 · 「子改动冒泡父」精确化为「仅已存在子章节的内容改冒泡、整段子章节增删不冒泡」（T1 实现细化）**：D-R3-03 字面「父区间内有改动行即 mod」会让「在 # A 下新增/删除整段 ## B」误把 A 标 mod（A 自身没改），使 per-section 指示失去意义（任意后代被增删→所有祖先 mod）。故 T1 精确化：① **配对存活的子章节内容改** → 冒泡标父 mod（符合 D-R3-03 设计意图，扫父节即知该大节有内容变）；② **整段子章节的增/删**（未配对标题的区间）→ 给其区间打掩码、`sectionHasChange` 跳过这些行 → **不冒泡**（由该子章节自己的 add/del 条目表达）。副带正确：标题改名（del+add 两端都不配对）也不再误标父 mod。实现见 `toc-diff.ts:buildExcludedMask`/`sectionHasChange`（lead 复核批准、记此）。**⚠️ 本条的「冒泡」部分已被 D-R3-10 推翻（见下）。**

- **D-R3-10 · 真浏览器实测后用户拍板：①归属彻底「不冒泡」②视觉改「标记条+类型符号」（取代 D-R3-05 下方色线 + D-R3-09 冒泡）**：T2 真浏览器（browser-e2e）实测大文档（1202 行）后，lead 跑 `computeTocDiff` 核对发现「过度标记」：文档根标题 h1 被标 mod（D-R3-09 仍把存活子章节内容改冒泡到根→根成全文改动总开关）、第 60 章被标 mod（新增的第 61 章前的空行分隔符落进 ch60 区间）。**用户拍板「都不要」（决策 C）**：
  - **归属改为「最内层精确归属、完全不冒泡」**（取代 D-R3-09 的冒泡）：每个改动行只归属到其前**最近的标题**（任意级别）；标题归属区间 = `[本标题行, 紧邻下一标题行)`（到下一个**任意级别**标题，非「同级或更高级」），父区间不再含子章节行。→ 文档根标题不因后代改动而 mod、per-section 最精确（贴合用户「看清改动位置」）。
  - **掩码扩展修「邻章牵连」**：给未配对（纯 add/del 整段章节）区间打掩码时，**额外把该标题行之前紧邻的连续空行/分隔行并入掩码**（从标题往上、连续空白到上一个非空行）→ 新增/删除整章不牵连前一章（修 ch60）。
  - **视觉「乙·标记条+类型符号」**（取代 D-R3-05 的 `border-bottom` 全宽细线，用户经 ui-ux-pro-max 设计建议 + 多轮拍板）：改动章节标题下方一条「**3px + 圆角(radius 2) + 内缩到标题宽度**」的子元素色标记条（非全宽 border，读作「该条目标记」而非分隔线）+ 行首一个**极小单色类型符号**（`+`新/`~`改/`−`删）——符号靠**形状**让色盲也能区分类型（满足 CRITICAL 规则 `color-not-only`，纯色线不达无障碍）。del 仍暗色 opacity 0.55 + 不可点。
  - 实测结果：11 处标记（+×1/~×9/−×1）全为真改动，h1 根 + ch60 不再误标，pageErrors=0。实现见 `toc-diff.ts`（归属/掩码）+ `ArtifactPanel.tsx:TocSidebar`（标记条+符号），commit `42d111e`。
