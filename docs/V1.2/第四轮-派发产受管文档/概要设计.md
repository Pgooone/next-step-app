# V1.2 第四轮 · 让文档型派发也能产受管文档 —— 概要设计

> 需求见 [`需求文档.md`](需求文档.md)；用户拍板 [`../QA/第四轮-派发产受管文档决策.md`](../QA/第四轮-派发产受管文档决策.md)；lead ADR [`../设计决策记录.md`](../设计决策记录.md) D-R4-*。

## 一、方案选型（调研 `wf_778dac19` 三视角对抗 holds=True）

| 方案 | 内容 | 结论 |
|---|---|---|
| A | 仅镜像 mode 分流、挂全 3 doc 工具、产物仍取 assistant 文本 | ❌ 否决：产物/链接/喂料三处脱节；propose_edit 在场会落死悬 pending |
| **B（采纳）** | 接线 + **create_artifact 闭环对账** + **dispatch 禁 propose_edit**，分两轮 | ✅ 真闭环、不引入死悬 pending、create 走 D-V4-02 结构操作豁免 |
| C | 挂全 3 工具 + dispatch 内**自动确认** propose_edit 的 pending | ❌ 否决：绕过 HITL、破 V2/北极星不变量② |

**headless/HITL 取舍（核心）**：dispatch 是 fire-and-forget、回合中无人按块确认 → 只走**无需确认的结构操作** `create_artifact`（立即落 v1 + 物化），把需确认的 `propose_edit` 挡在 dispatch 外（否则死悬 pending + 阻塞该文档后续所有提议）。「下游修订上游文档」推迟第二轮 HITL 后移。

## 三、数据流（顺序流水线，doc worker）

```
派发目标+子任务 ─▶ orchestrator.runDispatch（串行、acquireSlot ≤3）
                      │ projectId=task.projectId
                      ▼
              dispatch-runner.runWorker（mode=doc）
                      │ ① assembleProfileSessionOptions → options(tools=profile.tools)
                      │ ② docOptions = assembleDispatchDocSessionOptions(...)  // 6 项受限集、含 create_artifact/list_artifacts、无 propose_edit
                      │ ③ createAgentSession({...options, ...docOptions, ...codingToolsFallback, ...override})  // spread 顺序不可调(D-V2-04)
                      ▼
              worker 一回合：调 create_artifact ─▶ ArtifactService 物化受管文档(项目根 .md + managed 侧车)
                      │ agent_end.messages 含 toolResult{toolName:create_artifact, content:[{text:'{id,...}'}]}
                      ▼
              extractCreatedArtifactIds(messages) → artifactIds[]
              readCurrentContent(projectId, last) → createdContent（try/catch 兜底）
                      ▼
              orchestrator：assignment.artifactId=last；upstreamOutput=createdContent??output（喂下游）；
                          writeArtifact 文本保底仍写；判空放宽
                      ▼
              前端轮询终态 ─▶ DispatchPanel：产物按钮「受管文档」by-id 开 ArtifactPanel
                          ─▶ onArtifactsChanged → bump explorerRefreshKey → FileExplorer 受管分组重取
```

## 四、红线合规（三视角对抗核实，全 holds=True）

- **V2 提议红线**：create_artifact 立即落盘属结构操作、D-V4-02 明文豁免；propose_edit（需按块确认）被 dispatch 受限集**禁用**，从源头杜绝死悬 pending。
- **不改内核**：buildDispatchDocTools / assembleDispatchDocSessionOptions / extractCreatedArtifactIds 全是 Next-Step 自有代码，只经 `createAgentSession` 公开 options 装配。
- **D-V2-04 spread 顺序**：`options → docOptions → fallback → override`，docOptions 受限白名单覆盖 profile.tools（防 write/edit/bash 泄漏）；白名单显式含 2 个 customTool 名（否则内核按名过滤）。
- **≤3 并发**：`concurrency-gate` 一行未动。
- **客户端 bundle（D-R7B-07）**：DispatchPanel/AppShell 仅 `import type { Assignment }`、零新增 value-import 服务端模块（含 node:fs），artifactId 只是轮询 JSON 字符串。

## 五、承重墙

**T1 spike（lead 复跑）**：证 ① dispatch doc worker 工具集恰 6 项；② `create_artifact` 落的 id 能从 `agent_end.messages` 确定性抽出（`role:'toolResult'`/`toolName`/`content[].text`/JSON.parse/.id；与 listArtifacts/turn_end 三者一致）。默认 messages 抽取 + listArtifacts 兜底。**T1 未绿不开后续**。
