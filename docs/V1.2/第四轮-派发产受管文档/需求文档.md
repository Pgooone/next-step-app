# V1.2 第四轮 · 让文档型派发也能产受管文档 —— 需求文档

> 用户拍板见 [`../QA/第四轮-派发产受管文档决策.md`](../QA/第四轮-派发产受管文档决策.md)（D-V1.2-19/20）；lead ADR 见 [`../设计决策记录.md`](../设计决策记录.md)（D-R4-*）。

## 一、背景与根因

模拟新用户用「CS2 饰品监控网页」项目跑多 Agent 派发（dispatch）时发现：**文档型（mode=doc）派发的 worker 产不出「受管文档」**，只把正文当聊天回复、被 orchestrator 抓成 `.pi/artifacts/<dispatchId>/N-角色.md` 纯文本——**掉出受管体系**（无版本 / diff / 按块确认、不进 file panel「受管文档」分组、ArtifactPanel 看不到）。

**根因（lead file:line 复核）**：

- `lib/pi/dispatch-runner.ts:runWorker` 建 worker 会话只调 `assembleProfileSessionOptions`（仅 `tools: profile.tools`）+ `createAgentSession`，**全程无 `profile.mode` 分流、不挂任何 doc 提议工具、customTools 恒空**。
- 对照 `lib/pi/profile-session-wiring.ts:146-170`（交互式会话）：同样先 `assembleProfileSessionOptions`，但随后按 `profile.mode==='coding' ? undefined : assembleDocSessionOptions(...)` 合并 doc 工具（`create_artifact`/`propose_edit`/`list_artifacts`）。**dispatch 缺的就是这第二段**。
- 性质 = **有意分期 + 没回接**，非主动屏蔽：第一轮 `orchestrator.ts` 头注释 D-C-1「派发产物轻量普通文件、不版本化、留 Iter D」是当年刻意决策；`docs/QA/开发/受管文档入口并入filepanel决策.md` 用户拍板「派发产物**本期不纳入、后置**」；V2 提议工具只接 profile 会话、对 dispatch 零引用。本轮**兑现第五轮 `详细设计.md:185` §A「dispatch 产物受管化」**，非推翻旧拍板。

## 二、目标

让**文档型派发 worker** 也能产出**受管文档**（带版本 / diff / 按块确认、进 file panel 受管分组），使「多 Agent 派发」与「受管产物」两个支柱接通；coding 模式派发行为零变化；不破任何红线（不改内核 / V2 提议红线 / ≤3 并发 / 客户端 bundle 不拖 node:fs）。

## 三、范围

### 本轮做（第一轮）
1. dispatch worker 按 `profile.mode` 分流：doc → 装「派发专用受限集」（read/grep/find/ls + **create_artifact + list_artifacts**）；coding → 维持 `profile.tools`（空则退回全套内置工具含 bash）。
2. dispatch 专用受限集**禁用 `propose_edit`**（headless 无人按块确认）。
3. 产物对账：worker `create_artifact` 落的受管文档 id 回填 `Assignment.artifactId`、喂下游用受管文档正文、放宽判空（文本空但有受管文档不算失败）。
4. 展示层：进度弹窗产物链接按 id 打开 ArtifactPanel（非 FileViewer）；派发完成且产出过受管文档时，自动刷新 file panel 受管分组。

### 本轮不做（推迟第二轮）
- 下游 worker **修订上游 worker 产的同一份受管文档**（需 propose_edit + HITL 后移：dispatch 完成提示「N 处待确认」、用户事后在 ArtifactPanel 逐块确认）。
- dispatch worker 给被派发 agent 强制工具引导（agent.md / 目标里写「调用 create_artifact」）——本轮验收 fixture 显式写，生产引导属增强。
- 一个 agent 中途切 mode（mode 改阶段级 override）。

## 四、验收标准（AC）

| 编号 | 标准 |
|---|---|
| AC-1 | mode=doc 派发 worker 激活工具集**恰** `read/grep/find/ls/create_artifact/list_artifacts` 六项，**无** propose_edit/write/edit/bash（含 profile.tools 带 write/edit/bash 时被受限集覆盖剔除）。 |
| AC-2 | mode=coding 派发 worker：tools 空→含 bash（全套）；tools=['read']→恰 ['read']（不被 doc 子集污染、不退全集）。 |
| AC-3 | doc worker 调 `create_artifact` 后：受管文档物化（项目根真实 .md + `.pi/artifacts/managed/<id>/` 侧车 + `versions/1.json`），其 id 回填到 `Assignment.artifactId`。 |
| AC-4 | 顺序流水线：下游 worker 收到的上游产物 = 上游受管文档**正文**（非「已创建」回执）。 |
| AC-5 | 判空放宽：文本空但产出过受管文档 → 不判失败；文本空且无受管文档 → 仍失败（中止后续）。 |
| AC-6 | 进度弹窗产物按钮标「受管文档」、点击按 id 打开 ArtifactPanel（带版本，非 FileViewer）；派发完成后 file panel 受管分组**自动**出现新 .md（无需手动刷新）。 |
| AC-7 | `npm run lint` 净 / `npx tsc --noEmit` 零错误 / `npm test` 全绿；真浏览器 console pageErrors=0（客户端 bundle 不被 node:fs 污染）。 |
| AC-8 | coding 模式派发、纯文本 worker 旧行为零回归（`.pi/artifacts/<id>/N-角色.md` 文本保底仍写）。 |
