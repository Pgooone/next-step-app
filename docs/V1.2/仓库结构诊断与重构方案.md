## 一、现状说清楚

### 1. 为什么 `next-step-V1.2/docs/` 里有"双重轮次"

`next-step-V1.2` 是 V1.1 收官状态(commit `e5f97d4`)的精简工作副本。拷贝时把 V1.1 的整套 `docs/第N轮-*` 一并带进来当**祖先设计史**(只读,解释"现有代码 = V1.1 收官代码"的由来);V1.2 自己的新轮次则另放在 `docs/V1.2/第N轮-*`。于是同一棵工作树里出现两套都叫"第 N 轮"、嵌两层、**同号常常不同义**的目录。

实测坐实这层因果:
- `e5f97d4` 是当前 HEAD 的祖先(`merge-base --is-ancestor` = YES),HEAD 在它之上叠了 **37 个 first-parent commit / 38 个总 commit**(差额 1 个来自一次 merge `fe2a0ec`,见下方订正)——即 V1.2 自己的 7 轮迭代。
- 顶层 7 个祖先轮次目录经 `diff -rq` 与 `next-step-V1.1/docs/` 对应目录**逐字节相同**(抽查 `第一轮-基础迭代/` diff 输出为空)。
- 顶层 `设计决策记录.md` 首行就是 `# Next-Step V1.1 设计决策记录(ADR)`,517 行,全是 V1.1 时代的 ADR。

权威消歧依据是 `docs/V1.2/README.md`(当前已改动未提交),它明确点破"顶层 = 继承自 V1.1 的祖先设计史 / `docs/V1.2/` = V1.2 自身轮次"——但**目录结构本身没有任何视觉区隔**(没有 `历史-V1.1/`、`_archive/` 之类命名),纯靠这份 README 的文字拦着。新读者只看"第七轮"三个字必然混。

**三个 git 仓 = 同一远端 `Pgooone/next-step-app` 的三条分支**(`origin/master`、`origin/v1.1`、`origin/v1.2`),实测远端三分支取值:
- `origin/master = origin/v1.2 = aaf91b4`(同一 commit;每轮 v1.2 收官即 ff master)
- `origin/v1.1 = ecaafac`(已 push;落后 v1.2)
- Release tag `v1.0.0`、`v1.2.1`~`v1.2.6` **全部挂在 v1.2 这条线上**(`v1.2.6^{} = aaf91b4`),是历史锚点。

**历史形态(订正草案"严格线性直链"的说法)**:`e5f97d4..HEAD` 之间存在一次 merge commit `fe2a0ec`,所以**不是严格线性**——是"基本线性 + 一个 merge"。这正是"first-parent 距离 37、总 commit 数 38"差额的来源。`e5f97d4` 是 V1.1↔V1.2 的**共同基线 / HEAD 的祖先(first-parent HEAD~37)**,**不是**远端 v1.1 的 tip。

> ⚠️ 两个易踩的坑:
> 1. **`master` 语义已漂移**:名义"V1 发布线",实际在 V1.2 阶段已沦为"v1.2 的别名"(`origin/master = origin/v1.2`)。别误以为 master 还停在 V1。
> 2. **本地 V1 仓是停在 V1 时代的旧检出**(`cf904fc`),且**其本地未 fetch `aaf91b4`**(实测在 V1 仓内 `git merge-base aaf91b4` 直接报 "Not a valid commit name")——所以**无法在 V1 仓内用单条 `merge-base` 断言它与 v1.2 的祖先关系**。安全结论(旧检出、勿在其中 push/pull)依然成立,但不能说成"差整条历史/分叉",只能说"该仓本地不知道 v1.2 的存在"。

## 二、重构方案(给选项 + 推荐,本阶段只出方案不执行)

> **结论先行:推荐【最小档】,并按需叠加【中档】的两个低风险子项(外层 README 重写 + AFFiNE/node_modules 处理)。明确反对【激进档】。**
> 理由:最小档用一次 `git mv` 就能彻底消除头号困惑(C1/C2/C3/C4),完全在活跃的 V1.2 仓内部、保留全部历史、不碰任何别的分支与远端、零构建影响;而它恰好就是用户想要的"docs 内分 v1.1/v1.2、各分轮次"。激进档收益小、风险高,且与"分支本就是版本史"的现实相悖。

### 最小档 ⭐ 推荐:V1.2 仓内把祖先轮次收进 `docs/V1.1/`

**动什么**(全部在 `next-step-V1.2` 工作树内,用 `git mv`):
- 顶层 7 个祖先轮次目录 `docs/第一轮-基础迭代/` ~ `docs/第七轮-UI优化-diff内联与全屏/` → `docs/V1.1/第一轮-.../` ~ `docs/V1.1/第七轮-.../`
- 顶层 `docs/设计决策记录.md`(V1.1 版,517 行) → `docs/V1.1/设计决策记录.md`
- 顶层祖先 `docs/QA/` → `docs/V1.1/QA/`
- 顶层 `docs/images/` → `docs/V1.1/images/`(移动后须 `grep` 一遍引用这些图的祖先 md 相对路径是否仍成立;多为同目录相对引用、随整组一起移则不受影响)
- 跨版本方法论 `docs/新增功能开发流程.md` → **建议留在 `docs/` 顶层**(它跨 V1.1+V1.2,不属任一版本),或放 `docs/_方法论/` 单独标识

**收尾后目录形态**:`docs/ = V1.1/(祖先·只读基线) + V1.2/(本版·活跃) + 新增功能开发流程.md(跨版本方法论)`,"同号双重轮次"歧义彻底消失,目录名自带"哪套体系"语义。

**git 安全性**:✅ 全部 `git mv` + 一次普通 commit,**最安全的一类操作**。实测顶层祖先轮次**已被 git 跟踪**(`第一轮-基础迭代/` 有 10 个跟踪文件),`git mv` 保留历史,`git log --follow` 仍可追溯。**不触碰 `e5f97d4` 及更早任何已 push 的祖先 commit、不改写历史、不碰 `v1.1`/`master` 分支、不动远端、不动 tag**。属"在工作树内移动文件夹"。

**⚠️ 执行前置硬约束(实测的真实数据丢失风险)**:当前 V1.2 工作树脏,`git status -s` 实测有:
- 2 个已改未提交:`docs/V1.2/README.md`、`docs/QA/开发/通用多Agent配置-sf-mini反推.md`
- **未跟踪的整目录**:`docs/V1.2/第七轮-流水线与阶段看板/`、`tasks/V1.2/第七轮-流水线与阶段看板/`
- **未跟踪的单文件**:`docs/新增功能开发流程.md`
- 10 个未跟踪脚本:`scripts/r3-*.mts`、`scripts/verify-r2/r3/r5/r7b-*.{mts,mjs}`

这些**不在任何 commit 里**,重构移动若不先 `git add`/备份会被当游离文件处理而丢失。**务必:重构前先把这些 `git add` 落一个 commit(或单独备份),再开始 `git mv`。**

**风险**:低。唯一需留意的是祖先 md 内部的图片/交叉引用相对路径——整组连同 `images/` 一起移可规避大部分;建议移完 `grep -r "](.*\.png" docs/V1.1/` 与跨文档链接验证不空链。

**是否满足用户目标**:✅ 直接满足"docs 内分 v1.1/v1.2、各分轮次"。本报告判定**最贴合用户原话**的方案。

**验收方式**:① `git status` 确认改动全是 rename(`git mv` 的 R 状态)、无意外 delete;② `git log --follow docs/V1.1/第一轮-基础迭代/需求文档.md` 能追到搬家前历史;③ `grep -r "](.*\.png" docs/V1.1/` 抽查图片链接不空链;④ `npm run build` 兜底(docs 不参与构建,预期零影响)。**纯文档移动,不涉 UI,无需真浏览器。**

### 激进档 ❌ 不推荐:3 个版本目录/分支合并成单一仓库

**动什么**:把 V1/V1.1/V1.2 三个独立仓合成一个仓,代码取最新(V1.2),历史 docs 平铺成 `docs/v1`、`docs/v1.1`、`docs/v1.2`。

**git 安全性 / 坑(诚实评估)**:
- **会破坏 3 分支的现有历史语义**。三分支当前是同一远端的分支,且 `v1.1`/`v1.0.0`~`v1.2.6` tag 都锚在 v1.2 这条线上。要把"分支表达的版本史"塞进"单仓的子目录",要么 `filter-branch`/`filter-repo` 改写历史(**改写已 push 的祖先 → `origin/v1.1` 与全部 tag 失锚、强推冲突、协作者历史断裂**),要么放弃历史只搬文件(**丢掉三条分支的演进史,而这正是版本管理的价值**)。
- **本地 V1 仓(`cf904fc`)是停在 V1 时代的旧检出,且本地未 fetch v1.2 的 commit**;在它里面做任何 push/pull 极易踩坑(误把远端拉成 V1.2、或误 push 旧 master 覆盖)。合并操作一旦在错误仓里进行就是灾难。
- **`master` 已是 v1.2 别名**这一既有事实,会让"合并"过程中谁是真 main 更易混。

**风险**:高(不可逆的历史改写 + 强推冲突 + tag 失锚)。
**收益**:低——"按版本分目录"用最小档在 V1.2 单仓内做 `docs/V1.1/`+`docs/V1.2/` 即可达成;"代码 + 各版本史"用现有"分支 = 版本史"本就已经表达得很干净。
**结论**:**不推荐**。如果将来确有"单仓库 monorepo"诉求,应另起评估、用 `git subtree`/`read-tree` 等保历史手段单独立项,绝不在本轮顺手做。
