# V1.2 第三轮 · TOC diff 与文档性能 —— 需求文档（proposal）

> vibe-coding 第 1 步。把「做什么 / 为谁做 / 成功标准」钉死。
> 用户拍板见 `../QA/第三轮-TOC-diff与文档性能决策.md`（D-V1.2-15~18）；lead ADR `../设计决策记录.md`（D-R3-*）。

## 一、项目概述

- **目标**：在第二轮「版本间行内 diff + Diff 历史」基础上，①让左侧目录（TOC）在版本对比时**体现章节级 diff**（一眼看出哪些章节增/删/改）；②优化 file panel 渲染**大文档**时的卡顿。
- **性质**：本地单用户工具迭代（V1.2）。技术栈 / 红线不变（不改 pi 内核；artifact 内容修改必经 propose→按块确认；纯文件无数据库）。本轮两功能都是**只读展示 / 渲染优化**，不引入写路径、不碰红线。

## 二、两个功能

### 需求A · TOC 体现版本 diff

- 版本对比（选历史版 viewingHistory）时，左侧目录标出每个章节是**新增/删除/修改/未变**。
- **视觉**（D-V1.2-15）：改动章节标题**下方一条对应颜色横线**（绿=新增 / 黄=修改 / 红=删除），**线本身着色、无圆点**，改动行与上下行**留间距**。
- **范围**（D-V1.2-16）：仅 viewingHistory 态；时间线 / pending 态不带。
- **删除章节**（D-V1.2-17）：按 base 位置插入红线 + 暗色 + 不可点击的占位条目。
- 点击未变/改动章节仍能跳转（复用现有 data-slug + scrollIntoView）；del 占位不可跳转。

### 需求B · file panel 大文档性能优化

- 渲染很大很长的文档时消除卡顿。
- **实测定论**（lead chrome-devtools baseline）：滚动**已流畅**（0 掉帧）；卡顿是**渲染瞬间 ~600ms 主线程 long task**（react-markdown 解析全文），且**实例数不是放大器**（只读 1 实例 ≈ 版本 diff 11 实例 ≈ 600ms）。
- **优化**（D-V1.2-18）：**仅 memo 化**——让内容不变时的重渲染不再重解析；不虚拟化、不按长度降级。

## 三、成功标准（验收 AC）

- **AC1**（TOC diff·标记）：建多版本受管文档、选历史版 → 左侧目录改动章节显示对应颜色下划线（新绿/删红/改黄）+ 留间距、无圆点；未变章节无标记。
- **AC2**（TOC diff·归属正确）：新增章节标绿、删除章节红线暗色占位、改动章节（含子章节改动冒泡到父）标黄；改名章节呈现为删+增两条。
- **AC3**（TOC diff·跳转零回归）：点改动/未变章节仍滚到对应标题；del 占位不可点击；普通查看态（无对比）TOC 无标记、行为同现状。
- **AC4**（TOC diff·范围）：仅 viewingHistory 态带 diff；时间线模式 TOC 仍隐藏；pending 态不带。
- **AC5**（性能·memo）：chrome-devtools 实测——优化后「内容不变的重渲染」不再触发 ~600ms react-markdown 重解析（render long task 显著下降）；功能零回归。
- **AC6**（性能·零回归 + 稳定 key）：版本 diff / pending diff / 行内 ✓✗ / A3 跳转 / TOC 跳转全正常；改动块增删时段不串位。
- **AC7**（红线）：`GET /` 200、`pageErrors=0`、无 node:fs 进客户端 bundle；不产生 PendingChange/写路径。

## 四、非目标（本轮不做）

- ❌ 时间线 / pending 态的 TOC diff（仅 viewingHistory）。
- ❌ 虚拟化、按文档长度降级（实测滚动已流畅、虚拟化撞跳转）。
- ❌ 共享 ops / 改 lcs.ts / 拆聚块函数（不碰 D-R7B-04；性能瓶颈非 lcsDiff）。
- ❌ 标题改名的模糊匹配合并（接受拆 del+add）。
- ❌ 不改 pi 内核、不引数据库、不引虚拟化第三方依赖。
