# V1.2 第三轮 · TOC diff 与文档性能 —— 概要设计

> vibe-coding 第 2 步。模块划分 + 数据流 + 复用 + 红线。详细到算法/file:line 见 `详细设计.md`。

## 一、关键发现（决定本轮架构）

1. **TocItem 无行号**（`toc.ts:10-17`）：章节级 diff 归属需自行恢复标题行位置 → 新建 `parseTocWithLines`（不破 `parseToc` 契约 / `toc.test.ts`）。
2. **行级 diff 可归属到章节**：`lcsDiff`（`lcs.ts:34-67`）遍历时 i(base行)/j(target行) 双计数器是「把改动行锚到标题区间」的钥匙；`parseToc` 行序与 `splitLines` 同源。
3. **性能瓶颈实测**：滚动流畅、卡顿是 **~600ms react-markdown 渲染 long task**；**实例数非放大器**（只读 1 实例 ≈ diff 11 实例 ≈ 600ms）；lcsDiff 非瓶颈。→ 仅 memo 化、不虚拟化、不碰 D-R7B-04。
4. **base/target 现成**：viewingHistory 处 `historyBaseContent`（`ArtifactPanel.tsx:237-240`）+ `displayContent`（:217），TOC 归属零额外取数。

## 三、数据流

```
需求A（TOC 体现 diff）:
  viewingHistory → useMemo: tocDiff = computeTocDiff(historyBaseContent, displayContent)
    [新, 客户端安全; 独立跑一次 lcsDiff + useMemo 缓存, 不共享 ops]
  → TocSidebar(items=tocDiff)：每条按 diffKind 渲 border-bottom 色线(绿/黄/红)+间距;
    del 项暗色+不可点; 未变项同现状; 点击复用 data-slug + scrollIntoView

需求B（memo 化）:
  Markdown: heading工厂/components → 模块级常量; 包 React.memo(children 字符串浅比较)
  InlineDiffView segs.map: 每段 React.memo + 稳定 key(段内容/位置)
  → 内容不变的父重渲染不再重跑 react-markdown 解析(消除冗余 600ms)
```

## 五、红线核验

- ✅ **D-R7B-07 客户端边界**：toc-diff.ts 只值导入 lcs.ts/toc.ts、DiffBlock 仅 import type、绝不值导入 pending-change-service/artifact-service。
- ✅ **D-D3-10 订阅**：tocDiff 用 useMemo 从已订阅的 displayContent/historyBaseContent 就地算，**不建返回新数组的 store selector**；memo 化用稳定 key/常量，不引入新 selector。
- ✅ **D-R7B-04 聚块对齐**：本轮**不碰**（不共享 ops、不改 lcs.ts/聚块）。memo 段稳定 key 须保段顺序与 block.id 对齐不漂移。
- ✅ **只读 / 不破写盘红线**：TOC diff + memo 化纯展示/渲染优化，不产生 PendingChange、不调写路径。
- ✅ **不改内核 / 无数据库 / 不引虚拟化依赖**。
