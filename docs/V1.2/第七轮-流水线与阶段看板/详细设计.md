## 二、数据模型（完整字段定义）

> 全部 JSON、原子写文件、无 DB（D-V1.2-38）。`status` 沿用既有 `DispatchStatus`（`dispatch-store.ts:7` 实证 = `"pending"|"running"|"done"|"failed"`）。

### 2.1 PipelineBlueprint（`.pi/factory/pipelines/<id>.json`）

```ts
/** 流水线蓝图：一组有序阶段，一次定义、可复跑（D-V1.2-38）。 */
export interface PipelineBlueprint {
  id: string;            // randomUUID（node:crypto，仿 dispatch-store.ts:1/85）
  projectId: string;     // 归属项目（ProjectRegistry 反查 root，project-registry.ts:68 实证 get(id).root）
  name: string;          // 看板容器头 ⑂ 名称
  stages: PipelineStageSpec[];
  createdAt: string;     // ISO
  updatedAt: string;     // ISO
}

export interface PipelineStageSpec {
  order: number;          // 唯一权威执行序：1..N 连续、升序 = 串行顺序（钉死 F6「展示序≠执行序」，审计#12）
  agentId: string;        // 引用既有 AgentProfile；起 run 前预检存在（审计#11）
  subTaskTemplate: string;// 该阶段子任务（MVP 手填，无模板导入——D-V1.2-37 模板放二期）
}
```

**保存校验**（`PipelineStore.create/write`，与 dispatch `create` 同级 + 单测；对照 `dispatch-store.ts:68-82` 校验范式）：
- `stages.length >= 1`（拒 0 阶段；注意区别于 dispatch 的 2–3 限制，流水线 N≥1）；
- 每阶段 `agentId.trim()` 非空、`subTaskTemplate.trim()` 非空（拒空模板）；
- `order` 集合 = `{1,2,...,N}`（连续、无重复、无缺号）；落盘前按 `order` 升序归一排序。
- 校验失败抛领域错误（仿 `DispatchError`，`dispatch-store.ts:31` 的 `code: "NOT_FOUND" | "INVALID"`，API 层映射 422）。

### 2.2 PipelineRun（`.pi/factory/runs/<runId>.json`）

```ts
export interface PipelineRun {
  id: string;                 // runId（randomUUID）
  projectId: string;
  pipelineId: string;         // 来源蓝图（蓝图删后历史 run 仍可只读回看）
  pipelineName: string;       // 起 run 时快照蓝图名（蓝图改名/删后看板头仍可显）
  status: DispatchStatus;     // pending|running|done|failed —— run 级聚合态（复用 dispatch-store.ts:7）
  currentStageIndex: number;  // 0-based；看板全局进度条 ▰▰▱… 与 ④/N 头取此值
  createdAt: string;
  finishedAt: string | null;
  cancelRequested: boolean;   // run cancel 落盘标记（停止按钮 / POST cancel 翻；编排器 signal 同源）
  failedReason: string | null;// 失败/对账判 failed 的人读原因（看板失败态头「已失败·停在第N阶段」）
  stages: PipelineRunStage[];
}

export interface PipelineRunStage {
  order: number;
  agentId: string;
  agentName: string;          // 快照（agent 删后看板降级渲染 `agentName ?? agentId`，审计#11）
  subTask: string;            // 由 subTaskTemplate 实例化（MVP 直接 = 模板）
  status: DispatchStatus;     // pending|running|done|failed
  statusDetail?: "queued";    // 可选展示细分：排队等槽（审计#5）——不进权威状态机、仅看板渲染
  sessionId: string | null;   // worker 真实 sessionId（running/done/failed 才有；pending=null）
  artifactId: string | null;  // 该阶段 create_artifact 产物 id（产物链接落它、非 sessionId，审计#3）
  startedAt: string | null;
  finishedAt: string | null;  // 耗时 = finishedAt - startedAt（hover 浮窗显）
}
```

**累积上游**（D-V1.2-46 / 审计#8）：**不另存字段**，跑到第 i 阶段时由编排器实时拼接「所有 `order < i` 且 `status==='done'` 阶段的产物摘要」。产物正文取自 `result.createdContent ?? result.output`（与 `orchestrator.ts:197` 同源，只是从「只记上一跳 upstreamOutput」改为「累积全部已完成」，见 §三 runPipeline 主循环）。零额外存储。

**latestRunId / run 列表**（D-V1.2-35 / 审计#15）：`listRuns(projectId, pipelineId)` 扫 `.pi/factory/runs/*.json`、按 `createdAt` 倒序、**只列最近 N 条**。「最新 run」= 列表头，无需另存 latestRunId 指针（仿 `listVersions`）。

> **待 lead 实现期定（需求 §六待澄清项 2，记 ADR D-R7-*）**：①「只列最近 N 条」的 N 具体值（建议 N=20）；②`runs/*.json` 保留/清理策略（建议：listRuns 时若总数 > 保留上限 M，按 createdAt 删最旧的；M 建议 50）。本文不预设具体数值、留 ADR。

## 三、逐文件设计

### 3.1 `lib/domain/pipeline-store.ts`【新增·领域】

**职责**：蓝图 CRUD，原子写 `<projectRoot>/.pi/factory/pipelines/<id>.json`。**直接仿 `dispatch-store.ts` 结构**（同款 class + DI registry + atomicWrite + 领域错误 + findXxx 跨项目定位）。

**关键复用触点（已 Read 实证）**：
- 原子写 = `dispatch-store.ts:134-138` 的 `atomicWrite(filePath, content)`：`tmp = filePath + ".tmp-" + process.pid` → `writeFileSync(tmp)` → `renameSync(tmp, filePath)`。**逐字照搬**。
- 目录定位 = `dispatch-store.ts:56-58` 的 `dispatchDir(projectId) = join(registry.get(projectId).root, ".pi", "dispatch")`；本文改为 `factoryDir(projectId) = join(registry.get(projectId).root, ".pi", "factory", "pipelines")`。
- 跨项目定位 = `dispatch-store.ts:111-117` 的 `findTask(taskId)`（扫 `registry.list()` 所有项目）；本文同款 `findBlueprint(pipelineId)`（GET/PUT/DELETE 路由路径不带 projectId 时用，**但本文 API 路由路径已带 `[id]`=projectId，故 store 主用 `get(projectId, pipelineId)`，findBlueprint 仅 cancel/run 跨项目兜底备用**）。
- 校验 = 仿 `dispatch-store.ts:68-82`（见 §2.1）。
- `randomUUID` / `node:fs` import 同 `dispatch-store.ts:1-3`。

**关键函数签名**：
```ts
export class PipelineStore {
  constructor(private readonly registry: ProjectRegistry = new ProjectRegistry()) {}
  create(projectId: string, input: PipelineInput): PipelineBlueprint;   // 校验 + randomUUID + 落盘
  get(projectId: string, pipelineId: string): PipelineBlueprint;        // 不存在抛 NOT_FOUND
  list(projectId: string): PipelineBlueprint[];                          // 扫 factoryDir/*.json
  update(projectId: string, pipelineId: string, patch: PipelineInput): PipelineBlueprint; // 校验 + 落盘
  delete(projectId: string, pipelineId: string): void;                  // unlink（历史 run 仍可只读回看）
  write(projectId: string, bp: PipelineBlueprint): void;                // 原子写
}
type PipelineInput = { name: string; stages: { order: number; agentId: string; subTaskTemplate: string }[] };
```

**红线**：纯 `node:fs`、无 DB；属服务端领域层，**绝不被客户端组件 value-import**（避免 node:fs 进 bundle，D-R7B-07；客户端只经 fetch JSON 取，见 usePipelineStore）。

### 3.2 `lib/domain/pipeline-run-store.ts`【新增·领域】

**职责**：run checkpoint 读写 `<projectRoot>/.pi/factory/runs/<runId>.json`；`listRuns` 仿 listVersions；读时对账 `reconcileOrphan`。

**关键复用触点（已 Read 实证）**：
- 原子写同 `dispatch-store.ts:134`（同款 atomicWrite）。
- 对账计数源 = `concurrency-gate.ts:19-21` 的 `globalThis.__piSessions`（活会话 registry）。对账判据：`status==='running'` 且「当前阶段 `sessionId` 非空但 `globalThis.__piSessions?.has(sessionId)` 为假」→ 判 failed。
- 跨项目定位 = `dispatch-store.ts:111-117` findTask 同款 `findRun(runId)`（GET `pipeline-runs/[runId]` 路径不带 projectId，须扫 registry）。

**关键函数签名**：
```ts
export class PipelineRunStore {
  constructor(private readonly registry: ProjectRegistry = new ProjectRegistry()) {}
  create(projectId: string, run: PipelineRun): void;          // 起 run 时初始落盘
  get(projectId: string, runId: string): PipelineRun;         // 读盘（不对账）
  findRun(runId: string): { projectId: string; run: PipelineRun }; // 跨项目定位（GET 路由用）
  listRuns(projectId: string, pipelineId: string): PipelineRun[]; // createdAt 倒序、只列最近 N（§2.2）
  write(projectId: string, run: PipelineRun): void;           // 原子写（编排器读-改-写状态机）
  /** 读时对账（D-V1.2-45 / 审计#10，零新状态机）：running 但当前阶段会话已不在活 registry → 判 failed 回写。
   *  liveSessionIds 由 API 层注入（默认 globalThis.__piSessions 的 keys），保持领域层可单测、内核无关。 */
  reconcileOrphan(projectId: string, run: PipelineRun, liveSessionIds: Set<string>): PipelineRun;
}
```

**`reconcileOrphan` 逻辑**（纯函数式，仿 `session-agent-map.ts:64-72` pruneMissing 的「API 注入 live 集合」范式）：
```
if (run.status !== "running") return run;                 // 终态不动
const cur = run.stages[run.currentStageIndex];
if (cur?.sessionId && !liveSessionIds.has(cur.sessionId)) {
  run.status = "failed";
  run.failedReason = "进程重启，运行已中断";
  if (cur.status === "running") cur.status = "failed";
  run.finishedAt = new Date().toISOString();
  write(projectId, run);                                   // 回写盘、清僵尸
}
return run;                                                // 非续跑恢复，不越第三期崩溃恢复
```

### 3.3 `lib/domain/pipeline-orchestrator.ts`【新增·领域 · 承重】

**职责**：`runPipeline(run, deps, signal?)` —— 冻结模型 N 阶段串行编排器。**与 `runDispatch`（`orchestrator.ts:66-204`）同构**（DI 可注入、fire-and-forget、异常收敛为 run→failed 落盘、带 signal），增量 = ①不限超时 acquireSlot ②每阶段跑完主动 evict 释槽 ③累积喂下游 ④run checkpoint 落盘。

**关键复用触点（全部已 Read 实证）**：

| 复用件 | file:line | 用法 |
|---|---|---|
| `runWorker(args)` | `dispatch-runner.ts:81-98` | 起单阶段 worker；返回 `WorkerResult`（`:54-69`：`{sessionId,output,reason,artifactIds,createdContent?}`）；入参含 `projectId`/`profile`/`cwd`/`firstMessage`/`registerInnerSession`/`timeoutMs`/`signal?` |
| doc worker 受限集装配 | `dispatch-runner.ts:131-134` | runWorker 内部按 `mode!=='coding'` 调 `assembleDispatchDocSessionOptions`（`doc-session.ts:85`）套白名单 `DISPATCH_DOC_SESSION_TOOLS`（`doc-session.ts:44`、6 项 read/grep/find/ls + create_artifact + list_artifacts、**无 propose_edit**）——编排器不碰、白嫖 |
| `acquireSlot(opts?)` | `concurrency-gate.ts:32-50` | 排队等空位；本轮传「不限超时」选项（见 §3.5 T5） |
| `evictAgentSessions(root,agentId,deps?)` | `evict-agent-sessions.ts:33-50` | **每阶段跑完安全销毁 worker 会话释槽**（isStreaming→abort→destroy，`:45-46`；**只 destroy、不碰 bySession/removeOwner**，`:11-13` 红线注释） |
| `setOwner(cwd,sid,agentId)` | `session-agent-map.ts:94-99` | 写「会话→agent」归属（看板分组 + re-attach 反查靠它；`orchestrator.ts:161` 同款调用） |
| `DEFAULT_WORKER_TIMEOUT_MS` | `orchestrator.ts:27` | `5*60*1000` 单阶段超时兜底，照搬 |

**关键函数签名**（DI 仿 `OrchestratorDeps`，`orchestrator.ts:35-51`）：
```ts
export interface PipelineOrchestratorDeps {

  registerInnerSession: RegisterInnerSession;           // 必填（透传 runWorker）

}
export async function runPipeline(
  run: PipelineRun, deps: PipelineOrchestratorDeps, signal?: AbortSignal,

```

**主循环伪代码**（与 `orchestrator.ts:90-204` 对照，★ = 增量）：
```
doRunWorker = deps.runWorker ?? runWorker
doAcquireSlot = deps.acquireSlot ?? acquireSlot
doSetOwner = deps.setOwner ?? setOwner
doEvict = deps.evictAgentSessions ?? evictAgentSessions    // ★ DI 钩子
projectRoot = registry.get(run.projectId).root
current = 深拷贝 run; current.status="running"; runStore.write(...)

for (let i = 0; i < current.stages.length; i++):
  stage = current.stages[i]
  if (signal?.aborted || current.cancelRequested):          // ★ run cancel（顶检测）
      return failRun(current, stage, "已取消")
  profile = profileStore.get(run.projectId, stage.agentId)  // 预检在 POST 路由已做；此处兜底 try/catch→failRun
  current.currentStageIndex = i; stage.statusDetail = "queued"; runStore.write(...)  // ★先指向该阶段+排队态(在 acquireSlot 之前)：排队时全局进度条 ④/N 即指向正在排队的阶段(AC-5)
  try { await doAcquireSlot({ timeoutMs: Infinity }) }      // ★① 不限超时排队（AC-5）—— 见 T5 acquireSlot 扩展
  catch (e) { return failRun(current, stage, e.message) }
  stage.statusDetail = undefined; stage.status="running"; stage.startedAt=now; runStore.write(...)

  // ★③ 累积喂下游（AC-6）：拼所有 order<stage.order 且 done 的 stage 产物摘要
  upstream = current.stages.filter(s => s.order < stage.order && s.status==="done")
  firstMessage = stage.subTask
    + "\n\n请在完成后调用 create_artifact 产出受管文档。"   // ★ F10 保障(审计#9)：前置引导
    + (upstream.length ? "\n\n## 上游产物（累积）\n" + upstream.map(formatUpstream).join("\n\n") : "")
    // formatUpstream(s) = `### ${s.agentName}\n${该阶段产物正文}`；正文 = 编排器进程内缓存的 result.createdContent ?? result.output
    //   （与 orchestrator.ts:197 同源；累积全部已完成、非只链一跳 orchestrator.ts:126-129）

  result = await doRunWorker({ projectRoot, projectId: run.projectId, profile, cwd: projectRoot,
            firstMessage, registerInnerSession: deps.registerInnerSession,
            timeoutMs: workerTimeoutMs, signal, additionalSkillPaths, sessionManager, createOptionsOverride })
  stage.sessionId = result.sessionId
  doSetOwner(projectRoot, result.sessionId, stage.agentId)   // 看板分组 + re-attach 反查(orchestrator.ts:161 同款)

  ─────────── ★② 冻结释槽（F16 核心，AC-2/AC-3）───────────
  await doEvict(projectRoot, stage.agentId)                  // isStreaming→abort→destroy；只 destroy、不碰 owner map
  // 注：evict 按 agentId 逐出该 agent 名下存活会话（sessionsForAgent 反查，session-agent-map.ts:87-91）。
  //     串行下该阶段刚跑完的会话即被销毁、腾出 1 槽。同名 agent 跨阶段复用风险见「已知限制」。
  ────────────────────────────────────────────────────────

  if (result.reason==="timeout") return failRun(current, stage, "阶段超时")
  if (result.reason==="aborted") return failRun(current, stage, "已取消")
  if (!result.output.trim() && result.artifactIds.length===0) return failRun(current, stage, "阶段未产出")  // 同 orchestrator.ts:179
  stage.artifactId = result.artifactIds.at(-1) ?? null       // 产物链接落 artifactId(审计#3)
  // 缓存该阶段产物正文供下游累积：cache[stage.order] = result.createdContent ?? result.output（进程内 Map，零额外存储）
  stage.status="done"; stage.finishedAt=now; runStore.write(...)

if (全部 done) current.status="done"; current.finishedAt=now; runStore.write(...)
return current
// failRun(run, stage, reason): stage.status="failed"; run.status="failed"; run.failedReason=reason;
//   run.finishedAt=now; runStore.write; return run（仿 orchestrator.ts:207-219 failTask）
// fire-and-forget：起 run 的 POST 路由 void runPipeline(...).catch(()=>{})（仿 orchestrator 设计，:60-65 注释）
```

> **冻结释槽与「completed worker 不被销毁」现状对照**（已 Read 实证）：`runDispatch` 在 `result.reason==="completed"` 时**不 abort、不 destroy** worker 会话（`dispatch-runner.ts:161-163` 仅在 `reason!=="completed"` 时 abort；`orchestrator.ts:156-198` completed 路径全程无销毁调用）→ wrapper 留 registry 占槽到 `rpc-manager.ts:52` 的 10min idle 才回收（F16 痛点）。本编排器在每阶段（**含 completed**）跑完后主动 `doEvict` → 坐实「现 orchestrator 无销毁调用 → 新增 evict 后释槽生效」。**此即 T1-A 承重墙要证的命门**。

> **为何冻结后不饿死、不需单飞守卫**（D-V1.2-41 否决审计原推荐的单飞守卫，化解审计#4/#6）：每阶段跑完即 evict 还槽 → 单条串行流水线运行中**活 worker 会话恒 ≤1**（AC-2）；多 run 并发时各自活会话之和受 `acquireSlot` 限流、在上限内自然轮转排队（AC-4），不互饿、不死锁。`acquireSlot` 放行→worker 注册的 TOCTOU 微窗（`orchestrator.ts:113-115` 已记）在冻结模型下退回「可接受原状」，不引并发池。

### 3.6 `lib/pipeline/dot-matrix.ts`【新增·纯算法 T4】

**职责**：进度→点亮列数的纯函数。**抽出避免 node:fs 经 `"use client"` 链入 bundle**（D-R7B-07 教训）。
```ts
export const DOT_ROWS = 3;
export const DOT_COLS = 12;          // 3×12 = 36 块（D-V1.2-31）
/** progress(0~1) → 点亮列数（按列点亮、离散刻度，非百分比/进度环）。 */
export function filledCols(progress: number): number {

}

```

## 四、承重墙（T1 spike，lead 复跑；未绿不开后续）

### T1-A · 冻结释槽真能腾出并发槽（F16，AC-2/AC-3 命门）
- **证什么**：`runPipeline` 每阶段跑完调 `evictAgentSessions` 后，活会话 registry 计数**确定性回落**——≥4 阶段连跑，任一时刻活 worker 会话 ≤1，无须等 `rpc-manager.ts:52` 的 10min idle。

### T1-B · dispatch worker 会话能 re-attach 装回 doc 工具（AC-12 命门，D-V1.2-42）
- **证什么**：一个由 `runWorker` 起 + `setOwner` 写了归属、**且被 evict 销毁过** 的 dispatch worker 会话，经 `resolveOrReattachSession`（`session-reattach.ts:97`）能被 `lookupProfile` 反查命中、走 `reattachProfileSession` 重建**受限 doc 工具集**（含 propose_edit、人在场合法）而非 generic（反塞 write/edit/bash）。

## 七、已知限制（MVP，文档显式记）

- **同名 agent 跨阶段复用**：`evictAgentSessions(root, agentId)` 按 agentId 逐出该 agent **名下全部存活会话**（`session-agent-map.ts:87-91` sessionsForAgent）。串行单活会话下逐出当前即所需；若同一 agentId 在多阶段被复用且未来改并行，evict 可能误伤——MVP 串行，记限制、并行留二期。
- **改上游产物不回灌下游**（D-V1.2-47 / 审计#14）：MVP 仅 UI/文档显式写明「改上游不回灌下游」；「从此阶段重跑下游」留二期。
- **长 run token/算力护栏**：MVP 不建预算/速率护栏（未建模风险），**至少留 run cancel 兜底**（§3.11）+ 看板进度可见 → 跑偏可一键停。预算护栏留二期。
- **跨轮文档性能**：一 run 批产 11+ 受管文档；MVP 不在 runPipeline 同步刷 ArtifactPanel，产物经看板/受管分组按需 by-id 打开（一次只渲一篇，复用第三轮 memo 化基线）。受管分组列 11+ 项 = 既有 FileExplorer 列表能力，验收回归确认列 10+ 受管文档时 file panel 不卡。
- **崩溃后续跑恢复（resume）**：读时对账只「清僵尸」判 failed，非续跑（D-V1.2-33/45 不越第三期）。
- **F10 产物保障是引导非强制**：firstMessage 前置「请调用 create_artifact」是引导（审计#9）；模型不调时退回纯文本产物（判空放宽容忍，同 `orchestrator.ts:179`）；可选「done 但 artifactId 空」标 warn（实现期定）。
