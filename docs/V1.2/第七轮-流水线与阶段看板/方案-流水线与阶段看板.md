# 第二块：更新方案 —— 把「流水线 / 阶段看板」做成一等公民

## 2.0 设计哲学（一句话）

> **dispatch 现在是「一轮原子操作」（单次最多挂 2–3 个子任务、串行跑一轮），方案把它升级为「一条可保存、自动按序推进、跑完即释放槽、全局可视的流水线」——蓝图复用现有 Agent 档案，编排复用现有 `runWorker`/`acquireSlot`/`ArtifactService`/会话分组，pi 内核与 propose_edit HITL 红线一律不破。**

**砍掉调研蓝图的重型部分**（理由：sf-mini 那些机器本身默认全关，实测 F1~F16 没有一条痛点指向它们）：
- ❌ 13 阶段全家桶（用户自定义 N 阶段即可，模板第二期）
- ❌ RTM 0.85 覆盖率门 / 自治度 4 档真值表 / 协作模式 3 种
- ❌ DAG 画布（远期；MVP 用最简纵向列表）
- ❌ 真 SSE 总线（沿用现有 2s 轮询；实测 F14 痛点是「没有全局视图」，不是「轮询太慢」）
- ❌ **阶段级人确认 gate（stage.hitl → paused → 按块确认）—— 明确非目标**（详见 §2.5，原草案放第二期会撞红线）

## 2.1 先厘清「两个 3」（评审 P1，避免 lead 改错地方）

本方案反复涉及两个**完全不同**的「3」，先钉死：

| | 出处 | 含义 | 流水线如何对待 |
|---|---|---|---|
| **子任务条数上限 2–3** | `dispatch-store.ts:73`（`if (rawAssignments.length < 2 || > 3) throw INVALID`） | 「**一次派发最多挂几个子任务**」的输入校验 | **这才是「13 阶段塞不进一次派发」的真约束**。流水线突破它：蓝图不限阶段数 |
| **并发会话墙 = 3** | `concurrency-gate.ts:16`（`MAX_CONCURRENT_SESSIONS = 3`） | 「**同时活跃会话 ≤ 3**」的项目红线 | **绝不破**。串行流水线一次只起一个 worker，恒占 ≤1 槽 |

> 原草案在 F9/F16 把这两个 3 归到同一处文件行号、混用论证，是事实错误。本稿已分开：**F9 解的是子任务条数上限，F16 解的是并发槽不释放**。

## 2.4 编排器（`pipeline-orchestrator.ts`，新建，平行于 `runDispatch`）

> 为什么另起而非塞进 `runDispatch`：现有 `runDispatch` 的串行 `for` + 「任一失败中止后续」逻辑是为「2–3 串行」写死的，`DispatchTask` 数据模型扁平。硬塞会污染。但**编排原语大量复用**。

核心循环（伪代码，强调复用点 + 评审 P0 的关键销毁步骤）：
```
runPipeline(run, deps):
  run.status = running; writeRun(run)                      // 同 dispatch-store 原子写
  for stage in run.stages (按 order):                      // 同 orchestrator.ts 串行 for
    await acquireSlot()        ← 复用 concurrency-gate，默认行为不改（排队+60s兜底）
    stage.status = running; writeRun(run)
    firstMessage = stage.subTaskTemplate + 上游产物          // 复用 orchestrator.ts:125-128
    result = await runWorker({ profile, firstMessage, … })  ← 复用 dispatch-runner.runWorker
    setOwner(projectRoot, result.sessionId, stage.agentId)  ← 复用会话分组
    stage.sessionId  = result.sessionId
    stage.artifactId = result.artifactIds.at(-1)            // 复用 orchestrator.ts 回填
    upstreamOutput   = result.createdContent ?? result.output  // 复用 orchestrator.ts:196-197

    ★★ 关键（评审 P0 / 根治 F16）：本阶段跑完立即销毁该 worker 会话、释放并发槽 ★★
    releaseWorkerSession(result.sessionId)   // 见下「安全销毁」

    stage.status = (result.reason === "completed") ? "done" : "failed"; writeRun(run)
    if stage.status == "failed": run.status = "failed"; writeRun(run); return  // 同 runDispatch 失败即止
  run.status = "done"; writeRun(run)
```

**逐条对应需求「定义一次 N 阶段 → 自动按序跑 → 上游自动喂下游 → 并发满则排队不饿死 → 跑完即释放槽」**：

1. **定义一次 N 阶段**：读蓝图 `pipelines/<id>.json` 展开成 run。
2. **自动按序跑**：`for stage in stages`（同 orchestrator.ts 串行），不再受 2–3 限制。
3. **上游自动喂下游**：复用 `orchestrator.ts:125-128`（拼上游产物进下游首条 message）+ `:196-197`（`upstreamOutput = lastArtifactId ? (createdContent ?? output) : output`）。**这是实测「做得好、值得保留」的真有效能力，原样搬。**
4. **并发满则排队不饿死**：`acquireSlot` 现成排队语义（不改默认行为）。
5. **跑完即释放槽（根治 F16 的真药方，评审 P0）**：见下。

**安全销毁 worker 会话（根治 F16，复用既有验证过的配方）**：
- 现有 `dispatch-runner.ts:161-162` 只在 `reason !== "completed"`（timeout/aborted）才 `send({type:abort})`；**completed 路径完全不销毁**，会话留在 `__piSessions`，靠 10 分钟 `idleTimer`（rpc-manager.ts:52）才删 → 这正是 F16 的真根因。
- **复用 `lib/pi/evict-agent-sessions.ts` 已验证的安全销毁配方**：`if (wrapper.inner.isStreaming) await wrapper.send({type:'abort'}); wrapper.destroy();`——`destroy()`（rpc-manager.ts:230-235）会触发 `onDestroyCallback` → `registry.delete(realSessionId)`（:290），槽立即腾出。
- **⚠️ 红线守卫（evict-agent-sessions.ts 顶部明确警告，必须遵守）**：**只删 registry（destroy），绝不碰 `bySession`/`removeOwner`**——动 owner map 会让 re-attach 的 `getOwner` 返 null、误落 generic 分支、反塞回 write/edit/bash（第五轮修过的 bug）。因此 `setOwner` 保留（会话分组要用），**只 destroy 不 removeOwner**。
- **落点选择**：在 **pipeline-orchestrator 层** 每阶段后调用 `releaseWorkerSession`（直接复用 evict 配方的 destroy 逻辑），**不改 dispatch-runner 的 completed 行为**——避免影响既有 dispatch（既有 dispatch 用户可能依赖 completed 会话留存以便前端续聊）。

**为什么不改 `acquireSlot`（评审 P2 回归评估）**：
- `acquireSlot` 已是排队语义，无需改成「永不 reject」。原草案「去/放宽 deadline 让它永不 reject」会把既有 dispatch 在「前端开满 3 会话」时的「60s 后报错」变成「无限挂起」，对既有 dispatch 是行为回归。
- 既有 dispatch 的兜底（`orchestrator.ts` 把 acquireSlot 超时 catch 转 failTask）**保持不变**。
- 真正解决 F16 的是「跑完即释放槽」，不是「等更久」。

**复用清单（已逐一核实存在）**：`runWorker`（dispatch-runner.ts:187 返回 `{sessionId, output, reason, artifactIds, createdContent}`，`reason` 区分 completed/timeout/aborted）、`acquireSlot`（concurrency-gate.ts）、`ArtifactService.readCurrentContent`、`setOwner`（会话分组）、`evict-agent-sessions` 销毁配方、原子写模式。**不改 pi 内核**——起会话/取产物全经 `lib/pi` 封装，与现有 dispatch 同路。

**fire-and-forget 同构**：新增 `POST /api/projects/[id]/pipelines/[pipelineId]/run` 建 run 后 `void runPipeline(...).catch(()=>{})`（仿 `app/api/projects/[id]/dispatch/route.ts:34-40`），前端轮询 `GET /api/pipeline-runs/[runId]`。

## 2.5 红线与边界

**(1) 并发 ≤3 红线 —— 不破**
- 上限仍是 `MAX_CONCURRENT_SESSIONS = 3`（concurrency-gate.ts:16，**不改**）。
- 串行流水线一次只起一个 worker，恒占 ≤1 槽；「跑完即销毁」让它腾得更快。排队等的是「前端开满 3 个聊天」的场景——完全符合「同时活跃会话 ≤ 3」。
- 副作用提醒：若未来真做「阶段内并行跑多 worker」，多 worker 同时 `acquireSlot` 排队即**天然实现「并行度受 3 限流」**，红线自动守住。

**(2) propose_edit HITL 红线 —— 不破，且「阶段级人确认 gate」列为明确非目标**
- 现有 dispatch worker 用 `DISPATCH_DOC_SESSION_TOOLS`（doc-session.ts，**刻意排除 propose_edit**——headless 无人按块确认，propose_edit 会落下永远悬而未决的 PendingChange）。
- **流水线 worker 沿用此路**：产受管文档用 `create_artifact`，**不用 propose_edit**。
- **❌ 阶段级人确认 gate（stage.hitl → run paused → ArtifactPanel 按块确认 → continue）= 明确非目标（评审 mustFix P1 + 砍过度设计）**：
  - 原草案放第二期，但它要在 headless worker 里放开 propose_edit 产出 PendingChange，**这正是 doc-session.ts 注释明令禁止的事**——会撞「propose_edit 必须经人工按块确认」红线。
  - 且实测 F1~F16 **没有一条痛点指向 stage 级人确认**，无需求支撑。
  - **故彻底砍掉，不放任何期**。MVP 全阶段 auto，产物仍是受管文档；用户事后可在 ArtifactPanel 用既有 D3-D5 按块确认能力**手动改**任一阶段产物（这条路径本就存在、不经流水线、不破红线）。
  - 数据模型相应**不含 `hitl` 字段**（原草案 §2.3 的 `"hitl": false` 已删）。

**(3) 不改 pi 内核 —— 不破**
- 编排器全经 `lib/pi` 封装复用（runWorker / acquireSlot / registerInnerSession / evict 配方均为封装层，不触内核源码）。
- **本方案唯一涉及 `lib/pi` 下文件的改动 = 无**——`acquireSlot` 不改、销毁逻辑在 pipeline-orchestrator 层复用 evict 配方。既有 dispatch 零回归。

**(4) 不引数据库 / 鉴权 / 多租户 —— 不破**
- 纯文件落 `.pi/factory/`，仿 dispatch-store 原子写。

**诚实列出补不了 / 不做的**（与调研第 2 路一致 + 本次新增砍项）：
- 真并行 DAG（MVP 纯线性串行）
- Planner LLM 动态拆图（阶段由用户/模板静态定义）
- 真事件总线 SSE（沿用 2s 轮询）
- Critic 真评分阈值自动返工收敛
- 节点级单步 / 分叉回放（远期；MVP 连整 run 暂停都不做）
- stage 门 422 硬归一 / RTM 覆盖率门（砍）
- **stage 级人确认 gate（砍，非目标，见上）**
