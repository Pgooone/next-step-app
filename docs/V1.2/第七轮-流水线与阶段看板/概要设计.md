## 一、方案选型

> 总策略：**「附加式薄壳」**——蓝图 / 编排器 / 看板全在 Next-Step 代码侧新增，**复用既有 dispatch 链的全部内核封装**（`runWorker` / `acquireSlot` / `assembleDispatchDocSessionOptions` / `registerInnerSession`），pi 内核 0 改。新编排器 `runPipeline` 是 `runDispatch`（`lib/domain/orchestrator.ts:66`）的「N 阶段冻结模型」升级版，而非另起炉灶。

| 维度 | 备选 | 结论 |
|---|---|---|
| 蓝图持久化 | A 受管文档(版本/diff) / **B 原子写 JSON 文件** | **B 采纳**（D-V1.2-38）：配置 JSON 纳管为受管文档是过度设计；仿 `dispatch-store.ts:134` 的 `atomicWrite`（tmp+rename）即可。 |

## 三、数据模型

> 全部 JSON、原子写文件、无 DB（D-V1.2-38）。`status` 沿用既有 `DispatchStatus`（`dispatch-store.ts:7`）= `"pending"|"running"|"done"|"failed"`，再加阶段/排队/取消语义（见下）。

### 3.1 蓝图 PipelineBlueprint（`.pi/factory/pipelines/<id>.json`）

```jsonc
{
  "id": "uuid",                 // randomUUID
  "projectId": "uuid",          // 归属项目（ProjectRegistry 反查 root）
  "name": "CS2 监控网页流水线",   // 看板容器头 ⑂ 名称
  "stages": [
    {
      "order": 1,               // 唯一权威执行序：升序 = 串行顺序（钉死，杜绝 F6「展示序≠执行序」，审计#12）
      "agentId": "uuid",        // 引用既有 AgentProfile；起 run 前预检存在（审计#11）
      "subTaskTemplate": "..."  // 该阶段子任务（MVP 手填，无模板导入 —— D-V1.2-37 模板放二期）
    }
    // … N 阶段
  ],
  "createdAt": "ISO",
  "updatedAt": "ISO"
}
```

**保存校验**（`pipeline-store.create/write`，与 dispatch `create` 同级 + 单测，审计#12）：
- `stages.length >= 1`（拒 0 阶段）；
- 每阶段 `agentId` 非空、`subTaskTemplate.trim()` 非空（拒空模板）；
- `order` 为 `1..N` 连续整数、**无重复、无缺号**（拒 order 重复/缺号）；落盘前按 order 升序归一。

### 3.2 运行实例 PipelineRun（`.pi/factory/runs/<runId>.json`）

```jsonc
{
  "id": "runId-uuid",
  "projectId": "uuid",
  "pipelineId": "uuid",         // 来源蓝图（蓝图删除后历史 run 仍可只读回看，链 artifactId 非 sessionId）
  "pipelineName": "...",        // 起 run 时快照蓝图名（蓝图改名/删后看板头仍可显）
  "status": "running",          // pending|running|done|failed —— run 级聚合态
  "currentStageIndex": 0,       // 0-based，看板全局进度条 ▰▰▱… 与 ④/N 头取此值
  "createdAt": "ISO",
  "finishedAt": "ISO|null",
  "cancelRequested": false,     // run cancel 落盘标记（看板停止按钮 / POST cancel 翻；编排器 signal 同源）
  "failedReason": "...|null",   // 失败/对账判 failed 的人读原因（看板失败态头「已失败·停在第N阶段」）
  "stages": [
    {
      "order": 1,
      "agentId": "uuid",
      "agentName": "架构师",     // 快照（agent 删后看板降级渲染 `?? agentId`，审计#11）
      "subTask": "...",         // 由 subTaskTemplate 实例化（MVP 直接 = 模板）
      "status": "done",         // pending|running|done|failed（+ 衍生展示态：见 statusDetail）
      "statusDetail": "queued", // 可选展示细分：queued(排队等槽,审计#5) / 无 —— 不进权威状态机、仅看板渲染用
      "sessionId": "...|null",  // worker 真实 sessionId（running/done/failed/aborted 才有；pending=null）
      "artifactId": "...|null", // 该阶段 create_artifact 产物 id（产物链接落它、非 sessionId，审计#3）
      "startedAt": "ISO|null",
      "finishedAt": "ISO|null"  // 耗时 = finishedAt - startedAt（hover 浮窗显）
    }
    // … N 阶段
  ]
}
```

**累积上游**（D-V1.2-46 / 审计#8）：**不另存字段**，跑到第 i 阶段时由编排器实时拼接「所有 `order < i` 且 `status==='done'` 阶段的产物摘要」（产物正文从 `result.createdContent ?? result.output` 取，与 `orchestrator.ts:197` 同源，只是从「只记上一跳」改为「累积全部已完成」）。零额外存储。

**latestRunId / run 列表**（D-V1.2-35 / 审计#15）：`listRuns(projectId, pipelineId)` 扫 `.pi/factory/runs/*.json`、按 `createdAt` 倒序、**只列最近 N 条**（N 与 runs/*.json 保留策略具体值 = 实现期 ADR 定，需求 §六待澄清项 2，本文不预设）。「最新 run」= 列表头，无需另存 latestRunId 指针（仿 `listVersions`）。

**读时对账 reconcileOrphan**（D-V1.2-45 / 审计#10，**零新状态机**）：`GET pipeline-runs/[runId]` 读盘后，若 `status==='running'` 且「当前阶段 `sessionId` 非空但不在活 registry（`globalThis.__piSessions`，`concurrency-gate.ts:19` 同源）」→ 判 `status='failed'` + `failedReason='进程重启，运行已中断'` 并回写盘（清僵尸、非续跑恢复，不越第三期崩溃恢复）。看板轮询随即停（终态）。

## 四、关键流程

### 3.1 runPipeline 冻结模型主循环（F16 / D-V1.2-41）

> 与 `runDispatch`（`orchestrator.ts:66`）同构（DI 可注入、fire-and-forget、异常收敛为 run→failed 落盘、带 `signal`），增量 = ①不限超时 acquireSlot ②每阶段跑完主动销毁会话 ③累积喂下游 ④run checkpoint 落盘。

```
起 run (POST pipelines/[id]/runs)
  ├─ 预检：profileStore.get(每个 stage.agentId) 全部存在？否 → 不起 run、422 干净报错（审计#11）
  ├─ pipeline-run-store 建 run（status=running, currentStageIndex=0, 各 stage=pending）落盘
  ├─ new AbortController；登记 runId→controller（进程级 Map，供 cancel 翻，见 §3.5）
  └─ void runPipeline(run, { ...deps, signal: controller.signal }).catch(()=>{})   // fire-and-forget

runPipeline(run, deps, signal):
  for (order = 1..N):                          // 严格按 order 升序串行
    if (signal.aborted || run.cancelRequested) → failRun("已取消"); break
    run.currentStageIndex=order-1; stage.statusDetail = "queued"; 落盘   // ★先指向该阶段+排队态(在 acquireSlot 之前)：排队时全局进度条 ④/N 即指向正在排队的阶段（审计#5，与详细 §3.3 一致）
    await acquireSlot({ timeoutMs: Infinity })  // ★①不限超时：流水线下一阶段排队等槽、不被 60s 误杀(AC-5)
    stage.statusDetail = undefined; stage.status="running"; stage.startedAt=now; 落盘
    firstMessage = stage.subTask
                 + "\n\n## 上游产物（累积）\n" + 拼接(所有 order'<order 且 done 的 stage 产物摘要)   // ★③累积喂下游(AC-6)
                 // F10 保障(审计#9)：前置「请调用 create_artifact 产出受管文档」引导
    result = await runWorker({ projectRoot, projectId, profile, cwd:projectRoot, firstMessage,
                              registerInnerSession, timeoutMs: workerTimeoutMs, signal, additionalSkillPaths })
    stage.sessionId = result.sessionId
    setOwner(projectRoot, result.sessionId, stage.agentId)   // 看板分组 + re-attach 反查靠它(orchestrator.ts:161 同款)
    ──────────── ★②冻结释槽（F16 核心，AC-2/AC-3）────────────
    await evictAgentSessions(projectRoot, stage.agentId)     // isStreaming→abort→destroy；只 destroy、不碰 owner map
    // 注：evict 按 agentId 逐出该 agent 名下存活会话；串行下该阶段刚跑完的会话即被销毁、腾出 1 槽。
    //     同名 agent 跨阶段复用风险见「已知约束」——MVP 串行单活会话，逐出当前即所需。
    ─────────────────────────────────────────────────────────
    if (result.reason==="timeout") → failRun("阶段超时"); break
    if (result.reason==="aborted") → failRun("已取消"); break
    if (!result.output.trim() && result.artifactIds.length===0) → failRun("阶段未产出"); break   // 判空放宽(同 orchestrator.ts:179)
    stage.artifactId = result.artifactIds.at(-1) ?? null     // 产物链接落 artifactId(审计#3)
    stage.status="done"; stage.finishedAt=now; 落盘
  if (全部 done) run.status="done"; run.finishedAt=now; 落盘
  finally: 清 runId→controller 登记
```

**为何冻结后不饿死、不需单飞守卫**（D-V1.2-41 否决审计原推荐的 MVP 单飞守卫，化解审计#4/#6）：每阶段跑完即 `evictAgentSessions` 还槽 → 单条串行流水线运行中**活 worker 会话恒 ≤1**（AC-2）；多 run（或流水线+旧快派）并发时各自的活会话之和受 `acquireSlot` 限流、在上限内**自然轮转排队**（AC-4），不互饿、不死锁。`acquireSlot` 放行→worker 注册的 TOCTOU 微窗（`orchestrator.ts:113` 已记）在冻结模型下退回「可接受原状」，不引并发池。

## 六、承重墙（T1 spike，lead 复跑；未绿不开后续）

> 两条承重命门，对应需求 AC-2/AC-3（F16 冻结释槽）与 AC-12（续聊 re-attach）。spike 用 faux/桩脱离进程级状态，lead 独立复跑。

**T1-A · 冻结释槽真能腾出并发槽（F16，AC-2/AC-3 命门）**：证 `runPipeline` 每阶段跑完调 `evictAgentSessions` 后，活会话 registry 计数**确定性回落**（≥4 阶段连跑，任一时刻活 worker 会话 ≤1，无须等 `rpc-manager.ts:52` 的 10min idle）。验法：注入 faux registry + faux `runWorker`（注册一个假 inner、跑完留在 registry），断言每阶段 evict 后 `activeSessionCount()` 归零、第 i+1 阶段 `acquireSlot` 立即放行。**坐实「现 orchestrator 无销毁调用 → 新增 evict 后释槽生效」**。

**T1-B · dispatch worker 会话能 re-attach 装回 doc 工具（AC-12 命门，D-V1.2-42）**：证一个由 `runWorker` 起 + `setOwner` 写了归属的 **dispatch worker 会话**，经 `resolveOrReattachSession`（`session-reattach.ts`）能被 `lookupProfile` 反查命中、走 `reattachProfileSession` 重建**受限 doc 工具集**（含 propose_edit、人在场合法）而非 generic（反塞 write/edit/bash）。**T1-B 不绿 → 「进入完整对话」降级为只读快览**（不阻塞其余 MVP，但 AC-12 改判「降级达成」需用户确认）。

> 双层验收（沿项目纪律）：逻辑层 verifier（`pipeline-store`/`-orchestrator`/对账/校验 hermetic 单测）+ 端到端真浏览器（看板视觉 AC-10 / 进会话只读守卫 AC-11 / 失败态 AC-8 / cancel AC-9 / `pageErrors=0` AC-15，走 `browser-e2e`）。**UI 卡必走真浏览器**（SSR/hydration/node:fs 污染只在真浏览器暴露，D-R7B-07）。
