### 0. 环境与进入项目
- dev server 起在 :30141，浏览器打开后**应用自动恢复了上次的 CS2 项目**（直接进了工作台，体验顺）。
- ⚠️ 一进来就看到两件事：①工作台中央「开始使用」三步引导写的是「选/建项目 → 点 Agents 或主对话 → @ 转交」，**只字未提 Dispatch**（多 Agent 派发才是软件工厂的主线）→ **F1/C2**；②左侧会话列表堆了十几条历史测试会话（"echo NEXTSTEP_BASH_OK""只回复 OK"…），没有清理/折叠，真正的工作会话被淹没 → **F2**。
- 左下角四个入口 Models/Skills/Agents/Dispatch 仍是最小灰字图标，且开发态左下「N」浮标（Next.js DevTools）压着 Models（生产构建无此浮标，属 C13 dev-only）。

### 1. 配置阶段角色（Agents）
- sf-mini 13 阶段需要十多个角色，**Agent 列表默认空、无任何模板**，只能「新建 Agent」一个个手填 → **F4/C5**。本轮手捏了 3 个（资料调研员 / 开发工程师 / 代码审查员），加上已存在的产品经理 / 架构师。
- 新建表单字段清晰（名称*/角色描述/模型/技能/模式/工具/思考强度），模式默认「文档型」正合用。但：
  - 技能区赫然列着 **brainstorm / browser-e2e / cc-multi-agent-dev-flow / vibe-coding** 这 4 个**开发 Next-Step 本身**用的元技能，对 CS2 项目毫无关系 → **F3/C4**。
  - 模式两按钮等权、无「(推荐)」、helper 只列工具名（read/grep/…）不讲"什么角色该选哪个" → **F5/C11**。
- 每建一个：新建 → 填名称 → 填描述 → 保存（回列表）。9 个角色重复 9 遍，纯体力活。

### 2. 第 1 轮派发（资料 / 开发 / 审查）
- 复用已有的需求(prd)/设计(design)受管文档，本轮补 资料(1)/开发(4)/审查(5)。
- **F6/C6（🔴 最扎眼）**：Dispatch 表单里 Agent 按「最近创建」排序，显示成「开发→资料→审查」；而**实际执行是按「我点选的顺序」串行**。我必须**主动按 资料→开发→审查 的顺序去点**，UI 完全不告诉你"执行顺序=勾选顺序"。不懂这点的用户会以为按列表从上到下跑（开发跑在资料前面），流水线直接乱套。最终进度视图证明确实按"资料→开发→审查"跑，说明我的理解对，但这全靠经验、不靠 UI。
- **F7/C7**：「选择 Agent（N/3，至少 2 个）」只给数字、不解释为什么 2~3（3 是并发上限）。选满 3 个后产品经理/架构师按钮变灰，也没说为啥不能再选。
- **F8/C8**：子任务输入框**勾选后才冒出来**，事先看不到。
- **F9/C6+C15（🔴）**：13 个阶段塞不进一次派发（上限 3），只能人工拆成 4 轮，且**跨轮没有任何流水线衔接**——上游产物喂下游全靠我在子任务里手写"先 read 项目根目录的 xx.md"。这就是 sf-mini「顺序流水线/阶段」在 Next-Step 里的缺失。
- **F10/C10**：要让 doc 派发产出**受管文档**，必须在目标/子任务里**手写"调用 create_artifact"**，否则只落纯文本 `.pi/artifacts/...md`、不进受管文档体系。UI 无任何提示，纯靠我知道这个坑。
- **🐞 自动化坑（非产品 bug，但值得团队留意）**：用 chrome-devtools 的 `fill`/`fill_form` 填目标与子任务，**值进了 DOM 却没提交进 React 受控状态**，导致"明明填好了，`发起派发`按钮却一直 disabled、点不动"。普通 `fill` 失效，最后靠 `evaluate_script` 用"原生 setter 先清空再赋值 + 派发 input 事件"双步法才骗过 React tracker 解禁按钮。真人逐键输入不受影响——但「看着填好了却点不亮」这种状态若真人也偶发（如粘贴），会非常困惑。
- 发起后进度视图清晰：资料调研员「执行中」、其余「待执行」，按序推进。✅ 这点体验好。
- ⏱️ 第 1 轮耗时约 8 分钟（3 个 Agent 串行走 DeepSeek，开发那步最慢约 6 分钟）。完成后**受管文档组自动刷新**出新文档、**会话按 Agent 自动分组**（资料/开发/审查各一条）——✅ 这两点体验好（第四轮/第五轮的成果）。
- ⚠️ **F11**：完成后受管文档组里出现**两份「代码审查记录」**——审查 Agent 把 create_artifact 调了两次，系统不按标题去重。

### 3. 第 2/3 轮派发（单测/集测/部署 + 准备系测/收敛/发布）
- **F12（再次派发的坑）**：第 1 轮完成后点「Dispatch」，弹出的是**上一轮的「派发进度」摘要**（带「重新发起」按钮），**不是新的空白表单**。必须点「重新发起」才回到表单。新人很可能以为"派发完就不能再发了"。
- **F13（规模放大 C6）**：建到 13 个角色后，派发表单的 Agent 选择列表**又长、按最近创建倒序、无搜索无分组**。要在十几个里挑出本轮的 3 个、还得**按执行顺序逐个点**，体验随角色数增长明显变差。
- **F14（无全局视图）**：13 个阶段被迫拆成 4 轮派发，**没有任何一个地方能看到"13 阶段整体走到哪了"**（sf-mini 有阶段看板/泳道）。只能一轮一轮看各自的派发弹窗，跨轮进度全靠自己记。这正是 C15「缺流水线/阶段表达」的直接体感。
- 实操优化：第 2 轮起我把"建角色"和"填派发表单"改成脚本化驱动真实 UI（点真实按钮/填真实输入框、触发真实 React），避免每轮重复手填十几个框——但这是我作为自动化的取巧，**普通用户只能纯手工**，更累。
- ⏱️ 第 2 轮 3 个 Agent（单测/集测/部署）同样串行跑，耗时与第 1 轮相当。

### 4. 第 3 轮派发（系测/收敛/发布）——踩了两个真坑 F15 + F16
这一轮我**真实地跑错了一次**，暴露两个重要卡点：
- **F16（并发墙）**：R2 一结束我立刻发起 R3，结果整轮**「失败」**，报错 **「活跃会话已达上限 3，请关闭部分会话后重试」**。原因：R2 的 3 个 worker 会话还没释放，R3 的新 worker 抢不到并发槽（≤3 红线）。**多轮派发与 ≤3 并发上限天然打架**——而 >3 个阶段又必须分多轮，这是结构性矛盾。且失败后**没有自动重试**。
- **F15（重新发起预填旧配置）**：失败后我点「重新发起」想重发，发现表单**预填的是上一轮 R2 的角色（单测/集测/部署）+ 子任务**、3 个已占满上限、我要的"系测/收敛/发布"按钮全是**禁用**态。我第一次没察觉、用脚本直接发，结果**把 R2 的角色配着 R3 的目标又错跑了一遍**（幸亏被并发墙拦下、没产出垃圾）。正确做法是：**先逐个取消勾选旧角色**释放上限，再选新角色。这个"重新发起≠新建空白派发"的语义对新人极不友好。
- 修正后（取消旧 3 个 → 选系测/收敛/发布 → 重填子任务/目标 → 等旧会话释放后发起）R3 才正常跑起来。**对普通用户来说，这一段几乎必然踩坑**。

### 5. 第 4 轮派发（运维/反馈）——F16 升级：60 秒不够，被迫重启 dev
- 第一次：R3 结束后等了约 60 秒再发起 R4，结果**还是失败**（运维工程师 failed）——**worker 会话存活到 idle 回收（约 10 分钟），60 秒远不够**。「重新发起」又预填了 R3 的角色（再次印证 F15），我取消勾选后改选 运维/反馈，但发起即撞 ≤3 并发墙失败。
- 解决：**重启 dev server**（清空内存中所有活跃会话→并发墙归零 + dispatch store 重置）。重启 + 刷新后再开 Dispatch 是**干净空表单**（无 F15 预填），选 运维/反馈、发起，这次成功跑起来。
- 💡 这说明 F16 比想象更严重：**多轮派发之间，普通用户除了等 ~10 分钟会话回收、或手动逐个关闭上轮会话，几乎没有顺畅的衔接办法**（我是靠重启 dev 这种"开发者特权"绕过的，普通用户没有）。这是当前用 Next-Step 跑"多阶段长流程"最硬的结构性障碍。

## 总结：对优化 Next-Step 整体流程的建议

> 一句话：**Next-Step 把"捏角色 + 一次性派发 2~3 个"做得不错，但要承载 sf-mini 那样的"多阶段标准软件工程流水线"，当前是"硬拼"出来的——全程在和工具的结构限制搏斗。** 走完 13 阶段共发起 4 轮派发、手捏 11 个角色、踩了 1 次真错跑 + 1 次并发失败。

### 🔴 第一优先：把"多阶段流水线"做成一等公民（解 F9/F14/F15/F16，对应 C15 软件工厂蓝图）
这是最大的痛点集群。当前要跑完 13 阶段，用户被迫：手动拆成 4 轮 → 每轮手选 ≤3 个角色 → 手写"先 read 上游 .md"喂上游 → 轮次间撞 ≤3 并发墙 → 「重新发起」预填旧角色误导。建议（与 memory 里"软件工厂蓝图/方案 D"一致）：
1. **流水线蓝图**：一次性定义 N 个阶段（角色+顺序+子任务模板），保存为可复用蓝图；不再受"一次 ≤3"束缚。
2. **自动编排**：>3 阶段时由编排器**自动排队顺序推进**（活跃会话满了就等待，而不是报错失败），上游产物**自动喂下游**（无需用户手写 read 路径）。
3. **阶段看板**：一个视图看 13 阶段全局进度（泳道/状态徽章/当前在跑哪步），取代"逐轮看派发弹窗、跨轮全靠自己记"。

### 🟠 第二优先：修派发表单与轮次衔接的具体卡点
4. **执行顺序可见（F6）**：已选 Agent 用序号/拖拽显式表达执行顺序，并标注"按此顺序串行、上游喂下游"。**当前"显示按创建倒序、执行按勾选序"两者错位且零提示，是最隐蔽的坑。**
5. **「重新发起」语义拆分（F15）**：明确区分「重跑本次派发」（预填旧配置）与「新建一次派发」（空白表单）两个入口；预填时也应让用户能一键清空。
6. **并发墙友好化（F16）**：轮次间不要硬报错"活跃会话已达上限"，改为排队等待/自动重试；或在发起前提示"将先释放上轮会话"。
7. **上下限与子任务提示（F7/F8）**：解释"最多 3=并发上限"；未勾选时也提示"勾选 Agent 后可填子任务"。

### 🟠 第三优先：降低"从零配置"的门槛
8. **预置角色模板库（F4/C5）**：内置 sf-mini 那 13 个标准角色（产品经理/架构师/开发/审查/各类测试/部署/运维…）一键导入，免去逐个手捏。本次手捏 9 个角色纯体力活。
