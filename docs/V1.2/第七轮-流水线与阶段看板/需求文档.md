## 一、背景与目标

### 1.1 背景：sf-mini 全流程实测暴露的三类痛点

本轮做的是 V1.2 第一轮「多 Agent 管理」当年 deferred 的**机制层**（流水线 / 阶段编排 / 阶段看板，方案 D 纸面、从未实现；第一轮只做了视觉的 agent 卡片）。立项依据 = 用 Next-Step 网页真跑完 sf-mini 13 阶段全流程（CS2 项目）暴露的 F1~F16 缺口，其中本轮聚焦根治三类：

- **F9 / F14：跨轮无流水线、无全局进度。** 现有 dispatch 只能一次派 2~3 个子任务、上游产物只喂「紧邻上一阶段」一跳；13 阶段串行没有「蓝图（一次定义、可复跑）」概念，也没有一眼看清「跑到第几阶段 / 哪步炸了」的全局进度视图。
- **F16：completed worker 会话占槽到 10min idle 才回收。** 串行流水线每跑完一阶段，已完成的 worker 会话仍留在会话 registry 占着并发槽，多阶段累积下来要等到 ~10 分钟空闲回收才腾位，长流水线被并发上限（≤3）拖到几乎卡死。
  - F16 真根因（lead 复核，与边界审计一致）：`acquireSlot` 本就排队 + 60s 超时（非撞墙），真痛点是 completed worker 会话不被主动销毁、占槽到 idle 回收。修法 = 编排器每阶段跑完后主动销毁会话腾槽（复用第六轮已落地的安全销毁配方 `lib/pi/evict-agent-sessions.ts`，守「只 destroy、绝不碰 bySession/removeOwner」红线），**不动 `acquireSlot` 本身的排队语义**。
- **跑错的长流水线停不下来。** 跑偏的 run 持续占槽、烧 token，过去只能靠重启 dev 强杀——本轮要消灭这个痛点。

### 1.2 目标

让 Next-Step 能用「一张蓝图（N 阶段）→ 一键起 run → 自动串行编排 → 阶段看板看进度」表达 sf-mini 式多阶段流水线，并解决 F16 占槽问题；同时给最小的 run 级取消能力。具体：

1. **流水线蓝图**：用户手填 N 阶段（每阶段 = 一个已有 agent + 一段子任务模板），原子写文件持久化、可复跑。
2. **自动编排器（冻结模型）**：按 `order` 升序串行跑各阶段，每阶段跑完即销毁会话腾槽（F16），上游产物累积喂下游（F9）。
3. **阶段看板**：纵向列表 + Kimi 风格卡片，只读看全局进度、各阶段状态、产物链接；支持多 run 历史回看、空态引导建蓝图。
4. **进会话**：左键点卡片弹悬浮二级菜单，只读看该阶段冻结对话 + 产物 + 底部 agent 切换条；菜单内「进入完整对话」跳主对话窗解冻续聊。
5. **合并入口**（已定：一个入口·两 tab，D-V1.2-39）：左栏**一个「流水线」入口**；模态内**两 tab**——「流水线」（蓝图 N 阶段）/「快速派发」（旧 2~3 派发表单原样并入）。旧 `runDispatch` 逻辑几乎不动、F6/F8/F15 在「快速派发」tab 内就地修。
6. **run 级 cancel**：跑偏的 run 可一键停止，不再被逼重启 dev。

不破核心红线：不改 pi 内核、headless worker 不放 `propose_edit`、纯文件无 DB；**唯一一处红线变更**（≤3 并发改为「默认 3、用户可配」）为用户拍板、风险自担（见 §四）。

### 2.2 本轮不做（推迟第二期）

- **角色 / 流水线模板**（sf-mini 13 角色一键导入、预置角色库）—— D-V1.2-37。
- **阶段级 HITL（人确认 gate）** —— D-V1.2-40：MVP 全阶段 auto 产受管文档；要改产物事后在 ArtifactPanel 按块确认。「安全版可调节 HITL」（run 级暂停/继续 gate + `paused` 状态、worker 始终不放 propose_edit）保留为未来增项。
- **`dependsOn` 显式依赖声明** —— 累积全量上游已够，dependsOn 太重，留二期（D-V1.2-46）。
- **改上游产物 → 重跑下游 / 「从此阶段重跑下游」逃生口** —— D-V1.2-47：MVP 仅文档澄清「改上游不回灌下游」；重跑下游留二期。
- **崩溃后续跑恢复（resume）** —— 读时对账只「清僵尸」判 failed，非续跑（D-V1.2-33/45 不越第三期崩溃恢复）。
- **蓝图纳管为受管文档（版本/diff/回滚）** —— 对配置 JSON 过度设计，原子写文件即可（D-V1.2-38）。

## 三、功能需求

### F-1 流水线蓝图（pipeline blueprint）

- 蓝图 = 一组有序阶段，每阶段含：引用的 `agentId`、子任务模板 `subTaskTemplate`、执行序 `order`。MVP 用户手填 N 阶段（无模板导入）。
- 持久化：原子写 `.pi/factory/pipelines/<id>.json`（仿 dispatch-store 临时文件 + rename 模式，参考 `lib/domain/dispatch-store.ts` 的 `atomicWrite`）。**待核**：`.pi/factory/` 路径与「文件 API 只服务允许根（会话 cwd + `~/pi-cwd-\d{8}`）」边界是否冲突——写概要设计时纳入考量、双层验收回归。
- 保存校验（审计#12）：拒绝 0 阶段 / 空 subTaskTemplate / order 重复或缺号；钉死「**order 升序 = 唯一权威执行序**」，避免复活 F6「展示序 ≠ 执行序」。与现 dispatch `create` 同级校验 + 单测。
- agent 引用降级（审计#11）：起 run 前预检全部阶段 agentId 存在（缺则不起 run、干净报错）；看板/编辑器复用既有 `?? agentId` 降级渲染。

### F-2 自动编排器 · 冻结模型（D-V1.2-41 / F16）

- **冻结模型定义**：「冻结 = 释放会话活运行时腾槽 + 对话存盘保留（**非删对话**）」。每阶段跑完即还槽 → 串行流水线恒 1 个活会话、多 run 在并发上限内自然轮转排队，**不饿死、不死锁，连单飞守卫都不需要**（D-V1.2-41 否决了审计原推荐的 MVP 单飞守卫；审计缺口 #4/#6 由此化解）。
- **每阶段跑完主动销毁会话**：编排器在每个阶段（含 completed）跑完后主动销毁该阶段 worker 会话腾槽（复用 `lib/pi/evict-agent-sessions.ts` 安全配方：isStreaming→abort→destroy，**只 destroy、绝不碰 bySession/removeOwner**）。注：现编排器代码无此销毁调用，需新增（已核，非待核）。
- **上游累积喂下游**（D-V1.2-46 / 审计#8）：下游阶段首条消息拼入**全部已完成上游**阶段的产物摘要（非只链一跳）。
- **F10 产物保障**（审计#9，实现期处理）：拼 firstMessage 自动前置「请调用 create_artifact」，避免阶段只产裸文本被判 done；可选「done 但 artifactId 空」标 warn。
- **run checkpoint**：编排过程把 run 状态落 `.pi/factory/runs/<runId>.json`（复用已落盘文件「零额外存储」、listRuns 仿 listVersions）。

### F-3 阶段看板（PipelineBoard，只读）

视觉严格遵循 [`看板视觉定稿-kimi-v11.html`](看板视觉定稿-kimi-v11.html) 的 **Kimi(A)** 块（用户否决 Swiss(B)），支持亮/暗双主题：

- **形态**：纵向列表（D-V1.2-28），N 阶段按 order 从上到下 = 串行阅读顺序。
- **容器头**：`⑂ 流水线名 · ④/N` + 全局进度条（`▰▰▰▱▱…` 一行，数据取 `run.currentStageIndex`；格数随 stages 数动态，审计 §🟡）。
- **每阶段两行卡**：
  - 行1（身份）：dicebear notionists 头像（D-V1.2-30，`@dicebear/core` 本地离线生成、走 npm 依赖不依赖外网）+ 阶段名 + 角色名 + 右上两位零填充序号（`01`/`02`，纯文本灰字非圆圈徽章）。
  - 行2（任务+状态）：行首 `└` 树形连接符 + 单行截断子任务摘要 + **绿点阵 LED 进度**（D-V1.2-31，3×12 共 36 块、按列点亮、`filledCols = round(progress*12)`）+ 状态徽章 + 行尾 `›`。
- **各状态视觉**：待执行（灰/暗置/空点阵）· 运行中（点阵部分亮 + 左 3px emerald 强调条 + 呼吸动画 + 整卡高亮）· 完成（变灰退场 + ✓ + 满点阵）· **失败（红：失败徽章复用 STATUS_META 红 + 左红条 + run 级「已失败·停在第N阶段」头，审计#13）**。
- **排队态**（D-V1.2-44 / 审计#5）：阶段等会话槽时看板显「排队中·等会话槽」可见态。
- **只读**（D-V1.2-33）：看板只看进度，不做编排操作（不取消阶段、不重跑阶段）；要改产物事后走 ArtifactPanel 按块确认——不碰编排状态机、不破红线。只读 affordance 提示（避免 cursor/chevron 误导能控 run，审计 §🟡）。
- **空态**（D-V1.2-34）：无 run 时显示空态卡「+ 新建流水线」，点击直接进蓝图编辑（PipelineEditor）。
- **多 run 历史**（D-V1.2-35）：顶部下拉选历史 run 回看（仿第二轮版本历史）；run 下拉只列最近 N 条（审计#15）。

### F-4 进会话（悬浮二级菜单，D-V1.2-32/42）

- **左键点卡片 → 悬浮二级菜单**（非整页跳转）：只读快览该阶段 agent **真实会话内容**（冻结对话 transcript + 产物）+ 底部 agent 切换条（菜单内横向切换查看不同阶段 agent 会话）。
- **悬停卡片 → 浮窗预览**（约 330px、可向下滚动，与左键菜单**分开实现、勿合并**）：任务/范围/上游依赖/产物/验收清单/最近动作/耗时。
- **「进入完整对话」按钮 → 跳 Next-Step 主对话窗 ChatWindow 续聊**（复用既有 ChatWindow）：解冻 re-attach 占 1 槽（受并发上限；复用第五轮 re-attach 机制、已修 doc 工具 bug）。
- **边界守卫**（审计 #1/#2/#3）：
  - 已完成/已销毁阶段的会话页**纯只读**（不挂 ChatInput 的 send），不能在二级菜单里追问复活会话绕过 acquireSlot（#1）。
  - 底部切换条**只枚举 sessionId 非空**（running/done/failed/aborted）的阶段，pending 阶段置灰（#2）。
  - 历史 run 进会话纯只读；jsonl 缺失降级「记录已不存在」；产物链接落 **artifactId 而非 sessionId**（#3）。
- **续聊需先 spike**（D-V1.2-42）：「进入完整对话」解冻续聊依赖 dispatch 会话 re-attach（第五轮 deferred 的 §B），实现前先小 spike 验证可行。

### F-5 合并入口（已定：一个入口·两 tab，D-V1.2-39 / 审计#15②）

- 左栏新增**一个「流水线」入口** + 模态（与 Dispatch 同范式、无新路由，参考 `AppShell.tsx` 现有模态挂载 + 触发范式）。
- D-V1.2-39 用户已拍板「**合并成一个入口**」，落地 = **一个入口·两 tab**：模态内并列「流水线」tab（蓝图 N 阶段、走 `runPipeline`）/「快速派发」tab（旧 2~3 派发表单原样并入、走旧 `runDispatch`）。满足「合并一个入口」——左栏只有一个入口、UI 同模态；旧 `runDispatch` 逻辑几乎不动，F6/F8/F15 的修复在「快速派发」tab 内就地做。
- 看板轮询复用 `DispatchPanel` 现有轮询模式（`POLL_INTERVAL` 2000ms + 终态停轮询）、状态徽章复用 `STATUS_META`。

### F-6 run 级 cancel（D-V1.2-43 / 审计#7）

- 接最小 run 级 cancel：runPipeline 带 AbortSignal + 看板「停止」按钮 / `POST .../cancel` 翻 AbortController。微调 D-V1.2-33（只读 + 一个停止能力，run 级而非阶段级）。

### F-7 旧 DispatchForm 就地小修（审计 §🟡 / 讨论记录 §2.8）

随合并入口并入，低成本先行（不依赖流水线机制）：

- **F6**：DispatchForm 给已选 agent 标执行顺序号 ①②③ + 一句「将按此顺序串行执行，上游产物自动喂给下游」。
- **F15**：重跑入口拆两按钮——「重跑本次」（保留预填）/「新建空白派发」（清空 goal/选中 agent/子任务）。
- **F8**：未勾选 agent 时子任务输入框给占位提示「勾选 Agent 后可填子任务」。

### 4.2 红线变更（用户拍板、风险自担，D-V1.2-41）

- 项目原红线「**并发会话 ≤3（硬墙、不改）**」→ 改为「**默认 3、用户可在设置调整**」。冻结模型让 ≤3 从「硬墙」变「吞吐限流」。
- 落地三件套：① `concurrency-gate` 读配置（默认 3、给合理上限）② UI 给资源警示 ③ **同步更新根 `CLAUDE.md`「并发会话 ≤3」红线条目**。
- 调高 = 单进程更多活会话、资源耗尽风险用户自担；**冻结模型本身不变**。

### 4.3 待概要设计纳入的补漏批判（审计补漏 4 项）

- **`.pi/factory` 与「允许根」边界**：run 产物 / 蓝图 / run 文件路径是否在允许根内未核 → 写概要设计时纳入考量、双层验收回归。
- **跨轮文档性能**：一条 run 批量产 11+ 受管文档，叠加第二~三轮版本 diff / TOC diff / 性能 memo 化时，是否拖垮 file panel / ArtifactPanel 渲染（第三轮 perf 是单文档基线）→ 概要设计纳入考量、验收回归。
- **长 run 的 token / 算力预算护栏**：13 阶段串行长跑无预算/速率护栏，成本失控属未建模风险 → 概要设计纳入考量（MVP 至少留 cancel 兜底）。
- **蓝图编辑器 UX 完整性**：拖拽排序、阶段插入/删除中途态、subTaskTemplate 编辑体验、空态引导实际流程（D-V1.2-34 只验到「入口」）→ 概要/详细设计补全。

## 五、验收标准（AC）

> 务必覆盖边界审计补的边界。UI 卡走真浏览器（本项目 UI 验收必走真浏览器，见 browser-e2e）。

| 编号 | 标准 |
|---|---|
| AC-1（蓝图） | 蓝图原子写 `.pi/factory/pipelines/<id>.json`（临时文件 + rename）；保存校验拒绝 0 阶段 / 空 subTaskTemplate / order 重复或缺号；以 order 升序为唯一执行序。 |
| AC-2（冻结释槽·F16） | **每阶段跑完即销毁会话释放并发槽**：串行流水线运行中任一时刻活会话恒 ≤1（销毁经 isStreaming→abort→destroy，不碰 bySession/removeOwner）；可用会话 registry / 槽计数断言。 |
| AC-3（≥4 阶段不卡 ~10min） | **≥4 阶段一键起 run 能连续跑完、不被旧 10min idle 占槽拖住**（无须等 idle 回收即逐阶段腾槽推进）。 |
| AC-4（多 run 限流不饿死） | 多 run（或流水线 + 旧快派）并发时，受并发上限限流、在上限内自然轮转排队、**不互相饿死 / 不死锁**；不依赖单飞守卫即成立。 |
| AC-5（acquireSlot 不限超时 + 排队态） | 并发槽占满时流水线下一阶段**排队等待、不被 60s 超时误杀**（不复现违 AC「不报上限失败」）；看板显「排队中·等会话槽」可见态。 |
| AC-6（上游累积喂下游） | N≥4 流水线下游阶段首条消息含**全部已完成上游**产物摘要（非只紧邻一跳）。 |
| AC-7（孤儿 run 对账） | dev 重启后 run.status='running' 且当前阶段 sessionId 不在活 registry → GET run **读时对账判 failed**（看板不再永久「运行中」僵尸、轮询停）。 |
| AC-8（失败态可见） | 故意造一条会失败的流水线：看板该 run 显失败徽章（红）+ 左红条 + run 级「已失败·停在第N阶段」头，能一眼看出哪步炸了。 |
| AC-9（run cancel） | 运行中 run 点「停止」/ `POST .../cancel` 能中止编排（AbortSignal 生效）、不再被逼重启 dev；停止后会话被释放。 |
| AC-10（看板视觉） | 阶段看板纵向列表 + Kimi 风格卡片（亮/暗双主题）；全局进度条格数随 stages 动态；绿点阵 LED 3×12 按列点亮、`filledCols=round(progress*12)`；dicebear notionists 头像本地离线生成（断网可渲染）。MVP 阶段 progress 由离散 status 三档映射（done=满列 / running≈半列 / pending=空列），非连续进度；与详细设计 §3.6 一致。 |
| AC-11（进会话只读守卫） | 左键弹悬浮二级菜单只读看冻结对话 + 产物；已完成/已销毁/历史 run 会话页纯只读（无 send 输入、不复活会话绕过 acquireSlot）；底部切换条只枚举有 sessionId 阶段、pending 置灰；jsonl 缺失降级「记录已不存在」、产物链接落 artifactId。 |
| AC-12（进入完整对话续聊） | 二级菜单「进入完整对话」跳主对话窗 ChatWindow，能解冻 re-attach 续聊（占 1 槽、受并发上限）；续聊前 spike 已验 dispatch 会话 re-attach 可行。 |
| AC-13（合并入口） | 左栏**一个「流水线」入口** + 模态（无新路由）；模态内两 tab（「流水线」/「快速派发」），旧快派表单原样并入「快速派发」tab；F6 顺序号①②③ + 串行提示、F15 重跑/新建空白两按钮、F8 未选 agent 子任务占位提示均在「快速派发」tab 内生效。 |
| AC-14（并发上限可配） | `concurrency-gate` 读配置（默认 3、有合理上限）；UI 有资源警示；根 `CLAUDE.md`「并发会话 ≤3」红线条目同步更新。 |
| AC-15（红线/回归） | `npm run lint` 净 / `npx tsc --noEmit` 零错误 / `npm test` 全绿；真浏览器 console pageErrors=0（客户端 bundle 不被 node:fs 污染）；headless worker 不放 propose_edit（无永久悬而未决 PendingChange）；旧 dispatch 与既有受管文档/会话分组功能零回归。 |
