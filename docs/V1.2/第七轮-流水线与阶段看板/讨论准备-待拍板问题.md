## 1. 一句话目标

把多 Agent 派发从「一次最多挂 2–3 个子任务、一锤子串行跑一轮」升级为「**一条可保存、自动按序推进、跑完即释放并发槽、全局可视的流水线**」——即把「多阶段标准软件工程流水线 + 阶段看板」做成 Next-Step 的**一等公民**。这是第一轮「多 Agent 管理」当年明确 deferred 的机制层。

## 2. MVP 范围（方案建议，待用户确认）

只做三件 + 旧表单小修：
1. **流水线蓝图**：一次定义 N 阶段（角色 + 顺序 + 子任务模板），纯文件落 `.pi/factory/` → 根治 **F9**。
2. **自动编排器**：按序串行 + 上游产物自动喂下游 + **每阶段跑完主动销毁 worker 会话释放并发槽** → 根治 **F16**。
3. **阶段看板**：一个视图看全部阶段进度 + 当前第几步 → 根治 **F14**。
4. 旧 Dispatch 表单就地修补：执行顺序号（F6）、「新建空白派发」按钮（F15）、子任务提示（F8）。

**砍掉**（理由：sf-mini 这些机器默认全关、实测 F1~F16 无一指向它们）：阶段级 HITL gate（且撞 propose_edit 红线）/ 自治度 4 档 / 协作模式 3 种 / RTM 覆盖率门 / DAG 画布 / 真 SSE / paused 状态机 / 蓝图版本化。

## 3. 承重技术事实（讨论时勿推翻 —— 已 lead file:line 亲核 + 对抗评审纠偏）

- **F16 真药方**：当前 `acquireSlot` 本就是「排队等待 + 60s 超时」（`concurrency-gate.ts:43`），**不是撞墙报错**。真根因 = **completed 的 worker 会话不被销毁、占并发槽到 10 分钟 idle 才回收**（`dispatch-runner.ts:161` 只在非 completed 才释放；`rpc-manager.ts:52` idle=10min）。修法 = **编排器每阶段跑完主动销毁会话**（复用 `lib/pi/evict-agent-sessions.ts` 的 isStreaming→abort→destroy 配方，**严守「只 destroy、不碰 bySession/removeOwner」红线**，否则 re-attach 的 getOwner 返 null、反塞 write/edit/bash＝第五轮修过的 bug），**不动 acquireSlot**。
- **「两个 3」**：子任务数 ≤3（`dispatch-store.ts:73`，这才是「13 阶段塞不进一次派发」的真约束、F9 要突破的）≠ 并发会话 ≤3（`concurrency-gate.ts:16`，红线不破）。
- **红线（全程不破）**：`MAX_CONCURRENT_SESSIONS=3` 不改（只是腾槽更快）/ headless worker 不放 propose_edit（沿用 `DISPATCH_DOC_SESSION_TOOLS`）/ 不改 pi 内核 / 纯文件无数据库。
- **上游喂下游真有效、原样搬**：`orchestrator.ts:125-128`（拼上游产物进下游首条 message）+ `:196-197`（取 createdContent 优先）。实测有效（开发实现方案逐条引用了上游需求/技术方案）。

## 4. 待拍板问题清单（讨论议程）

> 每条方案里都有推荐默认，讨论时确认 / 调整即可。

1. **MVP 边界**：同意砍掉第 2 节那些重型部分吗（尤其**阶段级 HITL gate** —— 撞 propose_edit 红线 + 无实测需求，方案建议彻底不做）？还是有想保留 / 提前做的？
2. **阶段怎么定义**：MVP 让用户**手填 N 阶段**（选 agent + 填子任务模板 + 排序）。是否要顺带做「一键导入 sf-mini 13 阶段 / 标准角色模板」（方案放第二期、附带缓解 C5/F4）？
3. **新流水线 vs 旧 Dispatch**：新增独立「流水线」入口，旧「快速 2–3 派发」表单**保留并修补**（F6/F8/F15）—— 同意并存？还是要合并成一个入口？
4. **阶段看板形态**：MVP 最简**纵向列表**（每阶段一卡：序号 + 阶段名 + 绑定 agent + 状态徽章 + 产物链接）vs 横向时间线 / 泳道（方案建议 MVP 纵向、横向留第三期）。
5. **蓝图存储**：普通原子写覆盖（方案建议，仿 dispatch-store）vs 纳管为受管文档可版本/回滚/Diff（方案判为对配置文件的过度设计）。
6. **人确认时机**：MVP 全阶段 auto 产受管文档；用户**事后**可在 ArtifactPanel 用既有 D3–D5 按块确认能力手动改任一阶段产物（这条路径本就存在、不经流水线、不破红线）—— 可接受这种「事后确认」而非「阶段中途 gate」吗？
