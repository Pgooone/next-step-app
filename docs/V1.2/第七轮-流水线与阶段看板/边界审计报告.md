# 第七轮·流水线与阶段看板 — 边界/遗漏审计报告

> 2026-06-27，commit 前用 ultracode 做的对抗式边界审计（57 agent / 8 维度找 → 逐条对抗核实 → 分级汇总）。
> 目的：在写概要设计 / 开工前，补齐用户与设计可能没考虑到的边界情况。

**总体结论**：设计骨架稳、无破红线 / 会崩溃的 blocker，但有 **15 个 already_handled=false 的真缺口**需在「写概要设计 / 开 T1 前」补，其中 **7 条改动已定决策或重开取舍，须用户拍板**；其余 8 条属 lead 实现级取舍（记 ADR 即可）。无一条阻断 F16 核心机制本身。

## 🟠 中危真缺口

**会话生命周期 × 进会话（F16 内部矛盾，一组同源）**
1. **进已销毁 worker 会话追问会 re-attach 复活、绕过 acquireSlot** — F16 腾的槽被 D-V1.2-32「进会话」一键吃回。涉 D-V1.2-32/33 + ≤3 红线。→ **推荐方案 A**：已完成/已销毁阶段的会话页纯只读（不挂 ChatInput 的 send）。
2. **底部切换条会切到 pending 阶段（无 sessionId）→ 404** — 涉 D-V1.2-32 + run 数据模型。→ 切换条只枚举 sessionId 非空（running/done/failed/aborted）的阶段，pending 置灰。
3. **历史 run 进会话几乎必是死会话 / jsonl 被清则 404 死链** — 涉 D-V1.2-35/32 + F2。→ 历史 run 纯只读；jsonl 缺失降级为「记录已不存在」；产物链接落 artifactId 而非 sessionId。

**并发 ≤3**
4. **无 run 级互斥** — 两条流水线 / 流水线+旧 dispatch 并发，多 worker 叠加聊天瞬时挤满 3 槽互相饿死。涉 D-V1.2-35/39 + ≤3 红线。→ **MVP 单飞守卫**（起 run 前查 status=running 的 run，有则 422 + 按钮置灰）。
5. **acquireSlot 默认 60s 超时误伤流水线** — 用户开满 3 聊天时下一阶段排队 60s 即 failTask 整条中止，**直接违反方案自身 AC5「不报上限失败」**。→ pipeline 调 acquireSlot 显式传更大/不限 timeout（只动调用点）+ 看板加「排队中」可见态。
6. **TOCTOU**：acquireSlot 放行到 worker 注册之间多 run 并发可瞬时破 3 槽。→ 采纳 #4 单飞守卫则自动退回可接受原状。
7. **取消/中止 running 流水线无通道** — 跑错的长流水线停不下来、持续占槽烧 token，被逼回「重启 dev」（正是本轮要消灭的）。涉 D-V1.2-33（只读，当初否决了「只读+取消run」）。→ 接最小 cancel：runPipeline 带 signal + `POST .../cancel` 翻 AbortController。

**编排器与失败处理**
8. **上游只喂「紧邻上一阶段」产物** — N≥4 流水线下游拿不到更早的资料/需求，F9「自动喂下游」核心卖点悄悄退化成只链一跳。→ run.stages 累积所有已完成上游摘要，或 stage 声明 dependsOn；至少写明「MVP 只链一跳」降级。
9. **产文本但没产受管文档（F10）判 done 不失败** — 看板显示✓但产物链接空、下游只拿裸文本。→ 拼 firstMessage 自动前置「请调用 create_artifact」（最划算）；可选「done 但 artifactId 空」标 warn。
10. **run.status='running' 在 dev 重启后永不回收** — 看板永久「运行中」僵尸 run、轮询不停。→ GET run 时若 running 且 currentStage.sessionId 不在活 registry → 判 failed（零新状态机）。

**蓝图与 agent 引用 / 数据模型 / 看板**
11. **蓝图引用的 agent 被删后整条 run 静默坏死** — 调用时才中途 failTask、可复用蓝图悄悄腐烂。→ 起 run 前预检全 stage agentId 存在；看板/编辑器复用 DispatchPanel 的 `?? agentId` 降级渲染。
12. **蓝图保存缺校验**（0 阶段 / 空 subTaskTemplate / order 重复或缺号）— order 数组序 vs 字段值歧义会复活 F6「展示序≠执行序」。→ pipeline-store.create/write 补与 dispatch 同级校验 + 单测，钉死「order 升序=唯一权威执行序」。
13. **失败态(failed)在看板视觉里完全没设计** — failed 行兜底渲成 done/灰态，用户看不出哪步炸了。→ 补失败徽章（复用 STATUS_META 红）+ 左红条 + run 级「已失败·停在第N阶段」头 + 验收造一条故意失败的流水线。
14. **改上游产物后改动永远无法回灌下游** — 人工按块确认对流水线产物完全无效，人工把关沦为摆设；MVP 唯一重跑=整 run 从头且仍读重新产出的上游、不读手改版。涉 D-V1.2-40/33 + §2.6 第三期。→ 至少 UI/文档显式写明「改上游不回灌下游」；可选「从此阶段重跑下游」逃生口。
15. **runs/*.json 无清理 + run 列表枚举/对账缺位**：①只增不删、D-V1.2-35「回看」使删除路径焊死，run 下拉爆条；②无持久「活动/最新 runId」，刷新后看板默认指向不明 + 重复点「运行」起并行 run。→ MVP 定保留策略 + run 下拉只列最近 N 条；增持久 latestRunId 或用 #4 单飞守卫。

## 🟡 低危（lead 自主记 ADR，不必拍板）
- **re-attach 复活 doc worker 会装回 propose_edit**（人已在场、合法，仅需文档言明，勿改 reattachProfileSession）。
- **全局进度条硬编码 13 格** → gbar 格数=run.stages.length，>20 降级为「④/N + 比例条」。
- **agent 改 mode 后蓝图阶段工具语义被静默改写** → 看板/编辑器 stage 卡标 mode 徽章 + 写明「coding 阶段直接读写盘」。
- **底部切换条 13+ 阶段横滚无提示/不自动居中** → scrollIntoView 当前项 + 渐隐遮罩，真浏览器验收带一条 ≥13 阶段。
- **阶段名超长无截断** → `.rname` 补 ellipsis+title（同 DispatchPanel）。
- **只读看板 affordance 误导**（cursor:pointer/chevron 让人以为能控 run）→ 加「进度只读」轻提示或弱化点击视觉。
- **pipeline-store 照搬 dispatch「parse 失败即 throw」** → listRuns 走容错跳过坏文件（同 listArtifacts），单 run 详情再 throw + 单测。
- **下游不记 consumedVersion 指纹** → 仅在数据模型加一句「不追踪新鲜度」注释留痕，不预埋死字段。

## 补漏批判（本次审计可能没覆盖到的）
- **审计高度集中在「会话生命周期 / 并发 / 失败 / 存储」四个机制维度**，对**蓝图编辑器本身的 UX 完整性**（拖拽排序、阶段插入/删除中途态、subTaskTemplate 编辑体验、空态引导的实际流程）几乎未触——D-V1.2-34 只验到「入口」。
- **跨轮交互未审**：流水线产的受管文档与第二~三轮「版本 diff / TOC diff / 性能 memo 化」叠加时，一条 run 批量产 11+ 文档是否拖垮 file panel / ArtifactPanel 渲染性能，无人验证（第三轮 perf 是单文档基线）。
- **权限/允许根边界未审**：run 产物落 `.pi/factory/` 与既有「文件 API 只服务允许根（会话 cwd + `~/pi-cwd-\d{8}`）」是否冲突，蓝图/run 文件路径是否在允许根内未核。
- **并发模型只审了「≤3 槽争抢」，未审 token/算力配额**：13 阶段 ~45min 串行烧 DeepSeek，无任何预算/速率护栏，长 run 成本失控属未建模风险。
- **全部结论基于「待开工纸面」**：HEAD 无一行 pipeline 代码，多条 already_handled=true 依赖「实现者忠实复用既有 pattern」——一旦实现期偏离，低危项会回升为真 bug，**须在双层验收 + 真浏览器环节回归这些假设**。
