1. **第一轮「多 Agent 管理」只做了视觉**：经 git 实证只动 3 个纯前端文件（agent 卡片玻璃→Swiss），需求文档白纸黑字「先视觉后机制」「不改 agent 工作机制」。「流水线 / 阶段看板 / 编排」被列为「增量二+，本轮明确不做」。

## MVP 范围与核心修复（详见方案文档）

只做三件 + 旧表单小修，**砍掉**调研蓝图的全部重型部分（13 阶段全家桶 / RTM 门 / 自治度 4 档 / 协作模式 / DAG 画布 / 真 SSE / **阶段级人确认 gate**）：

1. **流水线蓝图**（一次定义 N 阶段，纯文件落 `.pi/factory/`）→ 根治 **F9**。
2. **自动编排器**（按序串行 + 上游产物自动喂下游 + **每阶段跑完主动销毁 worker 会话释放并发槽**）→ 根治 **F16**。
3. **阶段看板**（一个视图看全部阶段进度 + 当前第几步）→ 根治 **F14**。
4. 旧 Dispatch 表单就地修补：执行顺序号（F6）、「新建空白派发」按钮（F15）、子任务提示（F8）。

> **F16 真药方（对抗评审纠偏 + lead 亲核）**：当前 `acquireSlot` 本就是「排队等待 + 60s 超时」（`concurrency-gate.ts:43`），**不是撞墙报错**；真根因是 **completed 的 worker 会话不被销毁、占槽到 10 分钟 idle 才回收**（`dispatch-runner.ts:161` 只在非 completed 才释放、`rpc-manager.ts:52` idle=10min）。修法 = 编排器每阶段跑完主动销毁会话（复用 `evict-agent-sessions.ts` 安全配方），**不动 `acquireSlot`**。

## 红线（全程不破）
