# V1.2 第八轮 · 并发计槽改 in-flight 计量（根治跨 run evict 误杀）—— 需求文档

> 状态：**spec 草拟、待用户复审批准后开工**。承重轮（动并发计槽口径）。
> 由来：第七轮 T6 揪出「跨 run evict 误杀」pre-existing 缺陷（D-R7-07 决策2，留二期）；收官后资源实测（ADR D-R7-09）推翻「会话很重」前提 → 误杀修法去风险调查（ultracode 7-agent，2026-06-28）→ 用户拍板走**方案 B（改计槽语义）**而非 evict-by-sessionId。

## 一、背景

### 1.1 要修的 bug
流水线编排器每阶段跑完调 `evictAgentSessions(projectRoot, stage.agentId)`（`pipeline-orchestrator.ts:187`）腾并发槽（第七轮 F16）；evict **按 agentId 逐出该 agent 名下全部活会话**（`evict-agent-sessions.ts:41` 的 `sessionsForAgent`），非只逐本阶段 sessionId。T6「进入完整对话」能让被 evict 的 worker 会话复活续聊。于是：**用户复活了 agent X 的会话、又另起一条也用 X 的 run，新 run 跑完 X 阶段时把用户正聊的会话连根 destroy**（误杀）。单 run 测不出、T6 首次暴露。

### 1.2 资源实测推翻的旧前提（ADR D-R7-09）
T5 给 `concurrency-gate` 设并发上限的初衷基于「每会话 = 一个独立进程 + 工具子进程常驻 → OOM」——**已证伪**。实测：一个活会话 = 同一 Next 进程内两个 JS 对象（`AgentSessionWrapper` 包内核 `AgentSession`）、**0 独立进程/0 子进程/idle 时 0 常驻连接、~1MB 闲置堆**；并发槽 = `globalThis.__piSessions.size` 是**纯逻辑计数**。
→ 「完成态 worker 占槽」是**逻辑计数问题、非资源问题**；F16 每阶段 evict 的唯一动机就是腾这个逻辑槽。

### 1.3 用户拍板（D-V1.2-50，见 QA）
误杀去风险调查给两条修法，用户选 **方案 B**：
- **方案 B（本轮）**：让**完成态/idle 会话不计入并发数**（只数「正在跑回合」的）→ 编排器**不必每阶段 evict 腾槽** → 误杀因果链从源头断（不再有任何 evict-by-agentId 被触发）。顺带根治 legacy dispatch 占槽 + 未来装 spawn 类 pi 扩展也天然覆盖。
- （备选 evict-by-sessionId = 只把每阶段 evict 收窄为按本阶段 sessionId 逐出；更外科手术、改动更小，但不改全局语义、不顺带修 legacy/未来。**用户未选**。）

## 二、目标

把并发「占槽」的判据从「**registry 在册会话数**」改为「**同时在跑回合数（in-flight）**」，从而：① 根除跨 run evict 误杀；② 顺带根治 legacy dispatch 完成态占槽；③ 计槽口径与 ADR D-R7-09「真约束是 CPU 吞吐非内存」对齐。

## 三、需求条目与验收标准（AC）

| # | 需求 | 验收标准（可断言） |
|---|---|---|
| **AC-1** | 跨 run evict 误杀根除 | 用户「进入完整对话」复活 agent X 会话 → 另起一条也用 X 的 run 跑完 X 阶段 → 用户那条会话**仍活、可继续聊**（不被 destroy）。 |
| **AC-2** | F16 不复发 | ≥4 阶段（目标覆盖 sf-mini 13 阶段量级）串行流水线**能完整跑完**——完成态会话不计 in-flight 槽、不堵后续阶段；无须每阶段 evict。 |
| **AC-3** | 计槽改 in-flight | `acquireSlot` 只数 `isAlive() && inner.isStreaming` 的会话；完成态/idle 会话不计槽。「并发≤默认 3」语义 = 「**同时在跑回合 ≤3**」（语义放宽已用户拍板 D-V1.2-50）。 |
| **AC-4** | T6 cancel / re-attach 零回归 | cancel（abort→failRun）+「进入完整对话」re-attach（装回 doc 工具）行为不变。 |
| **AC-5** | 读时对账不破 | `reconcileOrphan`（`new Set(__piSessions.keys())`）不被破坏——**改的是 size 计量口径，不是 Map 成员**；完成态会话**绝不主动从 Map 删除**（靠 10min idle 自然回收），`keys()` 集合不变。 |
| **AC-6（承重）** | in-flight 绝不漏算 | 真正在跑的会话**永不被漏算**（否则 timeoutMs:Infinity 排队超额放行 N+1 并发、违 ≤3 红线）。T1 spike 坐实：worker 单回合全程 `isStreaming===true`（含 LLM 流 + 工具 + 多轮）+ agent_end 半拍仍 true（保守不漏放）+ N 阶段串行正确回落。 |
| **AC-7** | legacy 占槽顺带根治 | legacy dispatch（`orchestrator.ts`）完成态 worker 不再占 in-flight 槽（与 pipeline 同享新计量口径，无需单独改）。 |

## 四、红线（不可破）

- **不改默认上限来源**：`readMaxConcurrent` / `DEFAULT_MAX=3` / `HARD_CAP=100` 不动（本轮只改「什么算占槽」，不改「上限是多少」）。
- **不碰 owner-map**：`bySession`/`removeOwner` 绝不动（第五轮红线；方案 B 根本不调 evict，天然满足）。
- **completed 不主动从 Map 删**：只过滤 size，不 `registry.delete`（否则破 `reconcileOrphan` 的 `keys()` + re-attach `withStartLock` 快路径）。
- 不改 pi 内核、纯文件无 DB（`lib/pi` 只封装）。

## 五、非目标（本轮不做）

- 不做 evict-by-sessionId（备选方案、用户未选）。
- 不引入「按 sessionId 精确 evict」的内核改动。
- 不改 HARD_CAP 值（已于顺手小改 D-V1.2-49 放宽到 100）。
- 不做并发的 GUI 设置器（D-V1.2-48 已定配置文件 MVP）。

## 六、承重墙（开工第一步必先验）

**计量改造后真正 in-flight 的会话绝不被漏算**（致命路径 = `pipeline-orchestrator.ts:127` timeoutMs:Infinity 排队，漏算→超额放行且无超时兜底）。命门事实：`inner.isStreaming` 是否在 worker 单回合全生命周期恒 true、agent_end 半拍是否「保守偏不漏放」。**T1 活进程打点 spike 坐实后才开 T2 实现**（见详细设计 §承重墙）。
