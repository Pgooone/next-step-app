# V1.2 第八轮 · evict 收窄为 by-sessionId（根治跨 run evict 误杀）—— 需求文档

> 状态：**spec 重写完成（方案 B 经设计评审门 NO-GO、改选 evict-by-sessionId）、待用户复审批准后开工**。承重轮。
> 选型经过：用户初拍方案 B（计槽 size→in-flight）→ ultracode 设计评审门判 **NO-GO**（承重命门 isStreaming 在 retry 退避窗口漏算、违 ≤3 红线，详见 [`方案B设计评审-NO-GO记录.md`](方案B设计评审-NO-GO记录.md)）→ 用户改选 **evict-by-sessionId**（D-V1.2-50 轮次2）。

## 一、背景

### 1.1 要修的 bug（与方案 B 同）
流水线编排器每阶段跑完调 `evictAgentSessions(projectRoot, stage.agentId)`（`pipeline-orchestrator.ts:187`）腾并发槽（第七轮 F16）；evict **按 agentId 逐出该 agent 名下全部活会话**（`evict-agent-sessions.ts:41` 的 `sessionsForAgent`），非只逐本阶段 sessionId。T6「进入完整对话」能让被 evict 的 worker 会话复活续聊。于是：**用户复活了 agent X 的会话、又另起一条也用 X 的 run，新 run 跑完 X 阶段时把用户正聊的会话连根 destroy**（误杀）。

### 1.2 为何走 evict-by-sessionId（非方案 B）
误杀去风险调查 + 资源实测（ADR D-R7-09）后用户初选方案 B（改计槽语义为「在跑回合≤3」、删每阶段 evict）。但**设计评审门证伪了方案 B 的承重命门**：用 `inner.isStreaming` 作 in-flight 判据，在 worker 命中可重试错误（429/overloaded/timeout，DeepSeek 高发）→ `_prepareRetry` 指数退避 sleep 期间 `isStreaming===false`，而 worker 真在 in-flight（dispatch-runner 因 `willRetry=true` 未 finish）→ 漏算 → `acquireSlot(Infinity)` 超额放行第 4 个 worker → 违 ≤3 红线（详见评审记录）。
**evict-by-sessionId 完全规避此问题**：保留 F16 每阶段 evict 释槽、不动计量口径、不依赖任何内核时序信号，只把「按 agentId 一锅端」收窄为「只逐本阶段那个 sessionId」。

### 1.3 承重前提（lead 已亲核坐实）
「一 worker = 一 sessionId = 一槽」——worker 工具集**无任何 spawn 类工具**（`DISPATCH_DOC_SESSION_TOOLS` 6 项=[read,grep,find,ls,create_artifact,list_artifacts]、coding 7 项无 subagent/task）+ `__piSessions` 写入仅 `rpc-manager.ts:290-291` 两点 → 一个 worker 只开一个会话、占一个槽。故按本阶段 sessionId 逐出**绝不漏槽**（不像 by-agentId 要担心 inner 子会话）、且只 destroy 那一个、不碰用户复活的同 agent 其他 sid。

## 二、目标

把流水线编排器每阶段的 evict 从「按 `stage.agentId` 逐出该 agent 全部会话」**收窄为「只逐本阶段 `stage.sessionId` 这一个会话」**——根除跨 run evict 误杀，同时保留 F16 释槽机制、不动并发计量语义。

## 三、需求条目与验收标准（AC）

| # | 需求 | 验收标准（可断言） |
|---|---|---|
| **AC-1** | 跨 run evict 误杀根除 | 用户「进入完整对话」复活 agent X 会话 → 另起一条也用 X 的 run 跑完 X 阶段 → 用户那条会话**仍活、可继续聊**（因它是另一个 sessionId、不被本阶段按 sid 的 evict 命中）。 |
| **AC-2** | F16 不复发、释槽不变 | ≥4 阶段串行流水线能完整跑完——每阶段仍 evict 释槽（只是按 sessionId）、活 worker 会话恒 ≤1、不卡 10min idle。 |
| **AC-3** | evict 收窄为 by-sessionId | 每阶段 evict 只 destroy `stage.sessionId` 这一个会话；**绝不**逐出同 agentId 名下的其他会话（尤其用户复活的）。 |
| **AC-4** | T6 cancel / re-attach 零回归 | cancel（abort→failRun）+「进入完整对话」re-attach（装回 doc 工具）行为不变。 |
| **AC-5** | 计槽语义不变 | 「并发≤默认 3 = 在册会话≤3」语义保持；**不动** `concurrency-gate` 计量口径、不动 HARD_CAP/默认上限。 |
| **AC-6（承重）** | by-sessionId 逐出正确 | hermetic spike 坐实：`evictSession(sidA)` 只 destroy sidA、同 agentX 的 sidB **仍活**；流式时先 abort 再 destroy（防 jsonl 双写）；evict 后 `getOwner(cwd,sidA)` 仍返 agentX（**owner-map 零碰**、第五轮红线守住）。 |

## 四、红线（不可破）

- **不碰 owner-map**：`bySession`/`removeOwner` 绝不动（第五轮红线）——by-sessionId 逐出仍只 destroy registry wrapper、复用既有 abort-then-destroy 配方。
- **不改计量/上限**：`concurrency-gate` 的 size 计量、`readMaxConcurrent`/`DEFAULT_MAX`/`HARD_CAP` 零改动。
- 不改 pi 内核、纯文件无 DB。

## 五、非目标（本轮不做）

- 不改计槽语义（不走方案 B 的「在跑回合≤3」；评审 NO-GO，留 [`方案B设计评审-NO-GO记录.md`](方案B设计评审-NO-GO记录.md) §七「方案 B 修正版」作回溯备选）。
- 不顺带根治 legacy dispatch 完成态占槽（那是方案 B 才顺带的；本轮聚焦误杀根除）。
- 不动 HARD_CAP（已于 D-V1.2-49 顺手小改放宽到 100）。
- **不动** `app/api/projects/[id]/agents/[agentId]/route.ts:63` 的第二条 evict-by-agentId（第六轮改 mode 用、gated on mode 变化、不在 run 完成误杀链上）。

## 六、承重墙（开工第一步必先验）

**by-sessionId 逐出粒度正确**——纯应用层 hermetic spike（无需活进程，比方案 B 的全局计量时序命门轻得多）：`evictSession(sidA)` 只 destroy sidA、不误删同 agent 的 sidB、流式先 abort、owner-map 零碰 + 负对照 + 变异检查（退化回 by-agentId 则 sidB 被误删、断言必红）。详见详细设计 §承重墙。
