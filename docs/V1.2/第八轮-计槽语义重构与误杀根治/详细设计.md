# V1.2 第八轮 · 详细设计（evict-by-sessionId）

> 配套：[需求文档](需求文档.md) · [概要设计](概要设计.md) · [方案B评审NO-GO记录](方案B设计评审-NO-GO记录.md)。承重轮。
> 全部 file:line 经 ultracode 调查 + 设计评审门核实 + lead 亲核「worker 工具集无 spawn」「__piSessions 写入仅 rpc-manager.ts:290-291 两点」。承重命门轻（纯应用层、零内核时序依赖）。

## 〇、设计要点

保留 F16 每阶段 evict 释槽机制，**只收窄逐出粒度**：`evictAgentSessions(agentId)` → `evictSession(sessionId)`。承重命门 = 「by-sessionId 逐出粒度正确」，T1 hermetic spike 坐实后开 T2。**不动 concurrency-gate / 内核 / owner-map / 上限**。

## 一、逐文件设计

### 1.1 `lib/pi/evict-agent-sessions.ts`【改·新增 evictSession】

**现状（已 Read）**：`:33-49` `evictAgentSessions(projectRoot, agentId, deps)`：`:38` `lookup = deps?.sessionsForAgent ?? sessionsForAgent` → `:41 for (const sid of lookup(projectRoot, agentId))` 逐个 `:44-46`「若 `inner.isStreaming` 先 `send({type:'abort'})` → `destroy()`」。`:11-13` 红线注释「绝不碰 bySession/removeOwner」。

**改动**：新增 `evictSession(projectRoot, sessionId, deps?)`：
- 复用 `:44-46` 单会话配方：`const w = getSession(sessionId); if (!w || !w.isAlive()) return; if (w.inner.isStreaming) await w.send({type:'abort'}); w.destroy();`（getSession 经 DI 缝、生产取 `globalThis.__piSessions.get`）。
- **跳过 `sessionsForAgent` 反查**（直接对单个 sid 操作）——这正是「不误删同 agent 其他会话」的关键。
- **绝不碰** `bySession`/`removeOwner`（:11-13 红线）；`evictAgentSessions` 保留不删（别处可能引用，grep 确认）。

### 1.2 `lib/domain/pipeline-orchestrator.ts`【改·逐出收窄】

**现状（已 Read）**：`:82` `doEvict = deps.evictAgentSessions ?? evictAgentSessions`（⚠️ 评审订正：spec 旧写 :80，:80 实为 doAcquireSlot）。正常路径 `:181-182` `stage.sessionId = result.sessionId; doSetOwner(...)` → `:187` `await doEvict(projectRoot, stage.agentId)`；catch `:167` 兜底 evict。

**改动**：
- `:187`：`doEvict(projectRoot, stage.agentId)` → 按 `stage.sessionId` 逐出（此时 `:181` 已赋值 `result.sessionId`）。最小改法 = 把注入的 `doEvict` 换成 `evictSession` 签名、传 `stage.sessionId`（`deps.evictSession ?? evictSession`、`deps.evictAgentSessions` 注入口相应改名）。
- catch `:167`：⚠️ **评审订正（CP-1）**——`stage.sessionId` 在 `:181` 赋值、而 `:181` 位于 try/catch 块（`:165-178`）**之后**，故 worker 抛错触发 catch 时 `stage.sessionId` **恒为初值 null**（旧 spec 误写「在 catch 前 :181 赋值」是把控制流写反了）。改 `evictSession(projectRoot, stage.sessionId=null)` → `getSession(null)` 返 undefined → **no-op**。**生产安全且不漏槽**：`runWorker` 在 `registerInnerSession`（dispatch-runner.ts:151，唯一占槽点）之后到 return 之间**无任何 throw 路径**（`send` 'prompt' fire-and-forget 带 `.catch`、`await ended` 的 `waitForTurnEnd` 无 reject 形参、其余纯函数/try-catch），故 worker 若抛错必在**占槽之前**抛、catch 时根本没占过槽（premiseA + lead 复核坐实）；即便未来引入占槽后抛错的语句，该会话也靠 wrapper 10min idle 自销毁 + AC-7 reconcileOrphan 兜底。**承重 test⑥ 必须同步改断言**（见 §1.4 c）。
- **注释订正清单（评审 CP-3 补全，勿只改 :9-12）**：① 头注 `:9-12`「按 agentId evict」→「**按本阶段 sessionId evict**（防跨 run 误杀同 agent 用户接管会话）」；② `:147-149` catch 注释 + `:169` 内联——现写「catch 内 best-effort doEvict 兜底释槽…对已建会话是唯一释槽手段」，收窄后于 catch 语境永不为真（catch 时 sid 恒 null），改为如实陈述：「catch 传 `stage.sessionId` 此时尚为 null（`:181` 在 try 后）→ 此处 evictSession 对正崩 worker 会话是 no-op；该 worker 槽靠 10min idle + AC-7 reconcileOrphan 回收」。
- **保留** `setOwner`(:182)、`acquireSlot`(:127) 顺序、cancel signal 链(:102)。

### 1.4 `lib/domain/pipeline-orchestrator.test.ts`【改·承重断言 + catch 路径改造 + 加固误杀覆盖】

**承重 verify 主结构保留**（每阶段 evict、size→0、peak≤1、acq→set→evict 顺序、负对照）。但**评审 CP-1/CP-2/CP-5 揪出：这不是「只改第二参」**，有四处实质改造，勿照「结构不变」字面照搬：

**(a) evictSpy 内层真函数换 + 删反查缝（CP-5）**：承重 harness 的 `evictSpy`（`:173-179` / `:681-687`）内层现调真实 `evictAgentSessions(root, agentId, {sessionsForAgent, getSession})`；改为调 `evictSession(root, sessionId, {getSession})`——**删 `sessionsForAgent` DI**（evictSession 不反查 owner-map）。注入口 `deps.evictAgentSessions` → `deps.evictSession`。

**(b) 正常路径 evict 断言精确行（FL-2，按 describe 块枚举）**：真正把第二参 `agentId`→`sessionId` 的断言在——**承重块 `:212-215`**（四阶段 `toHaveBeenNthCalledWith`）+ **T6 cancel 块 `:738`（`NthCalledWith`）+ `:754`（`evictedAgents).toEqual(agentIds)` → 改按 sid）**。⚠️ spec 旧引的 `:253/:737/:753` 是 `toHaveBeenCalledTimes` 计数行、相差 1 行，勿漏改下一行的真断言。**失败分支 describe `:386` 起只有 `:424/:447` 的 `expect(evictCalls).toBe(1)` count 断言（注入泛型 `vi.fn(async()=>[])`、不绑 agentId）→ 改 sid 后仍成立、无需改第二参**。

**(c) catch 路径 test⑥ 必改（CP-1，命门）**：test⑥（`failFirstStage`，`:243-260`）faux 先 `fauxMap.set('s0')` 再 throw、断言 `:256 expect(fauxMap.size).toBe(0)` 依赖旧 by-agentId 经 `ownerByAgent` 反查删 Map。改 `evictSession(stage.sessionId)` 后——catch 时 `stage.sessionId` 恒 null（`:181` 在 try/catch 之后）→ `evictSession(null)` no-op → s0 留 Map → **`:256` 必 FAIL、`:254` 第二参也不再是 's0'**。**改造**：承认 catch 路径 size 不再回落——`:256` 改为断言 **no-op / size 不回落**（catch 漏的只是「正崩 worker 自己那一槽」、有 10min idle + AC-7 reconcile 兜底、可接受），`:254` 改为断言 `evictSession` 以 `null`（或不被有效调用）被调，保留 `failedReason`（worker 执行失败）+ 后续阶段 pending 断言。

**(d) 加固误杀根除的承重覆盖（CP-2，用户拍板①）**：承重 harness 现每 agent 只挂 1 会话（`ownerByAgent[agentId]=[sid]`，`:141/:646`）→ by-agentId↔by-sessionId 删同一条目、**变异检查（退化回 by-agentId）在承重块抓不住**（旧 spec「承重块变异必红」是过度声称）。故在某阶段的 agent **预置第二条「用户接管」会话 sidB**（写进 `fauxMap` + 同 agentId 名下，模拟跨 run 复活），加断言「**N 阶段 evict 跑完后 sidB 仍在 fauxMap**」。这样承重块才真覆盖 AC-1：变异退化回 by-agentId → sidB 被误删 → 断言红（证非 vacuous）。与 `T2.md` AC「承重测试断言同 agent 另一 sid 用户接管会话仍在 Map」一致（消除旧 spec §1.4「结构不变」与 T2.md 的自相矛盾）。

> 注：纯粹的「逐出粒度变异检查」（退化回 by-agentId → sidB 被误删 → 红）在 T1 spike（§二，sidA/sidB 同 agentX）已先行坐实；本 (d) 是把同一覆盖接进 orchestrator 集成承重块，双保险。

## 二、承重墙（T1 spike，lead 复跑；未绿不开 T2）

**命门**：`evictSession` 按 sessionId 逐出**只 destroy 目标 sid、不误删同 agent 其他活会话、流式先 abort 防 jsonl 双写、不碰 owner-map**。

**这是「单会话逐出粒度正确性」命门（纯应用层、复用现成 abort-then-destroy 配方、零内核时序依赖）**——比方案 B 的「全局计量时序正确性」（依赖内核 isStreaming 边界、retry 退避漏算）轻得多，**无需活进程打点、hermetic 单测即可**。

**spike 验法**（沿 T1/T3 faux Map「三处闭包同一对象 + 负对照 + 变异检查」纪律）：
- faux：`fauxMap`（唯一计数源）+ `ownerBySession` 反查。放 2 个会话 sidA/sidB 同属 agentX。
- **断言①（误杀根除直接证据）**：`evictSession(root, sidA)` 后 `fauxMap` 只剩 sidB（sidA.destroy 被调、真删 Map）、`sidB.isAlive()` 仍 true。
- **断言②（流式守卫）**：sidA.inner.isStreaming=true → 先 `send({type:'abort'})` 再 destroy（spy 顺序断言）。
- **断言③（owner-map 红线）**：evict 后 `getOwner(cwd, sidA)` 仍返 agentX（只 destroy registry、绝不碰 bySession/removeOwner）。
- **负对照**：`destroyDeletesMap=false`（destroy 只翻 alive 不删 Map）→ `fauxMap.size` 不回落，证「size 回落」断言真依赖 destroy 删 Map、非 vacuous。
- **变异检查**：把 `evictSession` 误写成「逐出该 agent 全部会话」（退化回 by-agentId）→ 断言① 必红（sidB 被误删），证测试能抓住误杀回归。

**go/no-go**：上述五条全绿 = **GO**（开 T2）。命门纯应用层、零内核时序依赖，预期高置信 GO。

> **实测落地（2026-06-28，commit `d090017`）= GO**：T1 spike `lib/pi/evict-session.spike.test.ts` 8/8 全绿（断言①只逐 sidA·sidB 仍活 + ②流式先 abort + ③owner-map 零碰〔候选连 ownerBySession 都不接收〕+ 负对照 + 变异检查）。**lead 四重确认**：队员自跑 8/8 + 亲读断言非 vacuous + 独立复跑 8/8 + 独立变异（抠候选 destroy → 断言①/②a/②b/负对照 4 红）。**探针已于 T3 收尾删**（命门事实迁本节 + ADR D-R8-01；永久回归保护迁 `evict-agent-sessions.test.ts` evictSession 单测 + `pipeline-orchestrator.test.ts` 承重块 sidB 接管断言）。另端到端补真 registry 集成 smoke（真 `__piSessions`+真 `registerInnerSession`+真 `evictSession`）lead 亲跑 3/3，详见 ADR D-R8-04。
