## 一、逐文件设计

### 1.1 `lib/pi/concurrency-gate.ts`【改·改动1】

**现状（已 Read 实证）**：
- `:21-23` `activeSessionCount() { return globalThis.__piSessions?.size ?? 0; }`
- `:40` `acquireSlot` 内 `const activeCount = opts?.activeCount ?? activeSessionCount;`
- `:41` `const limit = opts?.limit ?? readMaxConcurrent();`
- `:45-51` 轮询：`deadline = Date.now() + timeoutMs`（Infinity 时恒不超时）；`while (activeCount() >= limit) { 超时检查; await sleep(pollMs); }`

**改动**：
- 新增 `export function inFlightSessionCount(): number { return [...(globalThis.__piSessions?.values() ?? [])].filter((w) => w.isAlive() && w.inner.isStreaming).length; }`
  - 在 gate 内直接读 `globalThis.__piSessions`（同 `:22` 范式，**不 import rpc-manager 避循环依赖**）；`w.inner.isStreaming` 经 `AgentSessionWrapper.inner`（`AgentSessionLike`）取——确认 `AgentSessionLike` 类型暴露 `isStreaming`（rpc-manager 的 `inner` 类型，T2 实现期核）。
- `:40` 默认 `activeCount` 从 `activeSessionCount` 改为 `inFlightSessionCount`。
- **保留** `activeSessionCount`（导出不删，可能有其它引用——T2 grep 确认；若零引用可删，但保留更稳）。
- 头注 `:4-8`「闸门约束 = 在册会话总数」改写为「= 同时在跑回合数（in-flight）；完成态/idle 会话不计槽，靠 10min idle 自然回收（语义放宽 D-V1.2-50、对齐 ADR D-R7-09 真约束 CPU+LLM 非内存）」。

**红线**：不动 `limit`/`readMaxConcurrent`/`DEFAULT_MAX`/`HARD_CAP`（默认上限来源不改）；不 `registry.delete`（只过滤、不删 Map 成员）。

### 1.2 `lib/domain/pipeline-orchestrator.ts`【改·改动2】

**现状（已 Read 实证）**：
- `:80` `const doEvict = deps.evictAgentSessions ?? evictAgentSessions;`（:29 import、:46 注入口）
- 正常路径 step h `:181-182` `stage.sessionId = result.sessionId; doSetOwner(...)` → step i `:187` `await doEvict(projectRoot, stage.agentId);` → step j 结果判定 `:190+`
- catch `:166-170` worker 抛错时 best-effort `doEvict` 兜底
- `:127` step d `await doAcquireSlot({ timeoutMs: Infinity })`（致命路径：堵则死等不退）

**改动**：
- 删 `:187` 每阶段 `doEvict`。
- 删 catch `:166-170` 兜底 evict（worker 抛错也不再需要腾槽，完成态/异常态都不计 in-flight）。
- 删 `evictAgentSessions` 的 `import`(:29) + `deps.evictAgentSessions` 注入口(:46) + `doEvict`(:80)（成为孤儿，按红线「移除自己改动产生的 orphan」清掉）。
- 头注 `:9-12`「现 runDispatch 对 completed worker 不销毁…故本编排器新增 evict 后…」改写为「完成态不计 in-flight 槽（concurrency-gate inFlightSessionCount）→ 不再每阶段 evict → 跨 run 误杀根除 + 完成态靠 10min idle 自然回收（第八轮 D-V1.2-50）」。

**红线**：`setOwner`(:182) **保留**（看板分组 + re-attach 反查靠它，删了破 T5/T6）；`acquireSlot`(:127) 顺序/排队逻辑不动；cancel signal 链(:102)不动。

**⚠️ 改动 1/2 成对铁律**：只删 evict 不改计量 → 完成态累积到 limit 个 → `:127` Infinity 排队死等不退（F16 复发且更糟）。两改必须同卡同时落地，单测须同时覆盖。

### 1.3 `lib/pi/concurrency-gate.test.ts`【新增·测试】

`inFlightSessionCount` 单测：faux `__piSessions`（注入临时 globalThis Map）放若干 faux wrapper，`isAlive()`/`inner.isStreaming` 真假混合（在跑/完成/已销毁），断言只数 `alive && streaming` 的。含边界：空 Map=0、全 idle=0、混合精确计数。

### 1.4 `lib/domain/pipeline-orchestrator.test.ts`【改·承重 verify 重写】

**现状（已 Read）**：`:98-227` 承重 verify 断言「每阶段 evict 一次 / size→0 / peak≤1 / acq→set→evict 顺序」；T6 cancel `:716-757`。这些**作废**（不再每阶段 evict）。

**重写为新模型**（沿用 T1/T3 faux Map「三处闭包同一对象 + 负对照 + 变异检查」纪律）：
- faux `inFlightSessionCount`（数 fauxMap 里 streaming 的）+ faux runWorker（建会话进 fauxMap、回合内 streaming=true、完成后 streaming=false 但**留在 Map**）。
- 断言①：N 阶段串行能跑完（完成态留 Map 但不计 in-flight → 下阶段 `acquireSlot` 放行）。
- 断言②：全程 `inFlightSessionCount` peak ≤1（串行单活回合）。
- 断言③：run 结束**无任何 evict 调用**（grep/spy 证 evict 零调用）。
- 断言④（误杀根除）：构造「用户接管的同 agent 会话」（另一 sid、streaming=false 留 Map），N 阶段跑完后它**仍在 Map、未被 destroy**。
- 负对照：faux runWorker 完成后 streaming **永不翻 false**（模拟漏算 bug）→ 第 2 阶段 `acquireSlot({timeoutMs:小值})` 超时（证「漏算→堵死」可被测出）。
- 变异检查：`inFlightSessionCount` 误写成数 `size`（不过滤）→ N 阶段累积下断言①必红（completed 占满→堵死）。
- T6 cancel 块 `:716-757`：cancel 不依赖 evict（走 signal），断言 `failedReason==='已取消'` 保留；只删其中「evict 被调」类断言。

## 二、承重墙（T1 spike，lead 复跑；未绿不开 T2）

**命门**：计量改造后**真正 in-flight 的会话绝不被漏算**——致命路径 `pipeline-orchestrator.ts:127` `timeoutMs:Infinity` 排队（`deadline=now+Infinity` 恒不超时、堵则死等不退）。若 `inFlightSessionCount` 因 `isStreaming` 时序 bug 漏算真在跑会话 → 超额放行 N+1 并发 worker，违 ≤3 红线 + 资源/jsonl 风险。

**这是「全局计量时序正确性」命门（依赖内核 isStreaming 边界），比 evict-by-sessionId 的「单会话逐出粒度」命门重，须活进程打点（非纯静态）。**

**spike 验法**（本环境 `~/.pi/agent` 配 deepseek 凭证可真起会话，参 `verify-t6-drive` 法；在 onEvent 回调 + 轮询点打 `isStreaming` 时间戳）：
1. **全生命周期**：worker 单回合（createAgentSession→registerInnerSession→send(prompt)→`agent_end && !willRetry`）内 `inner.isStreaming` **全程 ===true**，覆盖 LLM 流 + 工具执行 + 多轮 follow-up。（静态 high confidence：内核 `agent.js:316` 置 true、runLoop while 含内层工具循环、`:347` finally 才 false——T1 须打点坐实。）
2. **半拍边界**：agent_end 监听回调内读 `isStreaming===true`、下一 microtask（`await Promise.resolve()`）后 ===false，坐实「多算半拍」是**保守不漏放**、非漏放。
3. **非 completed 路径**（对抗者点名盲区，**必须覆盖**）：timeout 阶段 + worker 抛错阶段两条——其会话 abort 后 `isStreaming===false`（`dispatch-runner.ts:161-163` abort 后落 false）、不计 in-flight，证 F16 在非 completed 路径也不复发。
4. **N 阶段串行**：`inFlightSessionCount` 在每阶段末（completed 会话故意留 Map）正确回落到「下阶段可放行」、全程 peak≤1。
- **负对照**：构造「假在跑」会话（streaming 永不翻 false）→ 证若判据错则下阶段死等超时被测出。
- **变异检查**：`inFlightSessionCount` 误写数 size → N 阶段累积断言必红。

**go/no-go**：①②③④全绿 + 负对照 + 变异检查能抓回归 = **GO**；任一打点显示 `isStreaming` 在回合内出现 false 窗口（漏算可能）= **NO-GO**（须退 evict-by-sessionId、回报用户）。

## 三、确认的风险（ultracode 对抗坐实）

| severity | 风险 | 缓解 |
|---|---|---|
| **blocker** | 改动 1/2 必须成对，且必须「过滤 size」而非「把 completed 移出 Map」 | 实测坐实：只删 evict 不改计量 → completed 累积到 limit 即 F16 复发 + Infinity 排队死等。绝不 `registry.delete` completed（破 `reconcileOrphan` 的 `keys()` 对账 + re-attach `withStartLock` 快路径 `rpc-manager.ts:319-320`）。详设钉死「filter-not-move」+ 单测守 `__piSessions.keys()` 集合不变。 |
| **major** | 承重测试整体重写（`orchestrator.test.ts:98-227`）= 承重网失效风险 | T1 spike 先独立坐实命门；T2 重写后 lead 独立复跑 + 变异检查（误写数 size 必红）证新承重网非 vacuous。 |
| **major** | `isStreaming` agent_end 半拍仍 true（finishRun 在其后 finally） | 已分析：保守偏不漏放、无害（§概要四）；T1 spike 实测②坐实是「多算半拍」非漏放。 |
| minor | 实现漂移去改 `evict-agent-sessions.ts` 须守第五轮红线 | 方案 B **根本不动 evict-agent-sessions.ts**；git diff --stat 收口确认它 + owner-map 零改动。 |
| minor | 方案 B 改全局计量顺带改 legacy dispatch（`orchestrator.ts:117` 默认 60s）排队行为 | 这是预期收益（AC-7 顺带根治 legacy 占槽）；legacy 完成态不再占槽是改善、非回归。T3 回归确认 legacy 派发仍正常。 |
| minor | 多 run 并发下完成态会话堆到 idle 才回收会推高同时**在册**会话数 | 在册数 ≠ in-flight 数；~1MB/会话、10min 自然回收，对 20G 可用内存噪声级（ADR D-R7-09）。不构成问题。 |
