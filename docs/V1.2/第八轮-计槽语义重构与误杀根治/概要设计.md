# V1.2 第八轮 · 概要设计

> 配套：[需求文档](需求文档.md) · [详细设计](详细设计.md)。承重轮。

## 一、核心机制（方案 B，两改成对、缺一即 F16 复发）

### 改动 1 · 计量口径：size → in-flight
`concurrency-gate.ts` 现 `activeSessionCount() = globalThis.__piSessions.size`（数整个 registry，含完成态/idle/聊天会话）。改为 **`inFlightSessionCount()`**：只数「真正在跑回合」的会话——`[...__piSessions.values()].filter(w => w.isAlive() && w.inner.isStreaming).length`。`acquireSlot` 的默认 `activeCount` 改指它（轮询逻辑不动）。

**判据 `inner.isStreaming` 是现成权威信号**：内核 `agent.js` run 起置 true、finally `finishRun` 置 false ⟺ 「该会话有一个回合在跑（含 LLM 流 + 工具执行 + 多轮 follow-up）」。本仓库已用它做 evict 的 abort 判据（`evict-agent-sessions.ts:45`）+ get_state 回前端（`rpc-manager.ts:89`），复用为计槽口径零新概念。

### 改动 2 · 去掉每阶段 evict
`pipeline-orchestrator.ts` 删 step i 的 `await doEvict(projectRoot, stage.agentId)`（:187）+ catch 兜底 evict（:166-170）。一旦完成态不计槽，编排器**不再需要主动腾槽**；删这行即从源头切断 `evict-agent-sessions.ts:41` 那个「按 agentId 一锅端」的 lookup → **跨 run 误杀因果链根除**（不再有任何 evict-by-agentId 被触发）。完成态 worker 会话留在 registry，靠 wrapper 的 10min `idleTimer`（`rpc-manager.ts:52`）自动 destroy→`registry.delete`，仅占 ~1MB 闲置堆 + 一个 setTimeout、不占真并发能力。

## 二、为什么误杀被根除（因果链对比）

| | 修改前 | 方案 B 后 |
|---|---|---|
| 每阶段跑完 | `evict(agentId)` 一锅端该 agent 全部会话 | **不 evict**（完成态本就不计槽） |
| 误杀触发 | 用户复活的同 agent 会话被连根 destroy | 无任何 evict-by-agentId 被触发 → 无从误杀 |
| 完成态会话回收 | 主动 evict（立即） | 10min idle 自然回收（不占 in-flight 槽、不急） |

## 三、红线天然满足（无需额外守卫）

- **owner-map 红线**：方案 B 根本不调 `evict`，`bySession`/`removeOwner` 零触碰（第五轮红线无虞）。
- **F16 不复发**：完成态不计 in-flight、永不堵，13 阶段串行可跑完（改动 1 保证；改动 1/2 必须成对，详见详细设计）。
- **T6 cancel**：走 `__piRunControllers` 的 signal（`getRunController.abort()`→编排器顶检测 + worker signal），不读 `__piSessions.size`、不受计量口径影响。
- **T6 re-attach**：靠 owner-map(bySession) + jsonl + `__piStartLocks` 反查，完成态被 idle 摘除正是其「miss→重建」设计场景，零回归。
- **读时对账**：`reconcileOrphan` 读 `__piSessions.keys()`——方案 B 改的是 size 计量口径而非 Map 成员，`keys()` 集合不变（**必须 filter-size，绝不 move-out-of-Map**）。

## 四、唯一须接受的边缘（保守、无害）

内核 agent_end 在 `finishRun` 置 `isStreaming=false` **之前** emit（内核注释明示「run 被视为 idle 是在所有 agent_end 监听 settle 且 finishRun 清完 runtime 态之后」）。故监听 agent_end 那一瞬读 `isStreaming` 仍为 true——刚结束的会话被**多算半拍槽**。这是「**偏保守不偏漏放**」（决不超额放行）、半拍后自然不计、对 F16 目标无害、**不必消除**。关键：这恰好保证改动 1 **永不漏算真在跑的会话**——而 pipeline 用 `timeoutMs:Infinity` 排队（堵则死等不退），漏算才是致命风险，`isStreaming` 的保守性正好规避它。
