# V1.2 第八轮 · 概要设计（evict-by-sessionId）

> 配套：[需求文档](需求文档.md) · [详细设计](详细设计.md) · [方案B评审NO-GO记录](方案B设计评审-NO-GO记录.md)。承重轮。

## 一、核心机制（evict 收窄粒度）

第七轮 F16 的「每阶段跑完 evict 释槽」机制**完全保留**，只改一件事：**逐出的粒度从「按 agentId 一锅端」收窄为「只逐本阶段那个 sessionId」**。

| | 修改前 | evict-by-sessionId 后 |
|---|---|---|
| 每阶段跑完 | `evictAgentSessions(root, stage.agentId)` → `sessionsForAgent` 逐出该 agent **全部**活会话 | `evictSession(root, stage.sessionId)` → 只 destroy **本阶段这一个** sid |
| 用户复活的同 agent 会话 | 被连根 destroy（误杀） | 是另一个 sid、**不被命中**（误杀根除） |
| F16 释槽 | 有（一锅端） | 有（按 sid，活 worker 恒 ≤1 不变） |
| 计量语义 | 在册≤3 | 在册≤3（**不动**） |

**为何 by-sessionId 不漏槽**（前提 lead 亲核）：worker 工具集无 spawn 类工具 + `__piSessions` 写入仅 2 点 → 一个 worker 只开一个会话、占一个槽。所以「只逐本阶段 sessionId」就把这个 worker 的槽腾干净了，不会像 by-agentId 那样需要担心 inner 子会话。

## 二、为什么误杀被根除（因果链）

| 步骤 | evict-by-sessionId |
|---|---|
| 每阶段跑完 | `evictSession(root, stage.sessionId)` 只 destroy 本阶段 sid |
| 用户接管的同 agent 会话（另一 sid） | 不在 `evictSession` 的目标内 → 不被 destroy |
| 误杀 | 无从发生（逐出按 sid、用户那条是不同 sid） |

## 三、红线天然满足

- **owner-map（第五轮红线）**：`evictSession` 仍只 destroy registry wrapper、复用 `evict-agent-sessions.ts:44-46` 既有「流式先 abort、再 destroy」配方，**绝不碰 `bySession`/`removeOwner`**——evict 后 `getOwner(cwd,sid)` 仍返原 agentId（re-attach 靠它）。
- **计量/上限不改**：不动 `concurrency-gate`（size 计量保持）、不动 `HARD_CAP`/`readMaxConcurrent`/`DEFAULT_MAX`。
- **F16 不复发**：每阶段仍 evict 释槽（只是按 sid），活 worker 恒 ≤1、不卡 idle。
