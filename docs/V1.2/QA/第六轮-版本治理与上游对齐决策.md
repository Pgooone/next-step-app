## 调研结论速览（已 lead 双重核实的事实）

- **首页版本机制本就完好**：`next.config.ts` 构建期自动读 `package.json.version`→`NEXT_PUBLIC_APP_VERSION`、读 `node_modules/@earendil-works/pi-coding-agent` 版本→`NEXT_PUBLIC_PI_VERSION`；`ChatWindow`/`SessionSidebar` 两处消费。**机制无需改**。
- **唯一缺口**：`package.json` version 自 `0.1.0` 起**全 git 历史从未 bump**（无 tag、无 CHANGELOG），首页 `web v0.1.0` 与「已到 V1.2 第五轮」严重不符。
- **原始 pi-web 基线 `0.6.16` 在 V1.2 任何可见处都没记录**（只活在 `next-step-V1/archive/`）—— 确认的溯源缺口。
- **上游更新实况**：pi-web `0.6.16`→最新 `0.6.18`（2 补丁，且上游把内核依赖抬到 `^0.79.10`）；pi 内核我们装 `0.79.0`、声明 `^0.79.0`（已允许 `0.79.10`，仅 `0.80.x` 超范围）、上游最新 `0.80.2`（含 pi-ai 入口重组破坏性变更、连上游 pi-web 都未上 0.80）。
- **上游两条有用修复（已 file:line / 真编译核实）**：① pi-web 0.6.17 修「非空工具预设把扩展工具过滤掉」——我方 `lib/rpc-manager.ts:351-366` 是逐字 pre-fix 代码；② pi-web 0.6.17「markdown 安全渲染」新增 `lib/markdown.ts`，我方 `MarkdownBody.tsx` 是未改基座可干净套用。

## D-V1.2-24 · 首页 web 版本号方案

- **背景**：首页 `web v0.1.0` 是死值（机制会自动读 package.json，但 version 从未 bump）。要让它随迭代更新，需约定「version 何时 / 怎么 bump」（机制不改）。
- **可选项**：
  - **A 纯 SemVer**：`0.MINOR.PATCH`，每轮收官 bump minor（第五轮→`0.6.0`），轮内 bugfix 走 patch。标准、npm/tag 友好；缺点：与「第 N 轮」话语体系差 1、需心算映射。
  - **B 轮次编码 `1.2.<轮次>`**：`1.2`=产品大版本，patch=已收官轮次（本轮即第六轮→`1.2.6`，进 V1.3→`1.3.0` 重起）。首页 `web v1.2.6` 直接自解释「V1.2 第六轮」、与 docs 轮次零心算对齐、改动最小；缺点：借用 patch 作轮次计数、非严格 SemVer（本地工具几乎不真 publish，代价可忽略）。
  - **C 构建期 `git describe`**：next.config 改读 git describe，版本=最近 tag+距离+短 hash。零维护；缺点：碰已稳定的 next.config 注入链（违 surgical）、带 hash 对用户不友好、构建期依赖 git。
- **推荐**：**B** —— 与项目「第 N 轮」话语体系天然同构、改动最 surgical、补现状缺失的发布留痕。
- **谁拍**：用户。
- **最终选择**：**B 轮次编码 `1.2.N`**。**本轮 = V1.2 第六轮 → version `1.2.6`、首页显示 `web v1.2.6`**（用户复核纠正：初次误设 1.2.5＝把编号当成了上一轮第五轮）。配套（lead 实现级，记 ADR）：每轮收官 bump version + 打 `git tag v1.2.N`；历史 5 轮收官点追溯补打 `v1.2.1~v1.2.5`（标注追溯、第五轮 `v1.2.5`@`31d7b73`），`v1.2.6` 标本轮收官。

## D-V1.2-25 · 原始 pi-web 基线版本记录落点

- **背景**：fork 基线 `@agegr/pi-web 0.6.16`（导入 commit `f923751`）在 V1.2 内零记录，是确认的溯源缺口。需持久化「fork 基线 + 当前内核范围 + 上次对照上游时间」以便日后 diff 上游升级基座。
- **可选项**：
  - **组合：`package.json.upstream` 字段（机器锚）+ 根 `UPSTREAM.md`（叙事）**：字段给脚本读（node_modules 无 pi-web 包，「当前 pi-web 版本」只能从此声明取）、MD 放溯源叙事 + 「如何 diff 上游」操作步骤。职责不重叠。
  - **仅 `package.json.upstream` 字段**：最省、机器可读；缺点：溯源叙事 / diff 操作指引无处安放。
  - **仅 README 脚注**：人读直接；缺点：机器不可读、未来 check 脚本拿不到 pi-web 基线版本。
  - （被否决）独立 `PROVENANCE.json` / docs ADR：信息分散或与 package.json 重复。
- **推荐**：**组合（package.json 字段 + UPSTREAM.md）** —— 机器锚 + 人读叙事各司其职。
- **谁拍**：用户。
- **最终选择**：**组合**。`package.json` 加顶层 `upstream` 字段（forkedFrom / 基线版本 / forkCommit / 内核 declared+trackingTarget / lastComparedUpstream）；根目录新建 `UPSTREAM.md`（叙事 + diff 步骤 + 本次对照结论），顶部注明「权威版本号以 package.json.upstream 为准」。

## D-V1.2-26 · pi 上游更新提醒机制（问题二）

- **背景**：`doctor.ts`（predev 自跑）目前不查上游，更新提醒空白。是否新增、做到什么程度。
- **可选项**：
  - **组合：`npm run check-upstream` 脚本（唯一联网点、写缓存）+ 启动 doctor 只读缓存软提示（零联网、非阻断）**：日常 `npm run dev` 被动可见、成本低；实测死网时启动同步联网会拖 ~10s，故启动路径绝不联网。
  - **仅 `check-upstream` 脚本**：想查才跑、启动零输出；缺点：不主动可见易忘。
  - **暂不做提醒**：维持现状。
- **推荐**：组合方案。
- **谁拍**：用户。
- **最终选择**：**暂不做提醒**。问题二本轮搁置（不新增 check-upstream 脚本 / 不动 doctor）；待日后需要再立项。
  > 注：故 D-V1.2-25 的 `package.json.upstream` 字段本轮仅作**溯源记录**用途，不接任何自动比对脚本。

## D-V1.2-27 · 本轮顺带回合并 / 升级范围

- **背景**：调研发现上游有「对我们有用且已核实」的更新，问是否顺带在本轮做。各项独立、可多选。
- **可选项（多选）**：
  - 回合并「扩展工具被过滤」修复（`lib/rpc-manager.ts`，已核实我方=上游 pre-fix）。
  - 回合并「markdown 安全渲染」（新增 `lib/markdown.ts` + 接线 `MarkdownBody`/`FileViewer` + 加 `rehype-raw`/`rehype-sanitize` 依赖）。
  - 内核 `npm update` 到 `0.79.10`（`^0.79.0` 已允许、零破坏、正是上游 pi-web 0.6.18 钉的版本，仅动 lockfile）。
  - 都先不做、只调研留痕。
- **推荐**：lead 在提问时已注明：rpc 修复「是否真影响取决于是否用扩展工具，建议先确认」。
- **谁拍**：用户。
- **最终选择（轮次1·初次拍板）**：**三项都做**（rpc 扩展工具过滤修复 + markdown 安全渲染 + 内核升 0.79.10）。
  > ⚠️ rpc 修复**不照搬上游**：它触 doc 会话受限工具集红线（doc 故意无 bash/扩展工具），实施方式由 **M4 承重 spike（ultracode 对抗式核实）** 裁决后定，记 ADR；若 spike 判定为 no-op（无扩展工具）将回报用户复议。

### 轮次2（2026-06-26，M4 承重 spike 裁决 + lead 复核后）

- **背景**：M4 spike（ultracode 7-agent 对抗式：测绘 + 红线泄漏/no-op/内核API 三视角怀疑者 + 裁决）+ lead file:line 复核，对两条回合并得出与初次拍板不同的事实：
  - **M4（rpc 修复）= provable no-op**：本环境**零扩展安装**（lead 亲验：`node_modules/@earendil-works/` 仅 pi-ai/pi-coding-agent、无 `~/.pi/agent/packages`、源码零引用），要加回的扩展工具集恒为空集，套用等价现状。红线安全（doc 会话物理不流经 `startRpcSessionInner`，退化分支 `session-reattach.ts:129` 只传 2 参 toolNames=undefined 不可达修复行）。**附带纠正报告硬错**：修复实为上游 pi-web **0.6.18**（非 0.6.17）——`npm pack` 三版实测 0.6.17 与 0.6.16 字节相同仍 pre-fix。
  - **M5（markdown）= 可选功能非 bug 修复**：lead 读真实上游 `lib/markdown.ts@v0.6.18` 确认其为 `rehype-raw`+`rehype-sanitize`（**渲染原始 HTML**），我们现用 remark-gfm 已正常渲染 markdown 表格、默认不渲染原始 HTML（安全）；引入它 = 加 2 依赖 + 新攻击面 + 对现有 KaTeX/mermaid 回归风险。
- **可选项（回报用户复议）**：M4 跳过+TODO / 跳过不留注释 / 仍实现（已知 no-op）；M5 跳过+TODO / 实现（接受回归验证）/ 先记 TODO 待议。
- **谁拍**：用户（M4 由 lead 按 no-op 事实判跳过+TODO；M5 回报复议）。
- **最终选择（订正）**：**M3 内核升 0.79.10 照常做；M4、M5 均跳过、各留一行 TODO 注释**（指向上游版本 + 触发条件 + 红线，零行为改动）。M5 用户明确选「先记 TODO 待议」。
  > 📌 经验：单 agent 调研报告把一个 no-op + 一个可选功能都包装成「已验证有用的修复」；是 ultracode 对抗式 spike（npm pack 负对照 + 零扩展实况核实）+ lead file:line 复核才戳破。承重事实必须对抗式核实 + lead 亲验。
