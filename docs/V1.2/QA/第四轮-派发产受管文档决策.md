## D-V1.2-19 · 文档型派发 worker 的工具集是否收窄为受限集

- **背景**：修复后 mode=doc 的 Agent 在派发中会装「受限工具集」（read/grep/find/ls + create_artifact/list_artifacts），即**失去 bash/write/edit**，与交互式 doc 会话一致。但现状（无 mode 分流时）doc 档案 worker 实际拿到的是 `profile.tools`（可含 bash）。是否接受此收窄？
- **可选项**：A 接受收窄（doc 派发对齐交互式 doc 会话、无 bash，需 bash 的派发角色改设 mode=coding，行为零变化）；B 不收窄、doc 派发保留 bash（破 V2 红线，doc 会话不应有直接写盘/执行通道）；C 实现前先检索现有项目是否有 mode=doc 且 tools 含 bash 的档案、确认无依赖再定。
- **推荐**：A + C —— 采纳 A 语义（保红线正解、与交互式 doc 会话一致），实现前做 C 检索确认无静默回归。
- **谁拍**：用户（答「收窄可以接受」）。
- **最终选择**：**A**（已做 C：lead 检索全机所有 `.pi/agents/*/agent.json`，现有 doc 档案 tools 全为空、无一带 bash → **零回归**）。需 bash 干活的派发角色设 mode=coding（coding 行为零变化）。
- **深层问题（用户问「一个 agent 先干 doc、后干 coding 怎么办」）**：`mode` 是**角色档案的固定属性**（doc/coding 是两套工具/权限范式，不混于一个会话）。正解 = **拆成两个角色/两个阶段**（如 `产品经理(doc)` 出需求 → `前端工程师(coding)` 实现），由流水线顺序跑不同 agent——这正是软件工厂的 stage×role 模型，且设计→编码间天然形成 HITL 卡点。「一个 agent 中途切模式」属把 mode 从档案级改阶段级 override 的更大改动，**未来项、本轮不做**。本轮修复与此正交（coding 派发本就直接写盘、不变）。

## D-V1.2-20 · 第一轮 dispatch 是否只给 create_artifact、禁用 propose_edit（分两轮）

- **背景**：dispatch 是 fire-and-forget、回合中无人按块确认。`create_artifact` 立即落 v1 + 物化（D-V4-02 结构操作豁免、无需确认）；但 `propose_edit` 产 PendingChange **必须人工逐块确认**才写盘。若派发 worker 挂 propose_edit，headless 下会留**永远悬而未决的 PendingChange**，还会经 `doc-tools.ts:161-169`「该文档已有待确认变更则拒绝」闸门**阻塞该文档此后所有提议（含交互式人类会话）**。代价：第一轮下游 worker 不能「接力修订上游产的同一份受管文档」（只能各自新建）。
- **可选项**：A 分两轮、本轮 create-only（做成可验收闭环、不引入死悬 pending；「下游接力修订上游文档」放第二轮 HITL 后移路线）；B 本轮就放开 propose_edit（落死悬 pending + 阻塞后续 + 需额外做 HITL 后移提示，风险高）；C dispatch 内自动确认 propose_edit 的 pending（绕过 HITL、破 V2/北极星不变量②，最高风险）。
- **推荐**：A。
- **谁拍**：用户（围绕「谁来决定 propose_edit」澄清后收敛到 A）。
- **最终选择**：**A**。第一轮 dispatch 受限集 = `DISPATCH_DOC_SESSION_TOOLS`（6 项，**去 propose_edit**）。否决 C（破红线）。「下游接力修订上游同一文档」**显式登记为第一轮已知限制**，走第二轮 HITL 后移路线承接。
- **深层问题（用户问「多 agent 协作里谁来决定 propose_edit」）**：**确认权永远在人**——agent 只负责「提议」(propose)，「确认哪些块写盘」是人在 UI 逐块 ✓/✗（V2 / sf-mini HITL 红线核心：agent proposes, human disposes）。多 agent 协作中下游可**发起** propose_edit，但确认时机后移：dispatch 完成提示「N 处待确认」、用户事后在 ArtifactPanel 逐块确认（HITL 后移不缺失）。这恰印证第一轮 create-only 对：第一轮还没建「事后批量确认」UX，故只放无需确认的 create_artifact。

## 第一轮（本轮）已知限制（登记，避免误解「顺序流水线」期望）

- 下游 worker **无法修订上游 worker 产的同一份受管文档**（propose_edit 禁用）——只能各自 `create_artifact` 新建独立受管文档。「甲产 PRD → 乙改同一份 PRD」推迟第二轮（HITL 后移）。
- 模型若**不主动调** create_artifact（firstMessage 仅子任务、无强引导），doc worker 会退回纯文本产物（走 `.pi/artifacts` 文件 + FileViewer），不产受管实体——判空放宽容忍此情况（不误判失败）。本轮真浏览器验收的 fixture 在派发目标里显式写「调用 create_artifact」以确定性看到受管文档路径；生产中建议在 agent.md / 派发目标里引导（属第二轮可增强）。
