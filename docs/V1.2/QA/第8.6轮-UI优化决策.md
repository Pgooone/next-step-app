# 第 8.6 轮 · UI 优化（按钮 + 流水线卡片交互）决策 QA

> **需求2**：优化按钮 + 流水线 agent 卡片交互。原诉求「从 React 组件库找」，经 ultracode 9-agent 调查（含对抗式怀疑者复验全部 file:line + 7 个对比度数字）**反转为「本轮不引库」**。

## D-V1.2-78 · UI 优化方向（引库 vs 自建）

**背景**：用户原话「优化按钮、从 React 组件库找；流水线 agent 卡片交互很烂」。ultracode 调查诊断真痛点 =「按钮零统一（182 个全内联手写、无统一 Button、borderRadius 16 种取值、danger 红 4 种写法）+ 可访问性近零（全项目仅 1 处 focus-visible、暗色主按钮文字对比度 2.54:1 不达 WCAG）」+ 卡片真 bug（queued 状态卡片/浮窗自相矛盾、hover 硬切无过渡、键盘不可达、动画无 reduced-motion 守卫）。

**可选项**：
- **A. 不引库·自建 .btn 语义类层 + 修卡片真 bug + 补 a11y**（P0+P1）；组件库留未来弹层族（届时 Base UI 而非 shadcn）。渐进、一个一个修。
- **B. 引 shadcn·primitive 工具箱渐进**（接 cn + Button 打样、逐步替换）。
- **C. 引 Base UI·headless 行为层兜底**。

**推荐 + 理由**：推荐 **A**。痛点是「token 离散 + a11y 近零」，非缺库的行为层；引库制造第二范式（内联 style vs className 实测 900:62）、违反三红线（简单优先 / 不为单次用途加抽象 / surgical-changes）、撞 token（`--border`/`--accent` 同名异义）、稀释多轮敲定的 Swiss 设计语言。**对抗式怀疑者亦「完全赞成不引库」**，并揪出调查自身一处水分（"Base UI 已在依赖树→引入零成本"是误导，实测 0 处 import、未进 bundle，已修正）。

**谁拍**：用户（经大白话沟通后）。

**最终选择**：**A·不引组件库·自建 .btn 模板 + 一个一个修**。用户原话「先建立自己的模板，一个一个修复」，正对齐调查的「P2 全量迁移移出本轮、渐进逐文件」。
