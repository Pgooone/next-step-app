## D-V1.2-21 · 进项目时如何恢复会话

- **背景**：进项目时停在「暂无会话」空态、不恢复主会话。需定「进项目默认恢复哪条会话」。系统目前**没有**「上次浏览会话」的记忆，主会话靠首条消息懒认定（D-V1.1-09）。
- **可选项**：A 默认恢复该项目 `mainSessionId`，无则进「主对话新建态」（URL `?session=` 深链优先级高于此默认恢复）；B 引入「上次浏览会话」记忆、进项目恢复上次看的那条（新增持久化状态、超出本缺陷范围）；C 维持现状空态、要用户手动从左栏点选（不修）。
- **推荐**：A —— 最小且符合「主对话 = 项目默认入口」直觉，不引入新持久化状态。
- **谁拍**：用户。
- **最终选择**：**A**。恢复优先级 = URL `?session=` > 该项目 `mainSessionId` > 无（→ 主对话新建态，T4）。**不引入**「上次浏览会话」记忆。

## D-V1.2-22 · 主会话是否加删除保护

- **背景**：主会话恢复后，若用户删掉主会话会怎样？是否给主会话加保护（禁删 / 二次确认），避免误删导致「项目失去默认入口」。
- **可选项**：A 不加保护，主会话可像普通会话一样删；删后再进走「主对话新建态」、发首条消息重新认定主对话；B 加保护（删主会话时二次确认 / 或禁止删主会话）。
- **推荐**：A —— 不过度设计；删后有明确兜底（新建态 + 重新懒认定），不会卡死。
- **谁拍**：用户。
- **最终选择**：**A**。不加保护。删主会话后再进 = 新建态、发首条重新认定（已由 T3 降级分支 + T4 新建态承接、真浏览器验过不崩）。

## D-V1.2-23 · dispatch 失败 / 超时会话是否归类、是否做视觉区分

- **背景**：Bug1 修复让 dispatch worker 会话归到对应 agent 名下。但派发可能失败 / 超时——这些 worker 的会话要不要也归 agent、要不要在分组里跟成功会话做视觉区分（如灰显 / 标失败）。
- **可选项**：A dispatch 失败 / 超时 worker 会话也归 agent 名下、也计入分组计数，**不做**成功 / 失败视觉区分；B 失败会话不归类（掉「其它会话」）；C 归类但加失败视觉区分（灰显 / 角标）。
- **推荐**：A —— 会话本身就是该 agent 跑出来的、归属语义正确；失败状态属 dispatch 进度弹窗的事，分组层不重复表达、不过度设计。
- **谁拍**：用户。
- **最终选择**：**A**。orchestrator 在 completed / timeout / aborted 三类终态都写 `setOwner` 归属、计入计数，分组层不做成功 / 失败区分。
