# 第八轮 · 计槽语义重构与误杀根治 —— 用户拍板 QA

> 建设性决策（设计/接线）。lead 实现级取舍记 `../设计决策记录.md` ADR D-R8-*。

## D-V1.2-50 误杀修法选型 + 计槽「活跃」语义

- **背景**：第七轮 T6 揪出「跨 run evict 误杀」（D-R7-07 决策2，留二期）；收官后资源实测（ADR D-R7-09）证一个活会话 = Next 进程内 ~1MB JS 对象、0 独立进程/子进程，内存非瓶颈 → 完成态 worker「占槽」是逻辑计数问题非资源问题。误杀去风险调查（ultracode 7-agent，2026-06-28）给两条修法，须用户定「`并发≤3` 里『活跃』指什么」才能选型。
- **可选项**：
  - **① evict-by-sessionId（调查推荐、lead 推荐）**：保持「并发≤3 = registry 在册会话≤3」语义不变，只把流水线每阶段 evict 从「按 agentId 一锅端」收窄为「只逐本阶段 sessionId」。承重命门轻（纯应用层「单会话逐出粒度」、复用现成 abort-then-destroy 配方、零内核时序依赖）、改动最小（不动计量、承重测试只改一句断言）、红线碰最少。但不改全局语义、不顺带修 legacy 占槽 / 未来 spawn 扩展。
  - **② 方案 B（用户选）**：把「活跃」从「在册≤3」放宽为「**同时在跑回合≤3**」（完成态/idle 会话不计槽）→ 编排器不必每阶段 evict → 误杀因果链从源头断。**更彻底**：顺带根治 legacy dispatch 完成态占槽 + 未来装 spawn 类 pi 扩展也天然覆盖。但代价更大：承重命门重（「全局计量时序正确性」、依赖内核 isStreaming 边界 + Infinity 排队放大漏算风险，须活进程打点 spike）+ 承重测试整体重写 + **是需求语义变更**（用户开 idle 聊天窗不再计数）须用户拍板。
- **推荐 + 理由**：lead 推荐 **①**——目标是修误杀，①直击根因、改动最小、红线最少、承重网不推倒；②的额外好处（legacy 占槽有 60s 超时不死等、未来扩展本环境零安装）当前都不是痛点，为它们承担更大改动面/语义变更/重 spike 不划算；②以后想做随时能做、①不挡路。
- **谁拍**：用户
- **初拍（轮次1）**：用户初选 **② 方案 B**（认为更彻底）。归类 **V1.2 第八轮**、建 spec 三件套。**但 spec 经设计评审门 NO-GO（见轮次2），最终落 ① evict-by-sessionId。**

### 轮次2（2026-06-28，设计评审门 NO-GO 后改选 → 最终）
- **背景**：方案 B spec 建好后，用户要求「使用 ultracode 复审」→ 设计评审门（7-agent、含**承重前提反驳者**）判 **NO-GO**：方案 B 承重命门「in-flight 不漏算」被证伪——`inner.isStreaming` 在 worker 命中可重试错误（429/overloaded/timeout，DeepSeek 高发）的 **retry 退避 sleep 期间为 false**，而 worker 真在 in-flight（dispatch-runner 因 `willRetry=true` 未 finish）→ 漏算 → `acquireSlot({timeoutMs:Infinity})` 超额放行第 4 worker 违 ≤3 红线，且**无内核字段可救**（retry 用独立 `_retryAbortController`、`isCompacting` 也盖不住）。lead 已 file:line 亲核坐实。完整记录见 [`../第八轮-计槽语义重构与误杀根治/方案B设计评审-NO-GO记录.md`](../第八轮-计槽语义重构与误杀根治/方案B设计评审-NO-GO记录.md)。
- **可选项**：① **evict-by-sessionId**（评审重申其优势：纯应用层、零内核时序依赖、规避 retry 漏算、改动最小、承重网不推倒）｜② **方案 B 修正版**（不读 isStreaming，改 **worker 生命周期层自报占槽**——计量与 `runWorker` 的「send→`agent_end && !willRetry`」对齐、天然覆盖 retry/compaction，但 dispatch-runner 自维护计数 + 前端聊天会话另需处理、改动面更大、须活进程 spike）。
- **推荐 + 理由**：**①**——评审把它的优势钉死了；方案 B 的「更彻底」代价具体化成了「内核 streaming 信号不可靠、要么漏算、要么自己重造一套 in-flight 计数」，不划算。
- **谁拍**：用户
- **最终选择**：**① evict-by-sessionId**（用户拍板）。用户要求把审查发现完整留痕（供回溯/讨论）→ 已建 `方案B设计评审-NO-GO记录.md`。spec 已重写为 evict-by-sessionId；**方案 B 修正版作回溯备选**（评审记录 §七，若将来仍想要「在跑回合≤3」语义则走它、不再用 isStreaming filter）。

## 备注

- **最终落地（evict-by-sessionId）的语义**：「并发≤默认 3 = registry 在册会话≤3」**保持不变**（不动 concurrency-gate 计量）——只把流水线每阶段 evict 收窄为按本阶段 sessionId 逐出，误杀根除而语义零变更。
- **回溯信息（方案 B / 修正版的语义影响，若将来采用）**：那条路会把「并发上限 N」从「同时存在 N 个会话」放宽为「同时 N 个在跑回合」——用户开多个 idle 聊天窗/看历史会话不再计入上限。这与 ADR D-R7-09「真约束是 CPU 吞吐 + LLM 限流、非内存」一致。本轮**未采用**。
