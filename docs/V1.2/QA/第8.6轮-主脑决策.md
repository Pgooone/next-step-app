# 第 8.6 轮 · 主脑（主对话作为 orchestrator 父 agent）决策 QA

> **第 8.6 轮新增需求**：在 Next-Step 引入「主脑」——主对话作为 orchestrator 父 agent，替用户理解需求、自动指派子 agent（手动创建的 / 流水线里的），并像 Kimi agent 集群那样在主对话里可视化子 agent 队员。
>
> 本文件记录 brainstorm 阶段的**用户拍板**（背景 → 可选项全列 → 推荐+理由 → 谁拍 → 最终选择）。lead 自主的实现级取舍记 [`../设计决策记录.md`](../设计决策记录.md) ADR（D-R8.6-NN）。
>
> ⚠️ brainstorm 进行中，本文件持续追加（已记 D-V1.2-74~76 三根骨架；可视化形态 / 配置项 等待续）。

## D-V1.2-74 · 主脑定位（与现有流水线 / dispatch 的关系）

**背景**：最根本的架构分叉——主脑和已做好的「流水线（第七轮）」「手动 dispatch」是什么关系，决定整个设计骨架，后面所有问题（配置项 / 自治度 / 可视化）都依赖它。

**可选项**：
- **A. 统一上层·主脑统领**：主脑理解需求后自主决定用「动态点名 agent」或「启动流水线」完成；流水线、手动派发都成为主脑的下属手段。
- **B. 并列·第三种编排模式**：主脑 = 主对话新增「AI 动态点名 agent」能力（理解→拆解→指派），与流水线平行独立，按场景二选一。
- **C. 智能发起器·只生成计划**：主脑只帮理解需求、生成派发计划 / 流水线草稿，执行仍走现有 dispatch/pipeline，不持续统领。

**推荐 + 理由**：推荐 **A**。最贴合用户原始描述「主脑是父、流水线/手动 agent 都是子 agent」；代价是最复杂、需分期落地。

**谁拍**：用户。

**最终选择**：**A·统一上层·主脑统领**。

## D-V1.2-75 · 主脑自治度（工作闭环）

**背景**：与「定位」独立的另一维——主脑接到需求后自己跑多远、要不要用户拍板。决定主脑「性格」与红线安全性（自主派发会撞并发≤3 墙 / 乱跑 / 资源失控）。

**可选项**：
- **A. 全自动·不打断**：理解→拆解→自主派发→自主监工→汇总，全程不打断（用户只在它提问/完成时介入）。最像 Kimi/Claude Code lead，但最需防呆。
- **B. 计划确认制**：理解→拆解→给「派发计划」→用户确认→执行→监工→汇总，大动作前用户拍板。
- **C. 逐步陪跑**：每一步（拆解 / 选 agent / 派每个任务）都确认。最可控但最啰嗦，接近现有手动 dispatch。

**推荐 + 理由**：推荐 **B**。在自动与可控间平衡；「派发计划」是结构化产物、能复用现有 dispatch/pipeline 执行；天然防呆（撞并发/乱跑前用户拦得住）。

**谁拍**：用户。

**最终选择**：**B·计划确认制**。

## D-V1.2-76 · 第一期 MVP 范围

**背景**：「完整统领」是个大工程（AI 拆解 + 指派 + 编排 + 队员可视化 + 守红线），不可能一期做完。按 YAGNI 先对齐第一期聚焦哪块。

**可选项**：
- **A. 闭环优先·端到端走通**：第一期做通「理解→拆解→计划→确认→执行→队员可视化→汇总」全链；执行手段先只支持「动态点名现有 agent」（启动预定义流水线作为手段、更深 AI 拆解留二期）。
- **B. 可视化优先·先有团队感**：第一期先搭「主脑 + 子 agent 团队可视化」框架（队员树 / 父子层级 / 实时态），编排薄复用现有 dispatch，AI 拆解从简。
- **C. 大脑优先·先做理解+计划**：第一期先做最有 AI 含量的「理解→拆解→生成结构化派发计划→确认」，执行 / 可视化复用从简。

**推荐 + 理由**：推荐 **A**。端到端真能用、能验证主脑价值；范围可控；A 的全链本身已含「队员可视化」（用户在意的 Kimi 感）。

**谁拍**：用户。

**最终选择**：**A·闭环优先·端到端走通**（执行手段先点名 agent，流水线接入留二期）。

## D-V1.2-77 · 队员可视化形态（Kimi 式「主对话显示队员」）

**背景**：用户要「像 Kimi agent 集群一样在主对话中显示 agent 队员」。需定可视化主场所，避免与现有 SessionSidebar（会话按 agent 分组）叠床架屋。

**可选项**：
- **A. 内联团队卡·长在主对话里**：主脑派活时在对话消息流内联渲染「团队卡」（队员头像+名+角色+实时状态），点队员进会话。与侧栏互补（侧栏=全部历史会话，内联=当前任务活跃队员）。
- **B. 侧栏团队树·复用升级会话分组**：左侧栏画「主脑→子 agent」树，复用 SessionSidebar 分组、成本最低，但「在主对话中」感最弱。
- **C. 独立团队看板·浮层总览**：单独「团队看板」（类似阶段看板），run 时网格展示队员实时态、按钮唤出。总览强但脱离对话。

**推荐 + 理由**：推荐 **A**。最贴合用户原话「在主对话中显示队员」、Kimi 感最强；与侧栏会话分组互补不冲突。

**谁拍**：用户。

**最终选择**：**A·内联团队卡·长在主对话里**。

## D-V1.2-79 · 队员（子 agent）来源

**背景**：看完用户提供的 Kimi 真实案例截图后浮现的关键岔路——Kimi 是「主脑临时现造一队专家角色」（如现编「日本市场研究员」、要 11 个就实例化 11 个），Next-Step 现状是「从已建 agent 池点名派」。

**可选项**：
- **A. 从已建 agent 池挑**：最快、复用现有 dispatch、第一期能用，但需用户先手动建好常用角色。
- **B. 主脑临时现造**：最像 Kimi、最灵活（无中生有召唤专家队），但需新做「临时造 agent」能力、工量大。
- **C. 两种都要**：最完整，第一期工期最长。

**推荐 + 理由**：推荐 A（第一期复用现有最快、B 留二期）。

**谁拍**：用户。

**最终选择**：**C·两种都要**（用户要完整 Kimi 体验、接受第一期工期最长）。

> ⚠️ **修正 D-V1.2-76 执行手段范围**：第一期执行手段不再只是「点名现有 agent」，新增「主脑临时造 agent」；闭环 MVP 范围相应扩大、工期变长（用户已接受）。
> 📌 衍生待定（留详细设计）：临时造出的 agent 用完即弃 or 存为正式档案进池复用（lead 倾向后者，呼应 C 的「两种都要」闭环）。

## D-V1.2-80 · 主脑的本体定位：主会话即总管（用户纠正）

**背景**：lead 初版设计小结措辞「主对话里的 AI 总管」，用户纠正——不是「对话里再嵌一个独立总管实体」，而是「**主会话本身就是总管**」。

**澄清结论**：
- 主会话（用户进项目默认对话的那个会话）**本身 = 主脑 / 总管**，不新建 orchestrator 实体、不另起名 / 造身份。
- 小任务主会话正常回；大任务它**自动进入「总管」状态**（理解 → 拆活 → 列计划 → 派队员 → 盯 → 汇总）。
- 「要设置的」随之从「创建主脑实体的身份」简化为「**主会话当总管时的行为规矩**」（能派哪些 agent + 允不允许临时造 / 派活前是否先给计划 / 并发上限；模型与说话风格沿用主会话既有设置）。
- 层级：主会话（总管）= 父；派出的队员 = **子会话**（= 对话里那排队员卡片）。

**谁拍**：用户（主动纠正 lead 措辞）。

**最终定位**：**主会话即总管**（非对话内嵌独立实体）。

> 队员卡片落地（复用第七轮卡片族 + 内联进 ChatWindow 消息流 + 接主脑派活数据 + GSAP 克制流程动效，与需求2 打磨统一）属 lead 实现级取舍，记 [`../设计决策记录.md`](../设计决策记录.md) **D-R8.6-02**。

## D-V1.2-81 · 主脑作为「完整 Agent」的 7 要素取舍

**背景**：用户用「完整 Agent 7 要素清单」（① 任务/边界/成功标准 ② 系统提示词 ③ think-act-observe 主循环 ④ 至少一个工具 ⑤ 短/长期记忆 ⑥ 护栏+trace ⑦ 评测用例）review 主脑设计，问哪些需加入。

**关键前提**：主脑 = 主会话（**已是 pi 内核驱动的 agent**）+ 总管能力，非从零造 agent → 7 要素大半内核/现有项目已提供，主脑只补「总管特有」少数项（避免重复造轮子 / 过度工程、守「简单优先」红线）。

**逐条取舍**：
- 🟢 **已有/复用（不新造）**：③ 主循环（内核）、④ 工具底座（主会话）、⑤ 短期记忆（会话上下文）、⑥ trace（transcript/run/版本，= Kimi「回放」对应物）、⑦ 测试框架（vitest + 双层验收）
- 🔵 **必做（已计划）**：② 总管系统提示、④ 派活工具、⑥ 计划确认闸
- 🟡 **用户拍板三处增量**：
  - **① 成功标准 → 第一期做**：主脑在派活计划里给每个子任务带验收点（像 Kimi 卡片「输出要求 + ✓完成项」），让「计划确认」更有意义。
  - **⑤ 主脑专属长期记忆 → 留二期**：第一期靠读项目现有文件（artifact/registry/run）足够。
  - **⑦ 评测 → 标杆场景**（不建复杂自动评测框架，LLM 判断质量自动评性价比低）：用 **sf-mini 全流程端到端跑一遍、看能否产出「好看、好用、可运行的 UI 原型」** 作验收 oracle；机制层（派活工具/卡片/并发/计划确认）另走确定性测试。

**谁拍**：用户。

**最终**：① 进第一期、⑤ 留二期、⑦ 标杆场景（sf-mini → 可运行 UI 原型）。已补进需求文档 §2.9。

## D-V1.2-82 · 设计完整性体检·7 要素外的 9 项缺口处置

**背景**：用户要求进设计前补完 7 要素之外、多 agent 编排常见的缺口。lead 列 9 项 + 建议，用户认可（「就这样很好」）。

**🔴 第一期定调（影响骨架）**：
1. **失败处理**：队员失败 / 超时 / 部分成功 → 主脑轻微自重试 1 次，不行**停下上报 + 给选项**（重试 / 换人 / 跳过 / 中止），不反复重试（防烧钱 / 死循环）。
2. **开工后中途控制**：复用第七轮 run cancel（随时喊停整任务）；「改方向」第一期 = 停掉重说，执行中插话改向**留二期**。
3. **计划可编辑**：第一期「同意 / 否决 / 打回重拟」（不逐行编辑）；执行中主脑想改计划**要再确认**。
4. **信息传递**：复用第七轮「上游累积喂下游」+ 主脑给队员下发任务卡（背景 + 成功标准）+ 主脑读产物汇总。**详细设计重点**。

**🟡 设计时定（有现成复用）**：
5. 产物归属 → 复用**第四轮**受管文档。
6. 队员工具 / 权限 → 复用**第六轮** doc/coding 模式。
7. 「自动判断当总管」边界 → 主脑判断 + 用户一句话覆盖（「你自己做」/「拆开派人」）。

**🔵 技术承重点（详细设计重点攻）**：
8. **长任务上下文管理**：主脑跑大任务对话会爆 → **承重点**，详设专门处理（内核 compaction）。
9. 成本 / 时间感知 → 计划确认时提示「派 N 个队员」。

**谁拍**：用户。

**最终**：🔴 4 条第一期定调、🟡 3 条复用现成、🔵 2 条详设攻坚。已补进需求文档 §2.10。

## D-V1.2-84 · 派活落地路线（技术调研后·对标 Claude Code）

**背景**：技术地基调研(workflow `whg6smqvc`)给出派活落地两路线开放问题：**A**(给主会话装派活 customTool、AI 自主调用；改主会话出生方式=最大风险/碰 V2「不碰 /api/agent/new」红线；计划确认闸难插) vs **B**(主脑产计划 JSON 回复→前端解析→确认→后端调编排 API；避开改主会话；但靠 LLM 正文产规整 JSON + 前端解析容错)。

**用户洞察**：参考 Claude Code 自身 agent team(Agent 工具 spawn 子 agent)/ultracode(Workflow 编排)/plan mode 这套成熟多 agent 派发方式改进 → 不二选一、融合。

**澄清（用户疑「workflow 不就是 JSON 吗」）**：JSON(结构化数据)哪条路都需要、跑不掉；A/C vs B 真正区别 = 结构化数据走「**工具调用参数**」(内核 schema 校验、可靠，= workflow schema 那套)还是「**回复正文夹 JSON**」(赌 LLM 产规整 + 前端解析容错)。

**可选项**：A·工具派活无 plan 闸 / B·产计划内容前端解析 / **C·对标 Claude Code(工具派活 + 计划模式确认)**。

**推荐 + 理由**：lead 推荐 **C**。融合 A(工具派活 = agent 原生、对标 Kimi、配 UI 乙)+ B(计划确认)的好处；计划与派活均走工具调用通道(schema 校验、比 B 正文夹 JSON 可靠)；正是 Claude Code 的 `ExitPlanMode→批准→工具执行` 流程。

**谁拍**：用户（确认路线 C + 追问 JSON 通道澄清后）。

**最终选择**：**C·对标 Claude Code**。流程：主脑调「提交计划」工具(参数=结构化计划 schema 卡)→系统暂停渲计划卡+确认→用户确认→主脑放行调「派活」工具触发编排。第一期聚焦动态派活(临时点名+造队员=agent team 式)；启动预定义流水线作手段留二期(D-V1.2-76)。

> ⚠️ 接受**承重点1**(改主会话出生方式装派活 customTool/resourceLoader、碰 V2 时代「不碰 /api/agent/new」红线)，须 **spike-1 命门**先验(含 D-V2-04 白名单：派活工具名进激活集否则静默失效)+ lead ADR 记打破红线决策(D-R8.6-05)。
