## 一、看板视觉（mockup v1→v11 迭代拍板）

### D-V1.2-28 看板形态
- **背景**：看板根治 F14（13 阶段无全局进度视图），形态决定信息怎么排。
- **可选项**：A 纵向列表 / B 横向时间线·步骤条 / C Kanban 状态分列
- **推荐 + 理由**：A 纵向列表——串行 order = 从上到下阅读顺序、信息密度高（序号/阶段/角色/状态/产物都放得下）、贴合模态窄竖空间；B 13 阶段太挤、放不下角色/产物；C 串行永远只 1 个执行中、丢失顺序、空间浪费。
- **谁拍**：用户
- **最终选择**：**A 纵向列表**

### D-V1.2-29 卡片视觉风格
- **背景**：用户喜欢 Kimi agent 集群卡片样式，要借鉴。
- **可选项**：Kimi 风格（紧凑·叠色·蓝强调·偏炫酷）/ Swiss 风格（留白·细描边·绿强调·克制）
- **推荐 + 理由**：lead 推荐 Swiss（融入现有浅色 app、克制）；用户偏好 Kimi 观感。两者都做了亮/暗 + 三窗口对比供选。
- **谁拍**：用户
- **最终选择**：**Kimi 风格**（支持亮 / 暗主题）

### D-V1.2-30 头像
- **背景**：Kimi 每 agent 一张不同的卡通线描头像；mockup 一度用 emoji（用户嫌丑）。
- **可选项**：彩色 emoji / 手画 SVG / dicebear notionists（外网 URL）/ dicebear notionists 本地生成
- **推荐 + 理由**：dicebear notionists **本地 `@dicebear/core` 生成**——真 notionists 风、每个不同、离线零外网（契合本地工具、不依赖用户网络）。
- **谁拍**：用户（点名要 notionists + 本地）
- **最终选择**：**dicebear notionists 本地生成**

### D-V1.2-31 进度表达
- **背景**：Kimi「序号下三点」实为点阵进度（用户点名值得学）。
- **可选项**：普通进度条 / 流动条纹 / 绿色点阵 LED（3×12 按列点亮）/ 百分比数字
- **推荐 + 理由**：绿点阵 LED——机器/工程感、离散刻度、与克制基线不冲突；图3 实证 Kimi 即此。
- **谁拍**：用户（点名）
- **最终选择**：**绿色点阵 LED（3×12 按列点亮）**

### D-V1.2-32 卡片交互（悬停 vs 点击）
- **背景**：要区分「轻预览」与「进会话」两种操作。
- **可选项**：合并成一个二级菜单（v8 做法）/ 悬停=浮窗预览 + 点击=进会话（分开）
- **推荐 + 理由**：分开——悬停弹浮窗预览（约 330px、可滚动：任务/范围/上游依赖/产物/验收清单/最近动作/耗时），不离开看板；左键点击进该 agent 详细会话（完整对话+产物），会话底部 = agent 切换条 + 顶部「进入对话」。
- **谁拍**：用户（明确纠正：浮窗要保留、勿被二级菜单取代）
- **最终选择**：**悬停浮窗预览 + 点击进会话（底部 agent 切换条）**

## 三、机制与范围

### D-V1.2-37 角色 / 流水线模板（sf-mini 13 角色一键导入）
- **可选项**：放第二期 / MVP 就做
- **推荐 + 理由**：第二期——MVP 用户手填 N 阶段；模板第二期附带预置角色（缓解 F4）。
- **谁拍**：用户
- **最终选择**：**放第二期**

### D-V1.2-38 蓝图存储
- **可选项**：原子写文件（`.pi/factory/pipelines/<id>.json`）/ 纳管为受管文档（版本/diff/回滚）
- **推荐 + 理由**：原子写文件——仿 dispatch-store 临时文件 + rename；对配置 JSON 套文档全家桶是过度设计。
- **谁拍**：用户
- **最终选择**：**原子写文件**

### D-V1.2-39 新流水线与旧 Dispatch 的关系
- **可选项**：并存 + 小修（F6 顺序号 / F8 子任务提示 / F15 新建空白）/ 合并成一个入口 / 流水线替换旧表单
- **推荐 + 理由**：lead 推荐「并存 + 小修」（风险小、改动局部）；用户选「合并成一个入口」。
- **⚠️ 实现影响**：合并意味着「快速 2–3 派发」要降级为流水线入口里的一种**轻量模式 / 预设**；需在概要设计里定清「单轮快派 = 一个临时蓝图 / 一次性 run」的映射，F6/F8/F15 的修复并入新入口。
- **谁拍**：用户
- **最终选择**：**合并成一个入口**（具体映射待概要设计）

### D-V1.2-40 阶段级 HITL（人确认 gate）
- **背景**：是否让阶段产物先暂停、等人确认再继续。
- **可选项**：砍掉（方案原推荐）/ 保留做 / **可调节（用户选某阶段是否暂停等确认）**
- **用户倾向**：可调节——让用户自行选择是否需要暂停等确认。
- **⚠️ 红线澄清（lead 必须先讲清再定稿）**：
  - **破红线的做法**（禁止）= 在 headless worker 里放开 `propose_edit` 产 PendingChange——`doc-session` 明令禁止，headless 无人值守会留永久悬而未决的 PendingChange。
  - **安全做法（可行，lead 提议）** = 把 HITL 做成 **run 级「暂停 / 继续」gate**：阶段仍用 `create_artifact` **auto 产**受管文档 → 编排器**暂停 run**（引入 `paused` 状态）→ 用户在 ArtifactPanel 审阅（要改就走既有「人在场的 `propose_edit` 按块确认」）→ 点「继续」再跑下一阶段。**worker 始终不放 propose_edit、红线不破。**
  - **代价**：引入 `paused` 状态机（方案原想砍），属范围增项。
- **谁拍**：用户
- **最终选择**：**MVP 先砍掉 HITL**（全阶段 auto 产受管文档；要改产物事后在 ArtifactPanel 按块确认）。**「安全版可调节 HITL」保留为未来增项**——MVP 后若要加，按上方「安全做法」（run 级暂停 / 继续 gate + `paused` 状态、worker 不放 propose_edit）实现，**勿走破红线的直接做法**，回到本条参考。

## 四、边界审计追加决策（2026-06-27 ultracode 审计后）

> 见 [`../第七轮-流水线与阶段看板/边界审计报告.md`](../第七轮-流水线与阶段看板/边界审计报告.md)（57 agent 对抗审计全文）。以下为审计后用户拍板的追加 / 修订决策。

### D-V1.2-41 并发模型（冻结模型）
- **背景**：F16 要释放并发槽；但 ≤3 红线 + D-V1.2-32「点击进会话」+ 多 run 的交互需理清（审计缺口 #1/#4/#6）。
- **可选项**：A 守红线·冻结模型 / B 放开 ≤3 上限（破红线、不销毁、自担资源）
- **推荐 + 理由**：A——「冻结」= 释放会话活运行时腾槽 + 对话存盘保留（**非删对话**）；每阶段跑完即还槽 → 串行流水线恒 1 活、多 run 在 ≤3 内自然轮转排队、不饿死不死锁、**连单飞守卫都不需要**；≤3 红线从「硬墙」变「吞吐限流」、保留不动。
- **谁拍**：用户（追问「会话不能冻结吗、为什么一定销毁」促成此模型；澄清销毁 ≠ 删对话）
- **最终选择**：**A 冻结模型 + 并发上限用户可配置（默认 3）**。冻结让 ≤3 从「硬墙」变「吞吐限流」；用户进一步要求**上限可自配**。
- **⚠️ 红线变更（用户拍板、风险自担）**：项目原红线「并发会话 ≤3（硬墙、不改）」→ 改为「**默认 3、用户可在设置调整**」。调高 = 单进程更多活会话、资源耗尽风险用户自担。落地需：① `concurrency-gate` 读配置（默认 3、给合理上限）② UI 给资源警示 ③ **同步更新根 `CLAUDE.md`「并发会话 ≤3」红线条目**。冻结模型本身不变。

### D-V1.2-42 进会话语义 + 解冻续聊
- **背景**：D-V1.2-32「点击进 agent 会话」遇上冻结模型（审计缺口 #1/#2/#3）。
- **可选项**：纯只读看冻结对话 / 只读 + 「继续此会话」解冻续聊
- **推荐 + 理由**：进会话默认**只读**看冻结对话 + 产物（0 槽、对话不丢）；**「继续此会话」= 解冻 re-attach** 占 1 槽（受 ≤3；复用第五轮 re-attach 机制、已修 doc 工具 bug）。
- **谁拍**：用户（解冻续聊**进 MVP**）
- **最终选择**：**只读看 + 「继续此会话」解冻续聊进 MVP**。⚠️ 需先小 spike 验 **dispatch 会话 re-attach**（第五轮 deferred 的 §B）。配套：底部切换条只枚举有 sessionId 的阶段、pending 置灰；历史 run 的 jsonl 缺失则降级「记录已不存在」、产物链接落 artifactId 而非 sessionId。

### D-V1.2-43 取消 run
- **背景**：跑错的长流水线停不下来、持续占槽烧 token，逼回「重启 dev」（本轮要消灭的痛点；审计缺口 #7）。
- **可选项**：接最小 cancel / 维持纯只读不接
- **推荐 + 理由**：接——runPipeline 带 AbortSignal + 「停止」按钮 / `POST .../cancel`。微调 D-V1.2-33（只读 + 一个停止能力）。
- **谁拍**：用户
- **最终选择**：**接最小 run 级 cancel**

### D-V1.2-44 acquireSlot 超时 + 排队态
- **背景**：现 acquireSlot 默认排队 + 60s 超时；流水线被 60s 误杀中止 → 违方案自身 AC5「不报上限失败」（审计缺口 #5）。
- **可选项**：流水线调用不限 / 超大超时 + 看板排队态 / 维持 60s
- **推荐 + 理由**：不限超时 + 排队态——只改流水线调用点（旧 dispatch 默认不变）；看板显「排队中·等会话槽」；配合冻结模型 + cancel，无限等也可控。
- **谁拍**：用户
- **最终选择**：**不限超时 + 看板排队态**

### D-V1.2-45 孤儿 running run 对账
- **背景**：dev 中途重启后 run.status='running' 永久残留 → 看板僵尸、轮询不停（审计缺口 #10）。
- **可选项**：读时对账判 failed / 不做
- **推荐 + 理由**：读时对账——GET run 若 running 但当前阶段 sessionId 不在活 registry → 判 failed。零新状态机、只清僵尸（非「续跑」恢复，不越 D-V1.2-33 第三期）。
- **谁拍**：用户
- **最终选择**：**读时对账判 failed**

### D-V1.2-46 上游喂下游范围
- **背景**：现方案下游只拿「紧邻上一阶段」产物，N≥4 拿不到更早的资料 / 需求，F9「自动喂下游」核心价值退化成只链一跳（审计缺口 #8）。
- **可选项**：累积全部已完成上游 / 只链一跳 / dependsOn 声明
- **推荐 + 理由**：累积全部已完成上游摘要——保住 F9 核心价值；dependsOn 太重留第二期。
- **谁拍**：用户
- **最终选择**：**累积全部已完成上游摘要**

### D-V1.2-47 改上游产物 → 下游
- **背景**：HITL 已砍（D-V1.2-40）+ 看板只读（D-V1.2-33），事后改上游产物无法回灌已跑完的下游（审计缺口 #14）。
- **可选项**：仅文档澄清「不回灌」/ 加「从此阶段重跑下游」逃生口
- **推荐 + 理由**：MVP 仅文档澄清——明确「改上游不影响已跑完的下游」；「从此阶段重跑下游」留第二期（与取消 / 重跑能力一起）。并纠正 `讨论准备-待拍板问题.md` §4 Q6「事后 ArtifactPanel 确认」的 overclaim（事后确认 ≠ 止损下游）。
- **谁拍**：用户
- **最终选择**：**MVP 仅文档澄清「不回灌」**；重跑下游留第二期
