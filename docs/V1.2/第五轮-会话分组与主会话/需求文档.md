# V1.2 第五轮 · 会话分组与主会话 —— 需求文档

> 用户拍板见 [`../QA/第五轮-会话分组与主会话决策.md`](../QA/第五轮-会话分组与主会话决策.md)（D-V1.2-21~23）；lead ADR 见 [`../设计决策记录.md`](../设计决策记录.md)（D-R5-*）。开工交接见同目录 [`开工交接.md`](开工交接.md)。

## 一、背景与根因

模拟新用户实际使用时报两个缺陷：

- **Bug1（会话分组）**：dispatch 派发的各 agent 会话**没按 agent 分组**、全堆进左栏「其它会话」区——本该「编码甲 / 编码乙」各自成组，却混作一团，看不出哪条会话属哪个被派发的 agent。
- **Bug2（主会话恢复）**：进项目时**不恢复主会话**、停在「暂无会话」空态——用户上次在该项目聊过的主对话不见了，每次进项目都像全新空项目。

**根因（lead file:line 复核）**：

- **Bug1 = 数据层缺一行**：`lib/domain/orchestrator.ts` 派完 worker 只把会话 id 写进 `assignment.sessionId`、**从不调 `setOwner`** 把「会话 → agent」归属写进 `ns-session-map.json` 的 `bySession`。而分组 UI（`lib/session-grouping.ts` / `SessionSidebar`）早已完备、就等 `bySession` 有数据。对照正确范式 `app/api/projects/[id]/agents/[agentId]/session/route.ts:45`（交互式起 agent 会话时已 `setOwner`）——dispatch 缺的就是这一笔归属落盘。
- **Bug2 = 进项目只刷 map、不读 mainSessionId 恢复**：`AppShell` 进项目时只 `refresh` 会话归属映射（拿来分组用），**不读 `mainSessionId` 去恢复主对话**。而主会话本身靠「首条消息懒认定」（D-V1.1-09 不预建空会话），既有的唯一恢复路径是 URL `?session=` 深链——但从项目墙进入时该参数被清空 → 落到空态。

## 二、目标

- **Bug1**：dispatch 派发的每个 agent（含失败 / 超时的）会话都归到对应 agent 名下、在左栏按 agent 分组显示并计入计数；不堆「其它会话」。派发到终态后左栏分组**自动刷新**（无需用户手动刷新）。
- **Bug2**：进项目时按优先级恢复会话——URL `?session=` 深链 > 该项目主对话 `mainSessionId` > 都没有则进「主对话新建态」（可直接输入、发首条懒认定主对话）。
- 不破任何红线：不改内核、不破 V2 提议红线、≤3 并发不变、客户端 bundle 不拖 `node:fs`、不预建空会话（守 D-V1.1-09）。

## 三、范围

### 本轮做

1. **Bug1 数据层（T1）**：orchestrator 派完 worker 无条件给该 worker 会话写 `setOwner`（会话 → 被派发 agent 归属）。completed / timeout / aborted 三类都写。
2. **Bug1 接线（T2）**：DispatchPanel 派发到终态时触发**独立回调**（不被 artifact 门控）通知左栏刷新会话分组；AppShell 有界重试拉取归属映射直到新归属出现。
3. **Bug2 恢复主会话（T3）**：进项目时按「URL > mainSessionId > 新建态」优先级恢复，恢复逻辑下沉 SessionSidebar 现有恢复 effect、不在 AppShell 另开。
4. **Bug2 新建态（T4）**：首次进新项目（无主会话）= 主对话新建态——经既有 `effectiveNewSessionCwd` 链天然成立，无代码（守 D-V1.1-09）。

### 本轮不做

- 主会话保护（删除二次确认 / 禁删）——用户拍板不加（D-V1.2-22）。
- dispatch 成功 / 失败会话的视觉区分——失败会话同样归 agent 名下、计入计数，不做区分（D-V1.2-23）。
- 「上次浏览会话」记忆——系统无此记忆、本轮不引入（D-V1.2-21）。
- doc-mode dispatch worker re-attach 续聊得 7 工具集（含 propose_edit）vs headless 跑时 6 工具集的非对称——文档化接受、机制零改（见 D-R5-06）。

## 四、验收标准（AC）

| 编号 | 标准 |
|---|---|
| AC-1 | dispatch 派发（coding）done 后，左栏**自动**（无需手动刷新、等 T2 重试窗口内）出现按 agent 分的分组（如「编码甲 / 编码乙」各 count=1），`ns-session-map.json` 的 `bySession` 含对应归属，**不堆「其它会话」**。 |
| AC-2 | dispatch 失败 / 超时的 worker 会话也归到对应 agent 名下、也计入分组计数（不做成功 / 失败视觉区分）。 |
| AC-3 | 进项目：发首条消息认定主对话后，回项目墙再进该项目 → **自动恢复**该主对话（主对话区高亮 + 中间区显示完整对话），URL 无 `?session=`。 |
| AC-4 | URL `?session=<id>` 深链优先级高于默认主会话恢复（带参进入恢复指定会话，不依赖 map 就绪）。 |
| AC-5 | 删除主会话（`mainSessionId` 陈旧 / 指向已删会话）后再进该项目 → 走「主对话新建态」（可输入、不崩），由 T4 接管。 |
| AC-6 | 首次进新项目（无任何会话、无主会话）= 主对话新建态：可直接输入、发消息前该 cwd 无会话；守 D-V1.1-09（临时 id 懒落盘、不调后端预建、不占并发槽）。 |
| AC-7 | `npm run lint` 净 / `npx tsc --noEmit` 零错误 / `npm test` 全绿；真浏览器 console pageErrors=0（无 JS 崩溃）。 |
| AC-8 | dispatch 既有行为零回归：`assignment.sessionId` 仍写、`.pi/artifacts/<id>/N-角色.md` 文本保底仍写、≤3 并发不变；交互式起 agent 会话 / 主对话懒认定行为不变。 |
