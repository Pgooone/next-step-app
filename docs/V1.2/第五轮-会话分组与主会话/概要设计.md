## 三、数据流

### Bug1（会话分组，dispatch worker）

```
派发目标+子任务 ─▶ orchestrator.runDispatch（串行、acquireSlot ≤3）
                      │ projectRoot=task.projectRoot
                      ▼
              逐 assignment：runWorker → result.sessionId
                      │ doSetOwner = deps.setOwner ?? setOwner
                      ▼
              orchestrator.ts:152（assignment.sessionId 之后、早于 timeout/aborted 判断）
                      │ 无条件 doSetOwner(projectRoot, result.sessionId, assignment.agentId)
                      │   → ns-session-map.json.bySession[sessionId] = agentId（completed/timeout/aborted 都写）
                      ▼
              前端轮询终态 ─▶ DispatchPanel.onSessionsChanged（独立回调、不被 artifact 门控）
                      ▼
              AppShell.handleDispatchSessionsChanged：有界重试 refresh（≤8×700ms）
                      │ before = 当前 bySession 键数；每次并 setRefreshKey 重拉 /api/sessions
                      │ 直到 bySession 增长（拿到新归属）或达上限
                      ▼
              SessionSidebar 分组 useMemo（groupSessionsByOwner）→ 左栏按 agent 分组、计数
```

### Bug2（进项目恢复，SessionSidebar 恢复 effect）

```
进项目（SessionSidebar key=currentProjectId 重挂、selectedCwd=null、restoredRef=false）
   │ AppShell 进项目 effect：refresh(currentProjectId) → map/loadedProjectId 异步到位
   ▼
恢复 effect（两阶段）：
   ① 会话恢复（restoredRef 守护、仅一次）
      effectiveMap = selectMapForProject(map, loadedProjectId, currentProjectId)
      target = pickSessionToRestoreOnEnter(effectiveMap, initialSessionId)   // URL > mainSessionId > null
      ┌─ target 非空：列表就绪→find→found ? setSelectedCwd+onSelectSession(_,true) : onInitialRestoreDone（已删/陈旧→降级）
      └─ target 空：mapReady=loadedProjectId===currentProjectId；未就绪且有 currentProjectId → return 等 map；就绪 → 标 restored
   ② 默认 cwd（恢复尝试已定后）：selectedCwd!==null 则跳过；否则 selectedCwdProp 优先、回退 getRecentCwds
   ▼
无主会话 → setSelectedCwd(currentRoot) → onCwdChange → AppShell setActiveCwd
   → effectiveNewSessionCwd=activeCwd（因 selectedSession===null）→ ChatWindow newSession 模式（T4 新建态）
```

## 四、红线合规（机制零改，lead file:line 已核）

- **不改内核**：T1 经 `OrchestratorDeps` 可注入钩子（生产默认 `setOwner` / 测试 spy）；所有改动为 Next-Step 自有代码，内核零 fork。
- **不预建空会话（D-V1.1-09）**：T4 走既有 `effectiveNewSessionCwd` 链，临时 id 懒落盘、不调 `startProfileSession` / 后端预建、不占并发槽。
- **客户端 bundle（D-R7B-07）**：DispatchPanel / AppShell / SessionSidebar 改动零新增含 `node:fs` 的服务端 value-import；`pickSessionToRestoreOnEnter` 是纯函数（仅 `import type SessionMap`）。
- **≤3 并发**：`concurrency-gate` / `acquireSlot` 一行未动。
- **V2 提议红线**：本轮不碰提议工具装配；doc-mode dispatch worker 仍走 `DISPATCH_DOC_SESSION_TOOLS`（6 项、无 propose_edit，第四轮成果）。re-attach 续聊得 7 工具集属续聊语义、文档化接受（D-R5-06）。
- **边角**：派发中改该 agent mode 会经 `evictAgentSessions` 逐出仍流式的 worker（先 abort 再 destroy、无损坏、仅令该 assignment 走失败分支）。

## 五、承重墙

- **T1（Bug1 数据层）= 承重墙**：会话归属落盘是 Bug1 的命门，分组 UI 全靠 `bySession` 有数据。经 `OrchestratorDeps` 可注入 `setOwner?` 钩子保 orchestrator faux 可单测（不引内核依赖、不真写盘）；+6 spy 用例断言 completed / timeout / aborted 三类都写、三类前置失败不写。
- **T2（Bug1 接线）= 承重墙**：必须新增独立 `onSessionsChanged`、**绝不复用** `onArtifactsChanged`——后者被 `if(!producedArtifact)return` 闸门挡，coding / 纯文本 worker（artifactId 恒空、占绝大多数）跑完不 fire、分组当场不显（M7·5.3 类 FAIL 复发，D-R5-02）。
