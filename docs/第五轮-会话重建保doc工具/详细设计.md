# 第五轮 · 会话 re-attach 重建受限 doc 工具集 —— 详细设计

> 修复 D-B4-4 已登记 gap：文档型 profile 会话在 re-attach（idle 销毁 / dev 重启 / SSE 重连）后丢失提议工具集、反获 write/edit/bash（违红线②）。
> **范围**：只修 profile/文档会话的 re-attach。主对话(`/api/agent/new`)、dispatch 的 §A/§B 更大 gap 不在本轮。
> 评审通过后再实现（用户拍板「先出详细设计再实现」）。

## 一、背景与根因（已确认 + 真模型复现）

文档型 agent 跟进轮调 `propose_edit` 失败（用户用新手引导 §三.8 复现）。

**根因**：follow-up 消息 / SSE 重连在内存 registry `__piSessions` 未命中活会话时，走通用
`startRpcSession(id, filePath, cwd)`（`rpc-manager.ts:301`）**重开 generic 会话**——
`createAgentSession({cwd,agentDir,sessionManager})` **不带 customTools**、默认全套 coding 工具
（`rpc-manager.ts:327/333-338`）。于是 re-attach 后的文档型会话：
1. **丢** `create_artifact`/`propose_edit`/`list_artifacts`；
2. **多出** `write`/`edit`/`bash` → 能直接写盘、**绕过按块确认（违红线②）**。

受限工具集只在初次 `startProfileSession`（`profile-session-wiring.ts:127-145`）注入一次，re-attach 不重建。
作者已在 `profile-session-wiring.ts:15-18` 登记此 gap（D-B4-4，「本卡不修」）。

**复现（真模型 DeepSeek 双相）**：
- turn1 create → turn2 propose **快连**（会话仍 alive）→ ✅ 落 PendingChange。
- turn1 后**重启 dev**（清空 `__piSessions`）→ turn2 propose → ❌ 无 PendingChange（工具已丢）。

**触发 re-attach 的时机**：① idle 10 分钟销毁（`AgentSessionWrapper` idleTimer，`rpc-manager.ts:52`）；
② dev 重启 / HMR 清空 `globalThis.__piSessions`；③ 进程级 cleanup。

## 二、re-attach 落点全集（blast radius）

`grep startRpcSession` → 三处调用：
| 落点 | 用途 | 是否本轮修 |
|---|---|---|
| `POST /api/agent/[id]` (route.ts:30) | 跟进轮发消息 | ✅ 修 |

> 关键：SSE 重连（events 路由）也走 generic 重建——所以**页面刷新/会话切回**就可能丢工具，不止「跟进轮」。两个 re-attach 落点必须共用同一套「profile 感知」重建逻辑，否则修了一处漏一处。

## 四、详细设计

### 4.1 profile 感知解析器 `resolveOrReattachSession(sessionId, filePath, cwd)`
```
1. existing = getRpcSession(sessionId); if existing?.isAlive() → return existing            // 活会话，快路径
2. agentId = readSessionMap(cwd).bySession[sessionId]                                        // session-agent-map
3. if (agentId) {                                                                            // 属某 agent → 文档型 profile 会话
     projectId = new ProjectRegistry().list().find(p => normalizeRoot(p.root)===cwd)?.id
     profile  = projectId ? tryGet(AgentProfileStore, projectId, agentId) : null            // 容错：删档/删项目 → null
     if (profile) return reattachProfileSession({ sessionId, filePath, projectId, projectRoot: cwd, profile })
   }
4. return startRpcSession(sessionId, filePath, cwd)                                           // 主对话/孤儿会话→generic（现状）
```
- **判定依据**：`bySession[sessionId]` 有 agentId ⟺ 它是「按档案起的会话」（M5 映射，`<root>/.pi/ns-session-map.json`）。`mainSessionId` 那条不在 `bySession` → 主对话走 generic（与 §B 现状一致）。
- **root→projectId**：`ProjectRegistry.list().find(root===cwd)`（无 getByRoot，list+find 足够；root 即 cwd，D-V1.1-01）。
- **容错**：档案/项目被删（agentId 在但 profile 取不到）→ 落 generic（会话不带 doc 工具，但属孤儿、可接受；不抛错）。

### 4.2 `reattachProfileSession({ sessionId, filePath, projectId, projectRoot, profile })`
与 `startProfileSession` 同源装配，**三点差异**：① 用 `SessionManager.open(filePath)` 重开既有会话（非 create）；② **不发首条 message**（D-B4-3 的「必发首条」是为**新建**会话触发真落盘防幻影空会话；re-attach 的会话文件已在磁盘、无此问题）；③ 重应用 model/thinking。
```
const sm = SessionManager.open(filePath, undefined);
const { options } = await assembleProfileSessionOptions({ projectRoot, profile, cwd: projectRoot, sessionManager: sm, agentDir, additionalSkillPaths: extraSkillDirs(projectRoot) });
const docOptions = assembleDocSessionOptions({ projectId, sourceActor: profile.name });   // 受限白名单 + buildDocTools
const { session: inner } = await createAgentSession({ ...options, ...docOptions, sessionManager: sm });
await applyProfileRuntime(inner, profile);          // 重设 deepseek 等（re-attach 否则回默认 faux）
return registerInnerSession(inner);                  // 进 __piSessions，接事件流
```
- 与 `startProfileSession:138` 同款 spread 顺序 `{...options, ...docOptions}`（docOptions.tools 覆盖 profile.tools，D-V2-04）。
- **⚠️ 待 spike 验证项**：opened 会话已带**持久化 systemPrompt**（注入块已落盘，见 D-B4-4 注释）；`assembleProfileSessionOptions` 的 resourceLoader 又会注入 systemPrompt——需确认 `createAgentSession(open + systemPrompt override)` 是**覆盖**还是**叠加**，避免角色块重复。若叠加，则 re-attach 只传 `docOptions`（tools+customTools）、**不传** profile 的 systemPrompt override（systemPrompt 靠 jsonl 持久化即可）。**关键必恢复的是工具集**（白名单+customTools），systemPrompt 已持久化。
- registerInnerSession 已含 idleTimer + onDestroy 摘除（沿用），无需新机制。

### 4.3 两路由接线
两路由 not-alive 分支由 `startRpcSession(id, filePath, cwd)` 改为 `resolveOrReattachSession(id, filePath, cwd)`。其余（cwd 取 SessionManager header、404、SSE/命令流）不动。`/api/agent/new` 不改。
