# Next-Step V1.1 设计决策记录（ADR）

> 本文记录 V1.1 实现阶段的 **lead 机制层决策**（详细设计里标 🔧 的关键几条）。
> **为什么由 lead 决定、而非问用户**：按北极星「冲突分层响应」—— 这些是**实现级取舍**（不触使命、不破任何不可破不变量），故 lead 自主拍板并在此留痕；只有触及使命 / 不变量的冲突才必须停下问用户。
> 每条记：背景 → 可选项 → 选定 → 理由（对照北极星）→ 影响。编号沿用 V1 `tasks/v1-history/decisions.md` 的 `D-` 风格，本期前缀 `D-V1.1-`。

## 对照基准（北极星浓缩）

- **使命**：本地 · 单用户的「多 Agent 软件工厂」；灵魂 = 多 Agent 自定义 + 协作产文档型产物 + **人对产物细粒度可控可回溯**。
- **不可破不变量**：①不改 pi 内核（只封装）；②产物改动必经按块确认；③纯文件落盘、本地单用户；④并发会话 ≤ 3；⑤复用 pi-web 既有能力、不另起炉灶。
- **设计哲学**：Simplicity First（最小代码、不投机抽象）、未决设计不预实现。

## D-V1.1-01 · M5 会话↔agent 映射的存盘位置

**背景**：会话数据结构无 `agentId`（`lib/types.ts:174-184`），且不可改 pi 内核的会话文件 → 必须在领域层另存「会话 → agent / 主对话」映射。存哪？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 项目本地 `<cwd>/.pi/ns-session-map.json` | 与项目其它 Next-Step 数据（`.pi/agents`、`.pi/artifacts`）同处；随项目走；契合既有 `.pi/` 约定 | 多会话并发写需原子写 |
| B | 中央 `~/.pi/agent/ns-session-maps/<encoded-cwd>.json` | 与内核会话存储（按 encoded-cwd）并行 | 复刻内核 encoded-cwd 间接层、贴内核内幕；项目移动即错配 |
| C | 塞进项目注册表 `projects.json` | 不新增文件 | 注册表单一职责被污染（项目列表 vs 每会话元数据）；每次会话操作都 churn 注册表 |

**选定：A —— `<cwd>/.pi/ns-session-map.json`（原子写，仿 V1 D-04）**

**理由（对照北极星）**
- **支柱「项目即工作区」**：映射是该项目工作区的元数据，应随工作区走；放 `<cwd>/.pi/` 下，项目即自包含。B 的中央存储会在项目移动时与 cwd 错配。
- **不变量③ 纯文件落盘**：A 是纯 JSON 文件，符合；且映射是 **Next-Step 领域层自己的数据**，而领域数据（agents/artifacts）一向落在 `<cwd>/.pi/` —— A 与既有约定一致。
- **不变量① 不改内核**：B 要复刻内核的 encoded-cwd 方案 = 贴着内核内幕走、易碎；A 只在领域层加一个文件，零内核耦合。
- **Simplicity First**：A 最小、最一致；C 让注册表越权背单一职责。
- **不变量④ 并发会话 ≤ 3**：A 用原子写（temp + rename）即可防自己的多会话并发，符合「乐观锁防自己」。

**影响 / 备注**：内核的会话**文件**存在中央 `~/.pi/agent/sessions/<encoded-cwd>/`（内核的事，不动）；我们的**映射**是领域层叠加物、项目本地。项目若移动路径，内核会话本就按旧 cwd 失联（内核既有行为），A 不会让此更糟。

## D-V1.1-02 · M6 项目首页的渲染方式

**背景**：要在进工作台前先有「项目卡片墙」首页。现状是单页 `AppShell` + 左栏项目下拉。

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 单页内按 `currentProjectId` 二选一渲染（`app/page.tsx`：空→`ProjectHome`，有→`AppShell`） | 保持 pi-web 单页架构；复用现有 currentProjectId 状态；最小改动 | 首页与工作台共一 URL（无项目深链） |
| B | 新增 `/projects` 路由 + `/project/[id]` 工作台 | 可深链到具体项目 | 引入 pi-web 本没有的路由层；偏离既有单页模型；过度工程 |

**选定：A —— 单页内按 `currentProjectId` 二选一渲染，不加新路由**

**理由（对照北极星）**
- **不变量⑤ 复用 pi-web 既有能力、不另起炉灶**：pi-web 是**刻意的单页应用**（全应用只有 `/` 一个路由，URL 顶多挂 `?session=`，见前端解析报告 §0/§1）。A 留在这个模型里；B 凭空加一层路由 = 另起炉灶。
- **Simplicity First**：A 只是「currentProjectId 为空就渲染另一个组件」，复用现成状态、几乎零新机制；B 要建路由文件、迁移导航 / URL 状态。
- **使命 本地 · 单用户**：本地单用户工具，项目深链不是核心诉求，为它引入路由层不划算。

**影响 / 备注**：若日后确实要项目深链，可仿现有 `?session=` 加一个 `?project=` query 参数实现，无需全套路由 —— 保留为非破坏性的未来选项（符合「未决设计不预实现」，现在不做）。

## D-V1.1-03 · M8 @agent 转交的数据打包与投递

**背景**：主对话 `@agent` 要把「全主对话历史 + 附件（发送时可勾选）」转交到该 agent 的**单独会话**。怎么打包、怎么投递？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 历史序列化为 `<context source="主对话">…</context>` 文本 + 文件按 M2 的 `<file>` 内联（浏览器上传无绝对路径，`name` 用文件名），作为目标会话**首条消息**经现有发消息链路投递 | 走内核自己的发消息路径、不碰会话文件内幕；与 M2 同一套内联格式；内容在目标会话里可见可回溯 | 历史序列化格式需实现时定细节 |
| B | 直接把结构化历史写进目标会话的 `.jsonl` | 保真度高 | 复刻内核消息格式、做会话文件手术 = 贴内核内幕、易碎 |
| C | 只发一个「见主对话会话 X」的指针 | 最轻 | 不满足需求（用户要把**内容**带过去） |

**选定：A —— `<context>` 文本 + `<file>` 内联，作目标会话首条消息，经现有发消息链路投递**

**理由（对照北极星）**
- **不变量① 不改内核**：A 用内核**自己的**发消息路径（`session.prompt` / `POST /api/agent`），消息格式由内核负责、我们零内幕耦合；B 自己写 `.jsonl` = 做内核会话文件手术，随内核升级易碎。
- **不变量⑤ 复用 pi-web 既有能力**：A 复用现成「向会话发消息」能力；B 绕过它另搞一套。
- **使命「细粒度可控可回溯」**：A 把转交内容作为一条**可见消息**留在目标会话里，透明、可回溯；B 可能把它藏进文件、不透明。
- **Simplicity First + 一致性**：A 复用 M2 已定的 `<file>` 内联格式，与对话框上传同一套，少一套机制。
- C 不满足需求（要带内容、非指针），淘汰。

**影响 / 备注**：`<context>` 的精确序列化（是否含工具调用块、角色如何标注）属 M8 实现期细节；本条拍板的是**机制**（文本内联 + 走正常发消息链路），非逐字格式。目标会话不存在时先按需起会话（M7 链路 + M5 标记归属）再投递。

## D-V1.1-04 · M8 转交的投递机制（起新会话 vs 复用既有会话）

**背景**：D-V1.1-03 定了「怎么打包载荷」，但「投递到该 agent 单独会话」一句过抽象。代码现实里只有两条真实链路：① 起**新会话** `POST /api/projects/[id]/agents/[agentId]/session`（载荷作 `firstMessage`，V1 已验收、不碰 `.jsonl`）；② 向**已存在会话** `sendAgentCommand(sid, prompt)`（但要求该会话 live 注册，落盘但未 live 的会话无 re-attach 链路）。选哪条？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | **转交 = 起新会话**：`POST /api/projects/[id]/agents/[agentId]/session`，载荷作 `firstMessage`；复用 `handleAgentSessionStarted` 切到该会话 + 接 SSE | 走 V1 已验收的起会话端点、不碰会话文件内幕；一举解决「投递路径」+「转交后接 SSE 流」；天然经 M5 标记归属 | 每次转交起一个新会话（符合「单独会话」语义） |
| B | **复用既有会话**：`sendAgentCommand(sid, prompt)` 向该 agent 已有会话追加 | 不新增会话 | 要求会话 live 注册；落盘但未 live 的会话**无 re-attach 链路**，需先补一个按 `sid` 的 re-attach 端点才可行 |

**选定：A —— 转交 = 起新会话（`POST /session`、载荷作 `firstMessage`、复用 `handleAgentSessionStarted` 切会话 + 接 SSE）**

**理由（对照北极星）**
- **不变量① 不改内核**：A 复用 V1 已验收的 `/session` 起会话端点，内核负责会话创建与消息格式，零内幕耦合。
- **不变量⑤ 复用 pi-web 既有能力**：A 复用现成的「起 agent 会话 + `handleAgentSessionStarted` 切会话接 SSE」链路，不另起炉灶。
- **Simplicity First**：A 一条端点同时解决投递与转交后接流；B 还得先补按 `sid` 的 re-attach 端点，多一套机制。

**影响 / 备注**：复用既有会话（B）列为**后续增强**——待补一个按 `sid` 的 re-attach 端点后再支持「向 agent 已有会话续接转交」。本期取 A：每次 `@agent` 转交起一条新会话，契合需求「该 agent 的**单独**会话」语义。**用户已确认**（先 A、B 留作后续迭代）；B 的来龙去脉详见 `后续迭代清单.md` 第 1 条。

**背景**：M6 卡片要显「项目名 + 路径 + 最近活动」，但 `Project` 类型只有 `id/name/root/createdAt`，无「最近活动」时间戳。

## D-V1.1-06 · M5 补 `removeOwner` + PATCH 删归属分支

**背景**：M5 任务卡只列 `set/get owner`，但详细设计 PATCH 要「增删 `bySession[sessionId]`」，缺显式「删归属」路径。

**可选项**：A 补领域 `removeOwner` + PATCH `{sessionId, agentId:null}` 删分支；B 只靠惰性清理兜底删。

**选定：A**。

**理由**：详细设计明确要「删」；惰性清理只兜底「外部直接删 `.jsonl`」的失活项，而显式改归属（如将来转交后重置）仍需主动删路径。对称接口、Simplicity。

**影响**：领域 / API 面多一个对称方法，无害；M7/M8 可用。

## D-V1.1-07 · M5 惰性清理放路由层注入存活集合

**背景**：惰性清理需判「会话是否存在」，存活判断依赖内核 `SessionManager.listAll()`；但领域层要求内核无关、可单测。

**可选项**：A 领域层 `pruneMissing(map, liveSet)` 纯函数 + 路由层经 `listAllSessions()` 注入存活集合；B 领域层直接调内核 `listAll`。

**选定：A**。

**理由（对照北极星）**：不变量①「不改内核」+ 领域层内核无关约定 —— 把内核耦合关在路由层，领域层只收 `Set<string>`、可纯函数单测；B 让领域层贴内核、难测。

**影响**：`getOwner/getMain` 单项读不做清理（无需全量存活集合）；惰性清理只在 GET 整图时发生，符合「读映射时丢弃已不存在项」。

## D-V1.1-08 · M4 二级菜单用浮层而非翻面

**背景**：M4 详细设计写卡片点击「就地展开二级菜单（翻面/浮层）」，含模型/技能/工具/思考强度现场编辑。实现选哪种？

**可选项**：A 浮层（点卡片弹独立浮层菜单，含完整配置表单）；B 翻面（卡片 3D 翻转，背面放菜单）。

**选定：A 浮层**。

**理由**：卡片 `aspectRatio:1` 仅约 150px，**塞不下技能多选等配置表单**；浮层是详细设计明列的分支之一（「翻面/浮层」），非越界；复用现有表单字段（`AgentFields`），Simplicity。

**影响**：卡片瘦成整卡可点（仅色块 + 真名），点击进浮层菜单三段（起会话 / 配置 / 删除）；新增 testid `agent-menu`/`agent-back-btn`，移除 `agent-edit-btn`/`agent-start-btn`（E2E 脚本须改用点 `agent-item` 进菜单）。

## D-V1.1-09 · M7 主对话懒创建 = 标首个落地会话为 main（不预建空会话）

**背景**：5.2 要每项目一个固定「主对话」。会话无 `agentId`、且普通会话**首条消息才落盘**（点 New 只前端记 cwd、不产生真实 id；发首条后 `POST /api/agent/new` 才生成）。主对话怎么来？

**可选项**：A 进项目时预先懒创建一条空 main 会话；B `mainSessionId` 为空时，把**首个落地的普通会话**标为 main。

**选定：B**。

**理由（对照北极星）**：A 在无首条消息时强建空会话 = 撞 B4「幻影空会话」坑、且违 pi-web「首条消息才落盘」设计；B 零新增建会话机制、完全复用现有链路（`handleSessionCreated` 是新会话拿真实 id 的唯一汇聚点）。Simplicity。

**影响**：判定抽纯函数 `pickMainOnSessionCreated(map, sid)`（空 main + 非空 sid → sid，否则 null）；挂 `AppShell.handleSessionCreated`。**关键边界**：`handleSessionCreated` 加 `claimMainIfEmpty` 参数——agent 起的会话（`handleAgentSessionStarted`）传 **false**，防项目首条恰是 agent 会话时被误标 main 而从 agent 分组消失；普通会话路径默认 true。
