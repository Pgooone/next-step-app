# Next-Step V1.1 设计决策记录（ADR）

> 本文记录 V1.1 实现阶段的 **lead 机制层决策**（详细设计里标 🔧 的关键几条）。
> **为什么由 lead 决定、而非问用户**：按北极星「冲突分层响应」—— 这些是**实现级取舍**（不触使命、不破任何不可破不变量），故 lead 自主拍板并在此留痕；只有触及使命 / 不变量的冲突才必须停下问用户。
> 每条记：背景 → 可选项 → 选定 → 理由（对照北极星）→ 影响。编号沿用 V1 `tasks/v1-history/decisions.md` 的 `D-` 风格，本期前缀 `D-V1.1-`。

## 对照基准（北极星浓缩）

- **使命**：本地 · 单用户的「多 Agent 软件工厂」；灵魂 = 多 Agent 自定义 + 协作产文档型产物 + **人对产物细粒度可控可回溯**。
- **不可破不变量**：①不改 pi 内核（只封装）；②产物改动必经按块确认；③纯文件落盘、本地单用户；④并发会话 ≤ 3；⑤复用 pi-web 既有能力、不另起炉灶。
- **设计哲学**：Simplicity First（最小代码、不投机抽象）、未决设计不预实现。

## D-V1.1-01 · M5 会话↔agent 映射的存盘位置

**背景**：会话数据结构无 `agentId`（`lib/types.ts:174-184`），且不可改 pi 内核的会话文件 → 必须在领域层另存「会话 → agent / 主对话」映射。存哪？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 项目本地 `<cwd>/.pi/ns-session-map.json` | 与项目其它 Next-Step 数据（`.pi/agents`、`.pi/artifacts`）同处；随项目走；契合既有 `.pi/` 约定 | 多会话并发写需原子写 |
| B | 中央 `~/.pi/agent/ns-session-maps/<encoded-cwd>.json` | 与内核会话存储（按 encoded-cwd）并行 | 复刻内核 encoded-cwd 间接层、贴内核内幕；项目移动即错配 |
| C | 塞进项目注册表 `projects.json` | 不新增文件 | 注册表单一职责被污染（项目列表 vs 每会话元数据）；每次会话操作都 churn 注册表 |

**选定：A —— `<cwd>/.pi/ns-session-map.json`（原子写，仿 V1 D-04）**

**理由（对照北极星）**
- **支柱「项目即工作区」**：映射是该项目工作区的元数据，应随工作区走；放 `<cwd>/.pi/` 下，项目即自包含。B 的中央存储会在项目移动时与 cwd 错配。
- **不变量③ 纯文件落盘**：A 是纯 JSON 文件，符合；且映射是 **Next-Step 领域层自己的数据**，而领域数据（agents/artifacts）一向落在 `<cwd>/.pi/` —— A 与既有约定一致。
- **不变量① 不改内核**：B 要复刻内核的 encoded-cwd 方案 = 贴着内核内幕走、易碎；A 只在领域层加一个文件，零内核耦合。
- **Simplicity First**：A 最小、最一致；C 让注册表越权背单一职责。
- **不变量④ 并发会话 ≤ 3**：A 用原子写（temp + rename）即可防自己的多会话并发，符合「乐观锁防自己」。

**影响 / 备注**：内核的会话**文件**存在中央 `~/.pi/agent/sessions/<encoded-cwd>/`（内核的事，不动）；我们的**映射**是领域层叠加物、项目本地。项目若移动路径，内核会话本就按旧 cwd 失联（内核既有行为），A 不会让此更糟。

## D-V1.1-02 · M6 项目首页的渲染方式

**背景**：要在进工作台前先有「项目卡片墙」首页。现状是单页 `AppShell` + 左栏项目下拉。

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 单页内按 `currentProjectId` 二选一渲染（`app/page.tsx`：空→`ProjectHome`，有→`AppShell`） | 保持 pi-web 单页架构；复用现有 currentProjectId 状态；最小改动 | 首页与工作台共一 URL（无项目深链） |
| B | 新增 `/projects` 路由 + `/project/[id]` 工作台 | 可深链到具体项目 | 引入 pi-web 本没有的路由层；偏离既有单页模型；过度工程 |

**选定：A —— 单页内按 `currentProjectId` 二选一渲染，不加新路由**

**理由（对照北极星）**
- **不变量⑤ 复用 pi-web 既有能力、不另起炉灶**：pi-web 是**刻意的单页应用**（全应用只有 `/` 一个路由，URL 顶多挂 `?session=`，见前端解析报告 §0/§1）。A 留在这个模型里；B 凭空加一层路由 = 另起炉灶。
- **Simplicity First**：A 只是「currentProjectId 为空就渲染另一个组件」，复用现成状态、几乎零新机制；B 要建路由文件、迁移导航 / URL 状态。
- **使命 本地 · 单用户**：本地单用户工具，项目深链不是核心诉求，为它引入路由层不划算。

**影响 / 备注**：若日后确实要项目深链，可仿现有 `?session=` 加一个 `?project=` query 参数实现，无需全套路由 —— 保留为非破坏性的未来选项（符合「未决设计不预实现」，现在不做）。

## D-V1.1-03 · M8 @agent 转交的数据打包与投递

**背景**：主对话 `@agent` 要把「全主对话历史 + 附件（发送时可勾选）」转交到该 agent 的**单独会话**。怎么打包、怎么投递？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | 历史序列化为 `<context source="主对话">…</context>` 文本 + 文件按 M2 的 `<file>` 内联（浏览器上传无绝对路径，`name` 用文件名），作为目标会话**首条消息**经现有发消息链路投递 | 走内核自己的发消息路径、不碰会话文件内幕；与 M2 同一套内联格式；内容在目标会话里可见可回溯 | 历史序列化格式需实现时定细节 |
| B | 直接把结构化历史写进目标会话的 `.jsonl` | 保真度高 | 复刻内核消息格式、做会话文件手术 = 贴内核内幕、易碎 |
| C | 只发一个「见主对话会话 X」的指针 | 最轻 | 不满足需求（用户要把**内容**带过去） |

**选定：A —— `<context>` 文本 + `<file>` 内联，作目标会话首条消息，经现有发消息链路投递**

**理由（对照北极星）**
- **不变量① 不改内核**：A 用内核**自己的**发消息路径（`session.prompt` / `POST /api/agent`），消息格式由内核负责、我们零内幕耦合；B 自己写 `.jsonl` = 做内核会话文件手术，随内核升级易碎。
- **不变量⑤ 复用 pi-web 既有能力**：A 复用现成「向会话发消息」能力；B 绕过它另搞一套。
- **使命「细粒度可控可回溯」**：A 把转交内容作为一条**可见消息**留在目标会话里，透明、可回溯；B 可能把它藏进文件、不透明。
- **Simplicity First + 一致性**：A 复用 M2 已定的 `<file>` 内联格式，与对话框上传同一套，少一套机制。
- C 不满足需求（要带内容、非指针），淘汰。

**影响 / 备注**：`<context>` 的精确序列化（是否含工具调用块、角色如何标注）属 M8 实现期细节；本条拍板的是**机制**（文本内联 + 走正常发消息链路），非逐字格式。目标会话不存在时先按需起会话（M7 链路 + M5 标记归属）再投递。

## D-V1.1-04 · M8 转交的投递机制（起新会话 vs 复用既有会话）

**背景**：D-V1.1-03 定了「怎么打包载荷」，但「投递到该 agent 单独会话」一句过抽象。代码现实里只有两条真实链路：① 起**新会话** `POST /api/projects/[id]/agents/[agentId]/session`（载荷作 `firstMessage`，V1 已验收、不碰 `.jsonl`）；② 向**已存在会话** `sendAgentCommand(sid, prompt)`（但要求该会话 live 注册，落盘但未 live 的会话无 re-attach 链路）。选哪条？

**可选项**

| 选项 | 做法 | 利 | 弊 |
|---|---|---|---|
| **A** | **转交 = 起新会话**：`POST /api/projects/[id]/agents/[agentId]/session`，载荷作 `firstMessage`；复用 `handleAgentSessionStarted` 切到该会话 + 接 SSE | 走 V1 已验收的起会话端点、不碰会话文件内幕；一举解决「投递路径」+「转交后接 SSE 流」；天然经 M5 标记归属 | 每次转交起一个新会话（符合「单独会话」语义） |
| B | **复用既有会话**：`sendAgentCommand(sid, prompt)` 向该 agent 已有会话追加 | 不新增会话 | 要求会话 live 注册；落盘但未 live 的会话**无 re-attach 链路**，需先补一个按 `sid` 的 re-attach 端点才可行 |

**选定：A —— 转交 = 起新会话（`POST /session`、载荷作 `firstMessage`、复用 `handleAgentSessionStarted` 切会话 + 接 SSE）**

**理由（对照北极星）**
- **不变量① 不改内核**：A 复用 V1 已验收的 `/session` 起会话端点，内核负责会话创建与消息格式，零内幕耦合。
- **不变量⑤ 复用 pi-web 既有能力**：A 复用现成的「起 agent 会话 + `handleAgentSessionStarted` 切会话接 SSE」链路，不另起炉灶。
- **Simplicity First**：A 一条端点同时解决投递与转交后接流；B 还得先补按 `sid` 的 re-attach 端点，多一套机制。

**影响 / 备注**：复用既有会话（B）列为**后续增强**——待补一个按 `sid` 的 re-attach 端点后再支持「向 agent 已有会话续接转交」。本期取 A：每次 `@agent` 转交起一条新会话，契合需求「该 agent 的**单独**会话」语义。**用户已确认**（先 A、B 留作后续迭代）；B 的来龙去脉详见 `第一轮-基础迭代/后续迭代清单.md` 第 1 条。

## D-V1.1-07 · M5 惰性清理放路由层注入存活集合

## D-V1.1-09 · M7 主对话懒创建 = 标首个落地会话为 main（不预建空会话）

**背景**：5.2 要每项目一个固定「主对话」。会话无 `agentId`、且普通会话**首条消息才落盘**（点 New 只前端记 cwd、不产生真实 id；发首条后 `POST /api/agent/new` 才生成）。主对话怎么来？

**可选项**：A 进项目时预先懒创建一条空 main 会话；B `mainSessionId` 为空时，把**首个落地的普通会话**标为 main。

**选定：B**。

**理由（对照北极星）**：A 在无首条消息时强建空会话 = 撞 B4「幻影空会话」坑、且违 pi-web「首条消息才落盘」设计；B 零新增建会话机制、完全复用现有链路（`handleSessionCreated` 是新会话拿真实 id 的唯一汇聚点）。Simplicity。

**影响**：判定抽纯函数 `pickMainOnSessionCreated(map, sid)`（空 main + 非空 sid → sid，否则 null）；挂 `AppShell.handleSessionCreated`。**关键边界**：`handleSessionCreated` 加 `claimMainIfEmpty` 参数——agent 起的会话（`handleAgentSessionStarted`）传 **false**，防项目首条恰是 agent 会话时被误标 main 而从 agent 分组消失；普通会话路径默认 true。

## D-V1.1-10 · M8 @agent 转交的 UI 落点与单测策略

**背景**：M8 机制已定（D-V1.1-03/04），但「@ 浮层 + 勾选 UI 放哪、怎么测」是实现取舍。

## D-V1.1-12 · P0 接线命门 spike —— profile.tools 白名单 × guard 共存（go/no-go）

**背景**：P0 档位1 接线时 `profile-session-wiring.ts:105` 会把 profileOptions（带 `tools: profile.tools` 白名单，`CODING_TOOL_NAMES` 子集）与 guardOptions（`{noTools:"builtin", customTools:[同名 7 工具]}`）按 `{...profileOptions, ...guardOptions}` 合并。D2 spike 当年是裸会话、没测过叠加白名单层——这是接线能否成立的命门，实现第一步须 go/no-go，绝不盲接。

**可选项**：A 先 spike 验证再决定合并策略；B 直接按 spread 接线、出问题再补；C 预先改造（显式 merge tools / 调 spread 顺序 / 不传 noTools）规避未知冲突。

**选定：A —— spike 验证（`spike/p0-profile-guard/harness.ts`），结论 = 按 spread 直接接线可行（GO），无需方案 C 任何规避。**

**结果（三重确认：build agent 11/11 + 对抗复核员含负对照判 sound + lead 亲跑 11/11 exit 0）**：矩阵 A（全 7）/A'（含 write/edit 子集）/B（只读）/C（空）全 PASS。命门 GO——白名单与 `noTools:"builtin"+customTools` 共存下，受管写仍被拦成 PendingChange、不落盘、不产新版本，非受管放行。

**根因（内核事实，查证 sdk.js / agent-session.js）**：
- `sdk.js:132`：`options.tools` 存在时 `noTools` 被忽略，active = `[...tools]`。
- `agent-session.js:1868-1871`：toolRegistry 先塞内置、再以 customTools 按名 `.set` 覆盖 → guard 同名 write/edit 覆盖内置。
- `agent-session.js:1823`：工具进激活集 ⟺ name ∈ 白名单。
- 综合：只要 write/edit ∈ 白名单，guard 版即胜；spread 顺序无关（键不冲突）。

**影响 / 带进 p0-wire 的约束**：
1. **白名单必须含 write/edit**，否则该工具不激活——profile 配置约束、**非 guard 失效**（Case B 只读 / C 空 反证）。只读 profile 的 agent 本就改不了 artifact，wire/verify 须明确此语义、勿误读为 bug。
2. spike 复刻的是「打算接」的形态；`profile-session-wiring.ts:105` 当前 `{...options, ...createOptionsOverride}` **尚未并入 guard**。wire 必须真把 guardOptions 并进此处、customTools 用本会话 cwd（`registry.get(projectId).root`）构造，否则结论不自动迁移到生产。
3. 「guard 胜出」只能靠行为证明（C 路线 write 与内置同名同 label 不可分辨）——harness 用「受管写→pending+磁盘无文件+无新版本」+ 负对照证伪。

### D-V2-05 · 同一 artifact 并发 propose = 拒绝并存
**背景**：propose_edit 每次 `old = readCurrentContent`(当前版快照)、无锁。同一文档连发两次 propose，第二次确认基于陈旧快照**静默覆盖**第一次、`ifMatch` 取当前版永不 409。V2 把 agent 变调用方后必然放大（一轮多次 propose / 用户"分多段提议"）。
**选项**：A 拒绝并存 / B 确认时检测冲突报 409 / C 不管登记。
**选定**：**A 拒绝并存**（用户拍）。

### D-V2-06 · 外部编辑保护 = materialize 前比对、不一致抛 EXTERNAL_MODIFIED
**背景**：propose 基于 version 快照算 diff、**不读真实 crd.md**。外部（编辑器/git/带 bash 会话）改真实文件后，下次确认 `submitVersion` 物化覆盖 → 外部改动**静默丢失（数据丢失，非仅"不进版本"）**。
**选项**：A 文档警示 + 代码比对 / B 仅文档警示 / C 不管。

### D-V3-01 · 受管=侧车不可删（技术定性）+ 方案二形态与前置约束登记
**背景**：用户质疑「受管文档」是否与已删的 guard 拦截层一样繁琐、该去掉，并提出「agent 产物直接进 file panel、像 Notion 列出、去掉手动点 Artifacts 按钮选文档」。lead 跑 ultracode 6-agent 工作流（4 并行精读 + 综合 + 对抗反方）核查。

**技术定性（核实，免后续重复质疑）**：
- 「受管文档」= 项目根真实 .md（物化文件、当前版投影）+ 旁挂侧车 `.pi/artifacts/managed/<id>/`（`artifact.json` + `versions/<n>.json` 永不覆盖快照 + `pending/<id>.json` 待确认）。用户想要的**版本 / 块级 diff / 按块确认 / 回滚 / 外部改动防覆盖全部由侧车承载**（artifact-service.ts:240-251,307-348）；真实 .md 自身存不下旧版全文与未决变更。**删受管 ≡ 删侧车 ≡ 删这些能力**，自相矛盾。

### D-V3-02~08 · 方案二详细设计的实现级取舍（ultracode 8-agent 设计 + 对抗定稿）
**来源**：2026-06-19 ultracode 设计 workflow（6 模块并行精读 + 综合 + 对抗）。完整设计见 `第三轮-受管文档入口合并/详细设计.md`。**对抗揪出 1 blocker + 2 major，已全部采纳修正。**

**D-V3-02 · 反查映射不新增端点/字段，前端用 listArtifacts 就地建集合**
`GET /api/projects/[id]/artifacts`（listArtifacts）一次请求已给全 `id+filePath?+title+kind+currentVersion`（artifact-service.ts:194-210）。受管分组点开直接用 `a.id`（**根本不经路径反查**）；反查只为「普通树去重」。弃因：新 by-path 端点是多余 I/O（服务端反查同样扫 managed/*）；后端下发 absolutePath 会冗余进存储且 projectRoot 改名漂移；用 title 做键因重名物化避让(-2.md)不可行。

**D-V3-03 · 受管分组 + 去重逻辑放 FileExplorer 内部**（非 SessionSidebar 独立区）
去重须在「同时持受管列表 + 普通树两份数据」处做，FileExplorer 是唯一同时渲染普通树者，自取 listArtifacts 最内聚。受管项是扁平列表、不塞 TreeNode 懒加载。形态 = 顶部受管分组 + 下方普通树两相邻 UI 块，吻合用户拍板的「切两层」。

**D-V3-04 · 去重 key = 绝对路径，非裸名（⚠️ 对抗 blocker 修正）；去重为本期必须**
`managedAbsPaths = Set(artifacts.filter(a=>a.filePath).map(a=>normalize(joinFilePath(projectRoot,a.filePath))))`，对普通树节点按 `node.fullPath` 精确相等剔除。**原综合选「裸名 node.name」是 blocker**：FileExplorer 的 cwd=`selectedCwd`（用户/会话可切，AppShell.tsx:407 + SessionSidebar:539/:650）**不恒等于 projectRoot**，裸名在 `cwd≠projectRoot` 时会误剔同名文件 / 张冠李戴打开错项目 artifact / 子目录 artifact 漏剔。绝对路径 key 一举解决三者且更简。`.filter(Boolean)` 跳过旧 artifact 的 undefined filePath。去重**必须**（不去重则 crd.md 物化在项目根必与普通树双现、且双入口致 SSE 冲突复活，见 D-V3-06 命门）。

**D-V3-05 · HITL「点开即受管」对纯浏览天然安全，零额外设计**
PendingChangeCard 判据 = `selectedArtifactId && pendingChanges.length>0`（双重与，PendingChangeCard.tsx:46），pending 由真实侧车数据驱动（listPendingChanges 空目录返回 []，pending-change-service.ts:347）而非「打开/选中」入口动作。新入口调同一 `open(artifactId)`、数据流与今天一致，故纯浏览不冒确认卡。反方担忧不成立、经核实证伪。

**D-V3-06 · SSE↔EXTERNAL_MODIFIED 冲突降级，ArtifactPanel 不引 live SSE；本轮默认 SSE 方案 A**
受管走 ArtifactPanel（无 SSE、读侧车）、普通走 FileViewer（有 SSE、读磁盘）两视图互斥（AppShell.tsx:960），「同一视图两套相反语义」不再出现 → 核心冲突消解。给 ArtifactPanel 加 SSE 会反噬机制层（要建 .md→侧车反向同步）且与「拒绝静默覆盖」再对冲，**绝不引**。残留缺口=侧车版 vs 磁盘 .md 可能静默偏离、仅 submitVersion/rollback 抛 409 暴露。**方案 A**=接受 409 toast 为唯一提示（现成链路 useArtifactStore.ts:211、最小、机制安全不丢数据）；方案 B（打开时只读探测 externalModified 挂黄条、零 SSE）**移出本轮、登记后续**（与主题正交、省 4 文件改动）。lead 定本轮走 A，用户可否决改 B。

**D-V3-07 · .pi 隐藏 = 后端 IGNORED_NAMES 加 ".pi" 一行；省 route 级单测**
type=list 是数据源收口点（route.ts:485），后端一处加最彻底（前端多消费者逐个滤易遗漏）。read/meta/watch 分支不查 IGNORED_NAMES，故派发产物 type=read 读 `.pi` 仍正常（DispatchPanel）。物化 crd.md 在项目根、与 `.pi` 判定不相交仍显示。不迁移侧车（违红线）、不通配 dotfile（误伤 .env/.gitignore）。全仓无 files route 既有测试，搭 route 级脚手架（NextRequest mock/临时夹具）不划算，靠 T4 真浏览器验收（树里无 .pi + 派发产物仍可开）。

**D-V3-08 · projectId/projectRoot 由 SessionSidebar 自取透传，不经 AppShell 冗余传（对抗 major① 修正）**

### D-V3-09 · T2 承重墙实现级取舍（实现 + ns-v3-review 只读对抗审查定稿）

### D-V4-01~04 · 删除受管文档（第四轮）实现级取舍
**来源**：2026-06-20，承接 QA `QA/开发/删除受管文档决策.md`（用户拍板 A1 彻底删除 / B2 两处入口 / C1 仅用户可删）。详设 `第四轮-删除受管文档/详细设计.md`。

红线「artifact 所有修改必经 PendingChange→按块确认」约束的是**内容修改**（propose_edit）。删除是生命周期操作，与 `createArtifact` 同属结构操作、不走 propose。明确登记防后人误判违红线。AI 不得直删（决策 C1）。

删一个 version 已被并发改过的文档 → 409、提示先刷新；Header 缺 If-Match 则宽松不校验。单用户低频但与既有写操作一致、成本低。入口②删非当前打开项可先不带 If-Match（登记）。
