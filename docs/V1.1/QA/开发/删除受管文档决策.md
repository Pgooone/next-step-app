# 删除受管文档功能 · 决策（用户拍板逐条留痕）

> 记录：2026-06-20。承接第三轮「受管文档入口并入 file panel」收官，用户提出受管文档生命周期缺「删除」。
> 格式：每条记 **背景 → 可选项（全列）→ 推荐 + 理由 → 谁拍 → ✅ 最终选择**。
> 配套：lead 实现级取舍见 `../../设计决策记录.md` D-V4 段；详设 `../../第四轮-删除受管文档/详细设计.md`；总索引 `../00-索引.md`。

## 背景

受管文档生命周期现状（lead 落码核查）：**生（createArtifact）+ 改（propose_edit→按块确认）+ 版本（submitVersion/rollback）齐全，独缺「删」**——`ArtifactService` 无 deleteArtifact、API 全是 GET/POST 无 DELETE、`useArtifactStore` 无 delete action、AI 提议工具无 delete。

存储对删除友好：一个受管文档的全部状态都在单一目录 `<root>/.pi/artifacts/managed/<id>/`（`artifact.json` + `versions/` + `pending/`），`rmSync(dir,{recursive})` 一刀清；物化 .md 单独在 `<root>/<filePath>`。

---

## 决策 A · 删除语义（是否销毁磁盘 .md）

「删除受管文档」中文歧义：删[受管文档]（文档没了）还是删[受管]（取消受管、文件留）。

**可选项**
- **A1 彻底删除**：删侧车 + 删磁盘 .md，文档整个消失。最贴「删除」直觉、无普通树重现困惑；代价=销毁文件 + 全部版本史，不可恢复。
- **A2 取消受管（降级）**：仅删侧车，.md 降级为普通文件、重回普通树。保住当前内容、只丢版本史；但有「删了怎么还在」困惑。
- **A3 两者皆给**：「删除」与「转普通文件」两个动作。覆盖两种意图；UI/决策更重、可能超本期必要。

**推荐**：A1。V3 已把物化 .md 从普通树去重隐藏，用户眼里受管文档就是「一个东西」，删它该整个消失；最符合 Notion「删除页面」直觉；靠二次确认 + 不可恢复警示兜底。

**谁拍**：用户。
**✅ 最终选择：A1 彻底删除（侧车 + 磁盘 .md 双删）。**

---

## 决策 B · 删除入口落点

**可选项**
- **B1 仅 ArtifactPanel 头部**：与版本下拉 / rollback 同处，复用 D-D5-5 两步二次确认；打开文档后删，语境清晰、UI 最省。
- **B2 头部 + 受管分组行悬停垃圾桶**：额外在受管分组行 hover 出垃圾桶，不打开也能删；更便捷但多一处 UI 与状态处理（行内删须本地刷新 + 处理「删的正是当前打开项」）。

**推荐**：B1（先最省）。

**谁拍**：用户。
**✅ 最终选择：B2 —— 头部按钮 + 受管分组行悬停垃圾桶都做。**（两处都接同一 `store.delete()`/同一二次确认范式，避免逻辑分叉。）

---

## 决策 C · AI 能否删除受管文档

create_artifact 是 AI 工具，对称看 AI 也可有 delete。

**可选项**
- **C1 仅用户可删**：AI 文档会话不给 delete 工具。删除破坏性大且无 HITL，本期范围最小、最安全。
- **C2 AI 也可删（经提议确认）**：给 AI delete 但须走「提议→用户确认」，与 propose_edit 对称；工作量更大、可能超本期。

**推荐**：C1。

**谁拍**：用户。
**✅ 最终选择：C1 仅用户可删。**（若日后要 AI 删，须走「提议→确认」而非直删，另立项。）
