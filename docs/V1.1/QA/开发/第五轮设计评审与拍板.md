## Q1 · major #3 并发去重接法

**背景**：dev 重启 / idle 回收后 `__piSessions` 空，前端「流式中刷新」会几乎同时发 `GET .../events` 与
`POST .../[id]`，两路由 fast-path 同时 miss → 各跑一次 reattach → 对同一 jsonl 并发建会话 → 后者覆盖前者，
前者成孤儿；孤儿 10min 后 onDestroy 闭包 `registry.delete(realSessionId)` 会**误删现归第二个 wrapper 的
registry 条目** → 级联反复重建，且重建出的可能是 generic（丢 doc 工具）→ **反噬本轮主线**。现状 generic 路径
用 `__piStartLocks` 去重，但本轮新增的 reattach 路径无锁。

**可选项（全列）**：
- **A 统一锁 + 抽内层函数**：resolver 层统一复用 `__piStartLocks`，把 generic 分支抽成不带锁的
  `startRpcSessionInner`，新老路径共用同一把锁。最稳，彻底消除双建 + 级联误删。代价：改动 `rpc-manager.ts`
  既有结构（抽函数，行为不变），轻微动「机制层一行不动」承诺。
- **B reattach 内自带 per-sessionId 去重**：只在 `reattachProfileSession` 内加锁，不碰 rpc-manager（更
  surgical）。代价：与 generic 路径不共享同一把锁，「档案被删→降级 generic」容错分支分叉时仍有极小双建窗口
  （而本轮恰要做该容错分支，正踩 B 的薄弱点）。
- **C 本轮不处理并发**：登记为已知 gap 留下一轮，本轮只做主线。代价：dev 重启后并发重建常见、且会反噬主线。

**推荐 + 理由**：**A**。① 与本轮主线强耦合（孤儿误删会让刚 reattach 好的 doc 会话又被删、重建成 generic，
等于白修）；② A 比 B 彻底（B 薄弱点恰在容错降级分叉，本轮正要做该分支）；③「抽函数」改动小且安全（原样提取、
行为不变），换来根本正确性；`__piStartLocks` 是会话生命周期基础设施，为正确性微调它不违背「机制层不动」精神
（受限集/提议工具/内核未动）。

**谁拍**：用户。
**最终选择**：**A 统一锁 + 抽函数**。（lead 先用人话解释三选项 + 推荐 A，用户采纳。）

## Q2 · 评审通过后如何推进

**背景**：评审为 `ready_with_fixes`，4 major + spike 死分支 + 7 minor 需订正。

**可选项（全列）**：
- **A 订正文档 → agent team 实现**：lead 先把订正回写详细设计 + 任务卡，然后用 agent team（非 subagent）
  串行实现 T1→T3，每卡门禁绿即细粒度 commit、T3 走真浏览器 + 重启复现回归。

- **C 设计需更大改动**：评审问题超出「实现时订正」范围，需重新设计。

**推荐 + 理由**：lead 倾向 A（无硬 blocker、订正均可随实现落实），但尊重用户「先出设计再实现」的一贯节奏。

**谁拍**：用户。
**最终选择**：**B 仅订正文档、暂停待复审**。lead 已完成订正（详细设计 §〇 + 任务卡 + progress），等用户
复审订正版后再开 T1。
