## 四、关键设计决策（为什么这么做）

### 1. 为什么「新写」而非改造现有引导？
两份受众不同、各司其职，**不要合并**：
- 根目录 `新手引导.md` = **面向使用者**（产品视角：能干什么、优势、怎么上手；去掉了迭代痕迹和内部实现）
- `docs/第一轮-基础迭代/新手引导.md` = **面向团队**（工程视角：实现原理、当前流程 gap），保留作进阶

### 2. 为什么改品牌只动前端两个文件？
`grep "Pi Agent Web"` 确认该字符串**只在前端组件**（`ChatWindow.tsx`、`SessionSidebar.tsx`），内核包 `node_modules/@earendil-works/*` 里没有。所以改前端显示字符串**不违反「不改 pi 内核」红线**。欢迎态打字机俏皮话数组（"fix that pesky bug" 等）**保留原样**——用户认为「挺好」。

### 3. 截图为什么要「干净数据」重做一遍？
初版截图夹带测试痕迹（项目名时间戳后缀、一堆「种子会话」、agent 自报「我是 pi 编程助手」与文档工厂定位冲突）+ 显示旧品牌。重做后：品牌统一 Next-Step、项目墙只留 3 个干净命名项目、05 背景是 agent 真实读文档整理要点的漂亮输出。

## 七、如需重新截图（配方 + 踩过的坑）

界面以后若有改动需要重截，参考这套方法（基于 `.claude/skills/browser-e2e`）：

**环境**：本机已缓存 playwright chromium **headless-shell**（必须用它、不是 full chrome，否则缺 GTK/X11 崩）；`run-e2e.sh` 会复用已在跑的 dev（端口 30141）、跑完不杀。

**造数据**：用 `lib/domain` 的 `ProjectRegistry` / `ArtifactService` / `PendingChangeStore`（参考 `scripts/d4-e2e-fixture.mts`），tsx 直接造项目 + 受管文档 + pending（del/mod/add 三块需用未变行隔开才不被 LCS 并块）。

**5 张要截的界面**：项目墙 / 工作台 / Agent 配置面板（点卡片弹出）/ 块级确认（受管文档行内三色高亮）/ @转交浮层。

**踩过的坑（重要）**：
- **项目墙**会显示 `~/.pi/projects.json` 里**全部**项目（含几十个历史测试项目）→ 截前临时过滤只留目标项目、`finally` 恢复。`ProjectRegistry.list()` 每次 `readFileSync`、**不缓存**，改文件后 reload 即生效。
- **块级确认面板**首开会卡「加载产物…」→ 必须等它消失 + 「N 处待确认」出现再截，别用定长 sleep。
- **@转交浮层**只在「主对话」语境挂载，主对话的 `sessionId` 记在 `<projectRoot>/.pi/ns-session-map.json` 的 `mainSessionId`。
- **主对话通用 agent 收到任务会跑 agentic loop**（去 `read` 文件），会卡在「Waiting for model」久不回；点 Stop 终止又会破坏 @浮层挂载。**最稳的办法**：让那个 loop 自然跑完，再「恢复该已完成会话 + 打 @」截图——背景反而是 agent 整理文档的真实漂亮输出。
- pi 会话存储在 `~/.pi/agent/sessions/<cwd 编码>/*.jsonl`。
- **OOM 约束**：本机 3.4G 无 swap，截图必须串行、单 dev + 单 chromium、**绝不发消息触发 agent 长流式**（会硬崩）。
- **验证 push 别信回显**：本机 git/工具回显偶有不可靠，确认远端状态一律以 `gh api` 或 `git ls-remote` 直查为准。
