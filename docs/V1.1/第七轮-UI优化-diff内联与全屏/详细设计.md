## 〇、缘起与原始需求（用户原话）

> **A1**　file panel 可以全屏化。因为文件内容日渐增多，右侧固定大小的已经满足不了需求。从侧屏切换到全屏时我想要比较好的动画效果，并且我想要悬浮于主界面的二级菜单样式，可以使用 gsap skill 进行优化。
>
> **A2**　全屏化后就可以在原文中展示 diff 等等功能，而不是孤零零的展示哪些 diff 可以确认删除。这样失去了 diff 参考上下文的作用。
>
> **A3**　优化 diff 在对话框内的性能表现，每次需要刷新或者需要等一会才能在对话框显示。并且链接上对话框内的 diff，左键鼠标点击在 file panel 中可以立即跳转到原文修改地，这样与上面第二点呼应上。

## 一、调研结论一句话 + 承重墙

**三需求高度可行、全是纯前端展示/交互层改动，逐条核验不破任何红线（不碰 `lib/pi` 内核、写盘仍唯一经 `resolveAndMaterialize`、不开新会话、不引 DB）。** 工作量约 4~5 人日。

**关键洞察：A2 的「原文内联 diff」内核早已存在**——`lib/artifact-view/anchor.ts:43 buildSegments` + `components/ArtifactPanel.tsx:415 InlineHighlightView`/`:456 HlSegment`（D3 时建好、经真浏览器验过）已能把 pending 块用「完全相等连续子序列 + 单调游标」就近锚到原文行做行内高亮（add 绿 / del 红删除线 / mod 黄）。本轮不是从零造内联，而是三件事：

> **承重墙 =「`DiffBlock.id` 作唯一 join key 贯穿三处 DOM」+「`Segment.hl` 补 `changeId` 上下文」+「store 单调 nonce 信号」**
>
> 1. **join key 现成且稳定**：`DiffBlock.id` 是 `randomUUID`（`lib/domain/pending-change-service.ts:131` 生成、落盘 JSON、跨刷新稳定），同一 id 出现在三处且一致——①对话框卡片 `PendingChangeCard`（`:294` key=b.id）②原文内联段 `HlSegment`（`anchor.ts` `Segment.hl` 携带整个 block）③并排 Diff 块 `DiffBlocksView`（`ArtifactPanel.tsx:521` key=b.id）。A2（建锚）与 A3（用锚）共用同一把钥匙，无需新算法。
> 2. **块↔原文位置 = `buildSegments` 实时算、用完即弃**：`findSubsequence` 算出 start 行号、切段后即丢、**不入持久层**。这刻意与写盘解耦。
> 3. **唯一数据结构补强**：`Segment.hl` 补 `changeId`（resolve 需 changeId+blockId）+ 新增保留 changeId 的 `useShallow` selector。纯渲染/锚定层，不进写盘路径。

### 铁律（红线守门）

- **只借 `block.id` 当 DOM 锚（渲染层只读），绝不给 `DiffBlock` 加行号/索引让写盘依赖。** 写盘 `applyResolvedBlocks`（`pending-change-service.ts:216`）靠「重放同一 LCS + 按 ordinal 对齐 blockIdx↔编辑组」、不读任何位置，且自带失配抛 `INVALID` 保护。一旦让写盘依赖新增位置字段，错位会污染重建内容。
- 内联段就地 ✓/✗ **仍走** `POST /api/artifacts/[id]/pending/[changeId]/resolve` → `resolveAndMaterialize`（`:411`，唯一写盘点），前端绝不直接改 content。

## 三、详细设计（按任务卡 T1~T5）

### 3.1 T1 · A3-性能「propose 后 diff 自动刷新」（独立、最先做）

**根因（已 lead 独立复验：`useAgentSession.ts` 全文 `refresh` 零命中）**：对话框 diff 卡片 `PendingChangeCard`（挂 `ChatWindow.tsx:456`）唯一数据源 `useArtifactStore.pendingChanges`，只由 `open()`（store:122）/`refresh()`（store:153）两条路径 fetch 填充。`propose_edit`（`lib/pi/doc-tools.ts:185`）会话内同步落盘 PendingChange JSON 后，**没有任何机制通知前端 store 重拉**——`agent_end`（`useAgentSession.ts:254`）不调 refresh，`AppShell.handleAgentEnd`（`:316`）只 bump 会话列表/文件树 refreshKey。故用户必须「重新点一下该 artifact」才看到 diff。

**方案（精准 gate 的 agent_end refresh）**，改 `hooks/useAgentSession.ts`：
- 新增 `proposedThisTurnRef = useRef(false)`；
- `agent_start`（:249）清零；
- `tool_execution_end`（:302，已有 `event.toolName`，见 `:294`）识别 `toolName === "propose_edit"`（注册名见 `doc-tools.ts`）→ 置位；
- `agent_end`（:254 末尾）`if (proposedThisTurnRef.current && useArtifactStore.getState().selectedArtifactId) useArtifactStore.getState().refresh();` 后清零。

理由：`refresh()` 已设计为静默（不重置 viewMode、不亮 loading，store:73-76），适合自动调用；仅「本回合确实 propose 且面板已开」才刷、零无关 fetch；复用 D-D5-2「自己触发后直接 refresh、不动 SSE switch」先例，零内核改动。时序保证：pendingStore.save 同步 → execute resolve → tool_execution_end → agent_end，故 agent_end 时 pending 必已落盘，无需轮询/setTimeout。

**边界**：artifact 未 open（`selectedArtifactId` 为 null）时 refresh 是 no-op、卡片本就 `return null`（:46）——维持现状（D-UI-06）。

### 3.2 T2 · 承重墙数据层（`anchor.ts` + `useArtifactStore.ts`）

- `lib/artifact-view/anchor.ts`：`Segment` 的 `hl` 变体增 `changeId` 字段；`buildSegments` 入参从扁平 `DiffBlock[]` 改为带上下文 `{block, changeId}[]`（或等价），使内联段就地 resolve 拿得到 changeId。**仅渲染/锚定层，写盘路径一行不动。** → verify：`anchor.test.ts` 锚定不回归。
- `lib/stores/useArtifactStore.ts`：新增保留 changeId 的派生 `selectPendingBlocksWithChange`（**订阅侧必须 `useShallow`**，否则 D-D3-10 无限重渲染、单测抓不到只在真浏览器崩）；新增 `focusBlockId: string|null`（初 null）+ `focusBlockNonce: number`（初 0）+ action `requestBlockFocus(blockId)`（`set` 切 `viewMode='inline'`、写 focusBlockId、nonce+1）。**完全复刻** `diffFocusNonce`/`requestDiffFocus`（store:54/187）。focusBlockId/Nonce 是标量字段、不返回新数组，无 useShallow 风险。

### 3.3 T3 · A2 内联渲染 + 就地确认 + A3 锚点消费（`ArtifactPanel.tsx`）

- `HlSegment`（:456）根 div 加 `data-block-id={seg.block.id}`（A2 内联块锚 + A3 跳转主落点）；`DiffBlocksView` 块 div（:521）也加 `data-block-id={b.id}`（兜底：降级态/diff 视图下定位）。
- `HlSegment` 角标区加就地 ✓/✗（pending 显示、已决显示状态标），onClick 调同一 resolve API + `refresh()`，与 `PendingChangeCard.resolve`（:127-166）同链路——抽 `useResolveBlock(artifactId)` hook 共用，避免两份 fetch 漂移（见 T4）。
- `effectiveMode`（:151）在**全屏态**默认/强制 `inline`（读 A1 的 `rightPanelFullscreen` 标志，D-UI-01）；全屏态降级阈值放宽到 ~80（D-UI-05）。
- 新增 effect 订阅 `focusBlockNonce`：`>0` 时 `contentRef.current.querySelector('[data-block-id="'+CSS.escape(focusBlockId)+'"]')` → `scrollIntoView({behavior:'smooth', block:'center'})` + 短暂高亮脉冲（复用 globals.css `saved-pop` 思路或临时 class 定时移除）。**照抄 `TocSidebar.jump`（:366-368）已验证范式。**
- 保留 `InlineHighlightView`（:424）顶部 `unaligned` 兜底提示（锚不到的块提示切并排 Diff）。
- 历史版（viewingHistory）分支保持只读（D-D5-4），就地 ✓/✗ 与 data-block-id 仅在「跟随最新版」分支生效。

### 3.4 T4 · A3 源端 + AppShell 接线（`PendingChangeCard.tsx` + `AppShell.tsx`）

- `PendingChangeCard`：订阅 `requestBlockFocus`，经 ChangeCard 透传到 `BlockRow`（:313/:294）；行容器 onClick 改 `onFocus(); requestBlockFocus(block.id);`（✓/✗ 按钮已 `e.stopPropagation()` 于 :380/389，不误触）。按 D-UI-04 保留卡片作总览，抽出的 `useResolveBlock` 供内联段与卡片共用。

### 3.5 T5 · A1 全屏 gsap 外壳（`AppShell.tsx` + `globals.css` + 新建 `lib/gsap-setup.ts` + `package.json`）

## 五、风险与红线

### 最大风险：A2 内联段级 react-markdown 解析保真

`InlineHighlightView` 对每个 Segment 各起一个独立 `<ReactMarkdown>`（`ArtifactPanel.tsx:446` plain / `:502` hl）。当一个 markdown 结构块（有序/无序列表、围栏代码块 ```、表格、引用块）的行被 LCS 切块切在 plain↔hl 边界时，半截结构被两个独立 parser 分别当完整文档解析——序号重置、围栏不闭合、表格断裂。这是「按段喂 markdown」范式的**内生限制、非 bug**，全屏放大正文后更显眼。

**本轮务实应对**（按代价从低到高）：(a) 文档化此限制 + 保留 `buildSegments` 的 `unaligned` 兜底（锚不到顶部提示切并排 Diff，`ArtifactPanel.tsx:424` 现状已有），验收承认跨结构块改动可能渲染降级；(b) 可选对围栏代码/表格 hl 段整块降级为纯文本高亮避免半截解析；(c) 彻底方案（放弃段级 markdown、改整篇 markdown + rehype 插件按 data-block-id 包裹高亮）工作量大、需重写 anchor→rehype，**本轮不做**。

### 次要风险

- **gsap FLIP 取态时机**：effect 内才 `getState` 会拿末态导致动画无位移 → 用 D-UI-09 受控触发范式。
- **CSS 钉宽冲突**：`globals.css:318 transition:width` + `:331 width:42vw` 须在全屏 class 内 `transition:none` + `width:100%!important` 让位 gsap。
- **D-D3-10 订阅死循环**：`selectPendingBlocksWithChange` 若每次返回新数组、订阅不 `useShallow` → 真浏览器一开即崩、单测抓不到。照 `selectPendingBlocks`（:85）既有 useShallow 范式。
- 所有 UI 集成类问题（Flip 残留 inline style、全屏切换卡死、hydration）只在真浏览器暴露。

### 红线逐条核验（全程不破）

| 红线 | 核验 |
|---|---|
| ① 不改 pi 内核 | 全部改动在 `components/`+`lib/artifact-view/`+`lib/stores/`+`hooks/`+`app/globals.css`+新建 `lib/gsap-setup.ts`+`package.json`；不碰 `lib/pi/**`、不让内核多发事件、不动 SSE switch（agent_end refresh 是前端钩子） |
| ② artifact 修改必经按块确认 | 内联就地 ✓/✗ 仍走 `POST .../resolve` → `resolveAndMaterialize`（唯一写盘点）；不新增绕过 PendingChange 的写路径、不动 DiffBlock 类型、不动 applyResolvedBlocks 的 ordinal 重放（只借 block.id 当 DOM 锚渲染层只读） |
| ③ 并发会话 ≤ 3 | 本轮不开新会话，无关 |
| ④ 无 DB/鉴权 | 跳转态是前端瞬态 store nonce（刷新归零、同 diffFocusNonce）；pending 仍落盘 `.pi/artifacts/managed/<id>/pending`，不引存储 |

## 六、验收要点（每张 UI 卡必走真浏览器，browser-e2e skill）
