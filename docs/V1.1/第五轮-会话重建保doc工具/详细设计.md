## 〇、评审修订记录（2026-06-20，ultracode 评审后订正）

本节记录「ultracode 多 Agent 评审（14 agent / 326 工具调用 / 6 维度对照真实代码核查 + 1 承重 spike + 对抗复核）」对**初版详细设计**的订正。初版被核实为 `ready_with_fixes`（方向与靶心精准、无硬 blocker），但有 4 处 major + 1 处 spike 前提错误，照初版伪码实现会**自破红线②**。本版已全部订正。

**承重 spike 结论（推翻初版前提，high confidence）**：
- **systemPrompt = 覆盖（override）**，不是叠加。内核**根本不持久化 systemPrompt**——`SessionManager.buildSessionContext()` 只返回 `{messages, thinkingLevel, model}`（无 systemPrompt 字段、全文无 appendSystemPrompt 写盘），`sdk.js` open 既有会话只 restore messages/model/thinkingLevel，systemPrompt 每次构造期由 `agent-session.js:_rebuildSystemPrompt` 从 resourceLoader **现算**（证据链：session-manager.js buildSessionContext / sdk.js:80-117,231 / agent-session.js:135→1893→555→622-651 / resource-loader.js:328-337）。
- 故初版 §4.2「覆盖 vs 叠加 spike」是**伪命题**，源头是 `profile-session-wiring.ts:17` 一条**错误旧注释**（「已落盘的 systemPrompt 仍在」）。
- **必删的有害预案**：初版「若叠加则 re-attach 只传 docOptions、不重注 systemPrompt」——jsonl 没有可回放的 systemPrompt，不重注会让**角色/记忆静默丢失**（退化成裸 generic+7工具），单测抓不到、只有真机内容对比才暴露。**reattach 必须照常走完整 `assembleProfileSessionOptions`（带注入块的 loader）。**

**4 处 major 订正**（详见各节）：
1. **§4.2 伪码 `assembleDocSessionOptions` 错层 spread + 漏 cwd** → 照抄会让受限白名单失效、泄漏 write/edit/bash。已订正为对齐 `startProfileSession:127-142`（cwd 必填、解构 `.options`、删冗余 sessionManager）。
2. **两个新函数缺 DI 注入缝** → faux hermetic 单测 AC 跑不起来。已补 `createOptionsOverride`/`docDepsOverride`/`sessionManager`/`registerInnerSession` 可选参（最硬约束：`docDepsOverride`，否则 buildDocTools 默认 `new ProjectRegistry()` 读 `~/.pi`）。

## 一、背景与根因（已确认 + 真模型复现）

文档型 agent 跟进轮调 `propose_edit` 失败（用户用新手引导 §三.8 复现）。

**根因**：follow-up 消息 / SSE 重连在内存 registry `__piSessions` 未命中活会话时，走通用
`startRpcSession(id, filePath, cwd)`（`rpc-manager.ts:301`）**重开 generic 会话**。两条 re-attach 路由
调用时 `toolNames` 为 undefined（`route.ts:30` / `events/route.ts:23` 均只传 3 参）→
`rpc-manager.ts:329` 的 `if (toolNames !== undefined)` 为假 → `:337` 不展开 `tools` 键 →
`createAgentSession({cwd,agentDir,sessionManager})`（`:333-338`）**既不带 tools 也不带 customTools** →
内核 `sdk.js:131/135` 落到默认激活集 `["read","bash","edit","write"]`。于是 re-attach 后的文档型会话：
1. **丢** `create_artifact`/`propose_edit`/`list_artifacts`（它们是 customTools，generic 路径不注入）；
2. **多出** `write`/`edit`/`bash` → 能直接写盘、**绕过按块确认（违红线②）**。

> 订正（评审 minor）：初版把 createAgentSession 标在 `:327`，实际 `:327` 是 `allCodingToolNames` 定义（toolNames 非空备用集，re-attach 走不到）；真实调用在 `:333-338`，前提是 `toolNames=undefined`。

受限工具集只在初次 `startProfileSession`（`profile-session-wiring.ts:127-142`）注入一次，re-attach 不重建。
作者已在 `profile-session-wiring.ts:15-18` 登记此 gap（D-B4-4，「本卡不修」）——但**该注释 :17「已落盘的
systemPrompt 仍在」是错误的**（见 §〇 spike 结论；本轮顺手订正）。

**复现（真模型 DeepSeek 双相）**：
- turn1 create → turn2 propose **快连**（会话仍 alive）→ ✅ 落 PendingChange。
- turn1 后**重启 dev**（清空 `__piSessions`）→ turn2 propose → ❌ 无 PendingChange（工具已丢）。

**触发 re-attach 的时机**：① idle 10 分钟销毁（`AgentSessionWrapper` idleTimer，`rpc-manager.ts:52`）；
② dev 重启 / HMR 清空 `globalThis.__piSessions`；③ 进程级 cleanup。

## 四、详细设计

### 4.1 profile 感知解析器 `resolveOrReattachSession(sessionId, filePath, cwd, deps?)`
```
1. existing = registry.get(sessionId); if (existing?.isAlive()) return { session: existing, realSessionId: sessionId }   // 活会话快路径
2. inflight = locks.get(sessionId); if (inflight) return inflight                                                        // 并发去重（方案 A，与 startRpcSession 共用 __piStartLocks）
3. const starting = (async () => {
     const agentId = getOwner(cwd, sessionId)                       // session-agent-map.ts:75，只读 bySession；main 不在此 → null
     if (agentId) {                                                 // 属某 agent → 文档型 profile 会话
       let profile = null, projectId = undefined
       try {
         projectId = new ProjectRegistry().list().find(p => normalizeRoot(p.root) === normalizeRoot(cwd))?.id   // 无 getByRoot，list+find 足够
         if (projectId) profile = new AgentProfileStore().get(projectId, agentId)                                // throw NOT_FOUND（档案/项目被删）
       } catch { profile = null }                                   // 容错：删档/删项目 → null（本轮新增的包裹逻辑）
       if (profile && projectId) {
         return reattachProfileSession({ sessionId, filePath, projectId, projectRoot: cwd, profile })
       }
     }
     return startRpcSessionInner(filePath, cwd)                     // 主对话/孤儿会话 → generic（不带锁内层，方案 A）
   })().finally(() => locks.delete(sessionId))
4. locks.set(sessionId, starting); return starting
```
- **判定依据**：`getOwner(cwd, sessionId)` 有 agentId ⟺ 它是「按档案起的会话」（M5 映射，`<root>/.pi/ns-session-map.json`，`bySession` 字段）。`mainSessionId` 是 `SessionMap` 独立顶层字段（`session-agent-map.ts:11/13`）、**永不进 `bySession`**（`setMain` 只写 mainSessionId、`setOwner` 只写 bySession）→ 主对话走 generic（与 §B 现状一致）。
- **root→projectId**：`new ProjectRegistry().list().find(...)`（确无 getByRoot，`project-registry.ts:52/68`；`normalizeRoot` 已 export，`:38`）。
- **cwd 两侧归一化（评审 minor）**：入参 cwd 来自 `SessionManager.open(filePath).getHeader()?.cwd`（route.ts:28 / events:21）未过 normalizeRoot；故比较时**两侧都过** `normalizeRoot`，避免尾斜杠/符号链接差异静默漏命中（漏命中 = profile 会话误落 generic = 悄悄丢 doc 工具，正是本轮要修的故障）。
- **容错（订正 major #4）**：`AgentProfileStore.get()` 在档案缺失抛 `AgentProfileError('NOT_FOUND')`，其内部 `agentsDir → registry.get` 在项目缺失抛 `ProjectError('NOT_FOUND')`（`agent-profile-store.ts:74-98` / `project-registry.ts:68-71`）。**仓内无 `tryGet`**——故必须 try/catch 包裹（建议只吞 NOT_FOUND 语义，IO/INVALID 等续抛，与 domainErrorResponse 分级一致），catch 置 null 后落 generic、不抛错（T2 AC③）。
- **返回类型统一（评审 minor）**：三分支（快路径 / reattach / generic）统一返回 `{ session, realSessionId }`（两路由 `const { session } = await ...` 解构，快路径不能返回裸 wrapper）。
- **DI 注入缝（订正 major #2）**：`deps?` 收 `{ getRegistry, getLocks, getOwner, lookupProjectId, getProfile, reattach, startGeneric }` 等可选依赖（生产默认走真实 import；测试注入 hermetic 实现）——否则 T2 单测要触网/污染真实 `~/.pi`。

### 4.2 `reattachProfileSession({ sessionId, filePath, projectId, projectRoot, profile, ...di })`
与 `startProfileSession` **同源装配**，**唯三差异**：① 用 `SessionManager.open(filePath)` 重开既有会话（非 create）；② **不发首条 message**（D-B4-3 的「必发首条」是为**新建**会话触发真落盘防幻影空会话；re-attach 的会话文件已在磁盘、无此问题）；③ 重应用 model/thinking。
```
const sm = sessionManager ?? SessionManager.open(filePath, undefined)
const agentDir = getAgentDir()
const { options } = await assembleProfileSessionOptions({
  projectRoot, profile, cwd: projectRoot, sessionManager: sm, agentDir,
  additionalSkillPaths: extraSkillDirs(projectRoot),
})
const { options: docOptions } = assembleDocSessionOptions({          // ← 解构 .options（订正 major #1）
  projectId, sourceActor: profile.name, cwd: projectRoot,            // ← cwd 必填（订正 major #1）
  ...docDepsOverride,                                                // ← DI 缝（订正 major #2）
})
const { session: inner } = await createAgentSession({
  ...options, ...docOptions, ...createOptionsOverride,               // ← spread 顺序 options→docOptions 不可调（D-V2-04）；删冗余 sessionManager: sm（options 已含同一 sm）
})
await applyProfileRuntime(inner, profile)                            // 重设 deepseek 等（re-attach 否则回默认 faux）
return (registerInnerSession ?? realRegisterInnerSession)(inner)     // 进 __piSessions，接事件流（DI 缝）
```
- **订正 major #1（照初版会自破红线②）**：`assembleDocSessionOptions` 真实签名（`doc-session.ts`）入参 `{ projectId, sourceActor, cwd, ...DocToolDeps }`（**cwd 必填**）、返回 `{ options: { tools, customTools } }`（**外层包 options**）。初版伪码漏 cwd（strict 编译不过）、且 `...docOptions` 未解构 `.options`（展开出字面 `options` 键，顶层 tools 仍是 profile.tools，7 名受限白名单根本不覆盖 → 若 profile.tools 含 write/edit/bash 即泄漏、3 提议工具又被内核按白名单过滤掉，`agent-session.js:1818-1831`，D-V2-04）。正确写法照搬 `profile-session-wiring.ts:127-142`。
- **spread 顺序**：`{...options, ...docOptions, ...createOptionsOverride}`——`docOptions.tools`（7 受限白名单）必须排在 `options.tools`（profile.tools）**之后**覆盖（D-V2-04 命门）。删初版末尾冗余的 `sessionManager: sm`（`assembleProfileSessionOptions` 返回的 options 已含同一 sm，`agent-profile-session.ts:206`）。
- **systemPrompt（订正 spike 死分支）**：内核 systemPrompt **不持久化、永远由 resourceLoader 现算**（见 §〇）。故 re-attach **必须**复用 `assembleProfileSessionOptions` 产出的同一 loader（带 `appendSystemPromptOverride` 重新注入 agent.md 角色块 + memory），得到与初次**完全一致**的 systemPrompt，无叠加、无重复。**初版「若叠加则只传 docOptions、不传 systemPrompt override」预案已删**——它有害（jsonl 无可回放 systemPrompt，不重注 = 角色/记忆静默丢失）。
- **DI 注入缝（订正 major #2）**：追加可选 `sessionManager` / `createOptionsOverride` / `docDepsOverride` / `registerInnerSession`（全 optional，默认走真实实现）。**最硬约束是 `docDepsOverride`**：`getActiveToolNames()` 断言绕不开 `buildDocTools` 的默认 `new ProjectRegistry()`（`doc-tools.ts` 读 `~/.pi`），不注入则 T1 单测非 hermetic。
- `registerInnerSession` 已含 idleTimer + onDestroy 摘除（沿用），无需新机制。

### 4.3 两路由接线
两路由 not-alive 分支由 `startRpcSession(id, filePath, cwd)` 改为 `resolveOrReattachSession(id, filePath, cwd)`。其余（cwd 取 `SessionManager.open(filePath).getHeader()?.cwd`、404、SSE/命令流）不动。`/api/agent/new` 不改（仍调 `startRpcSession`，新建主对话）。

### 4.4 并发去重（方案 A，用户拍板）
**问题**：dev 重启/idle 回收后 `__piSessions` 空，前端「流式中刷新」会几乎同时发 `GET .../events`（connectEvents）与 `POST .../[id]`（sendAgentCommand），两路由 fast-path 同时 miss → 各跑一次 `resolveOrReattachSession` → 对同一 jsonl 并发 `createAgentSession` → 各调 `registerInnerSession`，后者 `registry.set` 覆盖前者；前一个 wrapper 成孤儿但已 `start()` 订阅事件+起 idleTimer，且**孤儿 onDestroy 闭包 `registry.delete(realSessionId)`**（`rpc-manager.ts:290`）到点会删掉现归第二个 wrapper 的 registry 条目 → **级联反复重建**。

**方案 A（统一锁 + 抽内层）**：
1. 抽 `lib/rpc-manager.ts`：把 `startRpcSession` 的 `:316-353` 建会话主体（IIFE 内容）提取为**导出的不带锁内层** `startRpcSessionInner(sessionFile, cwd, toolNames?)`；`startRpcSession` 改为「existing 快路径 → inflight 检查 → `locks.set(startRpcSessionInner(...).finally(delete))` → return」（行为不变，`/api/agent/new` 继续用它）。
2. `resolveOrReattachSession` 在 §4.1 step1/step2 自行做 existing 快路径 + `locks.get` 检查，把整个「判 profile → reattach 或 generic」分流包进一个 Promise、`locks.set(sessionId, starting)` + `finally delete`；**generic 分支调不带锁的 `startRpcSessionInner`**（避免内外双锁）。
3. 二者复用**同一把** `globalThis.__piStartLocks`（按 sessionId 去重）——无论走 `/api/agent/new` 还是两 re-attach 路由，同一 sessionId 并发只建一个会话。

> 与记忆 D-V2-05「并发拒绝」精神一致。补一条并发去重单测（§六）。

## 五、不变量 / 红线核对

- **红线②产物改动必经按块确认**：本轮正是**修复**对它的违反（re-attach 不再给 write/edit/bash）。✅ 修复后红线在 profile 会话**跨 re-attach** 也成立。⚠️ 注意 §4.2 spread 顺序（docOptions 覆盖 options.tools）是受限集生效的唯一支点——订正 major #1 即守此红线，照初版错层 spread 反而会泄漏 write/edit/bash 自破红线②。
