**ultracode 核查关键结论（下列决策的事实基座）：**
- **双入口冗余确实存在**：同一 crd.md 既能从 file panel 文件树点开（走 FileViewer
  纯文本预览、够不到受管能力），又要从左栏 Artifacts 按钮 → ArtifactPicker 弹窗 →

  数据模型——侧车（`.pi/artifacts/managed/<id>/`：versions 快照 + pending + 元数据）
  承载用户想要的版本 / diff / 按块确认。删受管 = 删这些能力。用户的类比把两个不同层
  混为一谈（此区分 V2 反思轮已拍过，见 `V2/受管文档架构反思-AFFiNE对照.md`:18-23）。

## 决策 A · 「受管文档」概念去留

**背景**：用户质疑受管文档是否和 guard 一样繁琐、该去掉。

| 选项 | 说明 |
|---|---|
| **保留概念（推荐）** | 受管 = 侧车，是版本 / diff / 按块确认的物理载体；删了等于删用户明确想保留的四样能力 |
| 去掉受管 | 取消侧车 / 登记，回到普通文件——但版本 / diff / 确认随之消失，自相矛盾 |

**推荐**：保留。**理由**：ultracode 调查证实受管能力全由侧车承载、真实 .md 只是当前版
投影；读不到任何「不依赖侧车也有版本 / diff」的代码路径。受管与 guard 是不同层
（运行时拦截 vs 数据模型），不该混为一谈。

**谁拍**：用户（认可方案二＝「合并入口、保留机制」方向时默认接受）。
✅ **最终：保留「受管文档」概念**，本轮只动入口 / 展示层、不动机制层。

## 决策 B · artifacts 入口并入 file panel 的方向

**背景**：消除双入口冗余、让产物像 Notion 页在 file panel 列出。

| 选项 | 说明 | 改动 / 风险 |
|---|---|---|
| 方案一·只统一入口 | 受管 .md 加徽标、点开即进受管视图、删 Artifacts 按钮；受管仍显式登记 | 中小 / 低 |
| **方案二·Notion 式文档清单（用户选）** | file panel 顶部「受管文档」分组列出（kind·v 元信息），点开即受管；删弹窗删按钮 | 中 / FileExplorer 重构、须真浏览器 |
| 方案三·受管全量化 | 每个 .md 默认建侧车、任意文档都受管 | 大 / 高（引爆「外部改动自动进版本」+ 撞本机 OOM） |

**推荐**：lead 建议方案一（最小、不碰承重墙），方案二为可升级项。

**谁拍**：用户（**连续两次明确选方案二**）。
✅ **最终：方案二·Notion 式文档清单**。lead 已诚实告知方案二比方案一多付 FileExplorer
重构 + 5 条前置约束（见 D-V3-01），用户知情后仍选方案二（要更贴 Notion 的观感）。

## 由本会话决策派生的后续待办（登记不丢）

**方案二落地的已知前置约束**（实现级，详细设计须逐条处置；技术细节见 ADR D-V3-01）：
1. **反查映射缺失**：当前只有 `artifactId→filePath` 单向，无 `filePath→artifactId`；
   FileExplorer 分不清某 .md 是否受管——工作量集中点（D-V2-01 曾把 path 反查登记为
   「便利项后续加」）。
2. **live SSE ↔ EXTERNAL_MODIFIED 语义冲突**：FileViewer 文件一变就自动刷新，
   ArtifactService 却要外部改动就拒绝覆盖（D-V2-06）——合并到一个视图须调和。
3. **HITL 确认卡跨左右栏**：PendingChangeCard 挂左栏 ChatWindow、由 selectedArtifactId
   驱动；「点开即受管」不能让纯浏览误触确认现场。
4. **.pi 侧车泄漏**：`.pi` 不在 IGNORED_NAMES，versions / pending 裸 JSON 已可在文件树
   点开——并入前须先加忽略名单。
5. **派发产物物理隔离**：现状列不进 ArtifactPicker；本期不纳入。

**后续单独议题**：派发产物纳入文档清单、普通 .md 手动转受管、外部改动自动进版本
（方案三前置）。

**下一步**：lead 出方案二详细设计要点（反查映射 / FileExplorer 重构 / SSE 与 HITL
调和 / `.pi` 清理）→ vibe-coding 拆任务 → ns 团队串行实现 + lead 亲验 + 真浏览器验收。
