## 第 0 轮 · 开场确认（文件夹整理方式 + 报告深度）

### Q0-1　文件夹整理
**问题**：你想把「旧内容」怎么归到 next-step-V1？当前根目录有三类东西：① `next-step-app/`（真正的应用代码，自带独立 git 仓库、已上 GitHub，是 V1 的实现）；② `next-step/`（V1 的设计与规格文档）；③ 杂项（e2e 截图、原始需求 txt）。三种做法各有取舍：

| 选项 | 说明 |
|---|---|
| **A 应用原位不动，只收文档（推荐）** | 把 `next-step/`（文档）整理成 `next-step-V1/`，应用代码 `next-step-app/` 留在原位不动（它自带 git、已上 GitHub，移动有破坏构建和 git 历史的风险）。V1 的 README 说明 V1 是什么、并指向 GitHub 仓库。最稳妥。 |
| B 全部装进 next-step-V1/（含应用代码） | 把 `next-step/` 和 `next-step-app/` 都移进 `next-step-V1/`，根目录只剩两个版本目录，最整齐，但结构改动大、风险高一点。 |
| C 先不动旧文件，只建 next-step-V1.1/ | 暂不整理旧文件，先专注做需求文档，等做完再决定怎么归档 V1。 |

✅ **你的选择：A —— 应用原位不动，只收文档**

**问题**：原始需求第 1 条要一份「前端界面深度解析报告」（每个按钮的功能、每个页面的每个部分）。我已经在后台勘察代码了，你想要哪种程度的报告？它也会成为后面逐条确认需求的共同底图。

| B 精简清单版 | 只要一份「页面 + 按钮 + 各自作用」的速查清单，更快。 |
| C 暂时不用，直接进入提问 | 跳过报告，直接逐条问需求。 |

| 选项 | 说明 |

## 第 2 轮（功能#4 / 外观#1）

### Q2-1　功能#4 · 上传文件的目标
**问题**：现在只能传图片，是前端+内核两层限制（内核只认「文字」和「图片」两种内容）。所以关键不是「让你能选文件」，而是「选了文件后怎么处理」。你传文件的目标是？

| 选项 | 说明 |
|---|---|
| **A 让 Agent 读懂文件内容（推荐）** | 把文件转成文字喂给模型让它理解。需改前端、可能加后端文本提取。 |
| B 只是把文件存进项目当资料 | 不要求模型当场读。这其实是另一个「上传到工作区」功能，做法完全不同。 |

✅ **你的选择：A —— 让 Agent 读懂文件内容**

### Q2-2　功能#4 · 文件类型
**问题**：如果是「让 Agent 读懂内容」，你主要想传哪类文件？（决定工作量大小）

| 选项 | 说明 |
|---|---|
| **A 纯文本/代码/Markdown/CSV 就够（推荐先做）** | 档1：几乎只改前端、最省事。文件读成文字、按内核格式 `<file>` 内联进消息。 |
| B 必须包含 PDF / Word / Excel | 档2：二进制文件前端读不出文字，要加后端文本提取（PDF 还得新增解析库）。排版/图表会丢，只留文字。 |
| C 两者都要 | 先做档1（文本类），档2（PDF/Office）作为后续。 |

✅ **你的选择（自定义答复）：先做推荐的「档1·纯文本类」，另一个「档2·PDF/Office」记录下来、以后迭代用**

### Q2-3　外观#1 · 二级菜单是只读还是可改
**问题**：你要去掉编辑按钮，但现在改「模型/技能/工具/思考强度」全靠那个编辑表单。点开卡片后的二级菜单里这四项，是？

| 选项 | 说明 |
|---|---|
| **A 能现场改并保存（推荐）** | 把配置能力搬进二级菜单，点开就能改。最符合「去掉编辑按钮」初衷，但相当于把整个编辑表单塞进二级菜单，工作量更大。 |
| B 只看当前设置（只读） | 二级菜单只显示当前是什么，改配置还得另有入口。工作量小，但「去掉编辑按钮后改配置」要另想办法。 |

✅ **你的选择：A —— 能现场改并保存**

### Q2-4　外观#1 · 设计参考
**问题**：「玻璃感 + 卡片菜单」每个人想象不一样。你能给参考吗？

## 第 3 轮 a（功能#5 · 结构性决策）

### Q3a-1　项目首页
**问题**：你想要打开应用先看到一个项目管理首页、点项目才进工作台。现状是一打开就是工作台、靠左栏下拉切项目。你要哪种？

| 选项 | 说明 |
|---|---|
| **A 独立的项目首页（推荐）** | 打开应用先看项目列表/卡片墙（增删项目，参考 pi-web 风格），点某个才进工作台。最贴近你原话，也最清爽。 |
| B 只在「没选项目时」整屏显示项目列表 | 不新增独立页，只把现在的项目下拉扩成「没选项目时满屏列项目」。改动小。 |

✅ **你的选择：A —— 独立的项目首页**

### Q3a-2　@agent 的含义（最关键一问）
**问题**：在主对话里「@某个 agent 并把上下文+文件传给他」，你期望发生什么？（这决定要不要复用现有 Dispatch 机制）

| 选项 | 说明 |
|---|---|
| **A 转交到他的单独会话里** | 把你选中的几段话/几个文件，作为一条消息发到那个 agent 的单独会话里，让他在自己会话接着干（轻量手动转交）。 |
| B 就是「派发任务给他」 | 等价于现在的 Dispatch，只是入口更顺手。（那就复用现有派发机制改入口） |
| C @他在主对话里就地回答 | 不切到他的会话，就在当前主对话里用他的人设回一段（像群聊里 @ 某个专家）。 |

✅ **你的选择：A —— 转交到他的单独会话里**（即：新轻量功能，不等于 Dispatch，也不是就地回答）

### Q3a-3　Dispatch 去留
**问题**：你平时（或预期）会用到「多个 agent 接力」吗——让 A 先产出、把 A 的产出自动喂给 B 继续做这种流水线？

| 选项 | 说明 |
|---|---|
| **A 现在说不准，先保留不动（推荐）** | 先保留 Dispatch，等 @agent 新功能做出来、用一段时间确认接力场景真用不上，再回头删。最稳。 |
| B 会用到接力/批量 | 那 Dispatch 的核心价值就在这，建议保留。 |
| C 从来都是一个个 agent 单独叫来问 | 那 Dispatch 对我确实鸡肋，可考虑下线。 |

✅ **你的选择：A —— 现在说不准，先保留不动**

### Q3a-4　每个 agent 一个会话
**问题**：新增一个 agent 时，会话怎么处理？

| 选项 | 说明 |
|---|---|
| **A 建档案后按需起会话，左侧标明归属（推荐）** | 不一下子冒出一堆空会话。保持「先建 agent 档案、需要时再起会话」，但在左侧把会话显性标成属于哪个 agent（分组或标签）。 |
| B 建 agent 时就自动开一个空会话 | 每新建一个 agent 立刻给它开一个空会话。注意：会一次性冒出很多空会话。（贴近你原话「新增一个 agent 都将其建立一个单独会话」） |

✅ **你的选择：A —— 建档案后按需起会话，左侧标明归属**

### Q3b-1　主对话 / 进项目落地页

| 选项 | 说明 |
|---|---|
| **A 落地到聊天工作台；主对话=每项目固定一个总指挥会话（推荐）** | 进项目直接看到对话工作台，其中有一个固定的「主对话」作总入口；没设 agent 时默认就用它。 |

### Q3b-2　传给 agent 的内容（可多选）

### Q3b-3　左侧对话栏怎么区分

### Q3b-4　功能#4 文件大小上限

✅ **你的选择：C —— 不设上限**（实现侧建议加一句「大文件会消耗大量 token」的软提示，非硬限制）
