# 第六轮 · Agent 模式（doc/coding）—— 放开「编码型」会话的 bash 等内置工具 —— 详细设计

> 修复用户报缺陷「我真正需要 agent 有 bash 能力的时候无法开启 bash」。给 Agent 档案加 `mode` 维度区分「文档型 / 编码型」：编码型按需放开 bash/write/edit，文档型维持 V2 受限红线。
> **范围**：只动 profile 会话起会话 / re-attach 的工具集分流 + 档案 `mode` 字段 + AgentManager UI。**不碰 pi 内核、不动 doc-session 受限白名单定义、不碰主对话 `/api/agent/new` 与 dispatch**（这两条本就带 bash）。
> 与第三/四/五轮不同：本轮为**缺陷驱动、investigate→implement→verify 直做**（非先出三件套再实现），故本文为「设计 + 落地 + 验收」合一记录。**已实现并双层验收通过**（提交 `99f8917`）。

---

## 〇、缘起与复现（ultracode + 真浏览器 + 真实对话）

- **用户报**：编辑 agent 工具时勾了 bash，可起会话后 agent 用不了 bash；并提议「编辑工具时加『专门写文档的工具 vs 内置编码工具』的选择」。
- **复现**：ultracode workflow（8 agent 对抗校验）+ 真浏览器 + DeepSeek V4 Flash 真实对话双跑确认。UI 勾 bash → 存进 `agent.json` → 起会话 agent 调 bash → 内核返回 `Tool bash not found` → agent 自述「我没有 bash 工具」并列出 7 项受限工具（read/grep/find/ls + create_artifact/propose_edit/list_artifacts）。
- **定性**：是 V2「文档会话用受限工具集、改受管文档只能经 propose_edit → 按块确认」**红线的 by-design 实现 + UX「沉默陷阱」**（bash 能勾、能存盘、回读仍选中，起会话却被静默丢弃、全程零提示）。
- **对抗校验纠正直觉**：原以为「bash 全局开不了」是错的。**主对话（/api/agent/new）与派发 dispatch worker 本就带 bash**（勾进 agent.tools 即可，它们不套受限集），缺口**仅在 profile 文档会话这一处**。
- 完整调查 / 路径矩阵见 memory `next-step-bash-toolset-gap`；用户决策 QA 见 `docs/QA/开发/agent模式-bash能力决策.md`（索引 #12）。

## 一、根因（一句话）

`lib/pi/profile-session-wiring.ts` 起 profile 会话时
`createAgentSession({ ...options, ...docOptions, ...createOptionsOverride })`：
后面的 `docOptions.tools`（`doc-session.ts` 的 7 项受限白名单，**无 bash**）经 JS 对象 spread **整体覆盖** `options.tools`(=`profile.tools`)，且这段覆盖**无任何 if / flag / 字段开关**——只要走 profile 会话这条路，用户勾的 bash 必被替换。`reattachProfileSession` 同构。

## 二、目标与决策（方案A，用户 AskUserQuestion 拍板）

四方案全列（详见 QA）：
- **方案A（采纳）**：`AgentProfile` 加 `mode: doc|coding`，wiring 按 mode 决定是否套受限集。doc 维持现状、coding 用 `profile.tools`（含 bash）。机制零改、不碰内核、对红线零破坏（coding 等价 dispatch/主对话既有带 bash 会话）。
- 方案B：受限会话内并入 bash/write —— **削弱红线**（write 直接覆盖受管 .md、bash `echo>file` 绕过提议），否决作默认。
- 方案C：mode + doc 模式仅解锁 bash —— 折中、复杂度高。
- 方案D：仅补 UX 防呆，不放开 bash —— 治标、不满足诉求。

**最终选择**：方案A + 方案D 的 UX 防呆。实现级取舍见 ADR `docs/设计决策记录.md` **D-MODE-01~03**。

## 三、详细设计

### 3.1 数据层 —— `AgentProfile.mode`（`lib/domain/agent-profile-store.ts`）
- 新增类型 `AgentMode = 'doc' | 'coding'`；`AgentProfile` / `AgentProfileInput` 加 `mode`。
- `create`：`mode = input.mode ?? 'doc'`，非法值抛 `INVALID`。
- `update`：透传 `patch.mode`（含非法校验）。
- `readProfile`：**向后兼容归一化**——旧 `agent.json` 无 `mode` → `profile.mode !== 'coding' 即视为 'doc'`，存量 agent 行为完全不变。

### 3.2 接线层 —— wiring 两处按 mode 分流（`lib/pi/profile-session-wiring.ts`，承重）
`startProfileSession` 与 `reattachProfileSession` 各一处：
```
const docOptions = profile.mode === "coding"
  ? undefined                                  // 编码型：不套受限集
  : assembleDocSessionOptions({...}).options;  // 文档型：7 项受限白名单
await createAgentSession({ ...options, ...(docOptions ?? {}), ...createOptionsOverride });
```
- doc：docOptions.tools(7 受限白名单) 覆盖 profile.tools（spread 顺序 = 受限集生效唯一支点，D-V2-04 不可调）。
- coding：docOptions=undefined（spread 空对象）→ `options.tools`(=profile.tools，可含 bash) 即最终工具集，且**无 3 提议工具**（纯编码 agent）。

### 3.3 API 透传（`app/api/projects/[id]/agents/route.ts` POST + `[agentId]/route.ts` PATCH）
两路由 body 白名单 + input/patch 各加 `mode`（`typeof body.mode === 'string'` 守卫）。

### 3.4 前端 UI + 防呆（`lib/stores/useAgentStore.ts` + `components/AgentManager.tsx`）
- `AgentProfileInput` 加 `mode`；FormState/EMPTY_FORM/fromProfile/create&update payload 全接 `mode`（默认 doc）。
- AgentManager 新增「模式」选择器（文档型 / 编码型，按钮组样式同思考强度）：
  - **coding**：红字警告「使用下方勾选的内置工具（含 bash/write/edit），可直接读写磁盘、执行命令，不经提议确认」；工具按钮正常可勾。
  - **doc（防呆）**：工具按钮 `disabled` + 置灰（opacity 0.45）+ 标签「· 文档型不使用」+ 提示「使用受限工具集（read/grep/find/ls + 提议工具），改受管文档须经按块确认；不支持 bash/write/edit」。消除原「沉默陷阱」。
  - 锚点：`data-testid="agent-form-mode"`（`data-mode`/`data-selected`）、工具按钮加 `data-disabled`。

## 七、补丁 · 改 mode 即时生效到存活会话（D-MODE-04，方案B）

> 用户复测发现的残留缺陷：编辑 mode 只对**新会话**生效；**同一存活会话**内（编码型起会话 → 中途改文档型）bash 仍真可用，新会话才不可用。
> 根因：会话工具集在 `createAgentSession` 构造期固定；同会话发后续消息走 `resolveOrReattachSession` 的「活会话快路径」直接复用存活 wrapper、不重读档案。
> 经 ultracode 8-agent 调查 + 3 条对抗校验确证 go（re-attach 全链**现读磁盘 agent.json、无缓存**）。

**方案B（用户 AskUserQuestion 拍板）**：PATCH 改 mode 成功后、**仅 mode 真变化**时，把该 agent 名下**全部**存活会话从 rpc-manager 逐出（`getRpcSession(sid)?.destroy()`）→ 下一条消息走 re-attach、按新 mode 重建工具集（doc↔coding 对称即时生效）。

**落点**：`lib/domain/session-agent-map.ts` 加 `sessionsForAgent` 反查 + `lib/pi/evict-agent-sessions.ts`（新，含流式 abort 守卫、DI 可单测）+ `app/api/projects/[id]/agents/[agentId]/route.ts` PATCH 接线。+8 单测（sessionsForAgent 3 + evictAgentSessions 5）。

**关键约束**（对抗校验所得）：①只删 registry、绝不碰 `bySession`/`removeOwner`（动 map 会让 re-attach 误落 generic、反塞回 write/edit/bash）；②不可热改活会话工具集（与内核构造期 `_rebuildSystemPrompt` 冲突）；③逐出正流式的会话前先 `abort`（防无头孤儿双写 jsonl 致损坏）；④仅 mode 变化才逐出；⑤一 agent 可多会话（起会话+@转交）须全逐。详见 ADR `设计决策记录.md` D-MODE-04。

**验收**：lint + test(373) 全绿；**真浏览器往返双层 PASS**——同一会话 coding `echo`→toolResult 真输出 → 改 doc 后→`Tool bash not found`（bash 即时失效）→ 改回 coding→toolResult 真输出（即时恢复），pageErrors=0（SSE 经下条消息 `connectEvents` 自愈）。

## 八、补丁 · 编码型空 tools 退回全套编码工具（D-MODE-05）

> 用户复测发现：选「编码型」但**未勾任何工具** → agent 在对话中说「检查不到任何工具，工具列表 `(none)`」（连 bash 都没有）。
> 根因：coding 模式 `options.tools = profile.tools`；空数组 `tools:[]` 被内核当「白名单=零工具」（≠ `undefined` 的默认全套）。ultracode 7-agent + node 实证；与主对话「空 toolNames→默认全集」不一致。

**双层修复**：
- **wiring**（`profile-session-wiring.ts` start+reattach 两处）：coding + `profile.tools` 为空 → 退回全套 `CODING_TOOL_NAMES`（含 bash）。救**现有空 tools agent** + 用户手动清空全部，无需重建档案。
- **UI**（`AgentManager` 模式按钮 onClick）：切到编码型且未勾 → **自动全选** `CODING_TOOL_NAMES`，让用户看得见、新建即非空。
- `CODING_TOOL_NAMES` 提取到中性 `lib/pi/coding-tools.ts` **单一真相源**（client `useAgentStore` re-export + server `wiring` 共用；⚠️ vitest 不解析 `@/` 值导入，lib 内用相对导入 `../pi/coding-tools`）。doc 模式不受影响。

**验收**：+1 单测、lint + test(374) 全绿；**真浏览器双层 PASS**——① API 造「编码型+空 tools」agent 起会话 bash 真跑（WIRINGFIX 输出，证 wiring 兜底救现有 agent）；② UI 新建切编码型 bash/read 自动勾选、持久化全 7 工具。详见 ADR `设计决策记录.md` D-MODE-05。
