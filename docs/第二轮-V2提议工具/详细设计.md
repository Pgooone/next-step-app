## A. lead 实现级取舍（3 条，已拍定 → 记 `../../docs/设计决策记录.md` ADR）

> QA 轮次 5 把这 3 条列为"拆任务时定"。本轮由 lead 按 `lead-autonomy` 拍定（标 🔧）；用户 review 本文时可推翻。

### 🔧 取舍1 · `propose_edit` 收 **id**（不收 path）
- **背景**：artifact 既有 id(uuid) 又将有真实文件 path(crd.md)，propose 用哪个定位？
- **选项**：A 收 id；B 收 path；C 都支持。
- **决策：A 收 id。** **理由**：现有 API / PendingChangeStore / ArtifactService 全按 `artifactId`，收 id 零阻抗；agent 工作流天然两步——先 `list_artifacts` 拿 id → `propose_edit(id, newContent)`。path 反查（artifact.json 已记 filePath）作便利项，**后续按需再加**。

### 🔧 取舍2 · crd.md 物化到 **`<projectRoot>/<sanitize(title)>.md`**（项目根）
- **背景**：真实文件物化在哪，才"本地可开 + file panel 可见、类 Notion 页"？
- **选项**：A 项目根；B `docs/` 子目录；C `artifacts/` 子目录。
- **决策：A 项目根 `<projectRoot>/<sanitize(title)>.md`。** **理由**：最符合"类 Notion 页、一眼可见"；MVP 单/少 artifact 够用。物化路径在 `create` 时由 title 清洗生成、**存进 artifact.json 的 `filePath` 字段**，后续 submitVersion/rollback 以存的 filePath 为准（不随 title 改而漂移）。**多 artifact 拥挤 → 后续可改子目录**（登记，非本期）。

### 🔧 取舍3 · `create` 与 `propose` **分开两个工具**（不合一）
- **背景**：创建（无上一版、无 diff）与修改（有上一版、走 diff→确认）流程语义不同。
- **选项**：A 分开；B 合一（按 id 是否存在自动分流）。
- **决策：A 分开。** **理由**：两者流程语义不同——创建直接落 v1+物化、无确认闸门；修改必走 diff→按块确认。分开各自单一职责、schema 清晰；合一会让"要不要走确认"隐式依赖 id 是否存在，易错（违 Simplicity / 模块独立）。

## B. 实现前置 · 工具机制 spike（命门，必做且先做）

第一轮 P0 的教训：**接线命门先 spike、GO 再展开**（省大坑）。V2 有两个未验证命门，实现**第一步必做 spike**（任务卡 `V2-0`），GO 才进 V2-1~V2-6：

1. **`defineTool` 精确签名**：pi 包导出 `defineTool`（`dist/index.d.ts:8`），但本设计未固定其 `parameters`（zod? JSON schema?）/ `execute(args, ctx)` 签名。spike 跑通一个最小自定义工具（如回显 `list_artifacts` 雏形），确认签名。
2. **受限工具集组合行为**（关键安全命门）：确认「`tools` 白名单 = 只读集（无 write/edit/bash） + `customTools` = 3 提议工具」组合下，内核**既**激活只读内置工具、**又**挂上 3 提议工具，且 **write/edit/bash 确实不可用**。
   - 风险点（勘察发现）：P0 命门是"白名单**含** write + customTools 同名覆盖"；V2 是"白名单**不含** write + customTools 加新工具"，是**新组合**。若白名单遗漏排除 write、或 `noTools` 与白名单交互不如预期，内置 write 可能仍激活 → 绕过。spike 必须用**负对照**证明 write/edit/bash 调用失败。

> spike 产出：GO/NO-GO + 确认的 `defineTool` 调用范式 + 受限工具集的确切 options 形态。记 ADR。

## C. 模块详细设计

### V2-1 · 文档物化层（`lib/domain/artifact-service.ts` 扩展）

**数据结构变更**：`Artifact` 增 `filePath?: string` —— 物化真实文件相对 projectRoot 的路径（如 `需求规格.md`）。写进 `artifact.json`。

```ts
export type Artifact = {
  id: string; projectId: string; kind: string; title: string;
  currentVersion: number; version: number; status: "draft" | "finalized";
  filePath?: string;          // 新增：物化真实文件（相对 projectRoot）
  extra?: Record<string, unknown>;
};
```

**新增私有方法**：
- `private materializedPath(projectId, artifact): string` = `join(registry.get(projectId).root, artifact.filePath)`。
- `private materialize(projectId, artifact, content): void` —— 原子写（临时+rename，仿现有 `atomicWrite`）真实文件。

**改造现有方法**（在既有逻辑末尾追加物化，不动既有版本/侧车逻辑）：
- `createArtifact`：生成 `filePath = sanitizeFileName(title) + ".md"`（去非法字符、冲突加序号）；写进 artifact.json；建 v1 后 `materialize(content)`。
- `submitVersion`：提交新版后 `materialize(newContent)`（覆盖真实文件 = 当前版）。
- `rollback`：回退后 `materialize(回退版 content)`。

**依赖**：仅 node:fs + 既有 ProjectRegistry。**不依赖** 上层。
**验收要点**：建→真实文件存在且 content==v1；改提交→真实文件==新版；回退→真实文件==回退版；artifact.json 含 filePath；title 含非法字符（`/ : *` 等）被清洗；同 title 第二次建不覆盖第一份。

### V2-2 · 提议工具（`lib/pi/doc-tools.ts` 新建，`defineTool`）

**工厂**（仿 `assembleArtifactGuardOptions` 闭包注入范式，QA Q4 先例）：

```ts
export type DocToolDeps = {
  projectId: string;          // 闭包注入（execute 的 ctx 不带 projectId）
  sourceActor: string;        // 写进 version.author / PendingChange.sourceActor
  artifactService?: ArtifactService;   // 默认其文件后端
  pendingStore?: PendingChangeStore;
};
export function buildDocTools(deps: DocToolDeps): GuardToolDef[];  // 返回 3 个 defineTool 工具
```

**三个工具**（schema 与 execute 关键逻辑）：

| 工具 | parameters | execute 关键逻辑 | 返回 |
|------|-----------|------------------|------|
| `create_artifact` | `{ kind, title, content }` | `artifactService.createArtifact(projectId, {kind,title,content,author:sourceActor})`（V2-1 已含物化） | `{ id, filePath, version: 1 }` |
| `propose_edit` | `{ id, newContent }`（🔧 收 id） | `old = readCurrentContent(projectId, id)` → `change = buildReplacePendingChange({artifactId:id, sourceActor, oldContent:old, newContent})` → `pendingStore.save(projectId, change)` —— **不写盘** | `{ changeId, diffBlockCount, note:"已提交待按块确认" }` |
| `list_artifacts` | `{}`（MVP 不过滤） | `artifactService.listArtifacts(projectId)` | `[{ id, title, kind, currentVersion }]` |

**依赖**：V2-1（物化）+ 既有 pending-change-service（`buildReplacePendingChange` / `PendingChangeStore`）+ `defineTool`（pi）。
**关键点**：`propose_edit` 复用 guard 同款 `buildReplacePendingChange`，落的 PendingChange 与现有 resolve 路由 / PendingChangeCard 完全兼容（op=replace、diffBlocks 由 computeReplaceDiffBlocks 算）。确认流水线（resolve → applyResolvedBlocks → submitVersion → 物化）走既有 API 路由 + V2-1 物化，**零新增确认逻辑**。
**验收要点**：faux 驱动断言——create 落 v1+真实文件；propose 落 PendingChange（磁盘有 pending JSON、无新版本、真实文件未变）；list 返回清单含 id。

### V2-3 · 文档会话装配（`lib/pi/doc-session.ts` 新建，替 artifact-guard 装配位）

```ts
export function assembleDocSessionOptions(deps: DocToolDeps & { cwd: string }): {
  options: { tools: string[]; customTools: GuardToolDef[] };
};
```

**装配内容**：
- `tools`（白名单，只读探索集）= `["read", "grep", "find", "ls"]` —— **无 write / edit / bash**。
- `customTools` = `buildDocTools(deps)`（3 个提议工具）。
- skill：经既有 `assembleProfileSessionOptions` 的 `skillNames`（= profile.skills）启用，docSession 不重复处理。

**与 guard 的关键区别**（更简）：guard 要"拦 write/edit"故保留工具名 + 重建 operations；**V2 不要 write/edit/bash**，故白名单直接排除它们、customTools 只加 3 个新工具——**无需重建任何内核工具的 operations**。

**安全论证**（依赖 B·spike 验证）：白名单无 write/edit/bash → 内置写盘工具不激活；customTools 只加只读的提议工具 → **无任何直接写盘 / 执行路径 → 结构性无绕过**。这是 §F5 的实现。
**lead 取舍（受限集范围）**：给 read+grep+find+ls（只读探索全套，便于 agent 读材料/定位）+ 3 提议工具；比"只 read"更实用且仍无写盘能力。
**验收要点**：单测断言 options.tools 不含 write/edit/bash；customTools 含且仅含 3 提议工具；spike 负对照证 write/edit/bash 调用失败。

### V2-4 · 接线改造（`lib/pi/profile-session-wiring.ts` 改 1 处）

**现状**（勘察 wiring.ts:126-135）：`assembleArtifactGuardOptions({sourceActor, cwd})` → `createAgentSession({...options, ...guardOptions, ...createOptionsOverride})`。

**改造**：把 `assembleArtifactGuardOptions` 换成 `assembleDocSessionOptions`：
```ts
const { options: docOptions } = assembleDocSessionOptions({
  projectId,                  // 由 projectRoot 反查（registry.findByRoot）或入参透传
  sourceActor: profile.name,
  cwd,
});
const { session: inner } = await createAgentSession({ ...options, ...docOptions, ...createOptionsOverride });
```

**注意点**：
- `assembleDocSessionOptions` 需要 `projectId`（提议工具闭包要）；现 `startProfileSession` 入参有 `projectRoot`，需补 projectId（registry 反查或入参加 projectId）。
- **范围同 P0**：本轮把 **profile 会话**装配从 guard 换 docSession（P0 也只接 profile 会话）。dispatch / 主对话 / idle 不在本轮（本就未接 guard）。
- **🔧 登记后续**：是否区分"文档型 profile（受限工具集）vs coding 型 profile（完整工具）"——本轮 MVP 不区分（profile 会话 = 文档会话，承接 P0 范围）；若未来有 profile agent 需要 coding，再加 profile 标志位分流。登记 `后续`。
**验收要点**：经真实 `startProfileSession` + faux 起会话，工具集为受限集；create/propose 闭环可跑。

## D. 接线全景（数据流）

```
文档会话 agent
  │  create_artifact(kind,title,content)
  ├────────────────────────────────────────> ArtifactService.createArtifact ──> 写 v1 侧车 + 物化真实 crd.md
  │  propose_edit(id, newContent)
  ├──> readCurrentContent(old) ─> buildReplacePendingChange ─> PendingChangeStore.save（待确认，不写盘）
  │                                                                    │
  │                                              ┌─ ArtifactPanel/PendingChangeCard 渲染 diff 高亮（D3/D4）
  │  用户按块 ✓/✗ ─> resolve 路由 ─> applyResolvedBlocks ─> submitVersion ─┴─> 物化真实 crd.md（新版）
  │  list_artifacts() ──> listArtifacts ──> [{id,title,kind,version}]
  └─ read/grep/find/ls（只读探索） + skill（解析/转化）
       —— 全程无 write/edit/bash，无绕过路径
```
