> vibe-coding 第 2 步·概要设计。输入：`需求文档.md`。本文只做一件事：**把需求拆成职责单一、能独立测试的模块**，并排出依赖批次。详细接口 / 数据结构见 `详细设计.md`。

## 1. 模块职责表

| 模块 | 职责（一句话） | 主要落点 | 性质 |
|------|----------------|----------|------|
| **V2-0 工具机制 spike** | 验证 `defineTool` 签名 + 受限工具集组合行为（白名单只读 + customTools 提议工具，负对照证 write/edit/bash 不可用），GO 才展开 | `spike/` | 前置验证 |
| **V2-1 文档物化层** | 把受管 artifact 的"当前版"物化成项目里**真实 `.md` 文件**；artifact.json 记对应真实文件路径；create / submitVersion / rollback 时同步真实文件 | `lib/domain/artifact-service.ts`（扩展） | 新增（地基） |
| **V2-2 提议工具** | 用 `defineTool` 定义 `create_artifact` / `propose_edit` / `list_artifacts`，闭包注入 deps（projectId / service / store / sourceActor），调 V2-1 与既有 PendingChange 管道 | `lib/pi/doc-tools.ts`（新建） | 新增（核心） |
| **V2-3 文档会话装配** | `assembleDocSessionOptions`：产出受限工具集会话选项（`noTools:"builtin"` + customTools = `[read, create_artifact, propose_edit, list_artifacts]`，**无 write/edit/bash**；skill 经 skillNames 启用） | `lib/pi/doc-session.ts`（新建，替 artifact-guard 的装配位） | 新增（替换） |
| **V2-4 接线改造** | `profile-session-wiring.ts` 把 guard 接入点换成 V2-3 装配；文档会话走受限工具集起会话 | `lib/pi/profile-session-wiring.ts`（改 1 处） | 改造 |
| **V2-5 删 guard 层** | 删 `artifact-guard.ts` / `artifact-intercept.ts` + 对应 test；清理残留 import；保留 `resolveManagedTarget` 若 V2-1 仍需则迁移 | `lib/pi/`（删除 + 清理） | 删除 |
| **V2-6 端到端复用验证** | 提议工具 + D3-D5 UI/API 端到端跑通（逻辑层 faux + 真浏览器）；确认复用未回归 | `scripts/` E2E + 既有测试 | 验证 |

> PDF/Word 解析、转化 skill = **本轮不做**（需求 §6），不在模块表内。

## 2. 依赖关系与批次

```
V2-0 工具机制 spike（defineTool 签名 + 受限工具集组合，GO 才展开）
   └─> V2-1 文档物化层（地基，无依赖）
   └─> V2-2 提议工具（create/propose 要调物化层 + PendingChange 管道）
          └─> V2-3 文档会话装配（要把 3 个工具塞进 customTools）
                 └─> V2-4 接线改造（用 docSession 装配替 guard 接入点）
                        └─> V2-5 删 guard 层（接线切走后才能安全删，否则断）
                               └─> V2-6 端到端验证（全部就位后跑闭环）
```

**批次（串行为主——模块强依赖 + 本机 OOM 约束要求串行）**：

| 批次 | 模块 | 说明 |
|------|------|------|
| 批次 0 | V2-0 | 工具机制 spike（defineTool 签名 + 受限工具集组合行为），GO 才展开 |
| 批次 1 | V2-1 | 物化层地基，先做好供工具调用 |
| 批次 2 | V2-2 | 提议工具，依赖 V2-1 |
| 批次 3 | V2-3 → V2-4 | 会话装配 → 接线（紧密串行） |
| 批次 4 | V2-5 | 删 guard（接线切走后） |
| 批次 5 | V2-6 | 端到端验证 |

> 各模块**强依赖、近乎纯串行**：不像第一轮 M1~M8 有可并行的独立 UI 模块。这条链是一条接线主干，**逐个做、每个过门禁再下一个**最稳；也契合本机 OOM 约束（实现 agent team 串行）。

## 3. 复用清单（直用 / 零改，模块独立性的基础）

V2 的承重墙**早已建好**（Iter D），本轮新增的只是"调用方"与"物化层"：

| 既有能力 | 文件 | V2 怎么用 |
|----------|------|-----------|
| 切块纯函数 | `pending-change-service.ts:180` `computeReplaceDiffBlocks` | `propose_edit` 算 diff |
| 建 replace 型变更 | `pending-change-service.ts` `buildReplacePendingChange` | `propose_edit` 落 PendingChange |
| 重放确认结果 | `pending-change-service.ts:216` `applyResolvedBlocks`（只支持 op=replace） | 确认后重建新内容 |
| 变更落盘 | `pending-change-service.ts` `PendingChangeStore.save` | `propose_edit` 存待确认 |
| artifact 实体/版本 | `artifact-service.ts` `createArtifact/getArtifact/readCurrentContent/getVersion/listVersions/submitVersion/rollback/listArtifacts` | 工具直接调 |
| 视图纯函数 | `lib/artifact-view/{anchor,degrade,toc}.ts` | 渲染复用 |
| UI 组件 | `components/{ArtifactPanel,ArtifactPicker,PendingChangeCard}.tsx` | diff/版本/按块确认 UI 复用 |
| API 路由（8 个） | `app/api/artifacts/**` + `app/api/projects/[id]/artifacts` | 前端纯 id 操作，零改 |
| 工具定义 API | pi 包 `defineTool` | V2-2 定义 3 个提议工具 |
| 会话装配范式 | `agent-profile-session.ts` `assembleProfileSessionOptions`（tools 白名单 / skillNames 过滤 / noTools+customTools） | V2-3 仿此装配受限工具集 |

**结论**：D1-D5 是健康架构，无需重写、无需照搬 AFFiNE；本轮是"加物化 + 加 3 个工具 + 换装配 + 删 guard"，复用面最大。

## 4. 模块独立性自检

- **V2-1 物化层**：纯扩展 ArtifactService，输入(content)→输出(真实文件写盘)，可单测（建/改/回退后断言真实文件内容）。
- **V2-2 提议工具**：每个工具 = schema + execute(闭包 deps)，可用 faux deps 单测（断言 create 落 v1+真实文件、propose 落 PendingChange、list 返回清单）。
- **V2-3 会话装配**：纯函数 deps→options，可单测（断言 customTools 含 3 工具、不含 write/edit/bash）。
- **V2-4 接线**：1 处替换，可用既有 startProfileSession 测试路径验证。
- **V2-5 删 guard**：删除 + grep 无残留 import 即可验证。
- **V2-6 验证**：端到端，逻辑层 faux + 真浏览器。

---

## 5. 与第一轮 / Iter D 的关系

- **取代**：本轮取代 P0「逐路装 guard」档位 2/3（删 guard、换提议工具）。
- **复用**：Iter D（D1-D5）的 diff/版本/高亮/确认管道 + UI + API 全部复用。
- **不碰**：第一轮 M1~M8 的功能（命名/上传/项目首页/会话↔agent 映射/@agent 等）不受影响；dispatch/主对话本就未接 guard，V2 后亦不受影响（受管文档通过提议工具操作，非魔法路径）。
