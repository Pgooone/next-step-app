# Next-Step V2 需求文档 —— 文档实体 + 提议工具模型

> vibe-coding 第 1 步·需求。本轮（第二轮）从「逐路装 guard」转向「**文档层通用能力 + agent 提议工具**」。
> 需求/设计来源：经 5 轮用户拍板的讨论留痕 `../QA/开发/V2/受管文档架构反思-AFFiNE对照.md`（轮次 1~5 + 收敛模型）。
> 范围已拍板：**只拆核心闭环，PDF/Word 解析后置**（见 §6）。

---

## 0. 一句话

把「产物 diff / 版本 / 按块确认」从**给对话逐路装 guard 拦截**，重构为**文档层通用能力 + agent 提议工具**：`crd.md` 是项目里的**真实文件**，AI 通过 `create_artifact / propose_edit / list_artifacts` 三个提议工具操作它，每次改动走「diff → 按块确认 → 新版本」，agent 退化为众多调用方之一（仿 Notion / AFFiNE 的分层）。

---

## 1. 背景与缘起

P0 档位1（profile 会话接 artifact-guard）收官后，用户质疑「给每条对话路（profile / 主对话 / 派发 / idle）逐路装 guard」的别扭：能否**最小改动、覆盖所有路、更通用**？类似 Notion——对话是对话，对话产物变文档、自带 diff。

AFFiNE 源码核验给出了答案与纠偏（详见 QA 轮次 2）：

- **正解**：AFFiNE 的保护在「文档管道」、不在「对话」——AI 用工具吐"原文+新文"，固定代码算 diff、落盘，与"AI 怎么被调起"无关。diff / 版本 / 回退是**文档/存储层能力**，AI 只是众多调用方之一（Notion / AFFiNE 同构）。
- **纠偏**：对照文档曾断言"AFFiNE 命令 AI 保留 block_id（控制两端）"——**开源源码证伪**。开源真跑的是「模型吐整篇全文 → 后端按**内容 LCS** 重配 id」。即 AFFiNE 真正"稳"的机制 = 内容 LCS，**与我们的 LCS 文本对账同构**。我们不是弱替代、正是它的强机制本身。故 block_id 路线**后置**、整篇重写**首发**。

---

## 2. 核心理念

| 维度 | 旧（P0 guard 模型） | 新（V2 提议工具模型） |
|------|--------------------|------------------------|
| diff/版本归属 | 绑在「对话路」上，逐路装 guard | **文档层通用能力**，与对话解耦 |
| agent 角色 | 用通用 write/edit 写"魔法路径"，被拦截 | **调用方**：调 `propose_edit` 产内容，永不直接写盘 |
| 覆盖范围 | 每条路单独接（主对话/dispatch/idle 有 gap） | 文档会话用**受限工具集**，结构性无绕过路径 |
| 红线张力 | "拦截 agent 写文件"始终在闸门边缘 | 无 write/edit/bash → 无张力，guard 可删 |

**已验证为正确解耦**：Notion / AFFiNE 的版本/diff/回退都生长在文档/存储层、完全不知"AI"存在；AI 内容与手打内容走同一套（QA 轮次 3·Q6）。Next-Step 的 PendingChange / diff / 版本 / 按块确认管道**已经是**这个文档层能力（D3-D5 全建好）；唯一把它和"对话"耦在一起的就是 guard（路径拦截）。**把 guard 换成提议工具，agent 即成纯调用方**——正是用户想要的。

## 3. 目标用户场景（CRD 文档生产流）

用户的真实场景（QA 轮次 4）：

```
对话里丢 PDF / Word
  → 用 skill 解析（原始材料 → 文本）
  → 用 skill 转化（原始需求 → CRD 结构）
  → 创建 crd.md（项目里的真实文件）
  → 之后每一轮修改都带「版本 / diff / 高亮改动位置 / 确认（可为撤销键）」
     —— 手感类似 Notion / AFFiNE / Word 修订模式
```

- `crd.md` = 项目里**真实文件**（本地可用编辑器打开、pi-web file panel 可见，类 Notion 页），不是 app 内的抽象实体。
- "确认 = 撤销键" = **Word 修订模式**：按块 ✓接受 / ✗拒绝即 track-changes。差别：我们**先拦后确认**（更安全），Word/Notion 先落盘后撤销；对 AI 产出保持先拦后确认、UX 做成修订视图手感。
- skill（解析 / 转化）与文档机器**正交**：skill 管内容、文档机器管生命周期（版本/diff/确认），文档机器不关心内容来源。
- **🔎 关键澄清（用户钦点 + ultracode review 核实）**：用户原话——"我会在对话里**表明某一段要改、而非整篇改**"。实现上 `propose_edit` 仍由 agent **回传整篇新全文**（这是内部实现细节），但 **LCS diff 只切出变化的那段**、用户端**只看到 / 只确认那一块**。所以"局部修改"是**用户视角的体验**、由 diff + 按块确认层交付，与"整篇传输"**正交**，**本期无需局部 patch**（这正是 AFFiNE 真机制：模型吐整篇 → 后端 LCS）。详见详细设计 coreIssue 与 §C V2-2 的 description 硬约束。

## 4. 功能需求

| 编号 | 功能 | 说明 | 复用 / 新增 |
|------|------|------|-------------|
| **F1** | 文档实体物化为真实 `crd.md` | 受管 artifact 的"当前版"物化成项目里真实 `.md` 文件（本地/file panel 可见）；版本历史 + 待确认改动作为旁挂受管侧车（`.pi/artifacts/managed/<id>/`），artifact.json 记它对应哪个真实文件 | **新增**物化层（现状只存 versions JSON、无真实文件） |
| **F2** | `create_artifact` 工具 | agent 直接建文档：写真实文件 + v1 侧车，**无需 diff**（全新、无上一版）。诞生即受管文档 | **新增**工具（复用 `ArtifactService.createArtifact`） |
| **F3** | `propose_edit` 工具 | agent 改文档：**回传整篇新全文（内部实现）** → LCS 算 diff（**只切出变化段**）→ 落 PendingChange → 用户**只确认变化块** → 才写新版本 + 物化真实文件。**永不直接写盘**。工具 description 硬约束"必须完整新全文、未改段逐字保留"；同一文档已有未决改动时拒绝并存（D-V2-05） | **新增**工具（复用 `computeReplaceDiffBlocks` / `buildReplacePendingChange` / `PendingChangeStore` / `applyResolvedBlocks` / `submitVersion`） |
| **F4** | `list_artifacts` 工具 | agent 列出当前项目的受管文档（id + title + kind），供 `propose_edit` 取目标 | **新增**工具（复用 `ArtifactService.listArtifacts`） |
| **F5** | 文档会话受限工具集 | 文档会话工具集 = `read + grep + find + ls + create_artifact + propose_edit + list_artifacts`（7 项 tools 白名单，**须含全部 customTool 名** D-V2-04，否则 agent 调不到提议工具）+ skill，**不给 write / edit / bash** → 无直接写文件能力 → 无绕过路径 | **新增**装配 `assembleDocSessionOptions`（替换 `assembleArtifactGuardOptions`） |
| **F6** | 复用 D3-D5 的版本/diff/高亮/确认 | 用户要的"版本 / diff / 高亮改动位置 / 确认(撤销)"四样已建好——直接复用 | **复用**：ArtifactVersion(D5) / computeReplaceDiffBlocks(D2) / InlineHighlightView(D3) / PendingChangeCard 按块✓✗(D4) |

**流程两段、语义不同**（QA 轮次 4）：第 1 轮**创建** = 存 v1、无需 diff；第 2 轮+**修改** = diff → 确认 → 新版。**创建即受管文档**（诞生即有版本/可 diff），顺手消掉"先建普通文件再标记受管"的别扭。

## 5. 非功能需求（北极星不变量，不破）

- **不改 pi 内核**：提议工具用 `defineTool`（pi 公开 API）写成 next-step 代码，`lib/pi/*` 只做封装，不 fork 内核源码。不建 pi 包（QA Q5：本地单用户无分发需求、过度设计）。
- **复用 pi-web 既有能力**：会话 / SSE / 工具 / 技能 / 模型 UI 全复用。
- **纯文件、无数据库**：沿用 pi-web 哲学；真实 `crd.md` + 旁挂侧车 JSON，无 DB。
- **项目即工作区**：crd.md 是项目里真实文件，受管侧车留在项目 `.pi/` 下，不外迁（受限工具集已堵绕过，无需物理隔离）。
- **本机资源约束**：开发机 3.4G 无 swap；实现阶段 agent team **串行**、E2E 重启 dev 控内存（见 §实现约束）。
- **单用户**：不引入鉴权 / 多租户；回退维持整块替换、**不上 CRDT**（那是为多人协作）。

## 6. 明确不做（本期边界）

| 不做项 | 原因 | 去向 |
|--------|------|------|
| **PDF / Word 解析** | 本轮范围 = 只拆核心闭环（用户拍板）；解析是独立工作线、需引库（docx 有 mammoth、PDF 需引库） | 后续独立工作线单独拆 |
| **转化 skill（原始→CRD）** | 同上，prompt 型 skill、随解析工作线 | 后续 |
| **外部改动自动进版本** | crd.md 是真实文件，外部改它**不自动进版本/diff**——只有"AI 改（propose→确认）"走版本。用户决定本期不做、仅登记 | 后续增强（类 Notion 自动快照/文件监听）；见 QA 文档📌待考量 |
| **⚠️ 外部改动被静默覆盖=数据丢失（已加防护，非"不做"）** | 外部改真实 crd.md 后、AI 下次 propose 基于旧快照确认会**覆盖丢失**外部改动（数据丢失，非仅"不进版本"，review 揪出 §6 原表述低估了此风险）。**本轮加防护**：materialize 前比对、不一致抛 EXTERNAL_MODIFIED 拒绝物化（D-V2-06） | 本轮做防护；自动同步入版本仍后置 |
| **block_id 行内精确锚定（方案 C）** | 跨判定/提示/显示三层、改多个单测，与最小闭环冲突；AFFiNE 真机制=内容 LCS、与我们同构，A 整篇重写已够稳 | A 稳定后确有长文档噪声痛点再做 |
| **方案 B 物理隔离（受管移出 cwd + 断 bash）** | 仅当给文档 agent 通用 coding（write/bash）才需要；本期文档会话用受限工具集、无绕过路径 → 不需要 | 仅当未来要"同一 agent 既写文档又完整 coding"时再叠 |

## 7. 取代关系（给新窗口）

本模型**取代** P0「逐路装 guard」的档位 2/3 计划（dispatch/idle 逐个补 guard）——**不再逐路装 guard，而是删 guard、换提议工具**。P0 的 spike / wire / verify 不浪费（diff/版本/确认 pipeline + 接线机制复用），但 guard 本身被替换。详见 `../../docs/设计决策记录.md` 对应 ADR 与 QA 轮次 1~5。

## 8. 验收标准（成功长什么样）

1. **端到端闭环**：文档会话的 agent 能 `create_artifact` 建出真实 `crd.md`（本地可见 + file panel 可见 + v1 侧车）；再 `propose_edit` **回传整篇新全文** → LCS **只高亮变化段** → 用户按块 ✓/✗ 确认 → 写新版本 + 真实文件同步更新。**局部修改用例**：多段文档只改一段 → 仅产 1 个变化块、用户只确认那块。
2. **受限工具集无绕过**：工具集 = 7 项白名单 `read/grep/find/ls/create_artifact/propose_edit/list_artifacts`（+skill），**没有 write/edit/bash**；agent 无法绕过提议工具直接改 crd.md。
3. **并发 / 外部编辑防护**：同一文档已有未决 propose 时再 propose 被拒（D-V2-05）；外部改真实文件后确认抛 EXTERNAL_MODIFIED 不静默覆盖（D-V2-06）。
3. **guard 已删**：`artifact-guard.ts` / `artifact-intercept.ts` 删除，无残留 import；原 wiring guard 接入点换成文档会话装配。
4. **复用未回归**：D3-D5 的 diff/版本/高亮/按块确认 UI 与 8 个 API 路由行为不变（既有单测 + 真浏览器验收通过）。
5. **三类质量门禁全绿**：`npm run test` + `tsc --noEmit` + `npm run lint`。UI 相关走真浏览器验收。
