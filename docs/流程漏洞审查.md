# Next-Step 流程漏洞审查

> 配合 [`新手引导.md`](./新手引导.md) 产出。目的：诚实列出**当前实现**里流程上的断点 / 缺失，而非设计期风险。

## 怎么查的

4 路并行调查 + lead 亲自核实头号：
- **功能广度**（读需求/概要）+ **操作细节**（读详细设计/tasks）→ 功能与流程串联
- **实现核对**（读 components/app/lib 真实代码）→ 技术断点
- **UX 走查**（新手视角逐功能）→ 体验卡点

**两条关键纪律**：
1. **剔除"设计期待定项"**——一路调查从需求文档列了 15 条"漏洞"，但绝大多数是 M1–M8 实现时**已经解决**的设计待定（如"M5 映射没持久化"已存 `.pi/ns-session-map.json`、"M8 勾选交互未定"已是 AgentTransferPopover）。这些**不算当前漏洞**。
2. **区分性质**：🟦 **有意分期**（设计上留给 v2/后续，不是 bug）vs 🟥 **可补真 gap**（该补的疏漏）。

## P0 · 架构级：产物按块确认闭环在运行时断开 🟦【已亲自核实｜有意分期=v2 命门】

> 这是最重要的一条，也回答了新手引导里"Agent 怎么用上按块确认"的疑问。

**现象**：「Agent 干活产出产物 → 自动进按块确认」这条核心闭环，**运行时不成立**。

**根因（两段，均 lead 核实）**：
- **§B**（核心）：拦截 Agent 写产物的守卫 `assembleArtifactGuardOptions`（`lib/pi/artifact-guard.ts:149`）**全仓库运行时零调用者**——起会话的代码（`lib/pi/profile-session-wiring.ts` 等）从没把它传进 `createAgentSession`。所以真实 Agent 用的是普通 `write`/`edit`（直接写盘），不会被拦成待确认改动。按块确认 UI 当前只能显示**手动/测试 fixture 造的** pending。
- **§A**：Dispatch 派发产物用 `writeArtifact` 落成普通 `.pi/artifacts/<id>/*.md`（`lib/domain/orchestrator.ts` 注释 D-C-1「轻量普通文件」），**不进受管系统**、`ArtifactPicker` 也列不到 → 到不了按块确认/版本。

**影响**：北极星红线①「artifact 改动必经按块确认」在生产中**未被强制**。整套 D3/D4/D5 HITL（高亮/确认/版本）目前是"有 UI、缺真实数据源"。

**性质**：**有意分期**。记忆与决策（`next-step-v2-diff-blocker`、D-D2-*）显示：Iter D 已**实测拦截机制可行**（operations 自分流、零工具漂移），但"接线到真实会话"明确留作 **v2 主线**。不是疏忽，是分期。

**接线方案（v2 头号）**：起 profile 会话时调 `assembleArtifactGuardOptions({sourceActor, cwd, ...})`、把 `options` 并入 `createAgentSession`（注释 `artifact-guard.ts:143-147` 已写明用法）；Dispatch 产物改走 `createArtifact` 受管落盘。

## P1 · 技术流程断点（impl-audit 核对代码所得）

| 编号 | 断点 | 出处 | 性质 | 影响 / 建议 |
|---|---|---|---|---|
| **§C** | 「引用到对话框」(`editTarget`) 是**只读展示**，`ChatInput.handleSend` 从不读它 → 划选的引用**永远到不了 Agent** | ChatWindow QuoteBar / ChatInput | 🟥 可补 | D3 当时明确"只做引用展示，注入留 D4/HITL"，但 D4 未接 → 现成悬空。建议 handleSend 把 quoteText 拼进消息 |
| **§D** | 按块确认结果**不回馈发起的 Agent 会话**（单向 HITL）；"R/重新生成"是 stub | PendingChangeCard | 🟦 分期 | 闭环的"人审完→告诉 Agent"方向未做，属 v2 |
| **§E** | ArtifactPanel **无轮询/SSE**，后台产物更新不自动刷新，要手动重开 | ArtifactPanel | 🟦 分期 | = D5 scope B「无 SSE，前端直刷」（D-D5-2 选 A）。已知 |
| **§F** | 运行中的 Dispatch **刷新页面后不可见/不可恢复**（store 未持久化、无"列 dispatch"端点） | useDispatchStore / dispatch API | 🟥 可补 | 长派发中刷新就"丢"了。建议持久化 + 列表端点 |
| **§G** | **整个应用没有 toast 系统**：@转交失败、模型回退、技能缺失只 `console.warn` | 全局 | 🟥 可补（高杠杆） | 见 P2「操作无回执」——补一个 toast 能一次解决一大片 |
| **§H** | 首回合流式 token 可能丢：SSE 在 fire-and-forget 首 prompt **之后**才订阅、无 replay buffer | session 起会话链路 | 🟥 可补 | 偶发首条回答开头缺字。建议首 prompt 前先订阅或加重放缓冲 |

**§I（新手第一道坎，强烈建议修）🟥**：**经项目墙进全新项目时，主对话不可见、起不了会话。** 根因是**两条进项目路径不对称**（lead 已亲自核实）——`ProjectHome.handleEnter` 只 `select(p.id)`（`ProjectHome.tsx:69-72`）、**不设 cwd**；而 `ProjectSwitcher.handleSelect` 还调了 `onProjectSelected(p.root)` 设 cwd（`ProjectSwitcher.tsx:66-72`）。后果链：经墙进**全新空项目** → cwd 为空 → 「New」会话按钮 + Skills 按钮都 disabled、左栏只显英文 `No sessions found`、中间区 `Get Started` 反让用户"从侧栏选项目目录"（可他刚选完项目）；而「主对话」分区标题**只在已有 main 会话时才渲染**（`SessionSidebar.tsx:747`），**新手第一次进项目根本看不到主对话入口**，5.2「每项目固定主对话」卖点零可见性。修法极小：`handleEnter` 比照 `handleSelect` 补一句把 cwd 设到 `p.root`。
> 注：M7 真浏览器 E2E 当年用「种子会话锁 cwd」**规避**了这个坑（见 `next-step-m7-verified`），所以验收没暴露——但真实新手进全新项目会撞上。这是「测试规避了、产品仍在」的典型。

> impl-audit 还查出并**主动撤回**一个误报（DispatchPanel 的 `agentsLoadedId/refreshAgents` 是 selector 别名、非 bug），可信度较高。

## P2 · UX 流程漏洞（ux-audit 新手视角走查，多为 🟥 可补、低成本高收益）

### 1. 空状态零引导（贯穿全应用）
- 产物为空："当前项目暂无受管产物"——不说**产物从哪来**（`ArtifactPicker.tsx:97`）
- Agent 为空：只有"+ 新建 Agent"卡，无"**Agent 是什么/建来干嘛**"（`AgentManager.tsx:401-447`）
- 会话为空："No sessions found"（英文）——不说"**先建 Agent 还是可直接起会话**"（`SessionSidebar.tsx:724`）
- **建议**：每个空状态加一句"这是什么 + 下一步点哪"。

### 2. 操作无回执（最影响信任感）
转交 / 起会话 / 逐块确认 / 回滚 **成功后多数无任何提示**（根因 §G 无 toast）。新手反复怀疑"成没成"、容易重复点。
- **建议（高杠杆）**：补一个全局 toast，转交成功→"已转交给 @X"、确认/回滚成功→提示。一次投入覆盖一大片。

### 3. 发现性差
- 🔴 **`@` 转交零提示**：输入框 placeholder 只有"Message…"，新手 100% 不知道能打 `@`（`ChatInput.tsx:535`）。建议 placeholder 加"输入 @ 可转交给 Agent"。
- 文件按钮是**相册图标**，暗示只能传图、看不出能传文本（`ChatInput.tsx:644`）。
- Agent 字段文案看不懂：**思考强度 / 工具 / 角色描述**对新手是谜（`AgentManager`）。建议加一句帮助文案。

### 4. 高危操作无二次确认
"全部 ✓ / 全部 ✗"、版本"回滚"**直接执行**、无确认（`PendingChangeCard.tsx:180`、`ArtifactPanel` 回滚）。误点难回头。建议加二次确认。

### 5. 其它
- **「Get Started」引导文案过时**：AppShell 中间区占位仍是 pi-web 旧文案「① 选项目目录 ② 加模型」（`AppShell.tsx:888`），完全没提 V1.1 的 Agent / 主对话 / @转交 / 产物，新手照做完仍不知下一步。
- **三个切项目入口语义重叠**：项目墙 + 工作台「回到项目墙」按钮 + ProjectSwitcher 下拉，且经墙进 vs 经下拉进**行为不一致**（见 §I），心智负担重。
- **Agent 会话分组延迟 ≤5.6s**：起 Agent 会话后左栏分组要等几秒才出现。注：这是 **M7 修复缺陷②时加的有界重试**（≤8×700ms 轮询 land）的副作用——修好了"当场不显"、代价是几秒延迟。属可接受权衡，但可加个"加载中"指示。
- **主对话无视觉标记**：和普通会话长一样，看不出"哪个是主对话/不可删"。
- **Agent 删除后会话变孤儿**：归属名回退成 agentId 短串、色点变灰，看不出原属谁（`session-grouping.ts:66`）。

## 优先级建议（给下一轮迭代）

| 优先级 | 行动 | 理由 |
|---|---|---|
| **P0** | **v2 接拦截线**：`assembleArtifactGuardOptions` → `createAgentSession`；Dispatch 产物改受管落盘 | 让北极星核心闭环真正成立，否则按块确认是"空转 UI" |
| **P1** | **补全局 toast 系统** | 一次解决 §G + P2「操作无回执」一大片，成本低、信任感提升大 |
| **P1** | **修 §I：`ProjectHome.handleEnter` 进项目时同步 cwd**（一行，比照 `ProjectSwitcher`） | 新手进全新项目第一道坎：cwd 不设 → 主对话不可见、起不了会话 |
| **P1** | **补空状态引导文案 + 更新过时的 Get Started** | 纯文案、零风险，覆盖 V1.1 新流程（Agent/@转交/产物） |
| **P2** | `@` placeholder 提示 + 文件按钮/字段文案 | 提升发现性，纯文案 |
| **P2** | 高危操作二次确认 + Dispatch 持久化(§F) + §C 引用注入 | 防误操作 / 补悬空功能 |
| **P3** | §D 确认回馈 Agent、§H token 重放 | 偏机制，可随 v2 一起 |

> 关键认知：**绝大多数"漏洞"不是 bug，而是"分期未接线"或"低成本可补的 UX 文案"**。真正的承重墙只有一条——P0 的拦截线接线（v2 命门）。
