# 第四轮·删除受管文档 · 详细设计

> 背景与用户拍板见 QA [`../QA/开发/删除受管文档决策.md`](../QA/开发/删除受管文档决策.md)（A1 彻底删除 / B2 两处入口 / C1 仅用户可删）。
> 实现级取舍记 [`../设计决策记录.md`](../设计决策记录.md) **D-V4-01~04**。

## 一、目标（一句话）

给受管文档补上生命周期最后一环「删除」：**彻底删除**（侧车目录 + 物化 .md 双删），入口在 **ArtifactPanel 头部** 与 **受管分组行悬停垃圾桶** 两处、都走**两步二次确认**；仅用户可删（不给 AI）。机制层只**新增** `deleteArtifact`（与 `createArtifact` 对称的结构操作），既有三写方法 / 提议工具 / 侧车格式一行不动。

## 二、删除链路（端到端）

1. **触发**：① ArtifactPanel 头部删除按钮（文档已打开）；② 受管分组行 hover 出垃圾桶（无需打开）。两者都先弹**两步二次确认**（复刻 D-D5-5 rollback 两步按钮，非原生 confirm），文案警示「将永久删除该文档、全部版本历史与待确认变更、及磁盘文件，不可恢复」。
2. **确认** → 调 `useArtifactStore.delete(id)`（入口②传 id；入口①用 selectedArtifactId）。
3. **store.delete**：`DELETE /api/artifacts/[id]`（带 `If-Match: <version>` 乐观锁）。
   - ok → 若删的是当前打开项则 `close()`（清 selectedArtifactId → 右栏回 FileViewer/空）+ 触发 explorer 刷新（受管分组重拉丢掉它）。
   - 409（版本已变）/ 404（已删）/ 其它 → 现成 toast 链路（useArtifactStore.ts:211 范式）提示，不崩。
4. **后端**：`route.ts` DELETE → `parseIfMatch` → `findArtifact(id)` 反解 projectId → `service.deleteArtifact(projectId, id, {ifMatch})` → 204。
5. **service.deleteArtifact**：`readMeta`（不存在抛 NOT_FOUND）→ `assertVersionMatch`（ifMatch 不符抛 409）→ 容错删物化 .md（`materializedPath`，缺/已被外删则跳过）→ `rmSync(artifactDir, {recursive,force})`（meta + versions + pending 一并清）。

## 三、关键实现决策（ADR D-V4-01~04）

| ADR | 决策 | 要点 |
|---|---|---|
| **D-V4-01** | 删除语义 = **彻底删除**（决策 A1） | 删侧车目录 + 物化 .md 双删；存储单目录 → `rmSync` 一刀清。非「降级保留 .md」，避免普通树重现困惑 |
| **D-V4-02** | `deleteArtifact` = **结构操作，豁免 propose→按块确认红线** | 该红线约束**内容修改**（propose_edit→PendingChange→逐块确认）；删除是生命周期操作，同 `createArtifact` 不走 propose。详设/红线须写明，避免后人误判违红线 |
| **D-V4-03** | 清理顺序 = **先容错删 .md，再 rmSync 侧车**；.md 删除 best-effort | .md 缺失 / 被外部删 / 无 filePath（旧 artifact）→ 静默跳过（`force:true`），不阻断侧车删除。部分失败（.md 删失败但侧车已删）→ 退化为「.md 残留为普通文件」，可手动删、非数据丢失，可接受 |
| **D-V4-04** | 带 **If-Match 乐观锁**（与 submit/rollback 一致） | 删一个「版本已被并发改过」的文档 → 409，提示先刷新。Header 缺 If-Match 则不校验（宽松）。单用户低频，但与既有写操作保持一致、成本低 |

## 四、后端设计

### `lib/domain/artifact-service.ts`（新增一个 public 方法）
```ts
/** 彻底删除受管 artifact：删侧车目录（meta+versions+pending）+ 物化 .md（容错）。
 *  不存在→NOT_FOUND；ifMatch 不符→VERSION_MISMATCH(409)。结构操作、不走 propose（D-V4-02）。 */
deleteArtifact(projectId: string, id: string, input?: { ifMatch?: number }): void {
  const artifact = this.readMeta(projectId, id);           // NOT_FOUND
  this.assertVersionMatch(artifact.version, input?.ifMatch); // 409
  const matPath = this.materializedPath(projectId, artifact);
  if (matPath) { try { rmSync(matPath, { force: true }); } catch { /* best-effort, D-V4-03 */ } }
  rmSync(this.artifactDir(projectId, id), { recursive: true, force: true });
}
```
（`rmSync` 已可用 / 或与既有 import 一致；`readMeta`/`assertVersionMatch`/`materializedPath`/`artifactDir` 均现成私有方法。）

### `app/api/artifacts/[id]/route.ts`（现有 GET，新增 DELETE，仿 rollback 路由）
```ts
export async function DELETE(req: Request, { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  try {
    const ifMatch = parseIfMatch(req);            // 非数字→422
    const service = new ArtifactService();
    const { projectId } = service.findArtifact(id); // NOT_FOUND→404
    service.deleteArtifact(projectId, id, { ifMatch });
    return new NextResponse(null, { status: 204 });
  } catch (error) { return domainErrorResponse(error); }
}
```

## 五、前端设计

### `lib/stores/useArtifactStore.ts`（新增 delete action）
- `delete: async (id?) => {...}`：target = `id ?? get().selectedArtifactId`；version = 若 target===selectedArtifactId 用 `get().artifact?.version` 作 If-Match（入口②删非当前打开项时可省 If-Match 或先不带）。
- 调 `DELETE /api/artifacts/${target}`（带 If-Match）；ok → 若 `target===selectedArtifactId` 则 `close()`；失败走 toast。
- **explorer 刷新**：删成功后须让受管分组重拉（丢掉已删项）。机制 = 复用 `explorerRefreshKey` bump（AppShell，agent_end 后刷新同款路径）。入口①（ArtifactPanel）经回调/store 信号触发 AppShell bump；入口②（ManagedDocsGroup 在 FileExplorer 内）可直接本地重取 artifacts。**确切接线点 T2 落码时定**（grep `explorerRefreshKey` / `setExplorerRefreshKey`）。

### 入口①：`components/ArtifactPanel.tsx` 头部
- 状态栏（:151 区，title + 版本下拉 + pending 计数旁）加「删除」按钮（红色高危语义，仿 rollback 实心样式）。
- 点击 → 两步二次确认（复刻 rollback 的 `rollbackConfirmRef` 两步按钮 + 外点/Esc 关闭，D-D5-5 / BUG-04）。确认 → `store.delete()`。

### 入口②：`components/FileExplorer.tsx` 的 `ManagedDocsGroup`
- 行 hover（同 TreeNode hover 范式）右侧出垃圾桶图标按钮（`onClick` stopPropagation，避免触发打开）。
- 点击 → 同款两步二次确认（行内小气泡 / 复用确认组件）→ `store.delete(a.id)` → 本地重取 artifacts（FileExplorer 已持 artifacts state）。
- **删的若正是当前打开项**：store.delete 内 `close()` 会清 selectedArtifactId、右栏自动回退。

## 七、验收清单

**逻辑层·单测**（artifact-service.test.ts）：
- 删后 `listArtifacts` 不含该 id；`<root>/.pi/artifacts/managed/<id>/` 目录消失；物化 .md 消失；带 pending 的删后 pending 一并清。
- 不存在 id → NOT_FOUND；If-Match 不符 → VERSION_MISMATCH；无 filePath（旧 artifact）→ 不抛、只删侧车；.md 已被外删 → 不抛、正常删侧车。

**真浏览器·必走**（pageErrors 须空，repo-vendored browser-e2e）：
1. ArtifactPanel 头部有删除按钮；受管分组行 hover 出垃圾桶。
2. 点删除 → 出两步二次确认、**不立即删**；取消 → 文档仍在。
3. 确认 → DELETE 调用 → 受管分组该项消失（数量 -1）、侧车目录 + 物化 .md 双双不存在（API/fs 验）。
4. 删的是当前打开项 → 右栏回到 FileViewer/空（selectedArtifactId 清）。
5. **彻底删（A1）**：删后该 .md **不在普通树出现**（区别于降级会重现）。
6. 删带 pending 的文档 → 中栏确认卡随之消失、pending 目录清。
7. 两处入口删同一文档行为一致；删非当前打开项（入口②）→ 分组更新、右栏不误清他项。
8. （可选）并发：版本已变后删 → 409 toast、不误删。

**门禁 + 零回归**：`npm run lint && npm run test` 通过；tsc 0；复跑 `artifact-service / pending-change-service / doc-tools` test 全绿（既有三写方法 / 侧车格式未改）。

## 八、已知 gap（登记，本轮不修）

1. 物化 .md 删除 best-effort，部分失败时 .md 残留为普通文件（D-V4-03，非数据丢失、可手动删）。
2. 无「删除撤销 / 回收站」——彻底删即不可恢复（决策 A1 已知代价，靠二次确认兜底）。
3. AI 不能删（决策 C1）；如需 AI 删须走「提议→确认」另立项。
4. 入口②删非当前打开项时是否带 If-Match：实现可先不带（删的不是正在看的版本、并发面小），登记。
