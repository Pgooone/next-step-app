# Next-Step V1.1 需求文档

## 一、文档说明

这份文档是 Next-Step 第 1.1 版要做什么的「正式说明书」。它由 `Next-step改进原始需求.txt` 出发，经过一轮一轮的问答确认（全过程见 `QA/完整问答记录.md`，决策摘要见 `QA/需求确认记录.md`）整理而成。

**怎么读**：每条需求都按同一个套路写——

1. **现状 / 调研结论**：这块现在是什么样（基于对本项目源码的静态阅读——V1.1 即 V1 源码的工作副本，关键处带 `文件:行号`）。
2. **决策**：你拍板要怎么做。
3. **落地方案**：打算怎么改（给将来实现用）。
4. **验收标准（AC）**：做完后用什么来判定"算完成"。
5. **工作量 / 风险 / 优先级**：大致投入与注意事项。

> ⚠️ 一条贯穿全局的红线（项目既定）：对「受管产物」的任何修改必须经「按块确认 → 才写盘并生成新版本」，不允许直接编辑产物写盘。下面任何方案都不会碰这条红线。

## 三、需求详情

### bug#1 · agent 落盘命名变 UUID（观感修复）

**现状 / 调研结论**
- 这不是程序坏了，而是既定设计：`.pi/agents/<id>/` 用随机 UUID 作目录名（`agent-profile-store.ts:113/128` 用 `randomUUID()`），由 decisions.md 的 D-19/D-20/D-21 三条决策刻意拍板。
- 你的 agent 真名 `prd` **没丢**：它存在该目录的 `agent.json` 的 `name` 字段，以及 `agent.md` 的标题里。
- 驱动 agent 干活的 pi 内核**按文件路径读取 `agent.md`**，根本不看目录名，所以 UUID 不影响起会话/注入/模型等任何功能。
- 界面里其实已经显示真名（`AgentManager.tsx:565`）；目前唯一把 UUID 路径亮给用户看的地方是「删除确认」那句话（`AgentManager.tsx:506`）。

**决策**：低风险方案——**不动**文件夹的 UUID 命名设计，只把界面上仍显示 UUID 的地方改成显示真名 `prd`。

**落地方案**
- 把 `AgentManager` 删除确认文案从「显示 UUID 路径」改为「确定删除 Agent『prd』吗」。
- 全局排查界面上是否还有其它把 UUID 路径暴露给用户的位置，一并改成真名。

**验收标准（AC）**
- AC1：界面上任何用户可见处都不再出现 UUID 路径，统一显示 agent 真名。
- AC2：删除确认显示真名（如「prd」）。
- AC3：不改动 `.pi/agents/<id>/` 的落盘结构与既定设计。

**工作量 / 风险 / 优先级**：极小（约半小时）／风险极低／优先级低。

### 功能#2 · file panel 与 Iter D 按钮（体验优化）

**现状 / 调研结论**
- 「file panel」就是右边那块可收起的面板。Iter D 的产物视图 `ArtifactPanel` 现在就长在这个槽位里。
- Iter D 里跟「看」有关的按钮——Diff 视图切换、版本下拉、回滚、划选引用——**现在本来就在 file panel 上**（`ArtifactPanel.tsx:149/171/208/216`）。
- 真正还在左边对话框、没在 file panel 上的，**只有「按块确认」那组按钮**（`PendingChangeCard`，挂在 `ChatWindow` 底部）。
- 「右边看改动 + 左边对话里逐块确认」的同屏分工，是 D-D3-3 / D-D4-3 两条决策刻意设计的动线；并有联动：在对话框按 `D` 键，右面板会自动弹开并跳到并排 Diff（`AppShell.tsx:126`）。

**决策**：维持现状，**不**把按块确认搬到右面板；只加视觉/文案提示，让「右看改动 / 左逐块确认」的分工更清楚。

**落地方案**
- 当存在待确认改动时，在产物面板一侧给出指引（例如顶部状态栏的「N 处待确认」旁加一句「→ 在左侧对话框逐块确认」），或在左右两处加呼应标识，帮用户建立"左确认、右看"的心智。
- 保留 `D` 键弹 Diff 联动不变。

**验收标准（AC）**
- AC1：不移动按块确认按钮的位置。
- AC2：用户能一眼明白「改动在右边看、确认在左边做」。
- AC3：`D` 键弹并排 Diff 的联动不被破坏。

**工作量 / 风险 / 优先级**：小／低／低-中。

### 功能#3 · 识别项目 + 全局 skills（验证为主）

**现状 / 调研结论**
- 这条在代码里**看起来已经实现**：pi 内核 `DefaultResourceLoader` 同时扫全局（`~/.pi/agent/skills`、`~/.agents/skills`）和项目级（`项目/.pi/skills` 及祖先 `.agents/skills`）；`/api/skills` 一并返回；前端 `SkillsConfig` 已按 PROJECT / GLOBAL / PATH 三组展示；给 agent 配技能时取的是「项目级 + 全局级」的并集。
- 唯一隐藏机制是「项目信任（project trust）」会 gate 项目级技能，但默认 `true` 且全仓库没有任何地方把它关掉，所以实际运行不会被吞。

**决策**：在真浏览器里验证（而不是直接写代码）。

**落地方案（验证步骤）**
- 在 `项目/.pi/skills/<技能名>/SKILL.md` 和 `~/.pi/agent/skills/<技能名>/SKILL.md` 各放一个测试技能。
- 打开技能面板，确认 PROJECT / GLOBAL 两组都正确列出。
- **是** → 在本文档标「已具备（已验证）」，无需开发；**否** → 第一嫌疑是技能目录位置不对或 cwd 没指到项目真实根目录，按此排查，而非加代码。

**验收标准（AC）**
- AC1：真浏览器中，项目级与全局级测试技能都出现在对应分组。
- AC2：若未出现，明确定位为「配置/目录问题」还是「功能缺失」。

**工作量 / 风险 / 优先级**：以验证为主（若已具备则零开发）／低／中（先验证再定）。

> **状态：✅ 已双层验证（2026-06-17）。**
> - API 地基断言：`/api/skills?cwd=<项目根>` 返回 2 个技能 —— `e2e-proj-skill`(scope=project) + `e2e-global-skill`(scope=user/global)。
> - 真浏览器：Skills 面板正确分成 **PROJECT** 组（项目技能）与 **GLOBAL** 组（全局技能），两组都列出（证据图：`QA/功能3-skills验证-通过.png`）。
> - 依据：内核 `DefaultResourceLoader` 同时扫 `<cwd>/.pi/skills`（项目）与 `~/.pi/agent/skills`（全局）（`package-manager.js:1863/1869`）。
>
> **结论：功能#3 无需开发。** 验证时另撞到 2 个副发现（不阻塞、可选处理）：
> 1. `SkillsConfig` 安装面板把「项目」安装路径显示为 `.pi/agent/skills`，但实际项目自动扫描目录是 `.pi/skills`（显示文案与真实路径不符的小 bug，`SkillsConfig.tsx:260`）。
> 2. cwd / 项目 / 会话三者耦合混乱：选项目后 `SessionSidebar` 的「自动选最近 cwd」(`SessionSidebar.tsx:281`) 会覆盖项目 cwd，导致 Skills 面板一度用错 cwd —— 这正是**功能#5 要解决的问题**，验证时实地撞到，可作为功能#5 的佐证。

### 功能#4 · 对话框上传文本类文件（新功能）

**现状 / 调研结论**
- 现在只能传图片，是「前端 + 内核」两层限制：前端写死 `accept="image/*"` 并在 4 处过滤非图片（`ChatInput.tsx:293/126` 等）；更深一层，pi 内核 / pi-ai 的用户消息内容块**只有「文本块」和「图片块」两种**（`pi-ai types.d.ts:194`），不存在「文档/任意文件」块。
- 内核对非图片文件的官方做法：读成文字、用 `<file name="...">...</file>` 包起来拼进消息文字，而非当二进制附件。二进制文件（PDF/docx/xlsx）若按 UTF-8 读会乱码。

**决策**
- 目标：**让 Agent 读懂文件内容**（不是只存进工作区）。
- 范围：**先做档1（纯文本类）**；档2（PDF/Word/Excel）记录待后续迭代。
- 图片通道**不动**，新通道与之并存。
- **不设**文件大小硬上限。

**落地方案**
- 前端放开文件选择，接受文本类扩展名（`.md/.txt/.json/.csv/.log/.py/.ts/.html` 等）。
- 选中的文本类文件用 `FileReader` 读出文字，按内核格式 `<file name="文件名">内容</file>` 拼进这条消息再发送（与内核 `@file` 行为一致，**不碰内核**）。
- 图片仍走现有多模态通道，逻辑不变。
- 档2（PDF/docx/xlsx）需后端文本提取（docx 可复用项目已装的 `mammoth`，PDF 需新增解析库如 `pdf-parse`）——**本次不做**。
- 因为不设硬上限，**建议在 UI 上加一句软提示**：「大文件会消耗大量 token / 可能撑爆上下文」，让用户自己权衡。

**验收标准（AC）**
- AC1：能选择并发送 `.md/.txt/.json/.csv/.log/.py/.ts/.html` 等文本类文件，模型能读到其内容。
- AC2：图片上传功能不受任何影响。
- AC3：不设大小硬限制；附带「大文件耗 token」软提示。
- AC4：（档2）PDF/Office 暂不支持，已在文档记录为后续迭代。

**工作量 / 风险 / 优先级**：中（仅前端）／低-中（大文件 token 消耗）／中-高。

### 功能#5 · 项目首页 + 主对话 + 每 agent 会话 + @agent（大改造）

> 这是本次最大的一条，你也强调"很重要"。它实为 **6 个子需求**捆在一起。整体建议：**先把交互设计画清楚再动手**，并按风险从低到高分步实现。

**关键技术约束（先理解）**：当前「会话」数据结构里**没有 `agentId` 字段**（`lib/types.ts:174-184`），系统无法天然知道某个会话属于「主对话」还是「某 agent」。又因为**不能改 pi 内核的会话文件**，这个「会话↔agent / 主对话」的归属标记多半要在我们自己的**领域层另存一份映射**。子项 5.2/5.3/5.4/5.5 都依赖它。

#### 5.1 项目管理首页
- **现状**：没有「先看项目列表、点进才进工作台」的两级结构；一打开就是工作台，靠左栏 `ProjectSwitcher` 下拉切项目。但项目的增/删/列接口和 `ProjectRegistry` 都现成。
- **决策**：做**独立的项目首页**——打开应用先看到项目列表/卡片墙（可新增/删除项目，参考 pi-web 风格），点某个项目才进工作台。
- **落地**：新增一个项目首页（独立路由或"未选项目时整屏项目墙"），复用现成的项目增删列接口。
- **AC**：①打开应用先看到项目卡片墙；②可新增/删除项目；③点项目才进入工作台。

#### 5.2 主对话
- **现状**：不存在"固定主对话"概念。
- **决策**：进项目后**落地到聊天工作台**；每个项目有**一个固定的「主对话」**作为总指挥会话；**没设 agent 时默认就用主对话**。
- **落地**：在领域层为每个项目维护一个固定主对话会话（或标记某会话为"主对话"）；进项目默认打开它。
- **AC**：①进项目即进入聊天工作台并默认落在该项目的主对话；②主对话固定、每项目唯一。

#### 5.3 每个 agent 一个会话
- **现状**：「建 agent」和「起会话」是两步手动操作，建好的会话在左栏与普通会话无任何归属标记。
- **决策**：**建档案后按需起会话**（**不**在建 agent 时自动冒空会话）；会话需打上「属于哪个 agent」的标记。
- **落地**：保持"先建档案、需要时再起会话"；在领域层记录"会话↔agent"映射，供左栏显性标注。
- **AC**：①建 agent 不会自动产生空会话；②某 agent 起的会话被标记为属于该 agent。

#### 5.4 左侧对话栏区分
- **决策**：**分组 + 标签，两者都要**——左栏按「主对话 / 各 Agent」分区，每条会话又挂一个 agent 名字标签/头像。
- **落地**：改造左栏会话列表渲染，依据 5.3 的映射分组并加标签。
- **AC**：①左栏可见"主对话"与各 agent 的分区；②每条 agent 会话带 agent 标签。

#### 5.5 主对话里 @agent 并转交
- **现状**：完全没实现。当前唯一沾边的是文件树的"插入路径"按钮，只插一段文本，不选 agent、不跨会话传递。
- **决策**：@agent = **把内容转交到那个 agent 的单独会话**，让他在自己会话接着干。**这是新的轻量功能，不等于 Dispatch，也不是在主对话里就地回答。**
  - 转交的内容：**默认带「当前主对话的全部历史」+「用户所附的文件」**，并在**发送时允许用户现场勾选增减**。
- **落地**：在主对话输入框支持 `@` 唤出待选 agent 列表；选中后，把（全部主对话历史 + 所附文件，发送时可勾选调整）作为一条消息投递到该 agent 的单独会话。
- **AC**：①主对话输入框 `@` 能唤出待选 agent；②选定后内容被发到该 agent 的单独会话；③默认携带全部主对话历史 + 所附文件；④发送时可现场勾选要带哪些。

#### 5.6 Dispatch 去留
- **现状**：Dispatch（多 Agent 派发）是整个 **Iter C 里程碑**的产出，是项目"多 Agent 协作"卖点的唯一落地，已做过真实端到端验收。它与主对话是两套独立机制。
- **决策**：**先保留不动**。@agent（手动转交）与 Dispatch（自动接力流水线）**并存**；等 @agent 做出来、用一段时间确认"接力/批量"场景真用不上，再回头评估是否下线。
- **AC**：①本次不删除 Dispatch；②@agent 与 Dispatch 可并存使用。

**功能#5 整体工作量 / 风险 / 优先级**：大／中-高（5.2~5.5 依赖新增的会话归属映射）／**高**（核心体验，但需先设计后实现）。

### 外观#1 · agent 管理界面玻璃感重做（UI 重写）

**现状 / 调研结论**
- 当前 `AgentManager` 是「居中弹窗 + 横条列表 + 单独编辑表单页」的老式布局：每个 agent 一条横向长条，右侧并排 起会话/编辑/删除；点编辑或新建后整个弹窗切成长表单。
- 样式全是写死的纯色 CSS 变量（`globals.css`），全库 grep 不到 `backdrop-filter/blur/glass`——**目前零毛玻璃实现**。
- 底层数据与接口（`useAgentStore` 的 create/update/remove/startSession）都现成，**不用碰后端**。

**决策**
- 重做成**玻璃感 + 正方形卡片网格**；卡片只显 agent 名，作为**一级菜单**；点击**就地展开二级菜单**（模型 / 技能 / 工具 / 思考 / 起会话 / 删除），**去掉独立的编辑按钮**。
- 二级菜单里的「模型/技能/工具/思考」要**能现场修改并保存**（不是只读）。
- 没有参考图，**按通行玻璃感做法先出一版**，再据反馈调整。

**落地方案**
- 布局：弹窗 + 横条列表 → 正方形卡片网格（`grid` + `aspect-ratio:1`）。
- 玻璃感：卡片/面板用半透明 `rgba` 背景 + `backdrop-filter:blur`（全库目前无，需新增），并适配深浅色模式可读性。
- 交互：一级卡片只显名；点击就地展开二级菜单，内含**可编辑并保存**的模型/技能/工具/思考表单（保存走 `useAgentStore.update`）+ 起会话 + 删除；移除独立编辑按钮。
- 新建入口：网格中放一张带「+」的空卡片，点击进入新建（新建需填名称/角色等）。
- **重要提醒**：重写时**同步更新** E2E 测试模板依赖的 `data-testid`（`agent-manager` / `agent-new-btn` / `agent-item`），否则验收脚本会断。

**验收标准（AC）**
- AC1：agent 以正方形卡片网格 + 玻璃感呈现，卡片只显名。
- AC2：点击卡片就地展开二级菜单；其中模型/技能/工具/思考**可现场修改并保存**。
- AC3：二级菜单含 起会话、删除；无独立"编辑"按钮。
- AC4：新建 agent 入口可用。
- AC5：深浅色模式下均可读。
- AC6：E2E 的 `data-testid` 同步更新，验收脚本不断。

**工作量 / 风险 / 优先级**：中-大（UI 重写，不动后端）／中（测试 testid 同步）／中。
