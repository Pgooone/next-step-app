## 五、⚠️ 内存铁律（本机 3.4G 无 swap，已崩过一次 OOM）

**核心：一次只跑一个重活，少开 agent。**
- **不要并行重活**：实现跑门禁(vitest/tsc 峰值~500MB) 与 E2E 的 dev+chromium **绝不同时**。
- **dev server 会涨到 ~1.1GB**（Next dev 编译路由累积）→ E2E 重 AC 组间 **kill next dev 重起** 重置回 ~300MB。
- **E2E**：单 chromium、每用例 `close()`、跑完拆 dev+chromium 腾内存；用 faux 模型（轻、不依赖真凭证）。
- **挂内存看门狗**：`free -m` available 跌破 ~300MB 就减载（Monitor 转换式发事件）。
- **`pkill -f` 别用含 'next dev'/'chromium' 的模式**（会杀到自己的 shell，exit 144）；按端口（`ss`/`fuser -k 30141/tcp`）或 PID 杀。
- **建议编排（低并发）**：lead 直接做 M8 实现 + 1 个独立 verifier 跑 E2E（实现者不自证）；或最多 lead + 1 impl + 1 idle verifier。**别再起 5 个并行 teammate**。
- 崩了不慌：committed 安全、靠 `git status` 看落盘真相、`TeamCreate` 新名（旧名残留用 `-r2`/`-r3`）重建续做。详见项目记忆 `next-step-local-oom-constraint` / `next-step-agent-team-recovery`。
