## 三、本期修复（详细五段）

### BUG-01 · 进项目 cwd 错误 🔵 ✅🤖

**发现**：审查「新手视角」走查发现「经项目墙进全新项目时主对话不可见、起不了会话」。M7 真浏览器 E2E 当年用「种子会话锁 cwd」规避了它，5 项验收全绿没暴露。

**深究（根因，lead + agent team 双验）**——实为**三个相关缺陷**：
- **链路**：`AppShell.tsx:398` 把 `selectedSession?.cwd ?? newSessionCwd ?? currentRoot` 作 `selectedCwd` prop 传给侧栏；`SessionSidebar.tsx:209` 本地 `selectedCwd` 初始 `null`；`:264-266` 把本地值经 `onCwdChange` 推回 `AppShell.handleCwdChange→activeCwd`(`:174-175`)；门禁 `AppShell.tsx:446`（Skills）与侧栏 New 按钮都依赖它。
- **缺陷①·空 cwd**：`:269-288` 自动选 cwd 的 effect 在 `allSessions.length===0` 时直接 return、**全程不读 `selectedCwdProp`** → 机器无任何会话时，进新项目 cwd 恒为 null → New/Skills 禁用、主对话标题不渲染。
- **缺陷②·串项目 cwd（审查漏报，更危险）**：`/api/sessions`→`listAllSessions()`（`app/api/sessions/route.ts:6` / `lib/session-reader.ts`）返回**全局所有项目**的会话；`getRecentCwds`（`SessionSidebar.tsx:42-56`，注释 "across all sessions"）取全局最近 cwd。当机器**已有别的项目会话**时，进新项目 → `allSessions` 非空 → `setSelectedCwd(getRecentCwds[0])` = **另一个项目的目录** → `activeCwd`、New 会话、Skills **静默指向别人的目录**，新建会话会建到错的地方。
- **缺陷③·切项目不更新**：`SessionSidebar` 只收 `refreshKey`、**无 `key`**（`AppShell.tsx:390`）→ 切项目不 remount，本地 `selectedCwd` 留旧项目值；`selectedCwd===null` 的 guard 使任何「采纳新 prop」的 effect 都不会触发。

**解决方案（选 A 补强版；naive 选 A 被证伪）**：
- ❌ **naive 选 A**（仅加一个 `if(selectedCwd===null && selectedCwdProp) setSelectedCwd(prop)` 的 useEffect）**不可取**：会和 `getRecentCwds`(:286) 抢、可能打断 URL 会话恢复（`:270` 的 `allSessions.length===0` early-return 本是为「等会话加载再恢复」，naive 方案会让 `restoredRef` 提前置位）、且不解决缺陷②③。
- ✅ **定稿**（两处协同）：
  1. `AppShell.tsx:390` 给 `<SessionSidebar>` 加 `key={currentProjectId}` → 切项目干净 remount，本地 cwd 状态重置（修缺陷③）。
  2. 改 `SessionSidebar.tsx:269-288` auto-select effect：URL 会话恢复（需等 `allSessions` 加载）**之后**、`getRecentCwds` **之前**，插入「**优先采纳 `selectedCwdProp`（项目根）**」分支，并让该分支在 `allSessions` 为空时也能执行（修缺陷①②）。
- **边界/回归（必过）**：URL `?session=<id>` 恢复仍优先于项目根（恢复需等会话列表加载，勿被项目根抢先）；用户手动下拉选 cwd（`:532`）、自定义路径（`:307/323`）不被覆盖；无限循环检查（guard 充分，agent 复核确认）。

**决策 QA**：Q3 用户认可「选 A」；复核（Q6 要求的 agent team）证实 naive 选 A 有 3 处回归 → lead 精化为「补强版」，方向不变，记此。

**解决**：_待开工_。验收（真浏览器，OOM 防护，lead 自跑）：①无任何会话时进新项目 → New/Skills 可用、主对话渲染；②有别项目会话时进新项目 → cwd = 本项目根（不串别项目）；③A↔B 切项目 → cwd 跟随切换；④URL 恢复会话不被破坏。

### BUG-02 · 全局 toast 缺失 🔵 ✅🤖

**发现**：审查发现整个应用**没有 toast**——转交/起会话/确认/回滚成功多无回执，失败常只 `console.error` 或藏在局部态。

**深究（根因，lead 亲验 grep + agent team 逐点核实 16 处）**：全仓库仅 2 处 `toast` 字样，均为「toast 后置」注释（`useAgentStore.ts` / `profile-session-wiring.ts:34` 的 D-B4-5）——toast 从设计期就被有意延后、从未实现。各功能（M1–M8）增量开发时各做各的**局部反馈态**（`formError` / `setError` / `createError` / `rollbackError` / 卡片内 `hint`），导致：成功路径多为「关掉表单」无回执；失败提示只在用户可能没看的局部角落。

**16 处接入点**（agent team 全部确认 `rootCauseConfirmed`，按是否接 + 文案归纳）：

| 操作 | 位置 | 现状 | toast 方案 |
|---|---|---|---|
| 起会话 | `AgentManager.tsx:222-235` | startError 仅菜单内 | 成功「已起会话」/失败兜底（保留本地态） |
| 创建/更新/删除 Agent | `AgentManager.tsx:155-219` | 创建/更新成功无回执；删除吞错 | 成功 toast；失败：创建/更新已有 formError**不重复**，删除补 toast |
| @agent 转交 | `AppShell.tsx:301-308` | 仅 console.error | 成功「已转交到 X」/失败 toast |
| 创建/删除项目 | `ProjectSwitcher.tsx:76-106` | 创建成功无回执；删除吞错 | 成功 toast；失败：创建已有 createError**不重复**，删除补 toast |
| 逐块/全部确认 | `PendingChangeCard.tsx:96-196` | 成功无回执；失败仅卡片 hint | 成功 toast（全部→**单条**「已确认所有变更」）；失败 toast + 保留 hint |
| 版本回滚/选版本 | `useArtifactStore.ts:194-246` | 仅面板内 rollbackError | 成功「已回滚到 vN」/失败 toast（含 409 冲突文案） |
| 派发 | `DispatchPanel.tsx:92-120` | 仅面板内 error | 成功「派发已发起」/失败 toast |
| 刷新类 `.catch(()=>{})` | `AppShell:85,90`/`AgentManager:107,115,127`/`DispatchPanel:69,79` | 静默吞错 | **多数不接或防抖**（见边界） |

**解决方案**：自建最小 toast 基建（不引第三方，守「纯文件/精简」）——
- `lib/stores/useToastStore.ts`（zustand）：`{ toasts, show({type,message}), dismiss(id) }`，自动消失计时 + 手动关闭；
- `components/Toaster.tsx`（`"use client"`）：固定角落渲染、多条堆叠；挂载在 `app/layout.tsx`（确认 layout 为 server component，故 Toaster 必须独立 client 组件）；
- 分批接线：先接**高频成功回执 + 关键失败**（起会话/确认/回滚/@转交/增删项目与 Agent）。

**边界/回归（agent team 穷举，防刷屏是重点）**：
- **不双重提示**：已有局部 `formError/createError/setError` 的失败**只保留局部**，避免表单+toast 冗余；toast 仅补「成功回执」和「无局部反馈的失败兜底」。
- **防刷屏**：刷新类 `.catch(()=>{})`（尤其 M7 ≤8×700ms 重试、dispatch 2s 轮询）**默认不接 toast**；确需提示的（如轮询持续失败）加**去重/节流**（同类错误 30s 内只提示一次）。
- 「全部✓/✗」一次多块 → **单条** toast，非逐块。
- 卸载后 setState 防护；连点已被现有 `saving/starting` 禁用拦截。

**决策 QA**：toast API 形态属 lead 实现取舍（自建 vs 第三方），选自建最小，记入 `docs/设计决策记录.md`（D-V1.1-11，开工时落）。

**解决**：_待开工_。验收（真浏览器）：起会话/确认/回滚/@转交触发后均见 toast；连续刷新失败不刷屏。

### BUG-04 · 高危操作无二次确认 🔵 🤖

**发现**：审查发现「全部✓/✗」「回滚」直接执行、无二次确认，误点难回头。

**深究（根因，agent team 核实）**：`PendingChangeCard.tsx:178-197` 全部✓/✗ 直接 `resolve('confirm'/'reject')`（全部块）无确认；`useArtifactStore.ts` 回滚同样直接执行。而**删除 Agent/项目早有完整二次确认范式**（`AgentManager.tsx:644-687` 的 `confirmDelete` state / `ProjectSwitcher.tsx:46-64` 的 `confirmId` + 外点关闭）——新增的产物高危操作**没有复用**这套范式。

**解决方案**：复用现有 `confirmId/confirmDelete` 内联二次确认范式——

- **仅全部/回滚加**，逐块 ✓/✗ 等高频操作**不加**。

**边界/回归（agent team）**：外点 / Esc 关闭确认态（参考 `ProjectSwitcher:54-64`）；确认态与现有 `hint`/`error` 不抢空间（确认态替换按钮区、hint 仍在下方）；`busy` 时禁用；后端在确认态打开时已 resolve → 监听 pendingChanges 变化自动清确认态；新增确认态注意卡片行高不破裂。

**决策 QA**：纳入（Q4）。**解决**：_待开工_。验收：全部✓/回滚需二次确认；取消/外点可关；逐块操作不受影响。

### BUG-05 · 划选「引用」到不了 Agent 🔵 🤖

**发现**：审查 impl-audit 发现 `editTarget`（划选引用）只读展示、`handleSend` 从不读它 → 引用永远到不了 Agent（D3 留「展示，注入待 D4」，D4 未接 → 悬空半成品）。

**深究（根因，agent team 核实）**：`ChatWindow` 的 QuoteBar 读 `editTarget.quoteText` 仅做显示+清除按钮；真正发送的 `handleSend`（`useAgentSession.ts:334-402`）只收 `message`+`images`，**完全不读 `editTarget`**。对比 @ 转交（`AgentTransferPopover` 有 `onConfirm` 真正注入 agentId）——QuoteBar 无对应接线。

**解决方案**：
- `handleSend` 入口：若存在 `editTarget.quoteText` → 拼进消息（格式如 `【引用：{quoteText}】\n\n{userMessage}`）；
- **乐观清除防重复注入**：入口立即 `setEditTarget(null)`（防用户连点/steer 导致多条都带同一引用）；发送失败可不清（简化，因无重试机制）；
- Steer/Follow-up 路径同样接（`useAgentSession` 内 steer/followUp 也要读 editTarget）。

**边界/回归（agent team，本组风险最高）**：空引用跳过注入；引用+正文并存→引用在前拼接；引用+附件→文本拼消息头不影响图片块；**改 `handleSend` 行为可能影响 Steer/Follow-up**（须验证三条发送路径都正确读/清 editTarget）；乐观清除后发送失败的体验（须验证）。

> ⚠️ **本组触及核心发送链路（useAgentSession），回归面最大**。建议**最后做**、单独 commit、真浏览器验证三条发送路径（普通/steer/follow-up）+ 引用注入 + 附件并存。

## 五、说人话（这些 bug 是什么、为什么会出现、为什么当初会这么写）

> 这一章不讲代码细节，讲「人话」和「教训」。目标是复盘：为什么我写出来的东西会有这些问题。

### BUG-01 进项目 cwd 错误 —— 「老房子改造没换水管」
- **是什么**：进一个新项目，结果工作台要么按钮全灰点不了，要么**悄悄在别的项目目录里干活**（你以为在项目 B，新建的会话却落到了项目 A 的文件夹）。
- **为什么会出现**：这套侧栏代码是从 pi-web 搬来的，pi-web **没有「项目」这个概念**，它只有一个「最近用过的目录」全局列表。Next-Step 后来加了「项目」（每个项目有自己的根目录），但侧栏挑默认目录的逻辑**还在用那套「从全局最近目录里挑」**——于是它挑出来的可能是别的项目的目录。
- **为什么当初会这么写**：加「项目」这层新概念时，只改了「明显相关」的地方，没有回头审计「所有从全局状态挑默认值」的角落。加上 M7 的自动化测试**用种子会话把 cwd 锁好了**，测试路径根本没走「新项目空 cwd」，所以 5 项验收全绿，bug 却好端端躺在那。
- **教训**：**在继承来的代码上加一个新的「作用域」概念（这里是项目），必须把所有「取默认值 / 取最近 / 取全局第一个」的点全扫一遍重新作用域化。** 否则旧的全局逻辑会从你新画的边界里漏出去。还有——**测试用种子数据是「规避」不是「暴露」新手路径**，真实新手的第一步往往是测试从不走的那条。

### BUG-02 全局 toast 缺失 —— 「每个房间各装各的门铃，客厅却没有」
- **是什么**：你点了「起会话 / 确认 / 回滚 / 转交」，成功了却没有任何提示，失败了提示也藏在某个你没看的小角落，于是你反复怀疑「到底成没成」，忍不住又点一次。
- **为什么会出现**：M1 到 M8 一个模块一个模块做，每个模块都**各做各的局部反馈**（这个用 `formError`、那个用 `setError`、再一个用卡片角落的 `hint`）。成功路径大多是「把表单关掉」就完事，没有回执；失败提示则散落各处。从来没有一个**全局统一的通知**。
- **为什么当初会这么写**：toast 在最早的设计里就被标了「后置」（B4 的注释白纸黑字写着「toast 后置」），然后每做一个模块就「先用局部态顶一下」，一路顶到 M8。单看每个模块都「能用」，攒到一起就成了「操作无回执」一大片。
- **教训**：**「给用户的操作反馈」是横切关注点，要么一开始就定一个全局方案，要么你会攒出 N 个互不一致的局部实现。** 「先用局部态顶一下、toast 后置」这种话每说一次，就多一个静默操作——一次性补一个全局 toast，能把这一大片一起解决。

### BUG-03 文案/空状态 —— 「装修时屋里堆满样板家具，没人看见空房间」
- **是什么**：新手第一眼看到的是英文旧文案（`Get Started`、`No sessions found`）、中英混在一起的提示、以及空白面板上**一句引导都没有**，照着做完还是不知道下一步点哪。
- **为什么会出现**：英文串是从 pi-web 继承的，转成中文产品时没统一清扫；空状态则是因为**开发的时候屏幕上总是有数据**（种子项目、种子会话），「啥都没有」的那个画面开发者自己几乎从来没见过，于是只随手写了一行或干脆留英文。
- **为什么当初会这么写**：开发者和新手的视角天然错位——开发者永远「有数据、知道怎么操作」，看不见空状态和第一屏。
- **教训**：**空状态和首屏文案是开发者的盲区，却是新用户的第一印象。** 继承来的文案要成套清扫，别中英混杂；每个「列表/面板」都要专门设计「啥都没有时说什么、下一步点哪」。

### BUG-04 高危操作无二次确认 —— 「删项目要问三遍，删全部产物却一点就没」
- **是什么**：「全部✓ / 全部✗ / 回滚」这种一锤子下去难回头的操作，点一下就执行了，没有「你确定吗」。
- **为什么会出现**：删项目、删 Agent 早就有一套完整的「内联二次确认」（`confirmId` 状态 + 确认/取消 + 外点关闭），但后来给产物加「全部✓/回滚」时，**只顾着把功能跑通（happy path），没有回头复用这套已有范式**。
- **教训**：**「危险操作要二次确认」是个应当统一套用的范式；新增任何破坏性操作，第一反应就该是复用既有的确认模式**，而不是又裸奔一个。

### BUG-05 引用到不了 Agent —— 「先把按钮画上，接线留到下一期，然后忘了」
- **是什么**：你在产物里划选一段文字想「引用」给 Agent，输入框上方也确实显示了这段引用——但你发出去，Agent **根本收不到这段引用**。
- **为什么会出现**：D3 当时明确「**先只做引用的展示，真正注入留给 D4/HITL**」，结果 D4 没接这根线，于是它就成了一个**悬空的半成品**：看得见、用不上。
- **为什么当初会这么写**：「先展示、后接线」分期本身没错，错在「后接线」那一期没有把这根线**当成一个真任务**去关掉，只在代码里留了句注释，注释没人追，功能就烂尾了。
- **教训**：**「先展示、后接线」的功能，如果「后接线」不被当成一个明确的待办去追踪，就会悄悄烂尾成「看得见用不上」。** 分期未接线要登记成真任务（就像本轮的 P0 承重墙一样），别只留一句注释。

### 总结 · 五条可迁移的经验
1. **继承代码 + 新概念 = 重新审计所有「默认值/最近值」的来源**（BUG-01、03 都源于「pi-web 没有项目概念，旧的全局逻辑漏进了新边界」）。
2. **异步操作必须有回执；反馈是横切关注点，要早定全局方案**，别攒一堆局部态（BUG-02）。
3. **空状态和首屏是开发者盲区**——开发时永远有数据，要专门站在「啥都没有的新手」视角走查（BUG-01、03）。
4. **「先展示/先跑通、后接线」会烂尾成半成品**，分期未接线要当真任务追踪，不能只留注释（BUG-05、以及 P0 承重墙）。
5. **修 bug 前先穷举边界**：BUG-01 的「一行 naive 修法」会一口气引入「串项目 / 切项目不更新 / 打断会话恢复」三个新 bug——**对抗式复核 + 边界穷举，就是为了防止「修完一个、冒出三个」**。
