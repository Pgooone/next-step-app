# Next-Step V1.1 — 新窗口开工指引

> 新开一个窗口、用 **agent team** 做 V1.1 实现时，从这份开始。一页看懂「在哪、读什么、怎么起团队、别踩什么」。

## 你在哪

- 本目录 `next-step-V1.1/` = V1 源码的工作副本，**已是 git 仓库**：`next-step-app` 的 **`v1.1` 分支**（继承 V1 全部 41 个历史提交）。
- V1 已发布 **`v1.0.0`**（`master` + GitHub release）。**V1.1 的提交都提到 `v1.1` 分支**，将来出 `v1.1.0`。
- 提交身份已配好（pgone）。提交用**中文**、按 `CLAUDE.md` 约定（`type(scope): 描述` + 必要时结构化正文）。

## 先读（按序）

1. `prompt.md` —— **监工起始指令**（角色 / 质量门禁 / 开发批次）。
2. `docs/需求文档.md`（要什么）→ `docs/概要设计.md`（8 模块）→ `docs/详细设计.md`（每模块接口·逻辑·验收）。
3. `docs/设计决策记录.md`（**D-V1.1-01~04**，机制已拍板）、`docs/文档评审报告.md`（评审已回改）、`docs/后续迭代清单.md`（本期先不做的）。
4. `tasks/progress.md` + `tasks/M1~M8.md`（任务清单）。`tasks/v1-history/` 是 V1 历史，含 `decisions.md`。

## 怎么开工（agent team）

1. `npm install`（副本没带 node_modules，首次必跑）。
2. **TeamCreate** 建实现团队（如 `v1-1-impl`）。
3. 照 `prompt.md`：**你当监工**，按批次为每个模块派 **opus 子 Agent**（一个子 Agent 只做一个模块的实现 + 测试，上下文短）：
   - **批次 1（并行）**：M1 / M2 / M3 / **M5（承重墙，先做好供 M7/M8 接）** / M6
   - **批次 2（依赖 M5）**：M7 / M4（UI 重活，已拆阶段 A/B）
   - **批次 3（依赖 M5 + M7）**：M8
4. **每批跑质量门禁**（`npm run test` + `node_modules/.bin/tsc --noEmit` + `npm run lint`）全绿再进下一批；**UI 模块（M3/M4/M6/M7/M8）走真浏览器验收**（`../next-step-V1/next-step-app/.claude/skills/browser-e2e` 的 vendored 流程）。
5. 每个模块完成：勾 `tasks/progress.md` + 更新该区薄 README + 新决策记 `docs/设计决策记录.md`（沿 `D-V1.1-NN`）。
6. 提交到 `v1.1` 分支（中文 commit）；阶段性 `git push origin v1.1`。

## 红线（北极星不变量，别破）

不改 pi 内核（`lib/pi/*` 只封装）｜产物改动必经按块确认｜纯文件落盘·本地单用户｜并发会话 ≤ 3｜复用 pi-web 不另起炉灶。
